Horizontal vs Vertical Scaling
When each applies · limits · real cost
Definition & Core Purpose
Vertical scaling (scale up) means adding more resources — CPU, RAM, faster disk — to an existing machine. One node becomes a bigger, more powerful node.

Horizontal scaling (scale out) means adding more machines to a pool. The load is distributed across many smaller, commodity nodes.

Both exist because a single machine has a physical ceiling. The problem they solve: what do you do when one machine is no longer enough? The answer depends on your bottleneck, your budget, and how much complexity you can tolerate.
How It Works — Deep Mechanics
Vertical
Horizontal
1. Provision a larger instance. In AWS terms: move from m5.large (2 vCPU, 8GB) → m5.4xlarge (16 vCPU, 64GB). Downtime required unless the platform supports live resize (rare for major jumps).

2. OS picks up new resources automatically. The JVM, Postgres, or whatever runs on it can now use more RAM. But it won't automatically unless you tune: Postgres `shared_buffers`, JVM `-Xmx`, thread pool sizes all need reconfiguring.

3. Hard ceiling: the largest machine money can buy. AWS largest general-purpose is u-24tb1.metal — 448 vCPUs, 24TB RAM. Beyond that: no bigger box exists. You're forced horizontal.

4. Single point of failure remains. One machine, one failure domain. No redundancy by default.
1. Add a node to the pool. New instance registers with the load balancer or service discovery system (e.g., Consul, Kubernetes endpoints).

2. Application must be stateless — or state must be externalized (Redis, DB). If node A handles request 1 and node B handles request 2 from the same session, they must agree on user state.

3. Coordination overhead grows. Distributed systems need consensus (who's the leader?), data partitioning (which shard holds user X?), and replication (does node B have the write node A just did?). Each adds latency and failure modes.

4. Linear (or near-linear) cost scaling. 10 m5.large machines ≈ cost of 1 m5.10xlarge. But you get redundancy and can add/remove without downtime. Auto-scaling groups (ASG) do this dynamically based on CPU/RPS metrics.
Subtypes & Variants
Vertical — Storage scaling
Add larger SSD, NVMe arrays, or attach EBS volumes. Independent of compute scaling.
Vertical — Memory scaling
Add RAM to reduce disk I/O. Most impactful for databases (Postgres buffer pool, Redis).
Horizontal — Stateless
Pure compute workers (API servers, Lambda functions). Easiest — no coordination needed.
Horizontal — Stateful
Databases, caches, queues. Requires sharding, replication, consistent hashing.
Diagonal / hybrid
Scale up individual nodes while also scaling out. Common in Kafka (larger brokers + more brokers).
Auto-scaling
Horizontal scaling driven by metrics (CPU > 70%, RPS > threshold). AWS ASG, k8s HPA.
Scale Estimation
~96 cores
Max practical vertical on AWS (c7g.metal). Beyond this: no more headroom.

Trade-offs
✓ Gain (Vertical)

Simple — no code changes, no distributed systems complexity. Works immediately.

✗ Lose (Vertical)

Hard ceiling. Downtime during resize. No redundancy — one machine fails, you're down.

✓ Gain (Horizontal)

No theoretical ceiling. Redundancy built-in. Cost-efficient at scale using commodity hardware.

✗ Lose (Horizontal)

Application must be stateless or handle distributed state. Added latency from coordination. Ops complexity explodes.

✓ Gain (Vertical for DBs)

Avoids complex sharding logic. Postgres on a beefy machine handles millions of QPS before you need to shard.

✗ Lose (Vertical for DBs)

A single-master DB cannot scale writes horizontally without sharding. Vertical only helps reads (with replicas).

Failure Modes
Vertical SPOF: Machine dies → full outage. Mitigation: RDS Multi-AZ standby, warm standby replica.
Horizontal — split brain: Network partition causes two nodes to think they're primary. Write both → data divergence. Mitigation: use a consensus protocol (Raft, ZooKeeper) or single-primary DB.
Horizontal — thundering herd: All nodes restart simultaneously after a deploy, flood the DB. Mitigation: staggered deploys, circuit breakers, connection pools.
Auto-scaling lag: Spike hits before new instances are warm. Mitigation: predictive scaling, keep warm pool, over-provision slightly.
Real-World Examples
Netflix: Stateless API nodes scale horizontally via AWS ASG. Cassandra (horizontally sharded) stores user data. A single Cassandra node is also vertically sized to hold hot partition data in memory.

Uber: Started with a monolith on vertical iron. Migrated to horizontal microservices when driver-location writes exceeded what a single Postgres could handle (~100K writes/sec).

Razorpay: Payment processing is write-heavy. They vertically scale Postgres primaries and horizontally scale read replicas. API layer is stateless and horizontally auto-scaled.

WhatsApp: Famously ran 450 million users on ~50 engineers and ~few hundred servers. Heavy vertical scaling of FreeBSD servers (64+ cores, custom tuning). Eventually added horizontal sharding for message routing.
Interview Perspective
  1. Start with vertical — always acknowledge it's simpler and works fine until ~10–50K RPS depending on service type.
  2. Trigger for horizontal — "When I hit the machine ceiling, or when I need redundancy for SLA guarantees, I move to horizontal."
  3. State the constraint clearly: "Horizontal requires stateless services — so I externalize sessions to Redis, store uploads in S3."
  4. Probe question: "Your DB is the bottleneck — vertical or horizontal?" → Vertical for reads (read replicas), eventually horizontal sharding for writes. Never say "just shard it" without explaining the coordination cost.

Load Balancing
L4 vs L7 · Algorithms · Health checks · Sticky sessions
Definition & Core Purpose
A load balancer is a traffic distributor. It sits in front of a pool of servers and routes each incoming request to one of them, based on a chosen algorithm.

The problem it solves: a single server is a bottleneck and a single point of failure. Without a load balancer, adding server 2 means clients need to know about both — impossible at scale. The LB is the single, stable entry point; the pool of servers behind it is elastic and interchangeable.
How It Works — Deep Mechanics
L4 (Transport)
L7 (Application)
Algorithms
Layer 4 = TCP/UDP level. The LB sees: source IP, destination IP, source port, destination port — nothing else. It does not decrypt TLS, does not read HTTP headers, does not know the URL path.

How it routes: Opens two TCP connections — one from client→LB, one from LB→server. Uses NAT (Network Address Translation) to rewrite source/destination IPs on the fly. Ultra-fast: decision made in microseconds, no packet inspection.

When to use: Raw TCP throughput (game servers, IoT telemetry, non-HTTP protocols), latency-critical paths where header inspection overhead matters, high-PPS (packets per second) workloads. AWS NLB is L4.
Layer 7 = Application level. The LB terminates TLS, reads the full HTTP request — method, URL path, headers, cookies, body (partially). This costs CPU but buys intelligence.

What it can do that L4 cannot:
— Route /api/* to API servers, /static/* to CDN origin, /ws to WebSocket pool
— A/B testing: route 10% of requests to new service version based on header
— Sticky sessions via cookie: always send user 42 to server 3
— SSL termination: decrypt once at the LB, send plain HTTP internally (saves server CPU)
— Rate limiting per endpoint
— WAF integration (block SQL injection, XSS patterns)

AWS ALB, Nginx, HAProxy (mode http), Envoy are L7. This is the default choice for web services.
Round Robin: Requests cycle server 1→2→3→1→2→3. Simple, works when servers are identical and requests are similar in cost. Breaks down when some requests take 10ms and others take 5 seconds — fast servers idle while slow ones queue.

Weighted Round Robin: Server with 8 cores gets 2× requests of the 4-core server. Useful for heterogeneous hardware.

Least Connections: Route to the server with fewest active connections. Better for variable-length requests (file uploads, streaming). Requires the LB to track connection counts.

IP Hash: hash(client_ip) % n_servers → deterministic server. Same client always hits the same server. Good for "soft" stickiness without cookies, but breaks badly when servers are added/removed (all hashes shift). Use consistent hashing to fix this.

Consistent Hashing: Hash space is a ring. Servers are placed on it. Client hashes to a point on the ring, routes to the nearest server clockwise. Adding/removing a server only remaps ~1/n of clients. Used in distributed caches (Memcached, Redis Cluster).

Sticky Sessions — Deep Dive
Problem: If a user uploads state to server 1 (e.g., a shopping cart in memory), routing them to server 2 on the next request loses that state.

Cookie-based sticky sessions (L7 only): LB sets a cookie (e.g., AWSALB=srv1id). Every subsequent request from that browser includes the cookie → LB always routes to server 1. Expires after a configured duration.

The problem with stickiness: If server 1 crashes, all its sticky clients get errors (or must be re-routed, losing state anyway). Stickiness makes horizontal scaling less effective — you can't drain server 1 for maintenance without disrupting users pinned to it.

The real fix: Make servers stateless. Move session state to Redis. Now any server can handle any request. Stickiness becomes unnecessary.
Trade-offs
✓ Gain (L7 over L4)

Content-aware routing, SSL termination, WAF, A/B testing. Smarter and more powerful.

✗ Lose (L7 over L4)

Higher CPU cost (TLS decrypt, header parsing). Extra latency (~1–3ms). LB itself can become a bottleneck.

✓ Gain (Least Connections)

Fair distribution under variable-length workloads. No server piles up while others idle.

✗ Lose (Least Connections)

LB must track state (connection counts). Slightly more complex. Still doesn't account for request cost (CPU vs I/O).

✓ Gain (IP Hash)

Zero-overhead stickiness — no cookie, no session tracking at LB.

✗ Lose (IP Hash)

Breaks when server pool changes. Users behind NAT (same IP) all hit one server.

✓ Gain (Health Checks)

Automatic removal of unhealthy nodes. Self-healing routing without manual intervention.

✗ Lose (Health Checks)

Lag between failure and detection (interval × threshold = 30s typical). In-flight requests to the dead node fail during that window.

Failure Modes
LB itself as SPOF: If the load balancer dies, everything is down. Mitigation: LB HA pair (active-passive with VIP failover, e.g., AWS ALB is managed and internally redundant).
Cascading overload: Backend server dies → its load redistributes → surviving servers get overloaded → they die too → cascade. Mitigation: circuit breaker at LB, reject excess requests with 503 rather than forwarding them.
Health check gaming: Server returns 200 to health check but fails real requests (e.g., DB connection pool exhausted). Mitigation: deep health checks that actually test downstream dependencies.
Slow server problem: Server becomes slow (GC pause, I/O wait) but doesn't fail health checks. LB keeps sending it traffic. Mitigation: least-connections or outlier detection (Envoy's outlier ejection).
Real-World Examples
Netflix: Uses AWS ALB (L7) for HTTP traffic. Zuul (their own L7 LB) for cross-region routing with custom logic (chaos injection, canary routing).

Cloudflare: Uses consistent hashing at L4 for their Anycast network to route users to nearest PoP, then L7 routing within the PoP.

CRED: Nginx as L7 LB in front of Spring Boot microservices. Sticky sessions via Redis-backed token — not at the LB layer, making the design stateless at the LB and correct.
Interview Perspective
  1. Default to L7 ALB for web/API services. Only mention L4 NLB if you need ultra-low latency or non-HTTP protocol.
  2. Default algorithm: round robin or least connections. Mention consistent hashing only when the interviewer asks about caching or stateful routing.
  3. Sticky sessions is a red flag question. If you mention stickiness, immediately follow with "but the better solution is to externalize state so stickiness is unnecessary."
  4. LB as SPOF: Always note that the LB itself needs HA — managed services (ALB) solve this, self-hosted needs an active-passive pair.

Latency · Throughput · Availability
Definitions · How they conflict · SLA / SLO / SLI
Definition & Core Purpose
Latency = time for a single operation to complete. Measured in ms. "How long did request 1 take?"

Throughput = number of operations completed per unit time. Measured in RPS, QPS, MB/s. "How many requests can the system handle per second?"

Availability = fraction of time the system is operational and serving requests. Measured in nines (99.9% = 8.76 hours downtime/year).

These are the three axes of system performance. Every architectural decision trades one against another. Understanding how they conflict is the core skill of system design.
How It Works — The Fundamental Tension
Latency vs Throughput: At low load, latency is low and throughput scales linearly. As you approach system capacity, queues form. Queuing latency grows exponentially near 100% utilization (Little's Law: L = λW). You cannot simultaneously minimize latency AND maximize throughput — pushing throughput to the ceiling destroys latency.

Rule of thumb: keep utilization under 70–80% to maintain predictable latency.

Availability vs Latency: High availability often means replication. Replication adds coordination — synchronous replication forces the write to wait for all replicas to ack before returning. This doubles+ your write latency. Async replication is fast but risks data loss on crash (availability vs durability tension).

Availability vs Consistency (preview of CAP): To stay available during a network partition, you must accept stale reads. To stay consistent, you must refuse requests you can't verify. You cannot have both.
SLA · SLO · SLI — The Hierarchy
SLI — Service Level Indicator
The raw metric. "p99 latency of /checkout endpoint." Measured continuously from production traffic. It's a number.
SLO — Service Level Objective
Internal target for that metric. "p99 latency must be ≤ 200ms, 99.9% of the time, measured monthly." It's a contract with yourself.
SLA — Service Level Agreement
External contract with customers. "If availability drops below 99.9%, we credit 10% of monthly fee." Legal. Money changes hands.
Error Budget
SLO = 99.9% uptime → budget = 0.1% = 43.8 min/month. Spend it on risky deploys. Exhaust it → freeze changes.

Trade-offs
✓ Async processing

Decouple producer from consumer via queue. Throughput skyrockets — producer never waits.

✗ Async cost

Latency for the end-to-end operation increases. User gets "processing" instead of immediate confirmation.

✓ Caching

Dramatically reduces latency for reads. Throughput of DB is effectively amplified.

✗ Caching cost

Stale data. Cache invalidation is notoriously hard. Thundering herd on cache miss.

✓ More replicas

Read throughput scales linearly. Availability improves (one replica down, others serve).

✗ Replication cost

Write latency up (synchronous), or risk of data loss (async). Replication lag creates stale reads.

Failure Modes
Latency tail explosion: p50=10ms, p99=500ms, p999=10s. One slow operation (GC pause, lock contention, hot DB row) destroys tail latency. Fix: timeouts, async, connection pooling.
Throughput cliff: System handles 5000 RPS smoothly, then falls off a cliff at 5001. Queue backs up, timeouts cascade. Fix: load shedding, backpressure signaling to upstream.
Availability math mistake: Two 99.9% services in series = 99.9% × 99.9% = 99.8% availability. Chains of dependencies dramatically reduce overall availability.
Real-World Examples
Google: Internal SRE practice — 99.9% SLO internally, 99.99% external SLA. Error budget is ~43min/month. Teams that exhaust it freeze feature deployments until budget recovers.

Amazon: Every 100ms of additional latency = 1% decrease in sales. That's why every service has strict latency SLOs and aggressive caching.

Zepto: 10-minute delivery requires p99 checkout latency under 500ms. They cache product catalog aggressively, async inventory reservation, sync only payment confirmation.
Interview Perspective
  1. Always ask for the SLO first. "What's our latency target? What's the availability requirement?" This frames your entire design.
  2. Use the latency budget technique. Show you can decompose 200ms into sub-budgets per layer. Interviewers love this — it shows engineering maturity.
  3. Know the nines: 99% = 3.65 days/yr downtime. 99.9% = 8.76 hrs. 99.99% = 52 min. 99.999% = 5 min. Say these without hesitation.
  4. Error budget: Mentioning error budgets signals SRE knowledge — a huge differentiator at senior level.

CAP Theorem
Consistency · Availability · Partition Tolerance — pick 2
Definition & Core Purpose
Proved by Eric Brewer (2000), formalized by Gilbert & Lynch (2002): In a distributed system, you can guarantee at most 2 of 3 properties simultaneously:

Consistency (C): Every read receives the most recent write or an error. All nodes see the same data at the same time. This is linearizability — reads are guaranteed to see the last committed write.

Availability (A): Every request receives a (non-error) response — not guaranteed to be the latest data.

Partition Tolerance (P): The system continues operating even when network messages are dropped or delayed between nodes.

The key insight: Partitions happen in real networks. They're not optional. Therefore, you always choose P — and then you choose between C and A during that partition.
How It Works — Why You Can't Have All Three
Imagine two database nodes: Node A and Node B. A user writes to Node A. Before Node A can replicate to Node B, the network between them partitions (packets dropped).

Now a second user reads from Node B. You have exactly two choices:

CP (return error): Node B knows it might be stale. It refuses to serve the read → returns an error or blocks. Consistent (no stale data served) but unavailable.

AP (return stale data): Node B serves its last known value. The user gets an answer but it might be stale. Available but inconsistent.

CA (consistent + available but no partition tolerance) only works if the two nodes share the same physical machine — i.e., no distribution. Single-node Postgres is "CA" in this sense, but it's not a distributed system. The moment you distribute, partitions are possible and you must choose.
CP vs AP — Real Examples
CP Systems
HBase, Zookeeper, etcd, Consul, MongoDB (w/ strong consistency). During partition: blocks or returns error. Use when: financial transactions, distributed locks, leader election — stale data is dangerous.
AP Systems
Cassandra, CouchDB, DynamoDB (default), DNS. During partition: serves potentially stale data. Use when: shopping cart, user profiles, social feeds — staleness is acceptable, availability is critical.
CP in practice: ZooKeeper
Quorum writes — a write only succeeds if majority of nodes ack. If majority unreachable, writes fail. Reads from leader only → always consistent. Used by Kafka for leader election.
AP in practice: Cassandra
Every node can accept writes. During partition, both sides accept writes → divergence. On recovery, last-write-wins (LWW) or CRDT merge resolves conflict. Tunable with consistency levels (ONE/QUORUM/ALL).

Trade-offs
✓ Choosing CP

Data correctness guaranteed. No conflicts to resolve. Simple reasoning about state.

✗ Choosing CP

Service is unavailable during partitions. Timeouts spike. User experience degrades or breaks entirely.

✓ Choosing AP

Always responsive. Users always get an answer. Great for high-availability products where staleness is tolerable.

✗ Choosing AP

Data can diverge. Conflict resolution is hard (LWW loses updates, CRDTs only work for some data types).

✓ Tunable consistency

Cassandra's consistency levels let you choose per-operation. Critical writes use QUORUM, non-critical reads use ONE.

✗ Tunable consistency

Developer must understand the tradeoff and set it correctly. Wrong setting = silent data bugs.

Failure Modes
Assuming CA: Treating a distributed DB like a single-node system. Works until the first network partition, then data diverges silently. Common in teams new to distributed systems.
False CP: System claims CP but uses eventual consistency under the hood. Reads return stale data during normal operation, not just during partitions. Check: does the system use quorum reads?
Conflict explosion (AP): Two nodes both accept conflicting writes. LWW discards one silently. For money/inventory data, this is catastrophic. Fix: use CP for writes, AP only for reads where staleness is tolerable.
Real-World Examples
Amazon DynamoDB: AP by default (eventual consistency). Can opt into strongly consistent reads per-request (closer to CP but slower, costs more RCU). Shopping cart uses AP — if your cart briefly shows an item that's out of stock, that's caught at checkout.

Google Spanner: Achieves "CP with high availability" using TrueTime (GPS/atomic clock synchronization). Technically breaks the CAP impossibility result in practice by making partitions extremely rare and brief. Still CP — during a real partition, it blocks.

Razorpay payments: Payment ledger must be CP — a duplicate payment due to conflict resolution is worse than a timeout. They use Postgres with synchronous replication + quorum reads.
Interview Perspective
  1. Never say "we can have all three." This is the classic trap. P is always required. You choose C or A.
  2. Frame as a business decision: "For the payment service, I'll choose CP — a stale balance is worse than a timeout. For the user profile service, I'll choose AP — showing a slightly old profile is fine."
  3. Know CAP ≠ binary: Consistency is a spectrum (linearizable → causal → eventual). Mention this to show depth.
  4. Probe question: "What does Cassandra do during a partition?" → It stays AP — serves stale data. Divergent writes are resolved by LWW or read-repair after partition heals.

PACELC
Extends CAP — what happens even without partitions?
Definition & Core Purpose
Proposed by Daniel Abadi (2012). PACELC recognizes the limitation of CAP: CAP only describes behavior during partitions. But partitions are rare. What about normal operation?

During normal operation (no partition), a distributed system must still choose between Latency (L) and Consistency (C). Replication takes time. If you wait for all replicas to confirm a write (strong consistency), latency goes up. If you acknowledge immediately and replicate asynchronously (low latency), you risk temporary inconsistency.

PACELC = If Partition: choose A or C. Else (no partition): choose L or C.
How It Works — The Full Framework
PAC: During a Partition, choose between Availability (A) or Consistency (C). This is the CAP choice.

ELC: Else (normal operation), choose between Latency (L) or Consistency (C).

This gives 4 positions a system can occupy:

PA/EL: During partition → Available. Normal → Low Latency. E.g., Cassandra, DynamoDB (default). Most available, least consistent. Writes always go through, but replicas may lag.

PC/EC: During partition → Consistent. Normal → Consistent. E.g., HBase, Google Spanner, VoltDB. Write must be ack'd by all (or quorum). Highest latency but strongest guarantees.

PA/EC: During partition → Available. Normal → Consistent. E.g., MongoDB (default in later versions). Tries to be available during chaos, consistent otherwise.

PC/EL: During partition → Consistent. Normal → Low Latency. Rare. Hard to implement. PNUTS (Yahoo) approximated this with timeline consistency.
Subtypes & Variants — Real Systems Mapped
Cassandra — PA/EL
Always available, always low latency. Consistency is eventual. Tunable per-op. Use for: time-series, activity feeds, user events.
Google Spanner — PC/EC
Always consistent, even cross-globe. Latency is higher (~10ms commit). Use for: global financial ledgers, multi-region ACID transactions.
DynamoDB — PA/EL
Default eventual consistency. Strongly consistent reads cost 2× RCU. Typical: PA/EL; with strong reads: PA/EC.
MySQL / Postgres (single master) — PC/EC
Single master = no partition scenario in typical setup. Synchronous replication = PC/EC. High latency writes if replica is slow.

Trade-offs
✓ PA/EL

Sub-millisecond writes. Always available. Linear scale-out. Global write throughput.

✗ PA/EL

Stale reads. Conflicts must be resolved by application. Not suitable for money, inventory, anything where two conflicting writes must not both succeed.

✓ PC/EC

Correctness guaranteed. Easier to reason about. No conflict resolution logic in application code.

✗ PC/EC

Higher write latency (quorum round-trips). Lower write throughput. More expensive (Spanner: $0.90/million writes vs DynamoDB: $0.25).

Failure Modes
Applying PA/EL to financial data: Cassandra LWW silently discards a payment write. Money disappears. Fix: use PC/EC for critical writes, PA/EL only for analytics/activity data.
Treating PACELC as binary: Most systems are tunable. Cassandra with QUORUM consistency is much closer to PC/EC. Don't label systems as fixed — ask what consistency level is configured.
Latency misattribution: Blaming the DB for slow writes when the issue is cross-DC replication. PC/EC systems across regions have inherent physics-limited latency floors (~150ms for transcontinental commits).
Real-World Examples
WhatsApp message storage: PA/EL. A message delivered slightly out of order or a brief duplicate is acceptable. Availability and low latency are critical — a message store that refuses writes during network turbulence is unacceptable.

Razorpay payment ledger: PC/EC. A missed payment or double-credit due to conflict resolution is catastrophic. They accept higher write latency for correctness.

Uber driver location: PA/EL. Driver location updates 4×/sec. Serving a location 500ms stale is fine. A system that refuses to serve driver location during a partition is worse than serving a slightly stale one.
Interview Perspective
  1. Use PACELC to impress. Most candidates know CAP. Mentioning PACELC and explaining the ELC dimension signals senior-level thinking.
  2. Frame every DB choice with PACELC. "I'm choosing Cassandra here because this service needs PA/EL — it's event data where staleness is fine and write availability is critical."
  3. Probe: "Your Cassandra cluster has network turbulence — what happens?" → Replicas accept writes independently. After recovery, anti-entropy (read-repair, Merkle trees) reconciles differences. LWW applies for conflicts.

Fault Tolerance & Redundancy
Active-passive · Active-active · SPOF elimination · Resilience patterns
Definition & Core Purpose
Fault tolerance = the ability of a system to continue operating correctly even when components fail.

Redundancy = having duplicate components so that if one fails, another takes over.

The key distinction: fault tolerance is the property; redundancy is the mechanism that achieves it. A system without redundancy has a Single Point of Failure (SPOF) — one component whose failure brings down the entire system. Eliminating SPOFs is the primary job of redundancy design.

The problem it solves: hardware fails, software crashes, networks partition, datacenters flood. In any sufficiently complex system, component failure is not a question of if — it's when.
How It Works — Active-Passive vs Active-Active
Active-Passive
Active-Active
SPOF Elimination
Active-Passive (Primary-Standby): One node handles all traffic (active). One or more nodes sit idle, continuously receiving replicated state (passive/standby).

Failover flow: Health monitor detects primary failure → promotes standby → updates DNS/VIP to point to new primary → traffic routes to new primary. Takes 10–60 seconds typically (depends on detection + promotion time).

Warm standby: Passive is running, receives replication, can promote quickly (seconds). Extra cost: you're paying for an idle machine.

Cold standby: Passive is not running. Must be started on failure. Cheaper but slow failover (minutes). Used for DR (Disaster Recovery) scenarios where minutes of downtime is acceptable.

When to use: Databases (Postgres streaming replication, MySQL primary-replica), stateful services where coordination is complex. AWS RDS Multi-AZ is active-passive.
Active-Active: Multiple nodes all handle live traffic simultaneously. No idle standby. If one fails, the load balancer redistributes its traffic to surviving nodes.

Requirements: Nodes must be stateless (or state must be synchronized in real-time, which is very hard). Shared state (DB, cache) must handle the combined write load.

Benefits: No failover delay (traffic re-routes instantly). Better resource utilization (no idle standby). Higher throughput (all capacity is used).

Challenges: Write conflicts if state is distributed (split-brain). More complex coordination. Load balancer must detect failures fast.

When to use: Stateless API servers (always active-active), Redis Cluster (multi-primary for different hash slots), multi-region active-active (very complex — Google, Netflix do this).
Every layer can have a SPOF. Here's how to eliminate them layer by layer:

DNS: SPOF if single DNS provider. Fix: use multiple DNS providers (Route 53 + Cloudflare), TTL tuning for fast failover.

Load Balancer: SPOF if self-hosted single LB. Fix: LB HA pair with Virtual IP (VIP) — keepalived for failover. Or use managed LB (AWS ALB — internally redundant).

Application Servers: SPOF if single instance. Fix: always 2+ instances behind LB. Minimum viable: N+1 (can lose one and still handle peak load).

Database: Biggest SPOF risk. Fix: primary-replica replication (read scaling + failover), or multi-primary (Galera, CockroachDB) for active-active writes.

Cache: SPOF if single Redis node. Fix: Redis Sentinel (primary failover) or Redis Cluster (horizontal sharding + HA).

Message Queue: SPOF if single Kafka broker. Fix: Kafka replication factor ≥ 3, min.insync.replicas = 2.

Datacenter: SPOF if single AZ/region. Fix: multi-AZ deployment (99.99%), multi-region (99.999%).

Trade-offs
✓ Active-Passive

Simple failover logic. No write conflicts. Standby has full consistent state.

✗ Active-Passive

Failover takes 30–60s (downtime). Standby wastes resources. Single point of write throughput.

✓ Active-Active

No failover delay. Full resource utilization. Horizontal write scale (for stateless layers).

✗ Active-Active

Write conflicts if stateful. Split-brain risk. Much more complex coordination. Hard for DBs.

✓ Multi-AZ

Survives datacenter-level failure (power, cooling, hardware rack failure). 99.99% availability.

✗ Multi-AZ

Cross-AZ data transfer costs. Replication latency adds to write path. ~1–5ms AZ-to-AZ round-trip.

✓ Multi-Region

Survives region-level disasters (hurricane, major cloud outage). 99.999% availability possible.

✗ Multi-Region

150ms+ cross-region latency for synchronous writes. Massive operational complexity. 2–3× cost.

Failure Modes
Split-brain (active-active DB): Network partition causes both nodes to accept writes as primary. On reconnect, state diverges irreconcilably. Fix: use a quorum-based system (Raft), or accept that only one node can be primary (use fencing tokens).
Thundering herd on failover: Primary dies → all clients reconnect to replica simultaneously → replica is overwhelmed. Fix: exponential backoff with jitter on reconnect, circuit breakers.
Replication lag: Async replica is 10 seconds behind when primary fails. 10 seconds of data lost. Fix: synchronous replication (higher write latency) or accept the RPO (Recovery Point Objective) tradeoff explicitly.
VIP failover storm: DNS TTL too high → clients still hit dead LB for minutes. Fix: short DNS TTL (30–60s) for internal services, use connection-level VIP (keepalived) for instant failover.
Real-World Examples
Netflix: Runs active-active across 3 AWS regions. Every region can serve 100% of traffic. They regularly run Chaos Monkey to randomly kill instances and Chaos Kong to kill entire regions, validating their fault tolerance continuously.

WhatsApp: Multi-datacenter active-active for message routing. Messages are stored in Mnesia (Erlang distributed DB) with replication across nodes. Node failure is handled by the BEAM VM's built-in process supervision (OTP).

AWS RDS Multi-AZ: Active-passive. Primary in AZ-1, synchronous standby in AZ-2. Automatic failover in 60–120s. DNS endpoint updates to point to new primary. Used by Razorpay, CRED, and most Indian product companies for their DB tier.
Interview Perspective
  1. Always mention SPOF elimination as a first principle. Walk through your design layer by layer: "At the LB layer, I'll use managed ALB — it's internally redundant. At the DB layer, I'll use RDS Multi-AZ."
  2. Know your RTO and RPO: RTO (Recovery Time Objective) = how long can you be down. RPO (Recovery Point Objective) = how much data can you lose. Design your failover strategy around these.
  3. Active-active for stateless, active-passive for stateful — this is the safe default to state. Then qualify with "for the DB tier, active-active requires CRDTs or conflict resolution — I'd default to active-passive with a read replica unless we need global multi-region writes."
  4. Chaos engineering: Mention that you'd validate fault tolerance with chaos testing (kill random instances, inject network latency). Netflix Chaos Monkey shows you've read beyond textbooks.
