Core design decisions
ID generation
Snowflake 64-bit ID → base62 encode → 7-char short code. Globally unique, no coordination at generation time, monotonically increasing (DB-friendly). Never MD5/hash — collision risk and longer codes.
301 vs 302
301 = permanent. Browser caches redirect forever — no more requests hit your servers. 302 = temporary. Every click goes through your redirect server. Use 302 for analytics/click tracking. Use 301 if you want to reduce server load and don't need tracking.
Storage
Write: short_code → long_url in Postgres (primary key on short_code). Read: Redis cache in front (hot URLs stay in memory, p99 redirect = Redis GET = ~1ms). Cache miss: DB lookup + populate cache.
Custom aliases
User provides slug (e.g. "bit.ly/my-link"). Try INSERT; on unique constraint violation → return conflict error. No pre-reservation needed.
Architecture — write and read paths
WRITE (create short URL): Client → POST /shorten {url, expiry, custom_alias?} API → validate URL → generate Snowflake ID → base62(id) → 7-char code API → INSERT INTO urls (code, long_url, user_id, expiry) → Postgres API → return https://sho.rt/{code} READ (redirect): Client → GET https://sho.rt/abc1234 LB → Redirect Service Redirect Service → GET redis:url:abc1234 HIT → 302 Location: {long_url} (~1ms) MISS → SELECT long_url FROM urls WHERE code='abc1234' → Postgres → SET redis:url:abc1234 {long_url} EX 86400 → 302 Location: {long_url} (~15ms) Expiry: Postgres TTL column + cron job. Redis key TTL aligned. Analytics: async Kafka event {code, ip, timestamp, referer} → ClickHouse aggregation for click counts
Failure modes
Cache stampede: Popular URL expires; 10K concurrent requests all miss → 10K Postgres queries. Fix: mutex lock on first miss (only one DB fetch); others wait.
DB bottleneck on reads: 115K redirects/sec >> Postgres capacity. Fix: Redis handles 99% of reads. Postgres is cold path only. Read replicas as backup.
Short code collision: Impossible with Snowflake — the ID is globally unique before base62 encoding. Only custom aliases can conflict (handled by unique constraint).
45-minute interview walkthrough
  1. Min 0–5: Clarify scale (100M URLs/day?), analytics needed?, custom aliases?, expiry?
  2. Min 5–10: Estimates — DAU, read:write ratio, storage for 5 years, p99 latency target
  3. Min 10–20: API design. Core entities: url table. ID generation choice (Snowflake + base62). 301 vs 302 and why.
  4. Min 20–35: Architecture — write path (Postgres), read path (Redis → Postgres). Scale the redirect: Redis cluster, multiple stateless redirect servers, CDN for extreme scale.
  5. Min 35–45: Deep dive on one asked by interviewer. Likely: cache stampede, expiry implementation, or analytics pipeline.

The central problem — fan-out
When a user with 130M followers tweets, how does that tweet appear in 130M timelines? This is the fan-out problem and the answer defines your entire architecture.
Two approaches
Fan-out on write (push)
On tweet, push tweet_id to every follower's pre-computed timeline in Redis. Read = O(1) Redis list fetch. Cost: writing to 130M Redis lists takes minutes for a celebrity. Write amplification is enormous.
Fan-out on read (pull)
On feed load, fetch tweet_ids of everyone the user follows, merge and sort. Read = expensive N DB queries merged. Simple to implement. Kills at scale (300M users × 200 followees each = 60B DB queries/day).
Hybrid (Twitter's actual approach)
Fan-out on write for normal users. Fan-out on read for celebrities (> 1M followers). On feed load: pre-computed timeline (from written fan-out) + merge celebrity tweets in real-time. Best of both.
Celebrity threshold
Define celebrity: > 1M followers. Do NOT fan-out their tweets. On feed load, fetch from a "celebrity tweets" cache and merge with pre-computed timeline. The 1M-follower threshold limits celebrity count to manageable size.
Architecture
WRITE (tweet creation): Client → POST /tweet {text, media_ids} Tweet Service → INSERT INTO tweets (id, user_id, text) → Postgres Tweet Service → publish to Kafka "tweets" topic Fan-out Service (Kafka consumer): Get follower list from Follower Service (from Redis/Cassandra) IF user.followers < 1M: FOR each follower: LPUSH timeline:{follower_id} {tweet_id} (Redis, async) LTRIM to keep last 800 tweet IDs ELSE (celebrity): Store tweet in Celebrity Cache only (no individual fan-out) READ (load feed): Timeline Service: 1. LRANGE timeline:{user_id} 0 19 ← pre-computed, 20 tweets 2. Fetch celeb tweets for celebs the user follows (from Celebrity Cache) 3. Merge + sort by timestamp → top 20 4. Hydrate tweet_ids → full tweet objects (Redis tweet cache → Postgres) 5. Return enriched feed
Deep dives interviewers probe
  1. What about users who rarely log in? Don't maintain Redis timeline for inactive users. On login, compute timeline on-demand. Flag users as "cold" after 30 days of inactivity.
  2. Media (images/video)? Store in S3 via presigned upload. CDN in front of S3 for serving. Tweet stores S3 key; timeline hydration adds CDN URL.
  3. Trending topics? Kafka stream → Flink sliding-window count per hashtag per 5 minutes → top-K heap → Trending Service → Redis sorted set.
  4. Search? Kafka → Elasticsearch (index tweet text, user, timestamp). BM25 relevance + recency boost.

Core architecture decisions
Real-time connection
WebSocket — persistent, full-duplex. Each user connects to a Chat Server. Connection is sticky (same server for the session). 500M concurrent connections → ~50K Chat Servers (10K connections/server typical in BEAM/Erlang, fewer in Java).
Message routing
User A and User B may be on different Chat Servers. A's server receives the message, looks up which server B is connected to (from Redis: user_id → server_id), forwards via internal pub/sub (Redis pub/sub or Kafka) to B's server, which pushes to B's WebSocket.
Message storage
Cassandra — write-heavy (1M writes/sec), message lookup by (chat_id, timestamp). No random reads — always range scans. Partition key: chat_id. Clustering key: timestamp DESC. Stores last 30 days of messages. S3 for older messages and media.
Offline delivery
If B is offline: store message in Cassandra with status "pending". When B reconnects, fetch undelivered messages from Cassandra (WHERE delivered = false AND recipient_id = B ORDER BY timestamp). Deliver, update status. Push notification (FCM/APNs) to wake B's app.
Architecture — message flow
A sends message to B: A's client → WebSocket → Chat Server A Chat Server A: 1. Persist message → Cassandra (message_id, chat_id, sender, text, ts, status=SENT) 2. ACK to A's client: message_id + status=SENT (receipt 1) 3. Look up Redis: {B's server} = "chat-server-47" OR "offline" IF B is online: Chat Server A → Redis pub/sub "server:chat-server-47" → Chat Server 47 Chat Server 47 → push via WebSocket → B's client B's client → ACK → Chat Server 47 → update status=DELIVERED Chat Server 47 → notify Chat Server A → update status=DELIVERED Chat Server A → push DELIVERED receipt to A (receipt 2, double tick) B reads message → B's client → status=READ → push READ receipt to A IF B is offline: message status = PENDING in Cassandra Send FCM/APNs push notification to B's device When B reconnects → fetch pending messages → deliver → update status
Online presence
How it works: When user B connects, Chat Server registers B's heartbeat in Redis: SET presence:user_B {server_id} EX 30. Client sends heartbeat every 20s. If key expires (B offline), presence disappears. "Last seen" stored in Cassandra on disconnect. Privacy option: users can disable last-seen sharing.
E2E encryption (Signal protocol, brief)
The key insight: The server never sees plaintext. Each client generates a public/private key pair. Public keys are uploaded to the WhatsApp server. When A wants to message B, A fetches B's public key, encrypts the message with B's public key. Only B's private key (never leaves B's device) can decrypt it. WhatsApp's servers route ciphertexts they cannot read.
  1. Group messages: Store group member list. Sender encrypts message once per member (or use Sender Key protocol — one shared key per group session). Group messages fan-out to all online members via their respective Chat Servers.
  2. Media: Client encrypts media, uploads to S3 via presigned URL, sends the S3 key + decryption key via message. Server stores encrypted blob; only recipient can decrypt using the key in the message.

Two distinct paths
Upload path
Creator uploads raw video → multipart upload to S3 → triggers transcoding pipeline (async) → multiple resolutions produced → segments stored in S3 → CDN pulls segments on first request → video is "ready" when all resolutions processed.
Viewing path
Viewer requests video → video player fetches manifest file (HLS .m3u8 or DASH .mpd) listing all available resolutions and segments → player chooses initial resolution based on network → fetches video segments from CDN → adapts resolution as network changes.
Transcoding pipeline
Creator uploads raw video → S3 (presigned multipart upload) S3 event notification → SQS queue → Transcoding Workers (EC2 fleet) Transcoding Worker (per video): 1. Download raw video from S3 (or stream it) 2. Transcode to 5 resolutions: 360p, 480p, 720p, 1080p, 4K (FFmpeg: H.264 for compatibility, H.265/AV1 for efficiency) 3. Segment each resolution into 6-second chunks (.ts files for HLS) 4. Generate manifest file (master.m3u8 → resolution-specific .m3u8 files) 5. Upload all segments + manifests to S3 6. Update video metadata in Postgres: status=READY 7. Invalidate CDN cache for this video (if re-encoding) For a 2-hour movie: ~1200 segments × 5 resolutions = 6000 files in S3 S3 key pattern: s3://videos/{video_id}/1080p/{segment_number}.ts
Adaptive bitrate (ABR)
How the player adapts: Bandwidth estimation runs continuously. Player measures download speed of each segment. If the last segment took twice as long as expected → network is slow → switch to lower resolution. If consistently fast → switch to higher resolution. The manifest file lists all resolutions; the player picks segment by segment. Netflix and YouTube switch every 6–10 seconds. The result: no buffering on slow networks, best quality on fast networks — all automatic.
CDN strategy
Popular videos are in CDN edge cache globally. First request from a PoP: CDN fetches from S3 (origin) and caches. Subsequent requests: served from edge. For truly popular videos (Gangnam Style), segments are in every PoP worldwide with near-zero origin traffic. For long-tail videos: CDN cache miss → S3 fetch. Videos are immutable once transcoded — CDN cache TTL can be years (content-addressed keys change if video changes).
  1. Recommendations: Collaborative filtering (users who watched X also watched Y). YouTube uses a two-stage model: candidate generation (YouTube DNN) → ranking (wide + deep model). Features: watch history, search history, demographics, context.
  2. Comments/likes at scale: Write to Kafka → async to Postgres/Cassandra. Like count uses Redis INCR (approximate, not exact). Exact count from Cassandra for display.

Geospatial indexing — geohash
Geohash: Encodes a lat/lng into a string of variable length. Longer string = finer precision. Characters that share a prefix are geographically close (mostly — edge cases exist at boundaries). Geohash precision 6 (~1.2km × 0.6km cell) is the sweet spot for ride-matching.

Drivers are stored in Redis Geospatial sets: GEOADD drivers:live {lng} {lat} {driver_id}. Rider requests a driver: GEORADIUS drivers:live {rider_lng} {rider_lat} 2 km. Redis returns list of driver_ids within 2km radius. Real-time. O(N+log M) where N = results in radius, M = total drivers.
Architecture — ride request flow
Driver app → location update every 4s → Location Service Location Service → GEOADD drivers:live {lng} {lat} {driver_id} → Redis Geo → (async) Kafka "driver-locations" → ClickHouse for analytics Rider requests ride: Rider app → POST /ride {pickup_lat, pickup_lng, destination} Ride Service: 1. GEORADIUS drivers:live {pickup} 2km → [driver_1, driver_2, driver_3] 2. Filter: status=AVAILABLE, acceptable car type 3. Score by ETA (use routing service or haversine estimate) 4. Pick best candidate → send offer via WebSocket/push to driver 5. Wait 10s for accept. If reject/timeout → try next candidate 6. Driver accepts → create trip in Postgres → notify rider via WebSocket (driver_id, ETA, car details) 7. Trip status tracked: MATCHED → ARRIVED → IN_PROGRESS → COMPLETED Surge pricing: Flink window job: demand/supply ratio per geohash (last 5 min) supply = available drivers in cell, demand = open ride requests surge multiplier = f(demand/supply). Cap at 3×. Feeds into Pricing Service. Riders see surge before confirming.
Deep dives
  1. What if the matched driver's location is stale? Redis location is ≤4 seconds old. Driver may have moved. On accept, re-verify location and recalculate ETA. If driver is now too far, find next candidate. Driver-side app shows real-time routing.
  2. Geohash boundary problem: Rider at the edge of a geohash cell. Closest driver is 100m away but in a different cell. Fix: query the rider's cell AND all 8 neighbouring cells. Redis GEORADIUS handles this naturally with a radius query.
  3. ETA calculation at scale: Lightweight at request time: haversine distance ÷ average speed for the area. Routing API (Google Maps / Mapbox) only called after driver accepts — cost and latency are prohibitive at query time for all candidates.

Three sub-systems
Crawling
Distributed URL frontier (priority queue). Crawlers fetch HTML, follow links, respect robots.txt. Store raw HTML in distributed storage. Detect duplicate pages (SimHash fingerprinting). Crawl frequency based on page change rate.
Indexing
Parse HTML → extract tokens → tokenise → build inverted index: term → {docId, positions, TF}. Compute PageRank (link graph). Store index sharded by term hash. Replicated 3× for HA.
Serving
Query → tokenise → look up each term in the index → intersect posting lists → rank by BM25 × PageRank × 200+ signals → return top-10 results → SERP rendered. Must complete in <200ms including network.
PageRank — the key insight
PageRank models importance as votes: A link from page A to page B is a vote for B. High-quality pages that get many links from other high-quality pages rank higher. Computed iteratively: PR(B) = sum of PR(A)/out_links(A) for all pages A that link to B. Converges after ~50 iterations on the web graph. Google runs this over 60 trillion pages. PageRank alone isn't enough — combined with 200+ signals including content freshness, mobile-friendliness, page speed, user click-through rate, and domain authority.
Query serving architecture
Query: "system design interview" Frontend → Query Processor Query Processor: 1. Spell check + query expansion ("system design interview guide") 2. Tokenise → ["system", "design", "interview"] 3. Fan out to Index Servers (sharded by term) → fetch posting lists for each term in parallel 4. Intersection: docs containing all three terms 5. Score: BM25(query, doc) × PageRank(doc) × freshness × personalisation 6. Top-K merge (heap of size K) 7. Fetch doc metadata (titles, snippets) from Document Store 8. Return ranked results Index sharding: 60T pages can't fit in one machine. Shard by URL hash (document sharding) → each shard has full index. OR shard by term hash → each shard has subset of terms. Google uses document sharding: each index shard covers a random subset of all web pages. Query fans out to all shards, results merged.
  1. How do you handle 100K QPS at <200ms? Heavy caching of popular queries (same query asked millions of times). Index in-memory on serving machines (no disk I/O on hot path). Aggressive pre-computation. Thousands of serving machines globally.
  2. Fresh content (news)? Separate freshness-optimised index updated in near-real-time alongside the main index. News results fetched from freshness index; general results from main. Merged in ranking.

What makes Netflix architecturally unique
Open Connect CDN
Netflix built its own CDN appliances (OCAs) co-located in ISP datacenters. Pre-positions popular content before it's requested. ISPs serve Netflix content from their own network — no inter-network transit costs. Netflix provides the hardware; ISPs provide the rack space and power.
Multi-region active-active
Runs in us-east-1, us-west-2, eu-west-1 simultaneously. All three serve live traffic. Chaos Kong regularly kills one region to validate failover. Spinnaker for multi-region canary deployments.
Hystrix / Resilience
Netflix invented Hystrix (circuit breaker) and open-sourced it. Every service-to-service call is wrapped. Degraded responses (cached data, empty recommendations) always preferred over failures.
A/B testing at scale
Every UI change, algorithm change, encoding parameter is A/B tested. Hundreds of experiments running simultaneously. Users are randomly bucketed. Primary metric: streaming hours. Secondary: retention, reactivation. Results determine product direction — not opinions.
Recommendation system
Stage 1 — Candidate generation: For a user, generate ~500 candidate titles. Collaborative filtering: "users similar to you watched these." Matrix factorisation (SVD) on the user-item interaction matrix. Content-based: titles similar to what you've watched.

Stage 2 — Ranking: Score each candidate with a ranking model (gradient boosted trees or neural net). Features: predicted rating, novelty, context (morning vs evening), device, what others in your household watch, trending in your region. Return top-N.

Personalised thumbnails: Not just what to show — which thumbnail for each title to show YOU. A/B tested per user segment. Some users click more on action thumbnails; others on character-focused. Netflix generates multiple thumbnails per title and serves the optimal one per user.
  1. Encoding strategy: Netflix stores multiple encodings per title per device type. A 4K TV gets HEVC at 15Mbps. A mobile on 3G gets H.264 at 300Kbps. Per-title encoding: dark scenes encoded differently than bright scenes (Variable Bitrate). Reduces storage and bandwidth without quality loss.
  2. Billing service: Idempotent charges. Exactly-once via Kafka transactions. Stripe as payment processor. Reconciliation job nightly comparing Stripe ledger to internal ledger.
Payment system
Idempotency · exactly-once · double-spend prevention · ledger
The non-negotiable requirements
Payments have the highest correctness bar of any system. A missed notification is annoying. A missed payment or a duplicate charge is catastrophic — legal liability, customer loss, regulatory consequences. Every component must be designed around exactly-once semantics, zero data loss, and full auditability.
Core design decisions
Idempotency key
Every payment request carries a client-generated UUID as idempotency key. Server stores (idempotency_key, result) with unique constraint. Second request with same key returns stored result without re-executing. Prevents duplicate charges on retry.
Double-entry ledger
Every transaction is two ledger entries: debit (money leaves account A) + credit (money enters account B). Ledger is append-only — no updates, no deletes. Balance = sum of all debits and credits. This is how Razorpay, Stripe, and all banks work. Enables perfect audit trail.
Database choice
Postgres with SERIALIZABLE isolation. ACID is non-negotiable. No eventually consistent database for payment writes. Balance check and debit in one serializable transaction prevents double-spend.
Exactly-once delivery
Kafka producer with idempotent=true + transactions. Consumer: process payment event → mark idempotency key as used in same DB transaction → commit Kafka offset atomically. Duplicate events: idempotency key already exists → no-op return stored result.
Payment flow — complete
Merchant checkout → POST /payments {amount, currency, method, idempotency_key} Payment API: 1. Check idempotency_key in DB → if exists, return cached result 2. Validate: amount > 0, currency valid, merchant has permission 3. BEGIN SERIALIZABLE transaction a. SELECT balance FROM accounts WHERE id = payer_id FOR UPDATE b. IF balance < amount → ROLLBACK → return INSUFFICIENT_FUNDS c. INSERT INTO ledger (id, account_id, type=DEBIT, amount, ref_id, ts) d. INSERT INTO ledger (id, account_id, type=CREDIT, amount, ref_id, ts) e. INSERT INTO payments (id, status=PROCESSING, idempotency_key, ...) 4. COMMIT ← durability guaranteed by WAL fsync 5. Publish to Kafka "payments" topic: {payment_id, amount, ...} Payment Processor (Kafka consumer): 6. Forward to payment gateway (Stripe/bank network) 7. Gateway response: SUCCESS or FAILED 8. Update payment status in DB (idempotent update) 9. Publish "payment.completed" event for downstream (notifications, order service) Reconciliation (nightly): Compare internal ledger totals with bank statement Flag any discrepancy > ₹0 for manual review Target: zero discrepancy
Failure modes
Gateway timeout: We charged the user but the gateway response timed out. Did it succeed? Query gateway status API with payment_id. Always build a reconciliation job to catch these.
App crash after gateway success but before DB update: On restart, Kafka re-delivers the event. Idempotency key check: already processed → return cached result. Gateway charge not duplicated because gateway also has idempotency key.
Double-spend: Prevented by SERIALIZABLE + SELECT FOR UPDATE. Two concurrent requests for the same account both try to debit: one gets the lock, checks balance, debits. The second waits, then checks balance (now lower), may see insufficient funds.

Priority handling
Separate Kafka topics by priority. High-priority consumers are allocated more partitions and more worker instances. Low-priority topics can be deprioritised when the system is under load — marketing emails can wait; OTP codes cannot. This is the Kafka implementation of priority queues: not a single priority queue (Kafka doesn't support that), but multiple topics with different consumer capacities.
  1. Deduplication: Same notification_id delivered twice (Kafka at-least-once). Worker checks Redis: SET delivered:{notification_id} 1 NX EX 86400. NX = only set if not exists. If already delivered, skip.
  2. Rate limiting per user: Don't send more than 3 push notifications/hour to a user (notification fatigue). Redis counter: INCR notif_count:{user_id}:{hour} with TTL. If > 3, queue for next hour or drop if low priority.
Rate limiter
Token bucket Redis implementation · distributed rate limiting
Design scope
This problem is often framed as: "Design a rate limiter that can rate-limit API calls across a distributed fleet of 100 API servers, with per-user, per-endpoint limits."
The core challenge — distributed state
Local rate limiter (wrong): Each API server maintains its own counter. User sends 100 requests spread across 10 servers — each server sees 10 requests. With a limit of 50/min, the user is allowed through despite sending 100.

Centralised rate limiter (correct): All API servers query a shared Redis cluster for rate limit state. One source of truth. Slightly higher latency (one extra Redis call per request, ~1ms) but correct.
Implementation — token bucket in Redis Lua
Algorithm: token bucket Key: "rl:{user_id}:{endpoint}" Capacity: 100 tokens Refill rate: 10 tokens/second Lua script (atomic check-and-consume): ───────────────────────────────────── local key = KEYS[1] local capacity = tonumber(ARGV[1]) -- 100 local rate = tonumber(ARGV[2]) -- 10 tokens/sec local now = tonumber(ARGV[3]) -- current time ms local data = redis.call('HMGET', key, 'tokens', 'ts') local tokens = tonumber(data[1]) or capacity local last_ts = tonumber(data[2]) or now local elapsed_sec = (now - last_ts) / 1000 tokens = math.min(capacity, tokens + elapsed_sec * rate) if tokens >= 1 then tokens = tokens - 1 redis.call('HMSET', key, 'tokens', tokens, 'ts', now) redis.call('EXPIRE', key, 3600) return {1, math.floor(tokens)} -- allowed, remaining else return {0, 0} -- rejected end Response on rejection: HTTP 429 Too Many Requests X-RateLimit-Limit: 100 X-RateLimit-Remaining: 0 X-RateLimit-Reset: {unix_ts when next token available} Retry-After: 3
Layered rate limiting
Layer 1 — IP level
At the API gateway / Nginx: 1000 req/min per IP. Catches unauthenticated abuse and DDoS before it hits application servers.
Layer 2 — API key / user
In application code via Redis Lua: per-user, per-endpoint limits. Catches authenticated misuse. Enables tiered plans (free: 100/min, pro: 10K/min).
Layer 3 — tenant/organisation
For B2B APIs: shared limit across all users of the same organisation. Prevents one user starving others on the same plan.
Redis cluster HA
Rate limiter Redis must itself be HA. Redis Cluster with 3 primaries + 3 replicas. On Redis failure: fail open (allow requests) rather than fail closed (block everything).
Distributed key-value store
Consistent hashing · replication · quorum reads/writes
What you're building
Design a system like DynamoDB or Cassandra — distributed, highly available, eventually consistent. Stores arbitrary byte values by key. Supports GET, PUT, DELETE. Must survive node failures without downtime. Scale to petabytes.
Core components
Consistent hashing ring
Keys are hashed onto a ring. Each node owns a range of the ring (vnodes for even distribution). A key routes to the first node clockwise from its hash position. Adding/removing nodes remaps only ~1/N keys.
Replication
Replication factor N=3 (industry standard). A key is stored on the primary node AND the next N-1 clockwise nodes. All N replicas are in different AZs. On PUT, write to all N; on GET, quorum of R must agree.
Quorum reads/writes
W = 2, R = 2, N = 3. W + R > N → strong consistency guaranteed. Alternatively: W=1, R=1 for high availability + low latency (eventual consistency). Tune per operation.
Conflict resolution
Concurrent writes to same key across replicas. Vector clocks to track causality. LWW (last-write-wins) by timestamp for simple cases. Application-level merge for complex cases (shopping cart → union).
Storage engine
Write path (on each node): PUT key=user:42 value={"name":"Somesh"} 1. Write to WAL (commit log) → durability guaranteed 2. Write to MemTable (in-memory sorted structure) 3. Return ACK (fast — no disk I/O on hot path) 4. Background: flush MemTable → SSTable file on disk (immutable) 5. Compaction: merge SSTables, remove tombstones (deleted keys) This is the LSM Tree (Log-Structured Merge Tree) used by Cassandra, RocksDB: - Writes are always sequential (fast, no random I/O) - Reads: check MemTable → check Bloom filter → check SSTable(s) - Bloom filter: probabilistic check "is this key in this SSTable?" (no false negatives) Read path: GET user:42 1. Check MemTable (most recent writes) 2. Check Bloom filter for each SSTable (skip if key definitely absent) 3. Binary search within matching SSTable 4. Merge results across quorum nodes (R=2 of N=3 nodes) 5. Return value with highest vector clock version
Failure handling
Node failure: Hinted handoff — writes destined for the failed node are temporarily stored on the next clockwise node with a "hint" (deliver to node X when it recovers). On recovery, hints are replayed.
Anti-entropy: Background Merkle tree comparison between replicas. Root hashes differ → find divergent subtree → sync just the differing keys. Keeps replicas consistent without full data transfer.

The Trie data structure
A Trie (prefix tree) stores strings such that all strings with a common prefix share a common path from the root. Each node represents a character. Path from root to a node = a prefix. Nodes marked as "end of word" are complete search terms.

Searching for all words with prefix "sys": traverse from root → s → y → s. All descendants are valid completions. Time complexity: O(prefix_length) to reach the node, O(N) to collect all descendants (where N = number of completions under that prefix).

The problem: A prefix like "a" might have billions of descendants. Collecting and sorting all of them is too slow. Solution: store the top-K terms and their frequencies at each node — not just the children.
Optimised Trie with top-K at each node
Each Trie node stores: { children: {char → node}, top5: [(term, frequency), ...], ← precomputed top-5 for this prefix is_end: bool } GET suggestions for "sys": 1. Traverse root → 's' → 'y' → 's' 2. Return node.top5 → [("system", 8M), ("system design", 5M), ...] O(prefix_length) time — no traversal of descendants needed! Building the Trie (offline, periodic rebuild): 1. Collect all queries from past 7 days (Kafka → Hadoop) 2. Count frequency per (query, day) → aggregate total 3. Sort by frequency descending 4. Build Trie: insert each query, at each prefix node update top5 (min-heap of size 5) 5. Serialise Trie → distributed storage 6. Push to Typeahead Servers (hot reload, no downtime) Rebuild frequency: hourly for trending, daily for general
Distributed Trie
The Trie doesn't fit in one machine at Google's scale (petabytes of query data). Partition by first character (or first two characters). Node A owns all prefixes starting with 'a'–'f'. Node B owns 'g'–'m'. Etc. This is range sharding on the Trie.

A query for "sys" routes to the node owning 's'. Client is unaware of sharding — a router/consistent-hash layer directs the request to the correct Trie shard.

Replication: each Trie shard is replicated 3× for HA. Reads are served from any replica (all reads — no writes at query time). Updates are propagated during the offline rebuild cycle.
Personalisation layer
Global Trie gives population-level suggestions. Personalisation adds a user-specific component: recent searches, frequently used terms, location-based suggestions. Implemented as a client-side cache (last 20 personal searches) merged with the global top-5 at display time. No server round-trip for personal suggestions. Global Trie result → merge with client-side personal history → display top-5 combined.
  1. Caching at the edge: "sys", "syst", "syste", "system" are all popular prefixes. Cache them at CDN edge — the same suggestions served to millions of users don't need to hit Trie servers. Cache TTL: 1 hour. Hit rate for top-1000 prefixes: >80% of all requests.
  2. Trending topics: Break news causes "earthquake india" to spike. The offline rebuild is too slow. Separate "trending" layer updated every 5 minutes from real-time query stream (Flink top-K sliding window). Merged with Trie results at serving time.
  3. Interview framing: "I'd start with a single-machine Trie with top-K at each node. Scale by sharding on prefix first character. Cache popular prefixes at CDN edge. Add a trending layer for real-time signals. Rebuild the Trie every hour from aggregated query logs."
