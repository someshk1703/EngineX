Object vs block vs file storage
S3 · EBS · EFS — precise use cases and why each exists
The fundamental distinction
These are not three competing solutions to the same problem — they are three different abstractions built for three entirely different access patterns. Choosing the wrong one creates either correctness problems (file-locking semantics that don't exist) or performance problems (treating an API call like a filesystem read).
Block storage — raw addressable blocks
Block (AWS EBS) What it is
Exposes raw storage as a sequence of fixed-size blocks (typically 512B or 4KB). The OS mounts it as a device and formats it with a filesystem (ext4, XFS, NTFS). The storage layer sees only block reads and writes — no filenames, no directories, no metadata. The filesystem is entirely in the OS layer above.

Think of it as a very fast, durable SSD attached over a network. Your EC2 instance treats it exactly like a local disk.
Use when
Running databases (Postgres, MySQL) that need low-latency random I/O. Boot volumes. Any application that needs a mounted filesystem with POSIX semantics. EBS gp3: 16,000 IOPS, 1,000 MB/s throughput.
Do not use when
Sharing across multiple EC2 instances simultaneously — EBS is single-attachment (except EBS Multi-Attach, which is limited). Not for large unstructured data at rest. Not for global access.
Object storage — flat namespace of immutable objects
Object (AWS S3) What it is
Stores data as objects in a flat namespace (bucket + key). Each object is identified by a unique key and carries its data + arbitrary metadata. Objects are immutable — you don't append to them or seek within them at the byte level. To "update" a file, you PUT the entire new version. Access is via HTTP API (GET, PUT, DELETE), not a filesystem mount.

S3 is effectively infinite in capacity, globally durable (11 nines), replicated across 3+ AZs, with a pay-per-byte model. It is not a filesystem.
Use when
User uploads (images, videos, PDFs). Static web assets (CSS, JS, fonts). Backups and archives. Data lake raw storage. ML training datasets. Anything large, immutable, and globally accessed. ML model weights, Docker layer cache.
Do not use when
You need random byte-range writes (databases). Sub-10ms latency (S3 GET latency is ~20–100ms). POSIX filesystem semantics (rename is not atomic in S3). High-frequency tiny object operations (cost per API call adds up).
File storage — shared POSIX filesystem
File (AWS EFS / Azure Files) What it is
A network filesystem (NFS-based for EFS) that multiple instances can mount simultaneously and access with full POSIX semantics — directories, symlinks, file locking, atomic rename. Unlike EBS (single instance) or S3 (API, not filesystem), EFS mounts like a real shared folder across an entire fleet.

Throughput scales with storage consumed. EFS Throughput Mode: Elastic (recommended — auto-scales), Provisioned (predictable throughput), Bursting (default, credits model).
Use when
Multiple EC2 instances or ECS tasks need shared filesystem access simultaneously. CMS media libraries, shared config files, ML model serving (same checkpoint read by many inference workers). Kubernetes PersistentVolumes shared across pods (ReadWriteMany).
Do not use when
Latency-critical DB workloads (NFS overhead vs EBS). High-throughput sequential I/O (S3 is cheaper per GB and scales better for bulk). You only need single-instance access (EBS is faster and cheaper for that).
Decision cheat sheet
Question to ask → Answer → Use ───────────────────────────────────────────────────────────────────── Need to run a database? → Single fast disk → EBS (gp3/io2) User uploaded a photo/video? → Store immutably → S3 Multiple pods reading same files? → Shared filesystem → EFS Static website assets? → Global HTTP access → S3 + CloudFront Boot volume for EC2? → OS filesystem → EBS Data lake raw files (Parquet, CSV) → Cheap, infinite → S3 ML model checkpoint (100 pods) → Shared read → EFS or S3 Postgres WAL volume → Low-latency I/O → EBS io2 Cost (rough, us-east-1, 2024): EBS gp3: $0.08/GB-month EFS: $0.30/GB-month (elastic) S3: $0.023/GB-month (standard) S3 Glacier: $0.004/GB-month (archival)
Interview perspective
  1. S3 is not a filesystem — say this early. "S3 is an object store accessed via HTTP PUT/GET. It has no concept of directories, no byte-range writes, and no atomic rename. It's not suitable for running a database on."
  2. The YouTube upload pipeline answer: "Video files go to S3 (multipart upload for large files). Metadata (title, description, owner) goes to Postgres on EBS. Transcoded versions go back to S3. Static thumbnails served from S3 via CloudFront."
  3. EFS is the answer for shared filesystem across pods. "For EngineX's resume parser that runs on 10 ECS tasks and needs to read the same NLP model file — EFS with ReadWriteMany semantics. The model is loaded once to EFS, all tasks read it concurrently."
  4. Probe: "Your Postgres instance is slow. What storage would you upgrade to?" → EBS io2 Block Express — up to 256,000 IOPS, sub-millisecond latency. Never S3 or EFS for this. The bottleneck in Postgres is random 4KB page I/O — that's exactly what block storage is optimised for.

Data lake vs data warehouse
OLAP vs OLTP · columnar storage · when each applies
The core distinction
Both store large amounts of data for analysis. The difference is what you store, in what format, and what questions you ask of it.

Data warehouse: Structured, schema-on-write. Data is transformed and loaded in a defined schema before storage. Optimised for predefined business queries (dashboards, reports). SQL-native. High cost per GB but very fast for the queries it's built for.

Data lake: Raw, schema-on-read. Store everything in its original format (JSON logs, CSV exports, Parquet files, images, audio). Apply schema only when reading. Cheap per GB (S3). Flexible — the same data can be queried in many different ways by different teams.
OLTP vs OLAP — the critical split
OLTP — Online Transaction Processing
Powers your application. Short, frequent, transactional queries. SELECT one user, INSERT one order, UPDATE one balance. Optimised for low latency per query and high concurrency. Row-oriented storage (reads full row). Examples: Postgres, MySQL, DynamoDB.
OLAP — Online Analytical Processing
Powers business intelligence. Long-running, analytical queries. "Total revenue by region for the past 12 months across 1 billion rows." Optimised for high throughput scans. Column-oriented storage (reads only the columns needed). Examples: Redshift, BigQuery, ClickHouse, Snowflake.
Row storage (OLTP)
Data for one row is stored contiguously: [id=1, name="Alice", age=28, ...][id=2, name="Bob", ...]. Full-row INSERT/UPDATE is one sequential write. Reading all columns of row 1 is one seek.
Columnar storage (OLAP)
Data for one column is stored contiguously: [id: 1,2,3,...][name: Alice,Bob,...][age: 28,31,...]. Query "average age" reads only the age column — skips name, email, etc. Massive I/O reduction for wide tables. Compresses beautifully (same-type values compress 10–50×).
Columnar storage — why it's so fast for analytics
A table with 1 billion rows and 100 columns. Query: SELECT AVG(revenue) WHERE country='IN'. Only needs two columns (revenue, country).

Row store reads: Must scan all 100 columns × 1B rows = 100B values transferred from disk just to get 2B values we actually need. I/O wasted: 98%.

Column store reads: Read only revenue column (1B values) + country column (1B values). Skips all other 98 columns entirely. I/O reduction: ~98%.

Compression bonus: A column of country codes ("IN", "US", "GB") compresses with run-length encoding from 3 bytes/row to ~0.1 bytes/row on a sorted dataset — 30× compression. Row stores can't compress this efficiently because the column values are interleaved with other column types.

Vectorised execution: Modern OLAP engines (ClickHouse, DuckDB) process columns in SIMD-friendly batches of 1024 values at a time using CPU vector instructions. Processing 1024 integers in one CPU instruction vs one at a time = 1000× throughput.

Trade-offs
✓ Data lake

Store everything cheaply (S3 = $0.023/GB vs Redshift $0.25/GB). Schema flexibility — add new fields without migration. Multiple query engines can read the same data. Raw data preserved for future use cases not yet known.

✗ Data lake

Queries slower without materialised views. Schema-on-read means data quality issues only discovered at query time. "Data swamp" risk — raw data without governance becomes unusable.

✓ Data warehouse

Blazing fast for the queries it's designed for. Enforced schemas = data quality guarantees. Business users can use SQL without understanding Spark or Parquet.

✗ Data warehouse

Expensive per GB. Schema changes require migrations. Can't store unstructured data (images, audio, free text). Vendor lock-in (Snowflake, Redshift have proprietary features).

Interview perspective
  1. Always separate OLTP from OLAP. "I'd never run analytics directly on the production Postgres — it would kill OLTP performance. Analytics goes to a separate columnar store fed by CDC or batch export."
  2. Columnar = column-skip + compression. Know both benefits cold. "Reading 2 of 100 columns from a billion-row table is 50× faster in a column store because the other 98 columns are never touched."
  3. ClickHouse is the FAANG answer for high-throughput OLAP. "For real-time analytics at Razorpay (transaction volume dashboards, fraud rate by merchant), I'd use ClickHouse — sub-second queries on billions of rows, 10–100× faster than Redshift for high-cardinality GROUP BY."
  4. Probe: "How does Uber generate surge pricing based on real-time demand?" → OLTP path: driver/rider events written to Kafka. Stream processing (Flink): sliding-window aggregation of supply/demand per geohash. Output feeds the surge pricing service directly. Analytics: same events land in S3/ClickHouse for historical analysis and model training. Never touch the Postgres for this.

Search engines
Elasticsearch internals · inverted index · TF-IDF · BM25
The core data structure — inverted index
A regular database index maps row → columns. An inverted index maps term → list of documents containing that term. It "inverts" the document-to-word relationship into a word-to-document relationship. This is what makes full-text search fast — instead of scanning every document for your search term, you look up the term in the index and get the exact list of matching documents in microseconds.

How Elasticsearch indexes a document
1. Analysis pipeline: Raw text → Tokeniser (split on whitespace/punctuation) → Token filters (lowercase, remove stop words, stem). "Payment Gateway" → ["payment", "gateway"]. "Payments" → "payment" (stemming). This is why "payments" matches "payment".

2. Posting list: For each term, store: list of document IDs, position within each document (for phrase matching), term frequency per document (for relevance scoring).

3. Segments: Elasticsearch (built on Lucene) writes to in-memory buffers that are periodically flushed to immutable segment files. A search queries all segments and merges results. Background segment merging reduces the number of files over time. Near real-time (NRT) search: new documents visible in ~1 second (after the refresh interval).
Relevance scoring — TF-IDF and BM25
TF-IDF
BM25
TF-IDF (Term Frequency — Inverse Document Frequency):

TF (Term Frequency): How often does the search term appear in this document? More occurrences = more relevant. But raw count is not enough — a 1000-word document with 5 occurrences should not beat a 10-word document with 3.

IDF (Inverse Document Frequency): How rare is this term across all documents? "the" appears in every document — it's meaningless. "BM25" appears in few documents — it's highly discriminative. Terms that appear in fewer documents get higher IDF scores.

Score = TF × IDF: High score when the term appears often in this document but rarely across all documents. This is the exact signal you want for relevance.

Weakness: TF grows unboundedly. A document with 100 occurrences of a term scores 10× a document with 10 occurrences — but is it really 10× more relevant? In practice, relevance saturates. BM25 fixes this.
BM25 (Best Match 25) — Elasticsearch's default since version 5:

BM25 improves TF-IDF in two ways:

1. TF saturation: TF_bm25 = TF / (TF + k1). As TF grows, the score saturates. Default k1=1.2. A document with 10 occurrences doesn't score 10× a document with 1 — it scores about 2×. Reflects diminishing returns of repetition.

2. Field length normalisation: norm = (1 - b) + b × (field_length / avg_field_length). Default b=0.75. A term in a short document is more significant than the same term buried in a 10,000-word document. BM25 penalises longer documents relative to the corpus average.

Result: BM25 consistently outperforms TF-IDF in real search benchmarks (TREC, MS MARCO). It's the practical default for all modern search engines including Lucene, Elasticsearch, Solr, and Postgres's full-text search.
Elasticsearch in system design
Elasticsearch is NOT a primary database — it is a search index. Source of truth: Postgres (orders, products, users) │ sync via CDC (Debezium) or dual-write ▼ Elasticsearch index (search read model) │ query with full-text, fuzzy, aggregations ▼ Search results returned to user Index design for e-commerce product search: PUT /products/_mapping { "name": { "type": "text", "analyzer": "english" }, "description": { "type": "text", "analyzer": "english" }, "brand": { "type": "keyword" }, ← exact match, filter "price": { "type": "float" }, ← range queries "category_id": { "type": "keyword" }, ← faceted filter "stock": { "type": "integer" } } Query: "blue nike running shoes" under ₹5000 in "footwear" GET /products/_search { "query": { "bool": { "must": { "multi_match": { "query": "blue nike running shoes", "fields": ["name^3", "description"] ← name is 3× boosted }}, "filter": [ { "term": { "category_id": "footwear" } }, { "range": { "price": { "lte": 5000 } } }, { "range": { "stock": { "gt": 0 } } } ] } } }
Interview perspective
  1. Elasticsearch is always a secondary index. "I'd never use Elasticsearch as the source of truth. It syncs from Postgres via CDC. If Elasticsearch goes down, data is safe in Postgres and the index can be rebuilt."
  2. Know the inverted index by heart. "An inverted index maps term → posting list of (docId, positions, TF). A search looks up each query term in the index — O(1) per term lookup — then intersects the posting lists. That's why full-text search over a billion documents completes in under 100ms."
  3. BM25 over TF-IDF. "BM25 adds TF saturation and field length normalisation. A term appearing 100 times scores only marginally better than 10 times — relevance has diminishing returns, which BM25 models correctly."
  4. Probe: "How would you add search to Zepto's product catalog?" → Sync products from Postgres to Elasticsearch via Debezium CDC. Index name (text, English analyzer), brand (keyword for exact filter), price (float for range), category (keyword for facets). Query: BM25 on name and description, filters on category + price + in-stock. Rank by BM25 score × in-stock boost × seller rating.

Time-series databases
InfluxDB · Prometheus · downsampling · retention policies
Why time-series DBs exist
A time-series is a sequence of (timestamp, value) pairs, usually from a sensor, metric, or event stream. In a general-purpose relational DB, storing 1M metrics/second would mean 86 billion rows/day — inserts are the bottleneck, range queries are expensive (even with indexes), and disk consumption explodes.

Time-series databases are purpose-built to exploit temporal patterns: data arrives in timestamp order, old data is queried far less than recent data, queries are always time ranges, and precision requirements decrease with age (you care about the last minute at 1-second granularity but only care about last year at 1-hour granularity).
Prometheus vs InfluxDB
Prometheus
Pull-based — Prometheus scrapes /metrics endpoints every 15s. Stores metrics locally in a custom TSDB format optimised for recent data. Query language: PromQL. Designed for monitoring infrastructure and services. Integrates with Grafana for dashboards. AlertManager for alerting. Short retention (15 days default) — use Thanos or Cortex for long-term storage.
InfluxDB
Push-based — applications write metrics to InfluxDB via HTTP API. Time-series stored in "measurements" (like tables) with tags (indexed metadata) and fields (values). Query language: Flux (functional) or InfluxQL (SQL-like). Better for IoT sensor data, application metrics, business events. Built-in downsampling and retention policies.
TimescaleDB
Postgres extension. Converts Postgres into a time-series DB with automatic time-based partitioning (hypertables), compression, and continuous aggregates. Best when you already run Postgres and want time-series without a separate DB. Full SQL compatibility.
ClickHouse for time-series
Not a dedicated TSDB but handles time-series at extreme scale due to columnar storage + vectorised execution. Used by Cloudflare (1M metrics/sec), Uber (analytics). When you need OLAP-style aggregations over time-series data at massive scale.
Downsampling — the key technique
Problem: Raw data at 1-second resolution for 1 year = 31M data points per metric. Querying a year's worth of a metric for a dashboard takes forever and returns way more points than any graph can display.

Downsampling: Aggregate raw data into lower-resolution summaries as it ages.
Raw (last 7 days): 1-second resolution → keep every point.
Medium (last 30 days): 1-minute resolution → one point per 60 raw points (take avg/max/min).
Long (last 1 year): 1-hour resolution → one point per 3600 raw points.
Archive (beyond 1 year): 1-day resolution or discard.

A year of a metric at 1-day resolution = 365 points. Trivial to query and display.

Retention policies
Retention policy: Automatic deletion of data older than a configured age. In InfluxDB: define multiple RPs with different durations and shard group durations. New data goes to the default RP. A continuous query downsamples from the raw RP to longer-duration RPs automatically.

CREATE RETENTION POLICY "raw" ON "metrics" DURATION 7d SHARD DURATION 1d DEFAULT
CREATE RETENTION POLICY "monthly" ON "metrics" DURATION 30d SHARD DURATION 7d
CREATE RETENTION POLICY "yearly" ON "metrics" DURATION 365d SHARD DURATION 30d

Prometheus equivalent: --storage.tsdb.retention.time=15d. For longer retention, Thanos writes to S3 with downsampling at 5m and 1h resolution automatically.
Interview perspective
  1. Name the pattern: collect → downsample → retain. "I'd collect raw metrics at 10s intervals in Prometheus. After 7 days, downsample to 1-minute resolution. After 30 days, 1-hour. Store long-term in Thanos/S3. This reduces storage 99%+ while preserving analytical fidelity."
  2. Prometheus for infra, InfluxDB for business metrics. "Prometheus scrapes service health metrics. InfluxDB receives payment events and business KPIs where I need to push from the application."
  3. Probe: "Monitor 10,000 IoT devices sending 1 reading/second." → 10K readings/sec. Push to InfluxDB (push-based). Tag with device_id and location (indexed). Field: temperature/pressure. Retention: raw 7 days → 1-min downsample 30 days → 1-hour 1 year. Alert on PromQL-style rules for anomalies. Dashboard in Grafana.

Blob storage
Chunked upload · multipart upload · presigned URLs
The problem with large file uploads
Uploading a 5GB video file as a single HTTP request is fragile. One network hiccup at minute 45 of a 50-minute upload means starting from zero. The connection holds open for the entire duration, tying up server resources. If the server crashes mid-upload, the partial file is useless. And your API server shouldn't be handling large file data at all — it should focus on business logic.

Multipart upload solves all of this by splitting the file into chunks that upload independently and in parallel.
S3 multipart upload — step by step
1
Initiate. Client calls POST /bucket/key?uploads. S3 returns an UploadId. This is the session identifier for all subsequent parts.
2
Split and upload parts. Client splits file into N chunks (minimum 5MB each, except last part). Uploads each part: PUT /bucket/key?partNumber=N&uploadId=X. Parts can upload in parallel (8–16 concurrent uploads typical). S3 returns an ETag for each part.
3
Complete. Client sends all part ETags in order: POST /bucket/key?uploadId=X with body listing ETag of each part. S3 assembles the parts into a single object. The assembled object appears atomically — it's either fully present or absent.
4
Abort on failure. If the upload is abandoned, call DELETE /bucket/key?uploadId=X to clean up partial parts. Unaborted multipart uploads accumulate storage charges for the parts — set an S3 lifecycle rule to auto-abort incomplete multipart uploads after 7 days.

Presigned URLs — the right way to handle uploads
The wrong way: Client → API server → S3. The file goes through your API server. This ties up threads, wastes bandwidth (the file travels twice — client to server, server to S3), and makes your API server a bottleneck for large files.

The right way — presigned URLs: Client asks your API server for a presigned URL. Server generates a time-limited URL (signed with your S3 credentials) and returns it. Client uploads directly to S3 using that URL — no data passes through your server. S3 enforces the time limit and size constraints.

Presigned GET URL: Client can download an object from S3 without it being publicly accessible. URL expires in N seconds. Signed with your credentials to prove authorisation.

Resumable uploads
The problem: User is halfway through a 5GB upload. Connection drops. Restart from zero? Unacceptable.

Resumability with S3 multipart: Track which parts have been uploaded (store ETag list in your DB or client-side). On resume, query which parts are missing and upload only those. S3 keeps incomplete multipart uploads for up to 7 days — plenty of time to resume.

TUS protocol: An open protocol for resumable uploads. The client stores the upload offset. On resume, it sends a PATCH request with the current offset. The server continues from where it left off. Used by Vimeo, Cloudflare, and uploadcare. Worth mentioning for a YouTube-like system design.
Interview perspective
  1. Presigned URLs are mandatory for any file upload system. "Files never go through my API server. Client requests a presigned URL, uploads directly to S3. This keeps my API servers stateless and removes them from the data path entirely."
  2. Multipart upload minimum part size is 5MB. "S3 requires parts ≥ 5MB (except the last). I'd use 20–100MB chunks for large files, uploading 8 in parallel. For a 10GB video at 100MB/s with 8-parallel 100MB chunks, upload completes in ~12 seconds vs 100s sequential."
  3. Always set a lifecycle rule to abort incomplete uploads. "Abandoned multipart uploads accumulate storage charges. I'd add an S3 lifecycle rule: abort any incomplete multipart upload older than 7 days."
  4. EngineX resume upload for PDFs: "For EngineX, users upload resumes (typically small — < 5MB). For those I'd use a simple presigned PUT to S3, no multipart needed. For a video-based system like Zombie Hunt game replays, multipart upload with resumability would be essential."
