Cache fundamentals
Hit/miss · hit rate · cold start · TTL
Definition & core purpose
A cache is a fast, temporary storage layer placed in front of a slower data source. Its entire purpose: serve frequently requested data faster than the origin can. Every cache exploits temporal locality — the observation that data accessed once is likely to be accessed again soon.

The performance improvement is multiplicative. A Redis GET takes ~0.1ms. A Postgres query on a cold page takes ~5–50ms. A cache hit is 50–500× faster. At 1M RPS, a 90% hit rate means only 100K requests reach the DB — 10× less load.
Core metrics — know these precisely
Cache hit
Requested key exists in cache and is not expired. Returns immediately. No DB call.
Cache miss
Key not in cache (never set, expired, or evicted). Must fetch from origin, then optionally populate cache.
Hit rate
hits / (hits + misses). Target: 90%+ for production caches. Below 80% = cache is too small or TTL too short. Each % point matters at scale.
Cold start
Cache is empty (new deployment, restart, flush). All requests are misses → DB gets hammered. First 5–10 minutes can saturate origin. Warm with pre-population or accept the spike.
TTL (Time to Live)
Expiry duration. After TTL, key is automatically deleted. Controls staleness vs freshness. Tuning TTL is the primary knob for the consistency/performance tradeoff.
Stale data
Cache holds an old version after the DB was updated. Acceptable for feeds, profiles. Catastrophic for balances, inventory counts.

TTL strategy
Short TTL (seconds): Fresh data, high cache miss rate, DB still gets load. Good for: inventory counts, stock prices, seat availability.

Long TTL (hours/days): High hit rate, stale risk. Good for: product catalog, user profile, static content. Pair with explicit invalidation on write.

TTL jitter: If 1M keys all expire at the same second (e.g., set at midnight + 24h TTL), you get a thundering herd at midnight. Add random jitter: TTL = base + random(0, jitter_range). Spreads expiry across a window.
Failure modes
Cache stampede (cold start): All keys expire simultaneously → 10K concurrent requests all miss → 10K DB queries. Mitigation: staggered TTLs, mutex/lock on first miss (only one request fetches, rest wait), background refresh before expiry.
Cache as source of truth: Team starts reading from cache assuming it's always fresh. Cache restarts → data appears lost. Mitigation: cache is always a read-through layer, never the system of record.
Memory leak via no TTL: Keys set without TTL grow forever. Cache fills, evictions kick in erratically. Always set a TTL or an explicit eviction policy.
Interview perspective
  1. Quantify immediately. "At 100K RPS with a 95% hit rate, only 5K requests hit the DB — our DB needs to handle 5K QPS, not 100K."
  2. TTL is a design decision, not an afterthought. Interviewers probe: "how fresh does this data need to be?" — answer with a concrete TTL and justify it.
  3. Cold start = deployment risk. Mention cache warm-up strategy (pre-populate top-N keys on deploy, use a read replica to seed cache).
  4. Always ask: what's the cost of a stale read vs a cache miss? This frames every subsequent caching decision.

Cache strategies
Cache-aside · write-through · write-behind · read-through
The four patterns
Cache-aside
Write-through
Write-behind
Read-through
Cache-aside (lazy loading) — the most common pattern. Application code is in charge of the cache.

Read: check cache → hit → return. Miss → fetch from DB → write to cache → return.
Write: write to DB → invalidate or update cache entry.

The application manages both the cache and the DB. The cache is populated on demand (lazy). Keys that are never requested are never cached — good for large datasets where only a hot subset matters.

Risk: Cache and DB can diverge on concurrent writes (race condition). Thread 1 writes DB, Thread 2 reads stale cache before Thread 1 can invalidate. Mitigation: short TTL, or use CAS (compare-and-set) on invalidation.
Write-through — every write goes to both cache and DB synchronously before acknowledging to the client.

Write: write to cache → write to DB → return.
Read: always from cache (it's always up to date with DB).

Benefit: Cache is always consistent with DB. No stale reads.
Cost: Write latency doubles (cache write + DB write in series). Cold-start problem: keys written before a cache restart are lost until re-written. Unpopular items pollute cache with data that's never read (write amplification).
Write-behind (write-back) — write to cache immediately, acknowledge to client, then asynchronously persist to DB.

Write: write to cache → ack client → async job writes to DB later.

Benefit: Lowest write latency (cache write only on critical path). Can batch multiple writes to DB (efficiency). Cache absorbs write bursts.

Risk: If cache crashes before async write, data is lost. Durability is sacrificed. Not suitable for financial data. Use only when: write volume is very high, some data loss is acceptable (analytics events, counters), or writes to the same key happen rapidly (coalesce 100 counter increments into 1 DB write).

Real use: Memcached + MySQL at early Facebook for like counts. Redis as a write buffer for Cassandra with Kafka async flush.
Read-through — cache sits in front of DB and handles its own population. Application only ever reads from the cache.

On miss: cache itself fetches from DB and stores the result, then returns it. Application never talks to DB directly.

Benefit: Clean application code — no cache logic in the application layer. Cache is always the data layer. Works well with provider libraries (Ehcache, Spring Cache, NCache).

Difference from cache-aside: In cache-aside, the application fetches from DB and populates cache. In read-through, the cache does both — application is decoupled from DB entirely.

Limitation: First request for any key is always slow (cold miss, cache-fetches from DB). Cold-start is visible to users.
Architecture — cache-aside (most common)

Trade-offs by strategy
✓ Cache-aside

Only hot keys are cached. Resilient — cache can restart without data loss (repopulates). Simple to reason about.

✗ Cache-aside

Stale window between write and invalidation. Application code must manage cache. Cold start exposes DB.

✓ Write-through

Always consistent cache. No stale reads after writes. Simple read path.

✗ Write-through

Write latency up. Cache polluted with never-read keys. Miss on cold start still possible.

✓ Write-behind

Lowest write latency. DB protected from write bursts. Efficient batching.

✗ Write-behind

Data loss risk on cache crash. Complex async pipeline. Hard to debug.

Interview perspective
  1. Default answer: cache-aside for reads, invalidate on write. State this confidently and explain the race condition risk and how TTL mitigates it.
  2. Write-through if asked for strong consistency. "I'd use write-through for the user settings service — stale settings are confusing to users."
  3. Write-behind if probed on high-throughput counters. "For like counts or view counts where near-real-time is fine and some loss is acceptable, write-behind with periodic DB flush is the right call."
  4. Common trap: "Update cache on write" (write-through read path) — mention the race condition between concurrent writes. Deleting the key on write (cache-aside) is safer than updating it.

Eviction policies
LRU · LFU · FIFO · random — when each applies
The four policies
LRU
Least Recently Used
Evicts the key that hasn't been accessed for the longest time. Implemented as a doubly-linked list + hashmap: O(1) get and set. Every access moves the key to the front. Tail = eviction candidate.

Use when: working set fits in memory, access patterns have temporal locality (recently accessed = likely to be accessed again). Most common default.

Weakness: One-time large scan evicts the entire hot working set. A single batch job that reads 10M keys thrashes the LRU list, evicting data that was actually hot.
LFU
Least Frequently Used
Evicts the key with the lowest access count. Tracks frequency of access, not recency. Keys that are accessed rarely are evicted first regardless of when.

Use when: some keys are permanently hot (product homepage, top-10 SKUs), others are accessed once and forgotten. LFU keeps those hot keys forever.

Weakness: Frequency counts decay slowly. A key that was hot 6 months ago but is now cold has a high count and resists eviction. Fix: LFU with decay (Redis uses a logarithmic counter with time-based decay — allkeys-lfu).
FIFO
First In First Out
Evicts the oldest key regardless of access pattern. Simple queue. No tracking of recency or frequency.

Use when: all data has equal value (e.g., rate-limiting windows, session tokens), or data expires naturally anyway and eviction just needs to clear space predictably.

Weakness: No intelligence about hot vs cold data. A key inserted 30 days ago may still be very hot (homepage config), but FIFO evicts it anyway. Not suitable for general-purpose caching.
Random
Random eviction
Evicts a random key. Zero overhead — no bookkeeping. Surprisingly effective when all keys have similar access patterns (uniform distribution).

Use when: access is uniform (e.g., DNS cache where any domain is equally likely to be queried next), or when LRU/LFU overhead matters (very high throughput, tiny cache entries).

Redis: Uses approximated LRU (random sample of N keys, evict the least recently used among the sample). Avoids true LRU's O(N) overhead while approximating it well enough.

Interview perspective
  1. Default: LRU. It's the right answer for 90% of caching problems. Know its O(1) implementation (doubly linked list + hashmap).
  2. Upgrade to LFU when the interviewer says "some items are permanently hot." Then mention Redis's approximated LFU with time-decay.
  3. Probe: "What happens when a batch job scans the cache?" → LRU thrashing. Solutions: scan-resistant LRU (2Q or ARC algorithm), or run batch jobs against a separate read replica, not the cache.
  4. LRU implementation: "I'd implement it with a HashMap for O(1) lookup and a doubly linked list for O(1) move-to-front and tail eviction." Be ready to code this — it's a top 10 LeetCode problem (LRU Cache, #146).

How it works — Redis internals
Single-threaded event loop (command processing): Redis processes one command at a time — no locks, no race conditions inside. This is why Redis is safe to use for distributed locking (Redlock). Since Redis 6.0, I/O is multi-threaded but command execution remains single-threaded.

Persistence — RDB vs AOF:
RDB (snapshotting): fork() creates a child process, child writes a point-in-time snapshot to disk. Parent continues serving requests. Used for backups. Fast restart, but can lose last N minutes of writes.

AOF (Append-Only File): every write command is appended to a log file. On restart, Redis replays the log. Slower than RDB but near-zero data loss (can configure fsync every 1s or every write). File grows forever → periodic rewrite via BGREWRITEAOF.

Production: use both — RDB for fast restart + AOF for durability.

Redis data structure internals:
String: up to 512MB, SDS (Simple Dynamic String), O(1) GET/SET.
Sorted Set: skip list + hash map. O(log N) insert/remove. O(log N + M) range queries. Used for leaderboards, rate limiting.
HyperLogLog: probabilistic cardinality counter. Counts unique items with ~0.81% error using only 12KB of memory. Used for: DAU counting, unique URL visitors.
Redis use cases by data type
String
Session tokens, cached JSON blobs, counters (INCR for atomic increment), distributed locks (SET key value NX EX 30).
Hash
User profiles (field-level updates without deserializing whole object), product attributes. HSET/HGET/HGETALL.
Sorted Set
Leaderboards (score = points, ZREVRANGEBYSCORE), rate limiting (score = timestamp), feed ranking. ZADD/ZRANGEBYSCORE.
List
Message queues (LPUSH + BRPOP for blocking pop), activity feeds (bounded list of last N events), job queues.
Set
Unique visitors, tags, friend lists. SADD/SMEMBERS/SINTER for intersection (mutual friends).
Stream
Durable append-only log. Consumer groups. Like a lightweight Kafka inside Redis. Used for audit logs, event sourcing at smaller scale.
When to choose each
Choose Redis when

You need rich data types, persistence, replication, pub/sub, Lua scripts, or distributed locks. This is 95% of use cases.

Choose Memcached when

Pure simple string cache, extremely high throughput on multi-core (Memcached's multithreading wins here), or you need linear horizontal scaling without coordination overhead.

Interview perspective
  1. Default to Redis for every caching question. Memcached is the niche answer for specific multi-core throughput requirements.
  2. Know Redis data types by use case: Leaderboard = Sorted Set. Rate limiting = Sorted Set or INCR+EXPIRE. Session = String. User profile = Hash. Feed = List. Unique counters = HyperLogLog.
  3. Probe: "How do you implement a distributed lock with Redis?" → SET key uuid NX EX 30. NX = only set if not exists. EX = expire in 30s. Unlock: Lua script to check UUID matches before DEL (prevents another thread's lock from being deleted).
  4. Probe: "What's Redis Sentinel vs Redis Cluster?" → Sentinel = HA for single-primary setup (monitors, auto-promotes). Cluster = horizontal sharding across multiple primaries (16,384 hash slots, each node owns a range).

Cache invalidation
The hard problem — event-driven · TTL · version tags
Definition & why it's hard
Phil Karlton: "There are only two hard things in computer science: cache invalidation and naming things."

Cache invalidation is the problem of ensuring that when data in the source of truth changes, the cached copy is updated or removed in a timely and correct way — without either serving stale data or creating race conditions.

The difficulty: a cache is a copy. The moment a copy exists, it can diverge from the original. The question is: how quickly, and what are the consequences?
Three invalidation approaches
TTL-based
Event-driven
Version tags
TTL-based (passive expiry): Set a TTL on every cache entry. When it expires, the next read is a miss and repopulates from the source.

Simplest approach. No coordination needed. Works even if you can't reliably detect writes.

Staleness window = TTL. If TTL = 5 minutes, data can be up to 5 minutes stale. For product descriptions, that's fine. For inventory count, probably not.

Tuning TTL: Start with the longest TTL where staleness is acceptable. Shorter TTLs = fresher data but higher DB load. Find the business tolerance — "how stale is acceptable?" is a product decision, not just a technical one.

Jitter: Add ±10–30% random jitter to TTL to prevent simultaneous mass expiry (thundering herd).
Event-driven invalidation: When a write occurs, immediately invalidate or update the cache entry. No staleness window.

Approach 1 — Synchronous (in application code): After writing to DB, call cache.delete(key). Fast. Risk: if the app crashes after the DB write but before the cache delete, the cache has stale data until TTL. Also: race condition with concurrent reads.

Approach 2 — CDC (Change Data Capture): A separate service reads the DB's WAL (Write-Ahead Log) and publishes change events. A cache invalidation consumer subscribes and invalidates affected keys. Debezium + Kafka is the standard stack.

This decouples invalidation from the write path. No app code change needed. Works even for writes from legacy systems, batch jobs, or direct DB mutations.

Approach 3 — Message queue: On write, publish an event to a queue ("user:42 updated"). Cache invalidation workers consume and delete the key. Works well for fan-out invalidation (one write → many cache keys need invalidating).
Version tags / surrogate keys: Cache keys include a version identifier that changes when the underlying data changes. Stale entries are simply never accessed again — they expire via TTL naturally.

Example: Cache key = user:{id}:v{version}. User record has a version counter. When updated, version increments. New reads fetch user:42:v7 — old user:42:v6 is now orphaned and expires.

Tag-based invalidation (advanced): Tag cache entries with semantic labels ("product-catalog", "user-42"). On a product update, invalidate all entries tagged "product-catalog". Used in CDNs (Fastly, Cloudflare cache tags), and by Varnish.

Benefit: No coordination needed at write time. The version IS the invalidation mechanism. No DEL calls.

Cost: Cache fills with orphaned old versions until TTL. Requires version tracking in DB or application.
Architecture — CDC-based invalidation

Failure modes
Race condition (cache-aside write pattern): Thread A reads miss, fetches from DB (old value). Thread B writes new value, deletes cache key. Thread A then sets cache key with old value. Cache is now stale indefinitely (until TTL). Mitigation: use short TTL, or version the cache key, or use CAS (WATCH in Redis).
Invalidation fan-out explosion: One write invalidates thousands of cache keys (e.g., a category name change invalidates every product in that category). Mitigation: tag-based invalidation with lazy repopulation, or accept eventual consistency for large fan-outs.
Missed invalidation: App code has multiple write paths (API, admin panel, batch job). Developers forget to add cache invalidation to the batch job. Stale data persists. Fix: centralize writes through a single service, or use CDC which catches all DB writes regardless of source.
Interview perspective
  1. Start with TTL as the baseline. It handles 80% of cases. Then upgrade to event-driven for data where staleness matters.
  2. CDC is the senior-level answer. Mention Debezium reading Postgres WAL to trigger invalidation. This shows distributed systems maturity.
  3. The race condition question is almost guaranteed. "There's a race between the DB write and the cache invalidation. I handle this by always deleting (not updating) the cache on write, using short TTLs, and accepting a brief staleness window."
  4. Tag-based invalidation impresses when the problem involves many-to-many invalidation (e.g., "if a user changes their username, which cached pages need to refresh?").

CDN
Edge caching · cache-control headers · origin shield · invalidation
Definition & core purpose
A CDN (Content Delivery Network) is a geographically distributed network of edge servers that cache content close to end users. The goal: serve content from a server that's physically near the user (low latency) without hitting your origin server (reduced load).

Without CDN: a user in Mumbai requesting an image from a server in us-east-1 → ~200ms round-trip. With CDN PoP in Mumbai: ~5ms. For a page with 50 assets, this saves 10 seconds of asset loading.

CDNs cache: static assets (JS, CSS, images, fonts), API responses (with correct headers), HTML pages (for SSR sites), video segments, and even dynamic personalized content at the edge (Cloudflare Workers, Lambda@Edge).
How it works — mechanics
1. Cache-Control headers drive everything. Your origin server sets these headers on responses. The CDN reads them and decides how long to cache.

2. Cache key = URL (+ optional Vary headers). By default, two requests with the same URL hit the same cache entry. If you add Vary: Accept-Language, the CDN caches separate versions per language.

3. Cache HIT: CDN serves from edge. No request to origin.
Cache MISS (first request to this PoP): CDN fetches from origin (or origin shield), caches the response, serves to user.

4. Stale-while-revalidate: Serve the stale cached version immediately while asynchronously fetching a fresh version from origin. User gets fast response, cache refreshes in background.
Cache-control header anatomy
Cache-Control: public, max-age=86400, stale-while-revalidate=3600, s-maxage=604800 public — cacheable by CDN and browser (not private/sensitive) max-age=86400 — browser caches for 24 hours s-maxage=604800 — CDN caches for 7 days (overrides max-age for shared caches) stale-while-revalidate=3600 — serve stale for 1hr while refreshing Other key directives: no-cache — must revalidate with origin on every request (not "no caching") no-store — truly no caching anywhere (PII, auth pages) private — browser can cache, CDN must not immutable — content never changes (hashed filenames like app.a3f2e1.js) Strategy by asset type: CSS/JS with content hash: Cache-Control: public, max-age=31536000, immutable HTML page: Cache-Control: public, s-maxage=300, stale-while-revalidate=60 API response: Cache-Control: public, s-maxage=60 User data / auth: Cache-Control: private, no-store
Origin shield
Without origin shield: each CDN PoP (say, 200 globally) independently fetches from origin on a miss. For a low-traffic object, your origin gets 200 requests instead of 1.

Origin shield (shield PoP): A single designated "mid-tier" CDN node that acts as a cache in front of your origin. All other PoPs check the shield first. Miss at shield → one request to origin. Other PoPs get the response from shield.

Reduces origin load dramatically. Shield is typically located near your origin datacenter for low shield→origin latency. Used by Cloudflare (Tiered Cache), Fastly (shielding), AWS CloudFront (Origin Shield).
CDN cache invalidation
Invalidating cached content is the hardest CDN problem. Options:

URL-based purge: Send a purge request for a specific URL. Fast (~1–5 seconds to propagate globally). Works for targeted invalidations (update one product image).

Tag-based purge (surrogate keys): Tag cache entries with labels at response time (Surrogate-Key: product-42 category-shoes). Purge by tag: one API call removes all entries tagged "product-42" across all PoPs. Fastly and Cloudflare support this. Ideal for invalidating a product page + all its images + its listing page appearance simultaneously.

Versioned URLs (best practice): Never invalidate — instead, change the URL when content changes. CSS file: app.css?v=2 or app.a3f2e1.js (content-hashed). Old URL expires naturally. New URL is automatically fetched. Zero invalidation cost. This is what webpack/Vite does for asset fingerprinting.
Trade-offs
✓ Long CDN TTL

High hit rate. Low origin load. Users get fast responses globally.

✗ Long CDN TTL

Bug fixes / content updates take hours to propagate (unless you purge or use versioned URLs).

✓ Short CDN TTL

Fresh content. Rapid iteration. Easy to push fixes.

✗ Short CDN TTL

More origin requests. Lower hit rate. Higher origin cost and latency.

✓ Versioned URLs + immutable

Best of both worlds. Long cache, instant update via new URL. Zero stale risk.

✗ Versioned URLs

Requires build pipeline to generate content hashes. HTML must reference new URLs. More complex deployment.

Interview perspective
  1. Always mention CDN for static assets in any system design. "Static assets (JS, CSS, images) go behind CloudFront with max-age=1 year, immutable, using content-hashed filenames."
  2. Probe: "How do you update cached content?" → Versioned URLs for static assets (no invalidation needed). Tag-based purge for dynamic content. URL-based purge for emergency hotfixes.
  3. Origin shield is a senior-level detail. "I'd place an origin shield in us-east-1 near our origin to prevent cache-miss thundering herd from 200 global PoPs hitting origin simultaneously."
  4. stale-while-revalidate is an underrated answer for API caching. "The user gets the cached version immediately, and the CDN refreshes in the background. No user-visible latency on refresh."

Distributed cache
Consistent hashing · cache stampede · thundering herd
Why distribute the cache?
A single Redis node handles ~100K–1M ops/sec and is limited by one machine's RAM. At scale (Zepto, Razorpay, Netflix), you need:

1. More memory than one machine can hold
2. Higher throughput than one machine can serve
3. HA — one node failure shouldn't take down the entire cache

Distributing the cache means partitioning keys across multiple cache nodes. The hard problem: when you add or remove a node, how do you decide which node holds which key — without remapping everything?
Consistent hashing — the foundation
Naive approach (mod N): node = hash(key) % N. Adding/removing one node changes the mapping for (N-1)/N keys → massive cache invalidation.

Consistent hashing: Hash space is a ring (0 to 2³²). Nodes are placed on the ring by hashing their identifier (IP:port). Keys are also placed on the ring by hash. Each key routes to the first node clockwise from its position.

Adding a node: only the keys between the new node and its counter-clockwise neighbor move. ~1/N keys remapped. Removing a node: that node's keys shift to the next clockwise node. ~1/N keys remapped.

Virtual nodes (vnodes): Each physical node is represented by K virtual positions on the ring (e.g., K=150). This distributes keys evenly even with heterogeneous hardware and avoids hotspots when nodes join/leave. Redis Cluster uses 16,384 hash slots — a form of consistent hashing with fixed slot count.
Cache stampede — deep dive
What it is: A popular cache key expires. Multiple concurrent requests all see a miss simultaneously. All of them independently query the DB. DB is suddenly hit with N requests instead of 1.

Why it's dangerous: At 100K RPS with 90% hit rate, ~10K req/s hit the DB normally. If the top-10 hot keys expire simultaneously and each has 5K concurrent requests → 50K sudden DB queries. DB falls over. Cache now can't repopulate. System collapses.

Solution 1 — Mutex/lock: First thread to detect miss acquires a Redis lock (SET stampede-lock:key uuid NX EX 5). Only that thread fetches from DB and repopulates. Other threads wait (or return stale data). Lock expires → next miss triggers repopulation.

Solution 2 — Probabilistic early expiry (PER): Each request has a small probability of treating the entry as expired before its actual TTL. Earlier expiry → DB fetch happens before the crowd arrives. No locking needed. Used by Nginx.

Solution 3 — Background refresh: A background job (or Redis keyspace notification) detects keys approaching expiry and proactively refreshes them. Cache is always warm. No miss-driven DB hit.

Solution 4 — Serve stale on miss: When key expires, return the stale value immediately while triggering async refresh. Users see slightly stale data once; no DB stampede. Combine with background refresh.

Trade-offs
✓ Consistent hashing

Only 1/N keys remapped on node change. Smooth scaling. Minimizes cache invalidation on topology changes.

✗ Consistent hashing

Hot spots still possible (one key gets all traffic). Requires virtual nodes to distribute evenly. Client must be cluster-aware.

✓ Mutex on stampede

Exactly one DB request per miss. Clean and correct.

✗ Mutex on stampede

Other threads block, adding latency. Lock itself can fail. Complex to implement correctly (lock expiry vs DB query duration).

✓ Background refresh

Zero miss-driven DB hits. Always-warm cache. Best user experience.

✗ Background refresh

Complex infrastructure. Keys may refresh unnecessarily (low-traffic key refreshed 1000× before being accessed again). Harder to debug.

Interview perspective
  1. Consistent hashing is mandatory knowledge. Draw the ring, place nodes and keys. Explain virtual nodes. Know Redis Cluster's 16,384 hash slot implementation.
  2. Cache stampede is a high-value question. Most candidates say "use mutex." Senior candidates layer: mutex + background refresh + serve-stale-on-miss. Pick the right one for the data freshness requirement.
  3. Thundering herd ≠ cache stampede. Thundering herd = many clients reconnect simultaneously after a failure (e.g., after Redis restart). Fix: exponential backoff with jitter on reconnect. Cache stampede = many requests for the same expired key. Different problem, different fix.
  4. "What happens when a Redis Cluster node fails?" → The replica for that node's slot range is promoted (takes ~1–2 seconds). During promotion, requests to that slot range fail. Mitigation: clients should retry with backoff. For writes, the window of data loss = replication lag at time of failure (usually <1s).
