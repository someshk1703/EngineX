Consistent hashing
Virtual nodes · token ring · adding/removing nodes
The problem
Naive modulo sharding: node = hash(key) % N. Add one node: N becomes N+1. Almost every key remaps to a different node — massive data migration. At 1M keys, ~99.9% of keys move. Consistent hashing reduces this to ~1/N keys moving per node change.
How it works — the ring
Hash space [0, 2³²) wraps into a ring. Both nodes and keys are hashed onto this ring. A key routes to the first node clockwise from its position. Adding a node only steals keys from its immediate clockwise neighbour. Removing a node transfers its keys to the next clockwise node.
Interactive ring — add/remove nodes
3 nodes, 12 keys. Click add/remove to see remapping.
Virtual nodes (vnodes)
Problem with basic ring: 3 physical nodes placed by hash accident → uneven distribution. One node gets 60% of the ring, another gets 10%.

Fix — vnodes: Each physical node is represented by K virtual positions on the ring (K = 100–200 typical). Node A appears at positions A1, A2, … A150 spread around the ring. Keys are spread evenly across all virtual positions. Physical nodes end up with ~equal load regardless of hash placement.

Cassandra uses 256 vnodes per node by default. Redis Cluster uses exactly 16,384 fixed hash slots — a form of consistent hashing with discrete slots.
Architecture note
Key routing with 4 nodes (simplified, no vnodes): hash("user:42") = 2,147,483,648 → falls between Node B and Node C → routes to Node C (first clockwise) Add Node E at position 2,000,000,000: Keys between Node B and Node E → move to Node E Keys between Node E and Node C → stay on Node C Only ~25% of keys remapped (1 node / 4 nodes) With 256 vnodes per node: Keys remapped per add/remove ≈ 1/(N×vnodes) per vnode position Distribution variance < 5% vs naive placement > 40%
Trade-offs
✓ gain

O(1/N) remapping on topology change. Scales smoothly. Handles heterogeneous node capacity via weighted vnodes.

✗ cost

Slightly uneven distribution without vnodes. Client must be ring-aware (or use a proxy). Hotspot keys still hit the same node regardless of ring.

Interview perspective
  1. Draw the ring — literally. Nodes as dots on a circle, keys hashed between them. Interviewers want to see you can sketch this.
  2. Nail the "why vnodes" answer: even distribution + hot node avoidance + graceful rebalancing when a node joins.
  3. Redis Cluster vs Cassandra: Redis uses 16,384 fixed slots (simpler mental model, less flexible). Cassandra uses 256 vnodes per node (more adaptive).
  4. Probe: "A node goes down. What happens?" → Its vnodes' key ranges are now served by the next clockwise node. If replication factor = 3, one of the two remaining replicas takes over without any key migration needed.

Distributed ID generation
UUID problems · Snowflake · Instagram · ticket server
Why sequential auto-increment fails at scale
A single Postgres SERIAL column is the simplest ID. Works fine on one node. At scale: you cannot insert into multiple DB nodes simultaneously and guarantee uniqueness across them. A distributed ID generator must produce IDs that are unique across all nodes, ideally roughly time-ordered, and generated without coordination.
The four approaches
UUID v4
Snowflake
Instagram
Ticket server
UUID v4: 128-bit random. Zero coordination. Works everywhere.

Problems: Random UUIDs destroy B-tree index locality. New UUIDs insert at random positions in the index, causing page splits and cache misses. At 10M writes/day, this degrades write throughput by 30–50% vs sequential IDs. Also: 36-character string is 2–3× the storage of an 8-byte bigint. Cannot sort by creation time. Not human-readable in logs.

UUID v7 (2022 RFC): time-ordered UUID. First 48 bits = millisecond timestamp. Preserves index locality. Much better than v4 for DBs. Still 128 bits.
Twitter Snowflake — 64-bit integer:

| 41 bits timestamp | 10 bits machine_id | 12 bits sequence |

41 bits timestamp: Milliseconds since a custom epoch (e.g., 2020-01-01). Gives ~69 years before rollover. IDs are roughly time-ordered by default.

10 bits machine_id: Supports 1,024 nodes generating simultaneously without coordination. Each node knows its own ID at startup (via config or ZooKeeper).

12 bits sequence: Counter per millisecond per node. Resets to 0 each millisecond. Supports 4,096 IDs/ms/node = 4M IDs/sec/node.

Total: 64-bit integer. Fits in a BIGINT. Index-friendly (monotonically increasing per node). No coordination needed after node startup. Used by: Twitter, Discord, Instagram (variant).
Instagram approach: Generate IDs in Postgres itself using a custom function — avoids a separate ID service.

Schema: shard ID embedded in the ID itself. Sharding-aware. IDs are 64-bit: timestamp (41 bits) + shard_id (13 bits) + sequence (10 bits).

The PL/pgSQL function generates IDs atomically within Postgres transactions. Works without a separate service. Tightly coupled to Postgres but elegant — no extra infra.

Key insight: If you're already using Postgres and don't need cross-DB ID uniqueness, generating Snowflake-style IDs in Postgres functions is the simplest approach.
Ticket server (Flickr approach): A dedicated MySQL database with a single-row table. Each ID request runs REPLACE INTO Tickets64 (stub) VALUES ('a'); SELECT LAST_INSERT_ID();. MySQL auto-increment provides a globally unique sequential integer.

Pros: Truly sequential integers. Simple to understand. Works across any number of app nodes.

Cons: The ticket server is a SPOF. Mitigate with two servers (even/odd IDs), but failover adds complexity. Network round-trip per ID request adds latency. Throughput limited by the ticket server's write capacity (~50K IDs/sec).

Use when: You need human-friendly sequential IDs and can tolerate the SPOF risk (or run two servers). Simpler than Snowflake for lower-scale systems.
Snowflake anatomy
64-bit Snowflake ID: 1753024783001600001 Bit layout: [0][ 41 bits: ms timestamp ][ 10 bits: machine ][ 12 bits: seq ] ^ ^ sign bit (always 0) rightmost bit Decode example: timestamp_ms = (id >> 22) + custom_epoch_ms → "2024-03-15 10:23:01.234" machine_id = (id >> 12) & 0b1111111111 → node 7 sequence = id & 0b111111111111 → counter 1 Properties: - Monotonically increasing within a single node - Time-ordered globally (within a few milliseconds of drift) - 4,096 IDs per millisecond per node, 1,024 nodes - Total capacity: 4.2 billion IDs per second globally
Interview perspective
  1. Never suggest UUID v4 for high-write DB IDs without acknowledging the index fragmentation cost. Suggest UUID v7 or Snowflake.
  2. Snowflake is the default answer. Know the bit layout by heart: 41 timestamp + 10 machine + 12 sequence = 64-bit BIGINT.
  3. Probe: "Clock goes backward on a node." → Node must either wait until the clock catches up, or refuse to generate IDs during the anomaly. Twitter's implementation tracks last-used timestamp and panics if it goes backward. NTP should be tightly configured to avoid large adjustments.
  4. Probe: "We have 2000 nodes." → 10 bits = 1024 nodes max. Solution: increase machine_id bits (steal from sequence or timestamp), or use a tiered ID service where 10 bits = datacenter_id + machine_id within datacenter.

Distributed locking
Redis SETNX · Redlock · ZooKeeper · lease-based locks
Why distributed locks are hard
In-process locks (Java synchronized, ReentrantLock) work within one JVM. In distributed systems, the "lock" must be visible to all nodes. The challenge: the system holding the lock can crash, partition from the network, or experience GC pause — leaving the lock held forever.
Three approaches
Redis SETNX
Redlock
ZooKeeper
Redis SETNX (single-node):

SET lock:resource_id {uuid} NX EX 30

NX = only set if Not eXists. EX 30 = expire in 30 seconds (TTL prevents infinite lock hold on crash). Returns OK if lock acquired, nil if already held.

Unlock (must be atomic — use Lua):
if redis.call("get",KEYS[1]) == ARGV[1] then return redis.call("del",KEYS[1]) else return 0 end

The UUID check is critical: prevents Thread B from deleting Thread A's lock. Without it: A acquires lock, A's lock expires, B acquires lock, A finishes its work and calls DEL — deletes B's lock. UUID ensures only the lock owner can delete it.

Weakness: Single Redis node is a SPOF. If Redis crashes after the lock is acquired but before it expires, the lock is gone but the resource may be mid-operation.
Redlock (multi-node Redis): Proposed by Antirez. Uses N independent Redis nodes (N=5 recommended).

Acquire: try to acquire the lock on all N nodes with a short timeout. Lock is considered held if acquired on a majority (⌈N/2⌉ + 1 = 3 of 5) within the lock validity time.

The controversy: Martin Kleppmann proved Redlock is unsafe in the presence of process pauses (GC, OS scheduling): Thread A acquires lock, pauses for 10s (GC), lock expires, Thread B acquires same lock, Thread A resumes thinking it still holds the lock → both A and B are in the critical section simultaneously.

Fencing tokens fix this: Lock server returns a monotonically increasing token with each grant. Storage system rejects operations with a stale token. Even if A resumes with an expired lock, its token (42) is rejected in favour of B's fresher token (43).

Use Redlock for: advisory locks where brief double-execution is acceptable (job deduplication, cache warming). Do NOT use for safety-critical exclusive access to shared resources without fencing tokens.
ZooKeeper / etcd (CP consensus-based): The strongest distributed lock. Uses linearizable writes — a write to ZooKeeper is guaranteed to be seen by all nodes in order.

ZooKeeper ephemeral nodes: Create an ephemeral sequential znode: /locks/resource-0000000001. The client with the lowest sequence number holds the lock. Others watch the node just below theirs. When that node is deleted (owner released or crashed — ephemeral nodes auto-delete on session loss), the next client is notified and acquires the lock.

ZooKeeper session timeout (typically 3–10s) serves as the lease. If the client's session dies, its ephemeral node is deleted, releasing the lock automatically.

Pros: Truly linearizable. No split-brain. Session expiry is the TTL. No fencing token needed (ZooKeeper's Zxid serves as a monotonic ordering).

Cons: Latency higher than Redis (consensus round-trips). Operational complexity. Not appropriate for high-frequency short locks.
When to use what
Redis SETNX
Low-stakes advisory locks. Job deduplication ("only one worker processes job X"). Cache stampede prevention. Acceptable: occasional double-execution if Redis restarts.
Redlock + fencing
Medium-stakes. Multiple Redis nodes for HA. Must pair with fencing tokens for correctness. Seat reservation, rate limiting, distributed counters.
ZooKeeper / etcd
High-stakes leadership election, shard ownership, configuration management. Where correctness beats performance. Used by Kafka, HBase, Hadoop for coordination.
Database row lock
When your data is already in Postgres: SELECT FOR UPDATE is a perfectly good distributed lock if all nodes share the same DB. Simplest approach, often overlooked.
Interview perspective
  1. Start with Redis SETNX — show you know the NX+EX pattern and the Lua unlock script.
  2. Voluntarily mention the fencing token problem. "Redlock alone isn't safe under process pauses. I'd pair it with fencing tokens where the storage layer rejects stale token operations."
  3. ZooKeeper for leader election — show you know ephemeral nodes and the watch mechanism.
  4. Probe: "What happens if the lock holder crashes?" → TTL/EX ensures the lock expires automatically. ZooKeeper session expiry deletes ephemeral nodes. The lock self-heals without manual intervention.

Consensus — Raft & Paxos
Leader election · log replication · Paxos intuition
What consensus solves
Multiple nodes must agree on a single value (which node is the leader, what the next log entry is, what the committed state is) even when some nodes fail or messages are delayed. Without consensus, you get split-brain: two nodes both believe they are leader and accept conflicting writes.
Raft — designed to be understandable
Leader election
Log replication
Paxos intuition
Terms and roles: Time is divided into terms (monotonically increasing integers). Each term begins with an election. Nodes are in one of three states: Follower, Candidate, Leader.

Election trigger: Followers expect heartbeats from the leader. If no heartbeat arrives within the election timeout (150–300ms random), the follower becomes a Candidate and starts a new term.

Election process: Candidate increments its term, votes for itself, sends RequestVote RPCs to all other nodes. Each node grants at most one vote per term (first come, first served). Candidate wins if it gets votes from a majority (⌈N/2⌉ + 1). Becomes Leader, starts sending heartbeats immediately.

Randomized timeouts prevent ties: If two candidates start simultaneously (split vote), both time out again, but with different random delays — one starts the next election before the other, wins cleanly.

Safety invariant: A node only votes for a candidate whose log is at least as up-to-date as its own. Ensures the leader always has all committed entries.
Client writes go to the leader only. Leader appends the entry to its log (uncommitted), then sends AppendEntries RPCs to all followers in parallel.

Commitment: Entry is committed once a majority of nodes have written it to their logs. Leader then applies to its state machine, responds to client, and notifies followers to commit on next AppendEntries.

Log matching guarantee: If two logs have an entry with the same index and term, all preceding entries are identical. This allows Raft to guarantee consistency during leader changes.

Leader crash during replication: New leader is elected. On first AppendEntries to each follower, new leader sends its log and checks consistency. Followers whose logs diverge are overwritten to match the new leader. Committed entries are never overwritten — only uncommitted ones that weren't on a majority of nodes.

Read path: Leader must confirm it's still leader before serving reads (another leader may have been elected). Can use lease-based reads (within election timeout, leader is guaranteed to still be the only leader) for lower latency.
Paxos (intuition): The original consensus algorithm by Leslie Lamport. Raft was designed specifically because Paxos is notoriously hard to understand and implement correctly.

Two phases:
Phase 1 (Prepare): Proposer sends Prepare(n) to a quorum of acceptors. Acceptors respond with a promise not to accept any proposal with a lower number, and return the highest-numbered proposal they've already accepted (if any).

Phase 2 (Accept): If proposer receives promises from a majority, it sends Accept(n, value). Acceptors accept if they haven't promised a higher number. When a majority accepts, the value is chosen.

Why Paxos is hard: The basic algorithm only solves single-value consensus. Multi-Paxos (replicated log) requires significant additional protocol design. Leader lease, read safety, membership changes — all require extensions that are underspecified in Lamport's original paper.

Interview reality: Know Paxos at the phase level. Understand that Raft is essentially "Paxos made explicit" — same guarantees, clearer specification. etcd, CockroachDB, and TiKV use Raft. Google Chubby, Spanner use Paxos variants.
Architecture — Raft cluster write
5-node Raft cluster (leader = Node 1): Client → write "x=5" → Node 1 (Leader) Node 1: appends to log[idx=42, term=3, value="x=5"] (uncommitted) Node 1 → AppendEntries(idx=42, term=3, "x=5") → Node 2, 3, 4, 5 (parallel) Node 2 ack → Node 1 ✓ Node 3 ack → Node 1 ✓ ← majority (3/5), COMMIT Node 4 (slow)... Node 5 (partitioned)... Node 1: commit log[42], apply to state machine, respond to client "OK" Next AppendEntries carries commit notification → followers 2,3,4 apply Node 5 resync on reconnect: leader replays log entries since last known index
Interview perspective
  1. Know Raft, mention Paxos. "Raft is the practical implementation used by etcd and CockroachDB. Paxos is the theoretical foundation — same guarantees but notoriously difficult to implement correctly."
  2. Majority quorum = (N/2)+1. For 3 nodes, 2 votes needed. For 5 nodes, 3 votes. Always odd number of nodes to avoid tie.
  3. Probe: "Can a partitioned minority elect a leader?" → No. It can't get a majority of votes. The partition with the majority elects a new leader and continues. The minority partition is unavailable (CP).
  4. Probe: "How does Raft handle a network partition that heals?" → Old leader that was in the minority discovers a higher term number when it reconnects → steps down, becomes follower, syncs its log with the new leader, discarding uncommitted entries it made alone.

Leader election
Why needed · bully algorithm · Raft election
Why leader election exists
Many distributed systems require exactly one node to act as coordinator: the Kafka partition leader accepts all writes, the database primary handles writes, the Elasticsearch master manages cluster state. Without a clear leader: any node could accept writes, creating split-brain. Leader election is the mechanism for nodes to agree on which single node is in charge, and to replace it when it fails — automatically, without human intervention.
Bully algorithm
Simple algorithm. Each node has a unique numeric ID. The node with the highest ID becomes the leader.

Election trigger: Node P notices the leader is down (missed heartbeats). P sends an Election message to all nodes with a higher ID.

If any higher-ID node responds (OK), it takes over the election. If no higher-ID node responds (they're all down), P declares itself leader and sends Coordinator messages to all lower-ID nodes.

Problem: The node with the highest ID is always elected, even if it's actually the worst choice (low bandwidth, far from clients). Also: O(N²) messages in the worst case. Not used in modern production systems but appears in textbooks and older systems.

Modern replacement: Raft. Randomized election timeouts, log-based voting (most up-to-date node wins), O(N) message complexity.
Real-world leader election
Kafka
KRaft (Kafka 3.x): Raft-based controller election. Previously used ZooKeeper ephemeral nodes. Controller elected by Raft quorum manages partition leader assignments.
Elasticsearch
Master-eligible nodes hold elections using a bully-like algorithm with quorum. Master manages cluster state (index creation, shard allocation). Split-brain prevention via minimum_master_nodes = (N/2)+1.
Kubernetes
etcd (Raft) stores all cluster state. API server is stateless — any instance can serve. Controller manager uses leader election via etcd lease to ensure only one controller runs at a time.
Postgres
Patroni uses etcd/Consul/ZooKeeper for leader election. Winning node becomes the Postgres primary. Others become standbys. Automatic failover when primary's lease expires.
Lease-based election
Leader lease: The elected leader holds a time-bounded lease (e.g., 10-second renewable lease in etcd). While the lease is valid, the leader is authoritative. To remain leader, it must renew the lease before expiry.

If the leader crashes: lease expires after timeout. Next candidate acquires the lease via etcd compare-and-swap. New leader takes over. Maximum unavailability = lease TTL (~10s).

Lease TTL is the key tuning parameter: short TTL = faster failover but higher risk of spurious leader changes under network jitter. Long TTL = slower failover but more stable.
Interview perspective
  1. Know Raft election (covered in Topic 4). Bully algorithm is trivia — know it exists, know Raft replaced it.
  2. Lease + etcd is the practical answer for "how would you implement leader election?" in a system design interview.
  3. Probe: "Two nodes both think they are the leader." → Split-brain. Prevention: fencing tokens (storage layer rejects writes from old leader), etcd lease expiry (old leader can't renew → stops serving), STONITH (Shoot The Other Node In The Head — hardware fencing in high-stakes clusters).

Gossip protocol
Epidemic spread · eventual consistency · Cassandra's use
The core idea
Gossip protocols spread information through a cluster the way a rumour spreads through a network of people: each node periodically picks K random peers and exchanges state. Every node eventually learns everything, without any central coordinator.

Key properties: Decentralized (no single point of failure), eventually consistent (all nodes converge on the same state), robust to node failures (gossip routes around failures), scalable (O(log N) rounds to reach all N nodes).
How it works
Each node maintains a membership list: For every known node, it stores the node's address, a heartbeat counter, and a timestamp of when it last received that heartbeat.

Gossip round (every 1 second): Node A picks 3 random nodes from its membership list. Sends its entire membership list to each. Recipients merge the received list with their own — keeping the entry with the higher heartbeat counter for each node.

Failure detection: If Node B's heartbeat counter hasn't increased in T seconds (default ~5s in Cassandra), Node B is marked "suspect." After another timeout, it's marked "dead" and removed from the ring. This is eventually consistent — different nodes may mark B as dead at slightly different times.

Convergence: With N nodes and fan-out K, gossip converges in O(log N / log K) rounds. For 1,000 nodes with K=3: ~6 rounds ≈ 6 seconds for full cluster knowledge.
Cassandra's gossip implementation
Gossip in Cassandra serves three purposes:

1. Membership: Which nodes are in the cluster, which are alive, which are unreachable.
2. Token ownership: Which node owns which hash ring token ranges. This is how every node knows where to route any key without a central directory.
3. Schema: Gossip propagates schema changes (CREATE TABLE, etc.) to all nodes within a few seconds.

Cassandra uses Phi Accrual Failure Detector instead of a hard timeout — it computes a suspicion level (phi) based on the history of inter-arrival times of heartbeats. This adapts to varying network conditions rather than using a fixed timeout. Phi > 8 → node suspected. Phi > 12 → node marked down.
Trade-offs
✓ gain

No SPOF. Scales to thousands of nodes. Self-healing — gossip routes around failures automatically. No coordinator bottleneck.

✗ cost

Eventual (not immediate) consistency. Some nodes may have stale membership briefly. Network bandwidth grows O(N) (each node sends N-entry list to K peers). False positives on slow networks.

Interview perspective
  1. Gossip = epidemic broadcast. "Like a rumour spreading: each node tells a few random peers, who tell a few more. Reaches everyone in O(log N) rounds."
  2. Cassandra uses gossip for membership + token ownership. "Every Cassandra node knows the entire ring topology without a central directory. Gossip keeps it eventually consistent."
  3. Probe: "How does Cassandra detect a dead node?" → Phi Accrual Failure Detector based on heartbeat inter-arrival statistics. When phi exceeds threshold, node is marked suspect, then down after further timeout.

Vector clocks & conflict resolution
Causality tracking · Dynamo-style conflict resolution
The problem — ordering events in distributed systems
In a single machine, time is total order — event A before event B is unambiguous. In distributed systems, clocks drift, messages arrive out of order, and there is no global "now." Two events on different nodes can't be reliably ordered by wall clock time alone. Vector clocks capture causal relationships without relying on synchronized clocks.
How vector clocks work
Each node maintains a vector of counters, one per node in the system. Initially all zeros: [A:0, B:0, C:0].

Rules:
1. Before any internal event, increment your own counter: A's clock becomes [A:1, B:0, C:0].
2. Before sending a message, increment your own counter, then attach the full vector.
3. On receiving a message, merge: take the max of each position, then increment your own.

Causality: Vector V1 "happened before" V2 if every element of V1 ≤ V2 and at least one element is strictly less. If neither V1 ≤ V2 nor V2 ≤ V1, the events are concurrent — no causal relationship.

Dynamo-style conflict resolution
Amazon Dynamo stores multiple versions of a value when concurrent writes occur, using vector clocks to track them. On read, if only one version exists → return it. If multiple concurrent versions exist → return all to the application, which must reconcile.

Shopping cart (Dynamo's own example): User A adds item X on device 1. Concurrently adds item Y on device 2. Network partition keeps them separate. On merge: both X and Y must be in the cart. Union is the correct merge. Silently dropping either (LWW) would lose an item — unacceptable.

LWW (Last-Write-Wins): Cassandra's default. One write wins based on timestamp. Simple but lossy — the other concurrent write is permanently discarded. Acceptable for metrics, user profile updates. Unacceptable for shopping carts, inventory.

CRDTs (Conflict-free Replicated Data Types): Data structures designed so all concurrent operations can be merged without conflicts. G-Counter (grow-only), PN-Counter, OR-Set. Riak, Redis (some types) support CRDTs. The merge is always correct — no application-level conflict resolution needed.
Interview perspective
  1. Vector clocks show causality, not wall-clock time. "If V1 → V2 (happened-before), we know V1 caused V2. If V1 || V2 (concurrent), we have a conflict."
  2. Know the Dynamo shopping cart example cold. It's the canonical illustration of why LWW is wrong and why merge semantics matter.
  3. Probe: "Your AP system has concurrent writes. How do you handle conflicts?" → Depends on data type. For user profile (last-write-wins is OK), for counters (use CRDT G-Counter), for sets (use OR-Set CRDT), for shopping cart (merge/union semantics).

Two-phase commit (2PC)
Distributed transactions · coordinator failure
What 2PC solves
A transaction spans multiple databases (or services). You need atomicity: either all commit or all roll back. With independent DBs, one might commit while another fails → inconsistent state. 2PC coordinates a distributed atomic commit using a central coordinator.
The two phases
PHASE 1 — Prepare (Voting): Coordinator → "Prepare to commit TX-99" → Participant A, B, C Participant A: write to WAL, acquire locks, reply "YES, prepared" Participant B: write to WAL, acquire locks, reply "YES, prepared" Participant C: detects conflict, reply "NO, abort" All must vote YES to proceed. Any NO → abort. PHASE 2 — Commit (or Abort): If all YES: Coordinator → "COMMIT TX-99" → A, B, C Participants commit, release locks, ack coordinator If any NO (or timeout): Coordinator → "ABORT TX-99" → A, B, C Participants rollback, release locks BLOCKING PROBLEM — coordinator crash after Phase 1: Participants A, B are in "prepared" state (locks held, WAL written) Coordinator is down Participants CANNOT commit or abort unilaterally — they must wait Locks are held indefinitely → other transactions blocked This is the fundamental 2PC flaw: blocking on coordinator failure
Failure modes & mitigations
Coordinator crash after "Prepare": Participants are in limbo — they voted YES but don't know the outcome. They hold locks and must wait for the coordinator to recover. Recovery: coordinator uses its WAL to determine the intended outcome and re-sends Commit/Abort. Window of unavailability = coordinator downtime.
Participant crash after "Prepare": On recovery, participant checks its WAL — if it logged "prepared for TX-99," it must ask the coordinator for the outcome and commit or roll back accordingly.
3PC (three-phase commit): Adds a pre-commit phase to eliminate the blocking problem. Theoretically non-blocking but adds round-trips and is still unsafe under network partitions. Almost never used in practice.
Practical alternative: Saga pattern avoids 2PC entirely. Each step is a local transaction with a compensating transaction for rollback. No distributed locks, no coordinator SPOF, but no atomicity guarantee — eventual consistency only.
Where 2PC is actually used
Postgres PREPARE TRANSACTION + external coordinator. Google Spanner (using TrueTime for commit ordering). XA transactions in Java EE / JTA. Kafka exactly-once semantics (transactions across producer + consumer + offset commit). Most systems prefer Saga over 2PC for cross-service coordination.
Interview perspective
  1. Know the two phases by name and their purpose. Phase 1 = voting (can you commit?). Phase 2 = execution (do it).
  2. Coordinator failure is the key weakness. "2PC is a blocking protocol — if the coordinator fails after Phase 1, participants hold locks indefinitely. This is why most modern systems prefer Saga."
  3. Probe: "How does Kafka achieve exactly-once?" → Producer transactions + 2PC between the producer's transaction log and the consumer offset commit. Kafka's transaction coordinator is the 2PC coordinator.
  4. When to use 2PC: "Only when you absolutely need cross-service atomicity and can tolerate the blocking risk. For most microservice scenarios, Saga with compensating transactions is the better choice."

Saga pattern
Distributed transactions without 2PC · choreography vs orchestration
The core insight
2PC achieves atomicity by locking resources across services until all agree. This creates blocking, latency, and SPOF risk. Saga achieves eventual consistency differently: break the distributed transaction into a sequence of local transactions, each with a compensating transaction that can undo it if a later step fails. No distributed locks. No coordinator SPOF. Trade-off: the system is temporarily inconsistent during execution — this is the defining cost of Saga.
Two coordination patterns
Orchestration
Choreography
Orchestration Saga: A central orchestrator (coordinator service) drives the entire flow. It calls each participant in sequence, waits for responses, and triggers compensating transactions on failure. The orchestrator owns the saga state machine.

Flow for "place order":
Orchestrator → Reserve inventory → Success
Orchestrator → Charge payment → Fail
Orchestrator → Compensate: Release inventory (compensating tx)
Orchestrator → Mark order failed

Pros: Easy to reason about. Single place to see the full saga state. Easy to add new steps. Good for complex multi-step workflows.

Cons: Orchestrator is a new service to build and maintain. Centralizes logic — becomes a new coordination bottleneck. Risk of orchestrator being a SPOF (mitigate: make it stateless with state in DB).

Used by: Netflix (Conductor), Temporal, AWS Step Functions, Camunda. This is the dominant approach for complex sagas in 2024.
Choreography Saga: No central orchestrator. Each service reacts to events and emits its own events. Services are loosely coupled through an event bus (Kafka). Each service knows what to do in response to each event type.

Flow for "place order":
OrderService publishes → order.created
InventoryService listens → reserves → publishes inventory.reserved
PaymentService listens → charges → publishes payment.failed
InventoryService listens → publishes inventory.released
OrderService listens → publishes order.failed

Pros: No central coordinator. Fully decentralized. Services are loosely coupled — adding a new service just subscribes to the relevant events. Natural for event-driven architectures.

Cons: Hard to understand the full saga flow (no single place to see it). Debugging distributed failures requires tracing events across many services. Risk of cyclic event dependencies. Compensations are implicit and can be missed if a service doesn't implement them.
Compensating transactions — the hard part
A compensating transaction is semantically correct but not always a simple rollback. It runs after the fact, when other things may have happened.

Example: Payment was charged. Customer was notified. Now the inventory step failed. Compensating transaction: refund the payment + send a "sorry" email to the customer. You can't "un-send" the notification — you can only send a correction.

Not all operations are compensatable. If an email was sent, a physical document printed, or cash dispensed — you cannot un-do these. Design your saga steps to be as late-binding as possible (authorize the payment rather than charge; reserve inventory rather than consume) and only finalize irreversible actions at the very end.
Saga for order placement
Step 1: Create order (Order service)
Compensate: Mark order as cancelled
Local transaction. Order record created in PENDING state.
Step 2: Reserve inventory (Inventory service)
Compensate: Release reservation
Atomically decrement available stock, create reservation record.
Step 3: Charge payment (Payment service)
Compensate: Issue refund
Authorize and capture payment. If fails here → trigger steps 2C, 1C.
Step 4: Schedule delivery (Delivery service)
Compensate: Cancel delivery
If fails here → trigger steps 3C, 2C, 1C in reverse order.
Each step is a local transaction. Compensations execute in reverse order on failure.
Interview perspective
  1. Saga vs 2PC trade-off in one sentence: "2PC gives atomicity at the cost of blocking and SPOF. Saga gives availability at the cost of temporary inconsistency and compensating transaction complexity."
  2. Default to orchestration for interviews. "I'd use an orchestration saga with a dedicated workflow service — Temporal or a custom state machine backed by Postgres — to manage the saga state and retry failed steps."
  3. Idempotency is mandatory with Saga. "Every step must be idempotent because the orchestrator may retry on timeout. Payment must not double-charge if charged twice with the same idempotency key."
  4. Probe: "Payment succeeded but delivery scheduling failed. What happens?" → Reverse compensation: cancel delivery (no-op if nothing scheduled), issue refund for payment, release inventory reservation, mark order failed. Each compensation step must also be idempotent.
