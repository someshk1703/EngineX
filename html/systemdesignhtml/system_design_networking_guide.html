DNS
Resolution process · recursive vs iterative · TTL · DNS load balancing
Definition
DNS (Domain Name System) is the internet's distributed phone book. It translates human-readable domain names (api.razorpay.com) into IP addresses (34.93.21.48) that routers understand. It's a hierarchical, globally distributed system with aggressive caching at every layer — by design, most DNS lookups never reach the authoritative server.
Resolution — step by step
1
Browser cache. Chrome caches DNS entries (respect TTL). Check chrome://net-internals/#dns. If cached and not expired → done in microseconds.
2
OS resolver cache. /etc/hosts checked first, then OS DNS cache (nscd, systemd-resolved). TTL managed by OS independently of browser.
3
Recursive resolver (ISP or 8.8.8.8). If neither cache has it, the OS asks its configured resolver. This resolver does all the heavy lifting on behalf of the client — the client just waits.
4
Root nameserver. Recursive resolver asks a root server (13 sets, operated by IANA): "who knows about .com?" Root returns the address of the .com TLD nameserver.
5
TLD nameserver. Resolver asks the .com TLD: "who knows about razorpay.com?" Returns the address of Razorpay's authoritative nameserver.
6
Authoritative nameserver. Resolver asks Razorpay's nameserver: "what's the IP for api.razorpay.com?" Returns the A record (IPv4) or AAAA record (IPv6). Result cached by recursive resolver for the TTL duration.
7
Response to client. Recursive resolver returns IP to OS → OS returns to browser → TCP connection begins. Total time cold: 20–120ms. Warm cache: ~0ms.
Recursive vs iterative resolution
Recursive (what clients use)
Client asks resolver → resolver does all work → resolver returns final answer. Client makes one request and waits. Burden is on the resolver. All caching happens at the resolver.
Iterative (what resolvers use internally)
Resolver asks root → root says "ask .com TLD" → resolver asks .com TLD → TLD says "ask ns1.razorpay.com" → resolver asks that → gets answer. Each step returns a referral, not the answer. Resolver follows the chain.
TTL — the key design lever
TTL controls how long the DNS response is cached by resolvers worldwide. It's set by the domain owner in their authoritative DNS records.

Short TTL (30–60s): Changes propagate globally in under a minute. Essential before planned migrations, failover events, or blue/green deployments. Cost: every TTL expiry generates a fresh recursive query — higher load on your nameservers.

Long TTL (300–3600s): High cache hit rate globally. Reduced nameserver load. Changes take the full TTL to propagate — dangerous during incidents when you need to redirect traffic urgently.

Production strategy: Normal TTL = 300s. 24 hours before a planned migration, drop to 60s. Execute the migration. Raise back to 300s after validation. This is how companies do zero-downtime DNS cutovers.
DNS for load balancing & failover
Round-robin DNS
Return multiple A records for the same domain in rotating order. Clients pick the first one returned. Rough load distribution across IPs. No health checking — returns dead IPs too. Clients cache aggressively so rotation is unreliable.
Weighted routing (Route 53)
Send 90% of queries to primary region, 10% to canary. Gradual rollouts without code deployment. Change weights in seconds.
Latency routing
Route each user to the region with lowest measured RTT. AWS Route 53 maintains latency measurements and serves accordingly. User in Mumbai gets ap-south-1.
Health-check failover
Route 53 probes endpoints every 10s. If primary fails health checks → automatically switch DNS response to failover IP. TTL must be short for this to work quickly.

Interview perspective
  1. Know the hierarchy by heart: Browser → OS → Recursive resolver → Root → TLD → Authoritative. Explain each layer's role and caching behaviour.
  2. TTL is a design decision, not a detail. "Before a migration, I'd drop TTL to 60s 24 hours in advance, then restore after. This is the correct zero-downtime DNS cutover procedure."
  3. DNS ≠ reliable load balancing. "DNS round-robin distributes queries, not connections — clients cache aggressively, so distribution is uneven. Use an L7 load balancer for real traffic distribution."
  4. Probe: "How does Razorpay's API go down for 5 minutes during a failover?" → TTL was 300s. When the primary failed, it took 5 minutes for cached DNS entries to expire globally. Fix: pre-emptively lower TTL before incidents, or use Anycast routing.

HTTP/1.1 vs HTTP/2 vs HTTP/3
Head-of-line blocking · multiplexing · QUIC
The core problem HTTP/2 and /3 solve
HTTP/1.1 opens one TCP connection per request (or reuses connections with pipelining, which is widely broken). Loading a web page with 100 resources means 100 sequential round-trips — or opening 6 parallel connections per browser per domain (browser's workaround). HTTP/2 and HTTP/3 eliminate this by multiplexing many requests over a single connection.
HTTP/1.1 — the baseline
Connection model: One request at a time per TCP connection (with keep-alive, the connection stays open for reuse — but still one outstanding request). Pipelining (send multiple before waiting for responses) exists in spec but is disabled in most browsers due to broken proxy behaviour.

Head-of-line blocking (HOL): If request 1 is slow (large file), requests 2, 3, 4 wait behind it in the same connection. Like a queue where one slow customer holds everyone up.

Browser workaround: Open 6 parallel TCP connections per domain. Each connection can serve one request at a time. 6 parallel → some parallelism. Still 6 TCP handshakes + 6 TLS handshakes per domain. High overhead.
HTTP/2 — multiplexing over one TCP connection
Binary framing layer: HTTP/2 breaks messages into binary frames instead of text. Frames from different requests interleave over the same TCP connection. Each request/response pair is a stream identified by a stream ID.

Multiplexing: 100 requests can fly over 1 TCP connection simultaneously. No head-of-line blocking at the HTTP layer — request 1 being slow doesn't block request 2.

Header compression (HPACK): HTTP headers are repetitive (User-Agent, Cookie, etc.). HPACK compresses headers using Huffman coding + a shared dynamic table. Reduces header overhead by ~80%.

Server push: Server can proactively send resources before the client requests them. In theory: server pushes CSS and JS when HTML is requested. In practice: browser deprecation; rarely used today (removed in Chrome 106 for HTTP/2 push).

The remaining HOL problem: HTTP/2 eliminated HTTP-layer HOL. But TCP-layer HOL remains. TCP delivers bytes in order — if a packet is lost, all streams wait for retransmission. One lost packet blocks all 100 streams. This is TCP's fundamental limitation.
HTTP/3 — QUIC replaces TCP
QUIC (Quick UDP Internet Connections): HTTP/3 runs over QUIC instead of TCP. QUIC is implemented in user space over UDP. It reimplements the reliability guarantees of TCP but eliminates TCP's HOL blocking.

Why UDP? TCP is implemented in OS kernel — can't be changed without OS update. QUIC runs in user space, can be updated with the application. Deployed via Chrome and server software updates, not kernel patches.

Stream independence: QUIC tracks individual streams independently. A lost packet only blocks the stream it belongs to — all other streams continue. True elimination of HOL blocking.

0-RTT connection establishment: TLS 1.3 integrated into QUIC handshake. First connection: 1 RTT (vs TCP's 1 RTT + TLS's 1-2 RTT = 2-3 RTT total). Resumed connection: 0-RTT — can send data before the handshake completes (with replay attack risk, used carefully).

Connection migration: QUIC connections are identified by a Connection ID, not by IP:port. When a mobile user moves from WiFi to LTE (IP changes), the QUIC connection continues without interruption. TCP connections would drop.

Interview perspective
  1. Two kinds of HOL blocking. "HTTP/2 solves HTTP-layer HOL but TCP-layer HOL remains. HTTP/3 with QUIC eliminates both by making each stream independent at the transport layer."
  2. HTTP/2 adoption: ~65% of web traffic as of 2024. HTTP/3 growing (~30%). Still worth designing for HTTP/2 as the default, HTTP/3 as progressive enhancement.
  3. Probe: "Why did Google invent QUIC?" → TCP HOL blocking hurt latency on lossy mobile networks (high packet loss). QUIC's per-stream loss recovery meant a single dropped packet no longer stalled all streams. Measurable improvement in mobile latency for YouTube/Search.
  4. Probe: "Your API has slow endpoints mixed with fast ones. What does HTTP/2 fix?" → With HTTP/1.1, a slow request blocks the connection. With HTTP/2 multiplexing, slow and fast requests travel concurrently — fast responses arrive without waiting for slow ones.

WebSocket vs SSE vs Long polling
Real-time comparison · when each applies
The real-time problem
HTTP is request-response: the client always initiates, the server responds. But many applications need the server to push data to clients without the client asking — live chat, stock prices, collaborative editing, Uber driver location. Three patterns solve this with increasing sophistication.
Long polling — the HTTP workaround
Mechanism: Client sends a normal HTTP request. Server holds the connection open (doesn't respond immediately). When new data arrives, server sends the response. Client immediately makes another request. Repeat.

Latency: Near-real-time — delay is only the network RTT. No artificial polling interval.

Cost: Every "push" requires a full HTTP request-response cycle — headers, connection overhead. Under load, thousands of held-open connections consume server threads/sockets. Stateless servers require sticky sessions or an external pub/sub to route responses to the right connection.

Use when: You need real-time but can't use WebSocket (strict HTTP-only proxies, legacy environments). Simple to implement with any HTTP stack. Used by early Slack, early Facebook Messenger.
Server-Sent Events (SSE) — one-way stream over HTTP
Mechanism: Client sends one HTTP GET with Accept: text/event-stream. Server keeps the connection open and streams newline-delimited events (data: {json}\n\n). The connection stays open indefinitely. Built into the browser via the EventSource API.

Direction: Server → client only. No client-to-server messaging over the same connection.

Auto-reconnect: Browser EventSource automatically reconnects if the connection drops. Server sends id: field to enable resuming from last received event.

Infrastructure friendly: Plain HTTP — works through HTTP/2 multiplexing, CDNs, load balancers, proxies without special configuration. No protocol upgrade needed.

Use when: One-way server push — live dashboards, notifications, AI streaming responses (ChatGPT uses SSE), news feeds, real-time log streaming. Simpler than WebSocket when you don't need bidirectional.
WebSocket — full-duplex persistent connection
Mechanism: Client sends HTTP Upgrade request (Upgrade: websocket). Server responds with 101 Switching Protocols. Connection is now a persistent, full-duplex TCP channel — both sides can send at any time, independently.

Framing: WebSocket defines its own lightweight framing — no HTTP overhead per message. Small header (2–10 bytes) vs HTTP's 200–800 byte headers. Very efficient for high-frequency small messages.

Bidirectional: Client and server both push messages. True real-time chat, collaborative editing (cursor positions from all users), multiplayer games, trading terminals.

Infrastructure complexity: WebSocket connections are stateful and long-lived. Load balancers need sticky sessions (same client always to same server). L7 proxies must be configured to support the Upgrade. Horizontal scaling requires a message bus (Redis pub/sub) to route messages across server instances.

Use when: Bidirectional real-time — chat (WhatsApp Web), collaborative docs (Figma, Google Docs), multiplayer games, live trading. Your Zombie Hunt game almost certainly uses WebSocket.

Interview perspective

  2. WebSocket scaling needs Redis pub/sub. "With 10 app servers and 100K WebSocket connections, a message from user A (on server 1) must reach user B (on server 3). I'd use Redis pub/sub to fan out across server instances."
  3. Probe: "Build a live Uber driver location tracker." → Driver app → REST API updates location every 4s. Rider browser → WebSocket to receive location pushes. Server subscribes to driver location updates → pushes via WebSocket to all riders watching that driver. (Or SSE if the rider view is read-only.)

gRPC vs REST vs GraphQL
Protocol buffers · streaming · N+1 problem
REST — the default
REST (Representational State Transfer): Architectural style, not a protocol. Resources identified by URLs, manipulated via HTTP verbs (GET, POST, PUT, DELETE, PATCH). Stateless — each request is self-contained. JSON over HTTP/1.1 or HTTP/2.

Strengths: Universal — every language, framework, and tool speaks HTTP+JSON. Human-readable in browsers. Cacheable (GET responses can be CDN-cached). Simple mental model.

Weaknesses: Over-fetching: GET /users/42 returns 50 fields but you needed 3. Under-fetching (N+1 problem): GET /orders returns 100 orders but you need each order's user name — 100 more GET /users/{id} calls. No schema enforcement by default (OpenAPI/Swagger added separately).
GraphQL — query exactly what you need
Core idea: Client defines exactly what data it wants. Server returns exactly that — no more, no less. One endpoint (POST /graphql), one query language.

Solves N+1 problem: Query orders + each order's user name in a single request. The server resolves the graph of data relationships in one round-trip. DataLoader library batches N individual DB queries into one.

The N+1 problem in depth: A resolver for orders { user { name } } naively calls getUser(order.userId) for each order — 100 orders = 100 DB queries. DataLoader batches: collect all userIds during query resolution, make one SELECT WHERE id IN (...)>, cache results for the query's lifetime.

Weaknesses: Caching is hard (POST requests are not cacheable by default — use GET with query persisting or Persisted Queries). Complex queries can be expensive — clients can accidentally request huge graphs. Schema introspection exposed by default (security concern). Overkill for simple CRUD APIs.
gRPC — high-performance service-to-service
Protocol Buffers (protobuf): Binary serialization format. Define schema in .proto files — strongly typed, compact, fast. Generates client and server code in any language.

Why binary matters: A JSON payload {"user_id":12345,"name":"Somesh"} is ~35 bytes. The equivalent protobuf binary is ~12 bytes — 3× smaller. Parsing JSON is CPU-intensive (string scanning, unicode, number parsing). Parsing protobuf is mechanical bit-shifting — 10–100× faster.

HTTP/2 native: gRPC runs exclusively over HTTP/2. Gets multiplexing, header compression, and streaming for free.

Four streaming modes:
Unary: single request → single response (like REST).
Server streaming: one request → stream of responses (real-time sensor data).
Client streaming: stream of requests → one response (file upload).
Bidirectional streaming: both sides stream independently (real-time chat between services).

Weaknesses: Not human-readable. Browser support is limited (gRPC-Web proxy needed). Requires proto file coordination between teams — schema changes must be backward compatible.
When to use which
REST
Public APIs, mobile clients, browser clients, simple CRUD. Universal support. When team familiarity and ease of debugging matter more than performance.
GraphQL
Frontend needs flexible queries across related data. Multiple client types (mobile, web) need different data shapes. BFF (Backend for Frontend) pattern.
gRPC
Internal microservice-to-microservice. High-throughput, low-latency. Strong schema enforcement. Streaming use cases. Polyglot services where generated clients save work.
Real examples
Netflix: REST external, gRPC internal. GitHub: REST + GraphQL (both). Google Cloud: gRPC APIs. Uber: gRPC between microservices, REST for mobile apps.
Interview perspective
  1. REST for external, gRPC for internal. "Public API stays REST — universal client support. Microservice-to-microservice uses gRPC — binary efficiency, streaming, generated clients."
  2. N+1 is the GraphQL gotcha. "Any GraphQL resolver that makes a DB call per parent item has an N+1 problem. I'd use DataLoader to batch and cache within a request."
  3. Protobuf field numbering is critical. "In protobuf, field names don't appear in binary — only field numbers. Removing field 2 and reusing that number for a different field corrupts all existing serialized data. Always add fields, never reuse numbers."
  4. Probe: "Razorpay's dashboard shows order details with payment info for 500 orders. REST or GraphQL?" → GraphQL. REST would require GET /orders then 500 GET /payments/{orderId} calls — classic N+1. One GraphQL query returns all orders with nested payment in one roundtrip, DataLoader batches the payment queries.

TLS / HTTPS
Handshake · certificate chain · symmetric vs asymmetric · HSTS
Why TLS exists
HTTP sends data in plaintext. On any network between you and the server — your ISP, a coffee shop router, a nation-state — that data can be read or modified. TLS (Transport Layer Security) adds three guarantees: confidentiality (data encrypted, eavesdroppers can't read), integrity (data can't be modified in transit without detection), and authentication (you're actually talking to razorpay.com, not an imposter).
TLS 1.3 handshake — step by step
Client → Server ClientHello
Client sends: TLS version, list of supported cipher suites, client random (32 bytes of randomness), and key share (the client's Diffie-Hellman public key for the supported groups). This is the first message — round-trip 1 begins.
Server → Client ServerHello + Certificate + Finished
Server picks cipher suite, sends its DH public key (ServerHello). Sends its certificate (public key + identity). Derives session keys immediately using DH key exchange. Sends Finished (encrypted with the session key — proves server has the private key). All in one flight. Client now has everything needed to verify and derive session keys.
Client Certificate verification
Client verifies the server certificate: is it signed by a trusted CA? Is the domain name correct? Is it expired? Has it been revoked (OCSP check)? If all pass: client derives the same session key (DH is symmetric), sends Finished. Sends first encrypted HTTP request in the same flight.
Both Symmetric encryption begins
From this point: AES-256-GCM symmetric encryption. Both sides have the same session key (derived via ECDH — never transmitted). One RTT total for TLS 1.3. TLS 1.2 required 2 RTT.
Symmetric vs asymmetric — the key insight
Asymmetric (RSA, ECDH): Two mathematically linked keys. Anything encrypted with the public key can only be decrypted with the private key. Used in the handshake to authenticate the server and to establish a shared secret. Slow — RSA 2048-bit operations take microseconds on dedicated hardware, milliseconds on commodity CPUs.

Symmetric (AES): One shared key used for both encryption and decryption. 100–1000× faster than asymmetric. Used for all actual data transfer after the handshake.

The pattern: Use slow asymmetric crypto once to securely establish a fast symmetric key. Then use the fast symmetric key for everything else. This is TLS in a sentence.

Why ECDH over RSA for key exchange: ECDH (Elliptic Curve Diffie-Hellman) provides perfect forward secrecy — if the server's private key is compromised later, past session keys can't be derived. RSA key exchange embeds the session secret in the RSA-encrypted premaster secret — compromise the long-term key → decrypt all past sessions. ECDH generates ephemeral keys per session.
Certificate chain
A certificate says: "I am api.razorpay.com and here is my public key, signed by DigiCert." But how do you trust DigiCert? DigiCert's certificate is signed by DigiCert Root CA. The root CA certificate is pre-installed in your OS and browser trust store (there are ~150 trusted roots).

Chain: Leaf cert (api.razorpay.com) → Intermediate CA (DigiCert SHA2) → Root CA (DigiCert Global Root CA) → trusted by OS.

Why intermediates? Root CA private keys are kept offline (air-gapped) in HSMs. If compromised, all trust is lost. Intermediates are online and issue leaf certs. Intermediate compromise affects only certs signed by that intermediate — root can revoke the intermediate cert.

Certificate Transparency (CT): All publicly trusted certs must be logged to public CT logs. This prevents CAs from secretly issuing certs for your domain. Browsers reject certs not in CT logs.
HSTS — preventing protocol downgrade
Attack: User types razorpay.com — browser first tries HTTP (port 80). MITM intercepts, never upgrades to HTTPS. User is on plain HTTP without knowing.

HSTS header: Strict-Transport-Security: max-age=31536000; includeSubDomains; preload. Once received over HTTPS, browser remembers for max-age seconds. Future visits go directly to HTTPS — never HTTP first. Eliminates the first-visit attack window.

HSTS preload: Submit to Chrome's preload list — browsers ship with your domain hardcoded as HTTPS-only, so even the very first visit is secure. Cannot be unset without shipping a browser update — permanent commitment.
Interview perspective
  1. TLS 1.3 = 1 RTT. "TLS 1.3 completes the handshake in 1 round-trip, vs TLS 1.2's 2. For a mobile user with 150ms RTT, that's 150ms saved on every new connection."
  2. Perfect forward secrecy. "I'd use ECDH key exchange, not RSA, because ECDH provides forward secrecy — compromising the server's private key doesn't decrypt past sessions."
  3. HSTS preload for payment pages. "For a payments product like Razorpay, I'd absolutely add HSTS with preload. The cost of a protocol downgrade attack on a payment page is catastrophic."
  4. Probe: "What does the certificate chain verify?" → That the entity sending you their public key actually owns the domain (via a CA that verified domain ownership through DNS or file challenges), and that the CA is trusted (via chain to a root in your OS trust store).

TCP vs UDP
Reliability vs speed · when streaming uses UDP · QUIC
The fundamental distinction
TCP and UDP are both Layer 4 transport protocols, but they make opposite bets about what the application needs more.

TCP bets: correctness over speed. Every byte arrives, in order, exactly once. If a packet is lost, TCP retransmits it and blocks everything waiting for it to arrive. The cost: latency spikes on packet loss.

UDP bets: speed over correctness. Fire and forget. No connection setup, no ACK, no retransmission, no ordering. Packets can arrive out of order, duplicated, or not at all. The application handles this — or ignores it. The cost: unreliability that the app must manage.
TCP — how it actually works
Three-way handshake: SYN → SYN-ACK → ACK. One full RTT before any data can be sent. Establishes sequence numbers and window sizes.

Reliable delivery: Every sent segment is ACK'd. Unacknowledged segments are retransmitted after a timeout (RTO — Retransmission Timeout). RTO starts at ~200ms and doubles on each retry (exponential backoff).

Flow control: Receiver advertises a receive window (how much buffer space it has). Sender can't blast faster than the receiver can consume. Prevents overwhelming slow receivers.

Congestion control: TCP infers network congestion from packet loss (a lost packet = congestion signal). Responds by reducing send rate. Algorithms: CUBIC (default Linux), BBR (Google, latency-based rather than loss-based). BBR dramatically improves throughput on high-bandwidth, high-latency links (satellite, intercontinental).

Head-of-line blocking: TCP delivers bytes in order. Lost segment → all subsequent segments wait in kernel buffer until the lost one is retransmitted. This blocks even unrelated application-level streams (the HTTP/2 problem).
Why streaming uses UDP
For a video call, receiving a frame from 100ms ago is worthless — you need the current frame. A retransmitted packet from the past arriving late is worse than nothing: it uses bandwidth and arrives too late to display.

The UDP choice for media: RTP (Real-time Transport Protocol) runs over UDP. If a packet is lost — display the last frame (freeze), use error concealment, or skip it. A brief glitch is far better than a 200ms+ freeze waiting for TCP retransmission.

WebRTC: Browser-to-browser real-time media (video calls, screen share). Uses UDP with DTLS (Datagram TLS) for security + SRTP (Secure RTP) for encryption. ICE + STUN + TURN for NAT traversal. Google Meet, Discord video, Zoom — all WebRTC for the media plane.

Online gaming: Player position updates are sent via UDP. A missed position update is immediately superseded by the next one (100ms later). Retransmitting old position data would cause lag and stuttering worse than packet loss. Game state sync uses sequence numbers in the application layer to detect and discard out-of-order packets.

DNS uses UDP: DNS queries fit in a single UDP packet (512 bytes, or up to 4096 with EDNS). UDP adds zero overhead. If a query is lost, the resolver retries — simpler than TCP for one-shot queries.
QUIC — the best of both
QUIC (HTTP/3's transport) runs over UDP but reimplements reliability, ordering, and congestion control in user space — per stream. It gets TCP's correctness with UDP's flexibility:

Connection setup: no separate TCP handshake (saves 1 RTT). Reliability: per-stream — lost packet only affects that stream, not others. Congestion control: modern algorithms (NewReno, CUBIC, or BBR configurable). Encryption: mandatory (DTLS built-in). Connection migration: connection ID-based, survives IP changes.
Decision matrix
Use TCP when
Every byte must arrive: file transfer, email, HTTP, SSH, database connections. Correctness is non-negotiable. Latency spikes on packet loss are acceptable.
Use UDP when
Latency beats correctness: video/audio streaming (RTP), online gaming, DNS, IoT sensor data, real-time telemetry. Old data is worthless.
Use QUIC/HTTP/3 when
You want TCP reliability with UDP flexibility. Mobile clients (connection migration). High packet-loss environments. Modern APIs (if server/CDN supports it).
Numbers
TCP 3-way handshake: 1 RTT overhead. UDP: 0 RTT overhead. TCP retransmit timeout: 200ms–3s. QUIC connection: 0-1 RTT for new/resumed.
Interview perspective
  1. The key tradeoff in one sentence: "TCP guarantees delivery but HOL blocking under loss. UDP is fast but unreliable — the application handles reliability selectively."
  2. Real-time media always UDP. "For your Zombie Hunt real-time multiplayer, player positions go over UDP. A missed position packet is superseded in 50ms. Retransmitting it would cause the notorious 'lag spike' users hate."
  3. Probe: "How does Google Meet handle packet loss?" → WebRTC over UDP/RTP. On packet loss: FEC (Forward Error Correction) — pre-emptively sends redundant data so one loss can be recovered without retransmission. For video: concealment (freeze/extrapolate from last frame). For audio: comfort noise, PLC (Packet Loss Concealment).
  4. QUIC bridges the gap. "For HTTP/3 and modern APIs, QUIC gives you TCP's reliability per-stream but UDP's speed and flexibility at the transport level — connection migration is the killer feature for mobile."
