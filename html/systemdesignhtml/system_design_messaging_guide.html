Message queues vs event streams
RabbitMQ vs Kafka · pull vs push · when to use each
The fundamental difference
The distinction is not speed — it's about what happens to a message after a consumer reads it.

Message queue (RabbitMQ, SQS): A message is consumed and deleted. It's a task to be done, handed to exactly one worker. Once processed, it's gone. The queue is a pipeline, not a log.

Event stream (Kafka): Events are written to a durable, ordered, append-only log. Consumers read from any position and can re-read. The stream is a permanent record. Multiple independent consumers can each read the full stream at their own pace — nothing is deleted on read.

Pull vs push
RabbitMQ — classic message broker (AMQP):

Messages flow through exchanges → bindings → queues → consumers. Producers publish to an exchange with a routing key. The exchange routes the message to one or more queues based on binding rules (direct, fanout, topic, headers).

Push-based delivery: RabbitMQ pushes messages to consumers. Consumers register and receive messages as they arrive. Low latency for individual messages. The broker tracks which messages are in-flight and acked.

Message lifecycle: Consumer receives message → processes → sends ACK → broker deletes message. NACK or timeout → message goes back to queue (or to DLQ after max retries).

Best for: Task queues (image processing jobs, email sending), RPC patterns, complex routing (topic exchanges), when each message should be processed exactly once by exactly one worker, when message ordering is less critical than delivery.
Kafka — distributed event streaming platform:

Events are appended to topic partitions. Each partition is an ordered, immutable sequence of records. Records are retained for a configured period (default: 7 days, can be forever). Consumer offsets track where each consumer group is in the log.

Pull-based delivery: Consumers poll Kafka for new messages. They control their own read rate, replay, and position. The broker tracks no per-message delivery state — just the consumer's current offset.

Message lifecycle: Record appended to partition log → consumer reads at its own pace → consumer commits offset. Record remains in the log after reading — other consumer groups can read the same record independently.

Best for: Event sourcing, audit logs, activity tracking, change data capture, stream processing (Kafka Streams, Flink), fan-out to many independent consumers, replay and time-travel, data pipeline (ETL), when you need multiple independent consumers of the same events.
Push (RabbitMQ, SQS): Broker pushes messages to consumers as they arrive. Low per-message latency. Broker must manage backpressure — if consumers are slow, the broker must throttle producers or buffer messages. Consumer must be continuously connected.

Pull (Kafka): Consumers poll for batches of messages at their own pace. Consumer controls throughput. Broker doesn't need to track consumer state (just stores the log). Consumers can batch large reads for efficiency. Long-polling: consumer waits up to N ms for data to arrive, reducing empty poll overhead.

Why Kafka chose pull: With diverse consumers (some fast, some slow), push creates backpressure complexity at the broker. Pull lets each consumer run at its own optimal rate. The broker stores the log — consumers manage their own progress. This decoupling is what enables replaying events independently across consumer groups.

Pull downside: Slightly higher per-message latency (next poll interval). Not suitable for sub-millisecond response requirements.
Decision guide
Use RabbitMQ / SQS when
Task queue semantics (do this once). Complex routing rules. Per-message ACK/NACK. At-most-once delivery OK. Traditional worker pool pattern.
Use Kafka when
Multiple independent consumers need same events. Event replay required. Ordering within a partition matters. High throughput (millions/sec). Event sourcing, CDC, audit log.
Both wrong? Consider
Redis Streams: lighter than Kafka, heavier than Redis Lists. Good for moderate scale. AWS EventBridge: fully managed, serverless, routing rules without infra.
Scale numbers
RabbitMQ: ~50K msgs/sec per node. Kafka: ~1M msgs/sec per broker (sequential disk I/O). SQS: effectively unlimited (managed). Numbers assume defaults, tuning can 10× these.
Trade-offs
✓ Kafka strengths

Immutable log — replay, audit, CDC. Fan-out to unlimited consumers. Linear horizontal scale. Survives consumer downtime (log persists).

✗ Kafka cost

Operationally complex. Higher latency per message (ms vs µs). Ordering only within a partition. Overkill for simple task queues.

✓ RabbitMQ strengths

Per-message ACK semantics. Complex routing (exchanges, bindings). Lower operational complexity. Sub-millisecond delivery. Priority queues.

✗ RabbitMQ cost

Messages gone on consume — no replay. Fan-out requires message duplication to each queue. Memory-based → can fill up under load spikes.

Interview perspective
  1. Frame by semantics, not speed. "RabbitMQ is for task distribution — one worker, one message. Kafka is for event broadcasting — many consumers, permanent log."
  2. Notification system → Kafka. Multiple services (email, push, SMS) all consume the same notification event independently. Kafka fan-out is correct.
  3. Job queue → RabbitMQ/SQS. Image resize job should be processed by exactly one worker. No replay needed. RabbitMQ or SQS is correct.
  4. Probe: "User service updates user email. How do other services learn?" → Publish a user.email_updated event to Kafka. Profile service, notification service, and billing service each have their own consumer group and consume independently.

Kafka deep dive
Topics · partitions · offsets · consumer groups · replication
Core concepts — precisely defined
Topic
Named stream of records. Logically represents a category of events: orders, payments, user-events. Divided into partitions for parallel processing.
Partition
Ordered, immutable, append-only sequence of records within a topic. Unit of parallelism. Records in a partition are totally ordered. Records across partitions are not ordered relative to each other.
Offset
Sequential integer assigned to each record within a partition (0, 1, 2, …). Unique within a partition. Consumer tracks its current offset — the position it has read up to. Committed to __consumer_offsets topic.
Consumer group
A set of consumers that together consume a topic. Each partition is assigned to exactly one consumer in the group. Enables parallel consumption. Multiple independent groups can each consume the full topic.
Replication factor
Number of copies of each partition across brokers. RF=3: one leader, two followers. Leader handles all reads and writes. Followers replicate. On leader failure, a follower is elected leader.
ISR (In-Sync Replicas)
Set of replicas that are fully caught up with the leader. A write is acknowledged only after ISR replicas write it (configurable: acks=all). If ISR shrinks below min.insync.replicas, producers get an error.

Consumer group mechanics
The assignment rule: Each partition is consumed by exactly one consumer in a consumer group at any time. If the group has fewer consumers than partitions, some consumers handle multiple partitions. If more consumers than partitions — extra consumers are idle.

Optimal parallelism: consumers_in_group ≤ partitions_in_topic. Set partitions at topic creation based on your target consumer parallelism.

Rebalancing: When a consumer joins or leaves the group, Kafka's group coordinator triggers a rebalance — partitions are redistributed. During rebalance, consumption pauses (~seconds). Minimize rebalances: use static membership (group.instance.id) for long-lived consumers.

Key producer configs
acks=0
Fire and forget. No broker ack. Fastest, data loss risk. Use for metrics where occasional loss is OK.
acks=1
Leader acks. Fast. Risk: leader crashes before replication — data lost. Most common default.
acks=all (-1)
All ISR replicas ack. No data loss (if min.insync.replicas ≥ 2). Slower. Required for exactly-once.
Partition key
Records with the same key always go to the same partition → ordering guarantee per key. Null key → round-robin across partitions.
Interview perspective
  1. Partitions = unit of parallelism. "I'd set partitions = expected max consumers for this topic. For an order processing service with 10 workers, I'd create 10+ partitions."
  2. Partition key = ordering guarantee. "I'd use user_id as the partition key so all events for a given user are in the same partition and are processed in order."
  3. Replication factor = 3 always in production. "RF=3 means we can lose one broker with no data loss and one more with no availability loss."
  4. Probe: "Kafka consumer is slow, messages pile up." → Increase partition count (allows more parallel consumers). Add consumers to the group (up to partition count). Check for slow processing per message — is it waiting on DB? Cache expensive lookups. Consider async processing inside consumer.

Delivery guarantees
At-most-once · at-least-once · exactly-once — how each is achieved
The three guarantees
What it means
How achieved
At-most-once
Message delivered 0 or 1 times. May be lost, never duplicated. Fire and forget.
Producer: acks=0. Consumer: commit offset before processing. If crash mid-process → message lost, not retried.
At-least-once
Message delivered 1 or more times. Never lost, may be duplicated. Consumer must be idempotent.
Producer: acks=all + retries. Consumer: process first, commit offset after. Crash before commit → reprocess on restart.
Exactly-once
Message delivered exactly once. No loss, no duplicates. The gold standard, and the hardest.
Kafka: idempotent producer (dedup by sequence number) + transactional API (atomic write + offset commit). Requires consumer.isolation.level = read_committed.
Why at-least-once is the real default
The fundamental problem: Between processing a message and committing the offset, the consumer can crash. On restart, it re-reads and reprocesses. This gives at-least-once delivery.

The correct response: Make your consumers idempotent. Processing the same message twice must produce the same result as processing it once. Examples: INSERT ... ON CONFLICT DO NOTHING, UPDATE ... WHERE version = expected_version, SET key = value (idempotent by definition). An idempotent consumer with at-least-once delivery is effectively exactly-once from a business logic perspective — without the overhead of Kafka transactions.
Kafka exactly-once — deep mechanics
Idempotent producer: Each producer is assigned a PID (producer ID). Each message gets a sequence number. Broker deduplicates based on (PID, partition, sequence_number). Retries are safe — duplicates are silently dropped. Enabled with enable.idempotence=true.

Transactional producer: Wraps produce + offset commit in a single atomic transaction. Either both succeed or neither does. Consumer with isolation.level=read_committed only sees records from committed transactions — uncommitted records are invisible.

The two-phase commit under the hood: Kafka's transaction coordinator uses 2PC internally between the producer log and the __transaction_state topic. This is what guarantees the atomic commit — but it's fully managed by Kafka, not exposed to the application.

Interview perspective
  1. Default: at-least-once + idempotent consumers. "I design consumers to be idempotent so duplicate messages don't matter. This is simpler and cheaper than exactly-once Kafka transactions."
  2. Exactly-once for financial data. "For payment processing where a duplicate could double-charge, I'd use Kafka transactions with enable.idempotence=true and isolation.level=read_committed."
  3. Idempotency key pattern. "Each event carries a unique idempotency key. The consumer checks if that key was already processed before acting. Deduplication at the application layer is often simpler than exactly-once at the broker layer."
  4. Probe: "Payment processed twice." → Either not idempotent consumer, or exactly-once not configured. Fix: idempotency key stored in DB with unique constraint. Second process attempt hits unique constraint → no-op. This is the practical exactly-once for payments.

Event-driven architecture
Event sourcing · CQRS · eventual consistency
Event-driven architecture — the mindset shift
Traditional architecture: service A calls service B synchronously and waits. Event-driven: service A publishes an event to a bus and moves on. Service B (and C, D…) react when they're ready. The producer doesn't know or care who consumes the event.

Benefits: Decoupling — producer and consumer never need to know about each other. Resilience — if B is down, the event waits in Kafka; B processes when it recovers. Scalability — consumers scale independently. Auditability — the event log is a complete history.
Event sourcing
Core idea: Don't store the current state of an entity. Store the sequence of events that led to that state. The current state is derived by replaying events.

Example (bank account): Instead of a row account_id=1, balance=850, store: AccountOpened(initial=1000), Deposited(200), Withdrawn(350). Current balance = replay: 1000 + 200 - 350 = 850.

Benefits: Complete audit trail for free. Time travel — replay events up to any point in time. Event replay to rebuild read models. Debugging — reproduce any bug by replaying the exact sequence of events.

Costs: Reading current state requires replaying all events (use snapshots to mitigate — snapshot every N events). Schema evolution for old events is hard. Query patterns must work on the event log structure.
CQRS — Command Query Responsibility Segregation
Core idea: Split the write model (commands) from the read model (queries). They are separate code paths, separate data stores if needed.

Write side (command): Accepts commands (PlaceOrder, UpdateProfile). Validates, applies business rules, persists to the write store (event log or normalized DB), publishes events.

Read side (query): Optimized for specific query patterns. One read model per query type. Populated asynchronously from events. Can be denormalized, pre-aggregated, cached — whatever makes reads fast.

Example: Order write store = normalized Postgres (ACID, constraints). Order read model for "user's order history page" = a MongoDB document per user with embedded last-10 orders (pre-denormalized for fast page load). Updated by consuming order.placed events.

The consistency tradeoff: Read models are updated asynchronously → there's a propagation lag. User places order, immediately refreshes order history — might not see the new order yet. This is the cost of CQRS eventual consistency.

Interview perspective
  1. Outbox pattern is mandatory with event-driven. "I'd never write directly to Kafka in the same operation as writing to the DB — dual-write creates consistency risk. Instead, write to an outbox table in the same transaction, then relay to Kafka."
  2. CQRS separates scale concerns. "Writes hit the normalized Postgres. Reads hit purpose-built read models. Each scales independently."
  3. Eventual consistency is the tradeoff to name explicitly. "The read model is eventually consistent — typically 100–500ms behind. For this use case, that's acceptable. For balance checks before a payment, I'd read from the write model directly."
  4. Event sourcing ≠ Kafka. "Event sourcing is a storage pattern. Kafka is a transport. You can event-source to Postgres and publish to Kafka separately."

Dead letter queues
Poison messages · retry strategies · exponential backoff
What a poison message is
A poison message is a message that causes the consumer to fail repeatedly — crashing or throwing exceptions every time it's processed. Without a DLQ, the consumer retries indefinitely, blocking all other messages in the queue. A Dead Letter Queue (DLQ) is the safety valve: after N failed attempts, the message is moved out of the main queue into a separate DLQ where it can be inspected and handled without blocking normal processing.
The full retry pipeline
1
First delivery attempt. Consumer processes message. Succeeds → ACK → deleted. Fails → NACK → back to queue.
2
Retry with backoff. Wait 1s → retry. Fail again → wait 2s → retry. Fail → 4s → retry. Exponential backoff: wait = min(base × 2ⁿ, max_delay). Add jitter: wait = wait × random(0.5, 1.5) to prevent retry storms when many consumers fail simultaneously.
3
Max retries exceeded. After N attempts (common: 3–5), message is moved to the DLQ. Consumer moves on to the next message in the main queue. No more blocking.
4
DLQ inspection. Alert fired (DLQ depth > 0 = production incident). Engineers inspect the poison message. Root cause: bad data format? Downstream service down? Bug in consumer?
5
Remediation. Fix the bug. Either replay from DLQ (reprocess fixed messages), or dead-letter permanently (log and discard truly unprocessable messages). SQS: redrive policy re-enqueues DLQ messages to the source queue.
Exponential backoff with jitter — the math
Base delay: 1s | Max delay: 60s | Jitter: ±30% Attempt 1 fail: wait = min(1 × 2⁰, 60) × jitter = 1s × 0.85 = 0.85s Attempt 2 fail: wait = min(1 × 2¹, 60) × jitter = 2s × 1.20 = 2.4s Attempt 3 fail: wait = min(1 × 2², 60) × jitter = 4s × 0.92 = 3.7s Attempt 4 fail: wait = min(1 × 2³, 60) × jitter = 8s × 1.10 = 8.8s Attempt 5 fail: → DLQ Without jitter (thundering herd): 1000 consumers all fail at t=0 All retry at exactly t=1s → 1000 simultaneous retries → downstream crushed All fail again at t=1s → all retry at t=3s → same problem With jitter: Retries spread across [0.7s, 1.3s] window → 1000/0.6s ≈ 1667/s → manageable
DLQ in different systems
SQS + DLQ
Native support. Set maxReceiveCount on source queue. Messages auto-moved to DLQ. Redrive policy sends them back. CloudWatch alarm on ApproximateNumberOfMessagesVisible in DLQ.
RabbitMQ
Dead letter exchange (DLX). Set x-dead-letter-exchange on queue. Messages routed to DLX after NACK or TTL expiry. Separate DLQ queue bound to DLX.
Kafka
No native DLQ. Convention: consumer catches exception after N retries, produces message to {topic}-dlq topic. Spring Kafka: DeadLetterPublishingRecoverer does this automatically.
Retry topics (Kafka)
Multi-stage retry: fail → orders-retry-1 (wait 5s) → fail → orders-retry-2 (wait 30s) → fail → orders-dlq. Each retry topic has a consumer with built-in delay. Used by Spring Kafka's non-blocking retry.
Interview perspective
  1. Always mention DLQ when designing any queue-based system. "I'd configure a DLQ with maxRetries=3 and exponential backoff. DLQ depth > 0 triggers a PagerDuty alert."
  2. Exponential backoff + jitter is mandatory. "Fixed retries create synchronized thundering herds. Exponential backoff with jitter spreads retries across a window."
  3. Poison message vs transient failure: "A transient failure (DB briefly down) should recover on retry. A poison message (malformed data) will never succeed. The DLQ catches the latter and lets ops handle it without manual queue surgery."
  4. Probe: "DLQ is full. What do you do?" → Alert fires. First: determine if it's a bug (all messages bad) or a downstream outage (all messages fail because payment service is down). Bug → fix, redrive after fix. Outage → wait for service recovery, then redrive. Never blindly redrive without understanding the root cause.

Pub/Sub pattern
Decoupling producers and consumers · fan-out · real-world examples
Definition
Publish-Subscribe is a messaging pattern where publishers send messages to a topic (or channel) without knowing who will receive them. Subscribers register interest in a topic and receive all messages published to it — without knowing who published them. The topic is the decoupling point.

The core value: producer and consumer are completely independent. Adding a new consumer never requires changing the producer. Removing a consumer has zero effect on the producer or on other consumers.
Fan-out patterns
Topic fan-out
SNS + SQS fan-out
Kafka fan-out
Topic fan-out: One published message → delivered to all subscribers of that topic. N subscribers = N deliveries of the same message.

Use case: a payment.succeeded event must trigger: email notification service, fraud analytics service, accounting service, loyalty points service. All four are independent subscribers. Publisher (payment service) knows nothing about them.

The decoupling payoff: To add a 5th subscriber (say, a regulatory reporting service), you add a subscriber — zero changes to the payment service or any other subscriber. This is impossible with direct API calls.
AWS SNS + SQS fan-out pattern:

SNS topic is the pub/sub bus. Each subscriber has its own SQS queue subscribed to the SNS topic. SNS delivers a copy to each SQS queue. Each consumer processes from its own queue at its own rate.

Why SQS per consumer? Without a queue, if the subscriber service is down during the SNS delivery, the message is lost. An SQS queue buffers messages — the consumer catches up when it recovers. This makes the fan-out durable.

Pattern: Producer → SNS → [SQS-email, SQS-analytics, SQS-accounting] → independent consumers. Each SQS queue has its own DLQ.
Kafka fan-out via consumer groups:

Each consumer group is an independent subscriber to the topic. Group A reads from offset 0 at its own pace. Group B reads from offset 0 at its own pace. The same event is consumed by both groups — independently, without any broker-side duplication.

Add a new consumer group = new subscriber. Zero changes to producer or other groups.

Advantage over SNS+SQS: With Kafka, you can add a new consumer group after the fact and replay historical events from the beginning of the retention window. With SNS+SQS, you only receive events from the moment you subscribe. Kafka's log is a durable, replayable event history — pub/sub with time travel.
Notification system — pub/sub design
Notification system (WhatsApp / Razorpay style): Event sources (producers): Payment Service → publishes to Kafka topic "notifications" Order Service → publishes to Kafka topic "notifications" Kafka topic "notifications": Event: {type: "payment_succeeded", user_id: 42, amount: 1000} Consumer Group "email-notifications": Reads from "notifications" → filters type=* → sends email via SES Consumer Group "push-notifications": Reads from "notifications" → filters type=* → sends via FCM/APNS Consumer Group "sms-notifications": Reads from "notifications" → filters type=payment_* → sends SMS Consumer Group "in-app-notifications": Reads from "notifications" → writes to Redis pub/sub → WebSocket push Redis pub/sub: server-side pub, WebSocket connections subscribe → real-time in-app notification Fan-out numbers: 100K payment events/day → 4 consumer groups → 400K deliveries Zero producer changes to add a 5th notification channel
Pub/Sub vs point-to-point
✓ Pub/Sub

Producer-consumer decoupling. Fan-out to N consumers. Easy to add new consumers. Natural for event-driven systems.

✗ Pub/Sub

No guarantee of consumer order. Harder to track "did all consumers process this?" Debugging distributed event flows is complex.

✓ Point-to-point

Exactly one consumer processes each message. Simple reasoning about message ownership. Easy ACK/NACK per message.

✗ Point-to-point

Tight coupling between producer and specific consumer. Adding consumer = producer change. No fan-out without message duplication.

Interview perspective
  1. Pub/Sub is your default for cross-service communication. "Service A should publish an event. Services B, C, D subscribe. A never calls B directly — that's tight coupling."
  2. SNS+SQS fan-out on AWS. Know this pattern cold: SNS → multiple SQS queues → one consumer per queue. Each SQS acts as a durable buffer.
  3. Kafka for replayable fan-out. "Kafka lets me add a new consumer group and replay historical events — something SNS fan-out can't do."
  4. Probe: "Design a notification system." → Single Kafka topic "notifications". One consumer group per channel (email, push, SMS, in-app). Filter by notification type within each consumer. Scale each consumer group independently based on channel volume.
