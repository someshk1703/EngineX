SQL vs NoSQL
When to use each — not "NoSQL is faster"
The real question
"NoSQL is faster" is a myth. PostgreSQL on properly sized hardware with good indexes handles millions of QPS. The right question is: what does your data look like, and what queries do you need to run?

SQL wins when your data has relationships, you need joins, you need ACID transactions across multiple entities, and your schema is mostly known upfront. NoSQL wins when your data is document-shaped, your schema evolves rapidly, you need to shard horizontally across many nodes, or you need specific access patterns that SQL handles poorly at extreme scale.
Decision framework
Use SQL (Postgres, MySQL) when
Relational data with joins. ACID transactions (payments, inventory). Complex queries (aggregations, GROUP BY). Schema is stable. Team knows SQL. Data integrity matters above all.
Use Document DB (MongoDB, DynamoDB) when
JSON-shaped data where all fields accessed together (no joins). Schema changes frequently. Horizontal write scaling needed. Hierarchical data (orders with line items embedded).
Use Wide-Column (Cassandra, HBase) when
Time-series data. Write-heavy workloads (IoT, events, metrics). Need linear horizontal scale. Queries are always by a known partition key — no arbitrary queries.
Use Graph DB (Neo4j) when
Data IS the relationships. Friend-of-friend queries. Fraud ring detection. Recommendation engines where traversing N hops of relationships is the primary query pattern.
Use Search Engine (Elasticsearch) when
Full-text search with relevance ranking. Fuzzy matching. Log aggregation (ELK stack). Not a primary DB — always a secondary index over the source of truth.
Use Time-series DB (InfluxDB, TimescaleDB) when
Metrics, monitoring, IoT sensor data. Queries are always "range over time." Automatic downsampling and retention policies. TimescaleDB = Postgres extension — good if you already use Postgres.
The "NoSQL is schemaless" myth
NoSQL databases are not schemaless — they have schema-on-read rather than schema-on-write. The schema is enforced by your application code when it reads the data. This means: if you write documents inconsistently (some have "email", some have "emailAddress"), the DB accepts both. Your application breaks at read time.

SQL enforces schema-on-write — the DB rejects malformed data at insert time. This strictness is a feature, not a limitation.
Architecture — polyglot persistence
Real systems use multiple DBs for different access patterns: User service → Postgres (relational, ACID, user+account joins) Product catalog → MongoDB (flexible schema, embedded reviews) Search → Elasticsearch (full-text, relevance) Sessions → Redis (low-latency key-value, TTL) Driver location → Redis (sorted set by geo, 4×/sec writes) Analytics → ClickHouse (columnar, OLAP, fast aggregations) Event stream → Kafka (durable log, consumer groups) Metrics/monitoring → InfluxDB (time-series, downsampling) Rule: use the right tool for each access pattern. Never force all data into one DB to "keep it simple."
Trade-offs
✓ SQL strengths

ACID guarantees. Joins across tables. Mature query optimizer. FOREIGN KEY integrity. 40 years of operational knowledge.

✗ SQL weaknesses

Horizontal write scaling requires sharding (complex). Schema migrations on large tables are painful. JOIN performance degrades on very large tables without careful indexing.

✓ NoSQL strengths

Horizontal scale by design. No schema migrations. Fits document-shaped access patterns natively. Tunable consistency (AP or CP).

✗ NoSQL weaknesses

No joins (must denormalize). Eventual consistency is hard to reason about. No multi-document ACID (DynamoDB added it, but it's limited). Application must enforce data integrity.

Interview perspective
  1. Never say "NoSQL is faster." Say: "NoSQL is better suited for horizontal write scaling and flexible schemas. For relational data with transactions, SQL is the right choice."
  2. Default to Postgres for most services unless you can justify otherwise with a specific access pattern argument.
  3. Polyglot persistence is the senior answer. "I'd use Postgres for the order service (ACID), Elasticsearch as a read model for search, and Redis for session state."
  4. Probe: "Your product catalog has 500M SKUs. SQL or NoSQL?" → SQL with proper indexing first. If schema varies wildly per category (electronics vs clothing), MongoDB. If querying by arbitrary attributes, Elasticsearch.

Database indexing
B-tree · composite · covering index · selectivity
Definition & core purpose
An index is a separate data structure that maps column values to the row locations (page + offset) that contain those values. Without an index, every query does a full table scan — O(N). With an index, lookups are O(log N) for B-tree, O(1) for hash indexes.

The cost: indexes consume disk space and slow down writes (every INSERT/UPDATE/DELETE must also update all indexes on the table). The tradeoff: read-heavy workloads benefit enormously. Write-heavy workloads can be hurt by too many indexes.
B-tree index — deep mechanics
Postgres's default index type. A balanced tree where all leaf nodes are at the same depth. Internal nodes hold keys + pointers to child nodes. Leaf nodes hold keys + pointers to heap (actual table rows).

Properties: Self-balancing (height stays O(log N) even as data grows). Supports range queries (<, >, BETWEEN) — in-order traversal of leaf nodes. Supports ORDER BY and MIN/MAX cheaply. Page size typically 8KB; tree height for 1B rows ≈ 4–5 levels.

Why not a hash index? Hash indexes are O(1) for equality (=) but cannot do range queries, ORDER BY, or LIKE prefix matching. B-tree does all of these. Postgres hash indexes also weren't WAL-logged until PG10, making them unsafe.
Index types — key distinctions
Single-column index
CREATE INDEX ON orders(user_id). Lookup by user_id uses index. Any other column (status, created_at) still requires full scan or separate index.
Composite index
CREATE INDEX ON orders(user_id, status, created_at). Efficient for queries on (user_id), (user_id, status), (user_id, status, created_at). NOT efficient for (status) alone — leftmost prefix rule.
Covering index
Index includes all columns needed by a query — the DB never needs to touch the heap (table). SELECT status FROM orders WHERE user_id=42 with index on (user_id, status) = index-only scan. Zero heap I/O.
Partial index
CREATE INDEX ON orders(created_at) WHERE status='pending'. Only indexes a subset of rows. Smaller, faster for the specific subset. Perfect for "active records" queries.
Expression index
CREATE INDEX ON users(lower(email)). Indexes the result of a function. Query must use the same expression: WHERE lower(email) = .... Case-insensitive email lookup.
GIN index (Postgres)
Generalized Inverted Index. For JSONB, arrays, full-text search. Stores each element → row mapping. WHERE data @> '{"tag":"urgent"}' uses GIN on the JSONB column.
Index selectivity — the critical concept
Selectivity = fraction of rows returned by a given value. High selectivity = few rows returned = index is useful. Low selectivity = many rows returned = full scan may be faster.

High selectivity (good index candidate): email (unique), order_id (unique), UUID.
Low selectivity (poor index candidate): boolean column (only 2 values), status ENUM with 3 values, gender. A query WHERE is_active = true on a table where 90% of rows are active → index returns 90% of rows → Postgres chooses full scan anyway (correct choice).

Composite index selectivity: Index on (country, city, email) has low selectivity on country alone but high selectivity on (country, city, email) together. The optimizer evaluates the combined selectivity.
Architecture — query plans
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42 AND status = 'pending'; WITHOUT INDEX: Seq Scan on orders (cost=0.00..18450.00 rows=1 width=64) Filter: ((user_id = 42) AND (status = 'pending')) Rows Removed by Filter: 1844999 Planning time: 0.2ms Execution time: 890ms ← SLOW WITH INDEX ON (user_id, status): Index Scan using orders_user_id_status_idx on orders (cost=0.56..8.58 rows=1 width=64) Index Cond: ((user_id=42) AND (status='pending')) Planning time: 0.3ms Execution time: 0.08ms ← 11,000× faster COVERING INDEX ON (user_id, status) INCLUDE (created_at, total): SELECT status, total FROM orders WHERE user_id=42 → Index Only Scan ← never touches table heap, max speed
Failure modes
Index not used — implicit type cast: Column is INTEGER, query passes string: WHERE user_id = '42'. Postgres casts '42' to int, but the cast breaks index usage. Fix: use the correct type in your query.
Index not used — function on indexed column: WHERE YEAR(created_at) = 2024 wraps the indexed column in a function — index can't be used. Fix: WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'.
Too many indexes: Table with 20 indexes. Every write must update all 20 indexes. Write throughput collapses. Fix: audit with pg_stat_user_indexes — drop indexes with zero scans.
Leftmost prefix violation: Index on (user_id, status). Query on status alone → full scan. Developers assume all columns in a composite index are independently queryable.
Interview perspective
  1. Selectivity first. "Before adding an index, I check selectivity. A boolean column with 50/50 split has low selectivity — the optimizer won't use the index."
  2. Composite index column order: Put the highest-selectivity, most-frequently-queried column first. Equality conditions before range conditions.
  3. Covering indexes eliminate heap I/O entirely — mention these for read-heavy, latency-critical queries.
  4. Probe: "The query is slow even with an index." → Run EXPLAIN ANALYZE. Check: is it doing an Index Scan or Seq Scan? If Seq Scan despite an index: table statistics may be stale (run ANALYZE), or the selectivity is so low the optimizer chose correctly.

Transactions & ACID
Atomicity · Consistency · Isolation · Durability · Isolation levels
ACID properties — precise definitions
Atomicity
All operations in a transaction succeed or all are rolled back. No partial commits. If a payment debit succeeds but the credit fails → entire transaction rolls back. Implemented via undo logs.
Consistency
A transaction brings the database from one valid state to another valid state. All constraints (FOREIGN KEY, CHECK, UNIQUE, NOT NULL) are satisfied after the transaction. The DB rejects commits that violate constraints.
Isolation
Concurrent transactions execute as if they were serial. The degree of isolation is tunable via isolation levels — each level prevents specific anomalies at the cost of concurrency (and performance).
Durability
Once committed, data survives crashes. Implemented via WAL — the commit is written to durable storage (fsync) before the transaction is acknowledged. The in-memory data can be lost; the WAL ensures replay on restart.
Isolation levels — the full spectrum

Anomaly definitions — know these cold
Dirty read: Transaction A reads data written by Transaction B, which hasn't committed yet. If B rolls back, A read data that never existed. Prevented from Read Committed upward.

Non-repeatable read: Transaction A reads a row. Transaction B updates and commits that row. Transaction A reads the row again — it's different. A got two different values for the same row within one transaction. Prevented from Repeatable Read upward.

Phantom read: Transaction A queries SELECT COUNT(*) WHERE status='pending' → 10 rows. Transaction B inserts a new pending order and commits. Transaction A runs the same query → 11 rows. New "phantom" rows appeared. Prevented at Serializable.

Lost update: Two transactions read the same value, both compute a new value based on the old, both write. The first write is overwritten (lost). Classic race condition on a counter. Fix: SELECT FOR UPDATE, or atomic operations like UPDATE SET count = count + 1.
Architecture — payment transaction
BEGIN; -- Atomicity: both updates in one transaction UPDATE accounts SET balance = balance - 1000 WHERE id = 1; UPDATE accounts SET balance = balance + 1000 WHERE id = 2; -- Consistency: constraint check (balance must be >= 0) -- CHECK (balance >= 0) will reject if id=1 has < 1000 -- Isolation: other transactions can't see these changes until COMMIT -- Durability: on COMMIT, WAL flushed to disk before ack COMMIT; -- or ROLLBACK if either fails -- Serializable isolation for seat reservation: BEGIN ISOLATION LEVEL SERIALIZABLE; SELECT * FROM seats WHERE seat_id = 42 FOR UPDATE; -- lock the row -- if another transaction is modifying seat 42: wait or error UPDATE seats SET reserved_by = 99 WHERE seat_id = 42; COMMIT;
Interview perspective
  1. Know all 4 anomalies by name and example. "Dirty read: reading uncommitted data. Non-repeatable read: same row gives different values within a transaction. Phantom: new rows appear in repeated range queries."
  2. Postgres default is Read Committed — not Serializable. Most teams never change it and rely on SELECT FOR UPDATE for specific critical sections.
  3. Probe: "How do you prevent double-spending?" → Serializable isolation + SELECT FOR UPDATE on the balance row. Or: optimistic locking with a version column (check version on update, retry on mismatch).
  4. ACID vs BASE: NoSQL systems often use BASE (Basically Available, Soft state, Eventually consistent) instead of ACID. "Razorpay uses ACID (Postgres) for payments; Cassandra uses BASE for event logs."

Database replication
Sync vs async · primary-replica · multi-primary · replication lag
What replication solves
A single database node is a SPOF. Replication maintains one or more copies (replicas) of the primary's data on other nodes. This gives: high availability (replica can take over if primary fails), read scaling (read queries go to replicas, freeing the primary for writes), and geographic distribution (replica in another region reduces read latency for users there).
Synchronous vs asynchronous
Async (default)
Synchronous
Multi-primary
Async replication (Postgres default): Primary commits the write locally, acknowledges to the client immediately, then sends WAL records to replicas in the background. If primary crashes before replicas receive the WAL: up to a few seconds of data loss (the replication lag at the time of crash).

Replication lag: The delay between a write on the primary and it appearing on replicas. Under normal load: 10ms–100ms. Under heavy write load or network congestion: seconds. Reads from a replica during lag return stale data.

Detecting lag: SELECT now() - pg_last_xact_replay_timestamp() on the replica. Alert if > 30s. Route reads to primary if freshness is critical.
Synchronous replication: Primary waits for at least one replica to confirm it has received AND flushed the WAL record before acknowledging the write to the client. Zero data loss on primary failure — the replica has everything.

Cost: Write latency increases by the primary→replica round-trip + replica disk fsync. On the same rack: ~1ms extra. Cross-AZ: ~5ms. Cross-region: 50–150ms. Writes that normally take 2ms now take 7ms → 3.5× slower writes.

Postgres config: synchronous_commit = on (default: off). synchronous_standby_names = 'replica1'. If the named replica goes down, writes on the primary block until it reconnects or synchronous_commit is changed.
Multi-primary (multi-master): Multiple nodes accept writes. No single primary. Each node replicates to all others.

The hard problem: Concurrent writes to the same row on different primaries create conflicts. Who wins? Last-write-wins (LWW) silently discards one write. Conflict resolution logic must be in the DB or application.

Where it works well: Geographically distributed writes where each region primarily writes its own data (CockroachDB, Google Spanner, Galera for MySQL). Works poorly for global shared data (e.g., a single counter incremented by all regions).

Single-primary is almost always the right choice for most teams. Multi-primary adds enormous complexity for a problem you probably don't have yet.

Trade-offs
✓ Async replication

No write latency impact. Replica can lag without blocking writes. Primary failure doesn't cascade.

✗ Async replication

Data loss on crash = replication lag at time of failure. Stale reads from replicas. Replica can be seconds behind under load.

✓ Sync replication

Zero data loss guarantee. Replica always has complete data. Safe failover.

✗ Sync replication

Write latency up by replica RTT. If replica goes down: writes block (or must change config). Problematic for cross-region.

Interview perspective
  1. Default: async primary-replica. "For most services, async replication is correct — the replication lag is acceptable and write performance is not impacted."
  2. When to upgrade to sync: "For the payment service, I'd use synchronous replication or RDS Multi-AZ to guarantee zero data loss on primary failure."
  3. Read replica routing: Mention routing reads to replicas for analytics/reporting but always reading from primary for anything that needs to be immediately consistent after a write.
  4. Probe: "Your replica is 30 seconds behind. User writes data, immediately reads it — what happens?" → Read-after-write inconsistency. Fix: route the read immediately after a write to the primary, or track the replication LSN and wait until replica catches up.

Database sharding
Range · hash · directory · hotspots · cross-shard queries
Definition & why it's a last resort
Sharding = horizontally partitioning data across multiple independent database instances, each holding a subset of the data. Each shard is a fully independent database that has no knowledge of the others.

Sharding is a last resort. Before sharding: vertical scale, read replicas, caching, archiving old data, denormalization. Sharding adds massive operational complexity and destroys cross-shard transactions and joins. Most teams that think they need sharding actually need better indexing and caching first.
Three sharding strategies
Range sharding
Hash sharding
Directory-based
Range sharding: Partition key range divided into contiguous ranges. Shard 1: user_id 1–1M, Shard 2: 1M–2M, Shard 3: 2M+.

Pros: Range queries are efficient (all data for a date range is in one shard). Easy to reason about. Shard routing is simple arithmetic.

Fatal flaw — hotspot: Sequential inserts (auto-increment IDs, timestamps) all land on the last shard. New users → all go to shard 3. New orders → all go to the latest shard. One shard gets all writes while others are idle. You've added operational complexity without any write scaling benefit.

Use when: Queries are time-range based (logs, metrics, events). You can tolerate hot recent shards and archive old ones (HBase, ClickHouse time partitions).
Hash sharding: shard = hash(user_id) % N. Data is distributed uniformly across all shards — no hotspots for random data.

Pros: Uniform load distribution. No hotspot problem for write-heavy workloads. Simple to implement.

Fatal flaw — resharding: Adding a shard changes N → hash(id) % (N+1) remaps almost all keys to different shards. Massive data migration required. Consistent hashing (as used by Cassandra, Dynamo) reduces remapping to ~1/N of keys.

Use when: Write load is uniform across keys. You can predict your shard count in advance and rarely need to add shards. User-sharded social data (user A's data always goes to shard X).
Directory-based sharding: A separate lookup service (or table) maps each key → shard. shard_lookup(user_id=42) → shard_3.

Pros: Maximum flexibility. Can move individual keys between shards (rebalance hot users). No resharding problem. Supports arbitrary distribution.

Fatal flaw — lookup service as SPOF: Every DB request requires a directory lookup first. Directory becomes a high-QPS service (1B requests/day = 11K QPS). If it's down, all DB routing fails. Must be cached aggressively (cached in app memory), highly available.

Use when: You need the ability to rebalance specific "hot" tenants (large customers) to dedicated shards. Multi-tenant SaaS where each customer might have wildly different data volumes.
The cross-shard problem
Joins: User on shard 1, orders on shard 2 (if not co-located). SELECT * FROM users JOIN orders ON users.id = orders.user_id → impossible in one query. Must: fetch users from shard 1, fetch matching orders from shard 2, join in application. Slow and complex.

Cross-shard transactions: Atomic update across two shards requires 2-phase commit (2PC). Complex, slow, and a distributed systems landmine. Avoid by co-locating related data on the same shard (same user's orders always go to the shard that holds that user).

Aggregations: SELECT COUNT(*) FROM orders → must query all shards, sum results. Scatter-gather pattern. Only works if you can tolerate approximate counts or query all shards in parallel.
Architecture — shard routing
Shard Router (application or middleware layer) Request: GET /orders?user_id=42 │ ├─ shard_id = hash(42) % 4 → shard_id = 2 │ ├─ Route to Shard 2 (Postgres instance @ db-shard-2:5432) │ └─ Execute: SELECT * FROM orders WHERE user_id = 42 Co-location strategy (critical): User 42's profile → shard 2 (hash(42) % 4 = 2) User 42's orders → shard 2 (same shard key: user_id=42) User 42's cart → shard 2 (same) Result: ALL of user 42's data lives on one shard. Joins within a user are fast. Cross-user joins still scatter-gather. Hotspot problem solution: Hot user (celebrity with 100M followers): Override directory: user_id=1 → dedicated shard 99 (large instance) Regular users hash normally to shards 0-3
Interview perspective
  1. Propose sharding reluctantly. "I'd shard only after exhausting vertical scaling, read replicas, caching, and data archival. Sharding adds complexity that affects every query and transaction."
  2. Always propose a shard key with co-location in mind. "I'd shard on user_id and co-locate all user-related data (orders, sessions, preferences) on the same shard to avoid cross-shard joins."
  3. Hotspot is a mandatory mention. "Range sharding on timestamp creates a hot shard problem — all new writes go to the latest shard. I'd use hash sharding or bucket timestamps to spread load."
  4. Probe: "How do you handle a viral celebrity user on your social platform?" → That user creates a hotspot on their shard. Fix: directory-based routing to move them to a dedicated high-capacity shard, or cache their data aggressively at the application layer.

Database partitioning
Horizontal vs vertical · partition pruning
Partitioning vs sharding — the distinction
Partitioning splits data within a single database instance across multiple physical storage units (partitions). All partitions are managed by the same DB engine. Queries can span partitions transparently — the DB handles the routing.

Sharding splits data across multiple independent database instances. The application is responsible for routing.

Think of partitioning as an optimization technique within one DB. Sharding is a distributed systems architecture. You can use both simultaneously: partition each shard internally.
Horizontal vs vertical partitioning
Horizontal partitioning (row-based)
Split rows across partitions by a partition key. orders_2023, orders_2024. Each partition holds all columns for its subset of rows. This is what most people mean by "partitioning." Postgres PARTITION BY RANGE/LIST/HASH.
Vertical partitioning (column-based)
Split columns across tables. Hot columns (id, status, created_at) in one table, cold columns (long_description, metadata_json) in another. Reduces I/O for queries that only need hot columns. Essentially normalization taken further.
Range partitioning
PARTITION BY RANGE (created_at). Partition per month or year. Time-series data. Old partitions can be dropped instantly (DROP PARTITION vs DELETE with index churn). Queries by date range hit only relevant partitions.
List partitioning
PARTITION BY LIST (region). Partition per country or region. All EU users in eu_partition, US users in us_partition. Useful for data locality (GDPR: EU data stays on EU storage) and query isolation.
Hash partitioning
PARTITION BY HASH (user_id). Even distribution across N partitions. No hotspot for sequential inserts. Good for parallelizing large tables without a natural time or list dimension.
Partition pruning
Query optimizer detects which partitions could contain matching rows and skips the rest. WHERE created_at BETWEEN '2024-01-01' AND '2024-03-31' on a monthly partitioned table → scans only Q1 partitions. 9× faster than scanning all 12.
Architecture — time-partitioned orders table
CREATE TABLE orders ( id BIGSERIAL, user_id INT, created_at TIMESTAMP, total DECIMAL ) PARTITION BY RANGE (created_at); CREATE TABLE orders_2023 PARTITION OF orders FOR VALUES FROM ('2023-01-01') TO ('2024-01-01'); CREATE TABLE orders_2024 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2025-01-01'); -- Query hits ONLY orders_2024 (partition pruning): SELECT * FROM orders WHERE created_at > '2024-06-01'; -- EXPLAIN shows: Seq Scan on orders_2024 (not orders_2023) -- Drop old data instantly (vs DELETE which takes hours and bloats WAL): DROP TABLE orders_2022; -- instant, no row-level locking -- Indexes per partition (smaller, faster): CREATE INDEX ON orders_2024 (user_id, created_at);
Failure modes
Partition key in WHERE is mandatory for pruning: SELECT * FROM orders WHERE user_id = 42 (no created_at filter) → scans ALL partitions. Pruning only works when the partition key appears in the WHERE clause.
Too many partitions: 1 partition per day for 10 years = 3,650 partitions. Postgres opens a file handle per partition — at some point, planning overhead exceeds pruning benefit. Keep partitions in the hundreds, not thousands.
Cross-partition transactions: Insert into the wrong partition (partition constraint violation) → error. Always use the parent table for inserts and let Postgres route to the correct partition.
Interview perspective
  1. Partitioning before sharding. "I'd partition the orders table by month before considering sharding. Partitioning is transparent, keeps everything in one DB, and enables instant data archival."
  2. Partition pruning is the payoff. "With monthly partitions and a date filter, the query only scans the relevant month — 1/12 the I/O of a full scan."
  3. Archival use case: "Dropping a partition to archive old data is O(1) — it's just removing a file pointer. DELETE for the same operation would take hours and generate massive WAL."
  4. Probe: "GDPR requires EU user data to stay in EU." → List partitioning by region. EU partition stored on EU-region storage. Non-EU partition on US storage. Postgres tablespaces control where each partition is physically stored.

Connection pooling
Why it exists · HikariCP · pool exhaustion
Why connections are expensive
A TCP connection to Postgres is not cheap. The DB must fork (or use a lightweight process), allocate memory (~5–10MB per connection on the backend process), establish TLS, authenticate, and set up session state. This takes ~50–100ms. Doing this on every request is catastrophic at scale.

Postgres has a hard limit on max connections (~100–500 depending on RAM). At 100 connections × 10MB = 1GB of RAM used just for connection overhead, before a single byte of data is processed. A connection pool maintains a fixed set of long-lived connections and lends them to request threads for the duration of a query.
How HikariCP works
HikariCP is the fastest JDBC connection pool (Spring Boot's default). Internally: a lock-free ConcurrentBag of connection wrappers. When a thread requests a connection:

1. Check if the calling thread previously returned a connection (thread-local cache) → return it immediately (O(1), no contention).
2. If not, look in the shared bag for an idle connection → acquire and return.
3. If all connections are checked out → wait up to connectionTimeout (default: 30s) for one to become available.
4. Timeout → throw SQLTimeoutException (pool exhaustion).

Key config parameters:
maximumPoolSize: max connections to DB. Recommendation: start with 10, tune based on CPU cores on DB server. More is not better — DB CPU becomes the bottleneck.
minimumIdle: connections kept open when idle. connectionTimeout: max wait for a connection. idleTimeout: close idle connections after this duration. maxLifetime: recycle connections before DB's server-side timeout kills them (prevents "connection was closed" errors).

Architecture — PgBouncer + HikariCP
Why both HikariCP AND PgBouncer? App (Spring Boot, 50 instances) × HikariCP pool size 10 = 500 total connections to PgBouncer PgBouncer (connection pooler proxy, transaction mode) = 20 actual connections to Postgres Postgres max_connections = 100 Without PgBouncer: 50 app instances × 10 pool size = 500 connections to Postgres Postgres crashes (max_connections exceeded) With PgBouncer (transaction pooling): A connection is borrowed from PgBouncer only for the duration of a single transaction, then returned immediately. 500 app connections → multiplexed → 20 Postgres connections. Works because most app threads wait for response, not holding a transaction. CRITICAL: PgBouncer transaction mode breaks: - SET commands (session-level config is lost between transactions) - Prepared statements (must disable) - Advisory locks - LISTEN/NOTIFY
Failure modes
Pool exhaustion: All connections checked out, new requests queue. Queue fills → SQLTimeoutException → 503s to users. Root cause: slow queries holding connections too long. Fix: find slow queries (pg_stat_activity), add indexes, reduce query time. Short-term: increase pool size but this is treating symptoms.
Connection leak: Code opens a connection but doesn't close it (missing try-with-resources). Pool is gradually depleted. Fix: always use try-with-resources (Java), or Spring's @Transactional which handles connection lifecycle.
Pool size too large: 500 connections to a Postgres with 8 CPUs. DB spends more time context-switching between connections than executing queries. Throughput drops. The optimal pool size formula (HikariCP docs): pool_size = (core_count × 2) + effective_spindle_count. For 8-core DB with SSDs: ~17 connections per app instance.
Interview perspective
  1. Know why pooling exists and the cost of a raw connection (~50ms, ~10MB). "Without pooling, a 1000 RPS service would try to open 1000 DB connections per second — the DB falls over instantly."
  2. Pool size rule: "Bigger pool is not always better. Optimal size is roughly 2× CPU cores on the DB server. More connections than that and the DB spends time on context switching."
  3. PgBouncer is the senior-level addition. "At scale with many app instances, each with their own pool, I'd put PgBouncer in front of Postgres to multiplex hundreds of app connections down to a few dozen real DB connections."
  4. Pool exhaustion diagnosis: "If I see SQLTimeoutException, I check pg_stat_activity to find which queries are long-running and holding connections. Usually it's a missing index causing a slow query, not actually needing a bigger pool."

Write-Ahead Log (WAL)
Crash recovery · replication · change data capture
Definition & core purpose
The WAL is a sequential, append-only log file where every change to the database is written before the change is applied to the actual data pages. The guarantee: if it's in the WAL, it's durable, even if the process crashes a millisecond later.

The WAL is the implementation backbone of three critical features: crash recovery (Durability in ACID), streaming replication (replicas replay the primary's WAL), and Change Data Capture (tools like Debezium read the WAL to produce change events).
How it works — step by step
1. Write to WAL buffer (in memory): When a transaction modifies data, Postgres writes the change record (operation type, table OID, old/new values) to the WAL buffer in shared memory. This is fast — sequential memory write.

2. Flush WAL buffer to disk (fsync): At COMMIT, Postgres fsyncs the WAL buffer to the WAL segment file on disk. This is the moment of durability. The data file (heap) has NOT been updated yet — only the log.

3. Apply to heap (lazy): The actual data pages (heap files) are updated asynchronously by the background writer (bgwriter) when convenient. During peak load, the heap may be seconds behind the WAL.

4. Crash recovery: On restart, Postgres finds the last checkpoint (a known good point where heap = WAL). Replays all WAL records after the checkpoint. In seconds, the heap is consistent with the WAL again.

Sequential write = fast: Writing 1MB of WAL sequentially takes ~0.1ms on SSD. Updating random data pages (random I/O) would take much longer. WAL converts random I/O into sequential I/O — this is why it's fast.
WAL for replication & CDC
Streaming replication: The WAL sender process on the primary streams WAL records to the WAL receiver on the replica in near-real-time. The replica's recovery process replays these records, applying them to the replica's heap. The replica is essentially always in "recovery mode" — constantly replaying the primary's WAL.

Logical replication (Postgres 10+): Physical replication copies raw WAL bytes — only works between identical Postgres versions on the same architecture. Logical replication decodes WAL into logical change events (INSERT/UPDATE/DELETE at the table level) and streams those. Allows replication to different Postgres versions, or to non-Postgres targets (Kafka via Debezium, Elasticsearch).

CDC with Debezium: Debezium acts as a Postgres logical replication slot consumer. It reads the logical WAL stream, converts changes to JSON events, and publishes to Kafka. Zero impact on the application — the WAL is produced anyway.

Failure modes
WAL bloat from idle replication slots: A logical replication slot prevents WAL segments from being deleted (replica might need them). If the replica disconnects and never reconnects, WAL accumulates indefinitely. Disk fills → Postgres stops accepting writes. Monitor with: SELECT pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn)) FROM pg_replication_slots. Set max_slot_wal_keep_size limit.
Checkpoint too infrequent: Large gap between checkpoints means crash recovery must replay a lot of WAL → slow restart. Tune checkpoint_completion_target and max_wal_size.
fsync disabled: Setting fsync=off dramatically improves write performance but disables WAL durability guarantee. One power cut → DB corruption. Never in production. The performance gain is real but the risk is catastrophic.
Interview perspective
  1. WAL = durability in ACID. "Durability is implemented via WAL — the commit record is fsynced to disk before the client is acknowledged. Even if the heap hasn't been updated, the WAL guarantees recovery."
  2. WAL = replication mechanism. "Streaming replication works by the primary streaming WAL records to replicas, which replay them. That's why replication lag = WAL receiver being behind the WAL sender."
  3. WAL → CDC. "For cache invalidation, I'd use Debezium to read Postgres's logical WAL and publish change events to Kafka. Zero application code change needed — works for all write paths including batch jobs."
  4. Probe: "How does Postgres recover after a crash?" → Finds last checkpoint in pg_control. Replays all WAL records after that checkpoint. Achieves consistency in seconds (dependent on WAL volume since last checkpoint).

MVCC
How Postgres avoids read locks — multi-version concurrency control
The problem MVCC solves
Naive approach to concurrent transactions: readers take shared locks, writers take exclusive locks. Reader blocks writer. Writer blocks all readers. At high concurrency, everything queues. Throughput collapses.

MVCC's insight: readers never block writers, and writers never block readers. Instead of locking, Postgres keeps multiple versions of each row simultaneously. Each transaction sees a consistent snapshot of the database as it existed at the start of that transaction — even if other transactions are modifying data concurrently.
How it works — xmin/xmax
Every row in Postgres has two hidden system columns:

xmin: The transaction ID (XID) that created this row version.
xmax: The transaction ID that deleted (or updated) this row version. Zero if the row is currently live.

When a transaction reads a row, it checks: is xmin ≤ my snapshot's XID and xmax = 0 (or > my snapshot's XID)? If yes, this row version is visible to me.

When a transaction UPDATES a row: it marks the old version with xmax = my_XID (soft delete) and inserts a NEW row version with xmin = my_XID. The old version still exists on disk — other concurrent transactions that started before the update see the old version. The transaction that did the update (and later transactions) see the new version.

VACUUM — cleaning up dead rows
MVCC creates "dead rows" — old versions that are no longer visible to any active transaction. These consume disk space and slow down scans. VACUUM reclaims this space.

Autovacuum: Postgres runs autovacuum automatically as a background process. It periodically scans tables for dead rows and removes them. You can tune aggressiveness via autovacuum_vacuum_scale_factor (default: 20% of table = vacuum trigger) and autovacuum_vacuum_cost_delay.

VACUUM FULL: Rewrites the entire table to a new file without dead rows. Reclaims maximum space but requires exclusive lock on the table. Avoid in production. Use regular autovacuum instead.

Table bloat: High-write tables with slow autovacuum accumulate dead rows. Table grows to 10× the live data size. Queries slow down proportionally. Fix: tune autovacuum aggressively for hot tables, or use pg_partman to rotate partitions.
MVCC and isolation levels
MVCC is how Postgres implements isolation levels:

Read Committed: Each statement gets a new snapshot at statement start. A re-read within the same transaction can see committed changes by others (non-repeatable read is possible).

Repeatable Read: Snapshot taken at transaction start. All reads within the transaction see the same consistent state. Committed changes by others are invisible (no non-repeatable reads). Postgres also prevents phantom reads at this level (stricter than SQL standard).

Serializable (SSI): Postgres's Serializable Snapshot Isolation. Uses the MVCC snapshot mechanism but additionally detects "serialization anomalies" — situations where the result couldn't have been produced by any serial execution order. On detection: transaction is aborted with a serialization error and must retry. Zero false positives (unlike 2PL which causes unnecessary blocking).
Interview perspective
  1. "Readers don't block writers, writers don't block readers" — say this verbatim. It's the core MVCC elevator pitch and interviewers recognize it as accurate.
  2. Explain xmin/xmax if asked how it works. "Every row has a creation XID (xmin) and a deletion XID (xmax). A transaction sees a row if xmin is before its snapshot and xmax is after (or zero)."
  3. VACUUM is the MVCC gotcha. "MVCC leaves dead row versions that must be cleaned up by VACUUM. On high-write tables, if autovacuum can't keep up, table bloat accumulates and performance degrades."
  4. Probe: "How does Postgres implement Repeatable Read without locking?" → MVCC snapshot taken at transaction start. All reads in the transaction see the same version of data as of that snapshot, regardless of concurrent commits. No shared locks required.
