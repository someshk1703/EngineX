Topic 1 of 6 — Block 5: State Management

Client state vs server state

Different problemsOwnershipStale dataRedux misuse
Definition

Client state and server state are fundamentally different problems masquerading as the same one. Client state is UI state that lives exclusively in the browser: is this modal open, which tab is selected, what text is in this search input. You own it, it's always synchronous, it doesn't expire, and only your app can change it. Server state is data that lives on the server and is fetched asynchronously: the user's profile, the list of products, the current order status. You don't own it — the server does. It can change while your user is looking at it. It can become stale. It requires loading/error/success states. It needs caching, deduplication, background refresh, and pagination. Managing these two with the same tool (Redux) is the most common architectural mistake in large React codebases.

How they differ internally

Client state characteristics

Synchronous. You control when it changes. Never stale — it only changes when your code changes it. No loading states. No network errors. Small, transient. Examples: modal open/close, selected filter, form draft, sidebar collapsed, dark mode preference.

Server state characteristics

Async — requires fetch. Can change on the server at any time without your knowledge. Needs caching (don't re-fetch unchanged data). Needs deduplication (10 components mounting = 1 request, not 10). Needs background refresh. Can fail. Needs pagination and infinite scroll support. Examples: user data, API lists, order status, prices.

The Redux-for-server-state anti-pattern

Redux was designed for client state. When teams use it for server state, they end up writing: manual loading flags, manual error states, manual cache invalidation, manual deduplication, manual background polling, manual optimistic updates — thousands of lines of boilerplate solving problems that TanStack Query solves in 3 lines.

The right split

Server state → React Query / TanStack Query / SWR / RTK Query. Client UI state → useState / useReducer / Zustand / Jotai. Global client state that's truly global (theme, auth status) → Context or Zustand. Don't conflate layers.

The problems server state creates that client state doesn't

Staleness

Server data fetched at T=0 may not reflect server reality at T=60s. Client state never stales — only you change it. Server state requires a TTL and background revalidation strategy.

Request deduplication

10 components all call useUser(id) simultaneously on mount. Without deduplication: 10 network requests. React Query: 1 request, 10 subscribers to the result. Redux: you have to implement this yourself.

Cache synchronization

User updates their profile. Now GET /user cached 30s ago is stale. You need to invalidate the cache. React Query: queryClient.invalidateQueries(['user']). Redux: manually dispatch to clear + refetch, update reducer, update selectors.

Real-world architecture use case

Razorpay / CRED-style dashboard architecture

A payments dashboard separates concerns cleanly. Server state (transactions, user KYC status, wallet balance, recent settlements): React Query, auto-refreshed every 30s, invalidated on any mutation. Client state (selected date range filter, sidebar open/closed, which transaction row is expanded, multi-select state): Zustand store, never touches the network. Auth state (session token, user role): Zustand with persistence, seeded from server on login, treated as client state after that. This separation means the server state layer can be swapped (React Query → SWR → RTK Query) without touching a single UI component.

Common mistakes
Storing API response data in Redux/Zustand and writing a manual cache layer on top of it. This is reinventing React Query poorly. If data lives on a server, use a data-fetching library that understands server state semantics.
Using React Query for UI state — useQuery with a local-only key and no queryFn (or a function that doesn't fetch). This misuses the cache layer for something that doesn't need caching or network semantics. Use useState.
Interview perspective
"Why is managing server state with Redux problematic?"
Redux is designed for synchronous, you-own-it state. Server state has async lifecycle, staleness, deduplication, background refresh, cache invalidation — none of which Redux handles natively. Teams end up writing thousands of lines of middleware, reducers, and selectors to solve problems that TanStack Query solves out of the box. The real issue: Redux doesn't know your data is stale. React Query does.
"What belongs in global client state vs server state vs local component state?"
Local component state: UI that doesn't need to be shared (form inputs, toggle states, hover effects). Global client state: UI state shared across distant components (theme, locale, sidebar collapsed, multi-step wizard progress). Server state: anything fetched from an API (use React Query / SWR). Don't over-globalize — component state is free; global state has coordination overhead.

Redux internals

Store / reducerAction flowMiddlewareSelector memoisation
Definition

Redux is a predictable state container based on three principles: single source of truth (one store, one state tree), state is read-only (only actions can trigger changes), and changes are made with pure functions (reducers). The unidirectional data flow — action → dispatcher → reducer → new state → UI re-render — makes state changes traceable, reproducible, and time-travel-debuggable. Redux's primary valid use cases today: complex client-side state with many interdependencies, applications requiring strict audit trails, or teams already on Redux who haven't migrated. New projects should default to Zustand + React Query.

How it works internally

The Redux store is an object with four methods: getState(), dispatch(action), subscribe(listener), and replaceReducer(nextReducer). When dispatch(action) is called, the store passes the current state and the action to the root reducer, receives a new state object, stores it, then calls all subscriber listeners. React-Redux's useSelector subscribes to the store — when the store notifies listeners, it runs the selector function and re-renders the component if the selected value changed (using strict equality by default). Middleware is a higher-order function that wraps dispatch — it intercepts every action before it reaches the reducer, enabling async logic (thunk), logging, analytics, and sagas. The middleware chain is dispatch → middleware1 → middleware2 → reducer.

Subtopics

Reducer purity requirement

Reducers must be pure functions: (state, action) → new state. No side effects (no API calls, no random values, no Date.now(), no mutations of the existing state). Purity enables: time-travel debugging (replay actions on pure functions), predictable testing, and concurrent renders in React 18.

Redux Thunk

Allows dispatching functions (thunks) instead of plain objects. The thunk middleware checks if the dispatched value is a function; if so, calls it with (dispatch, getState). Enables async action creators. Simple, synchronous-looking async code. Limited: can't cancel, can't model complex async flows cleanly.

Redux Saga

Uses ES6 generators to model complex async flows as readable sequential code. Sagas run as long-lived background threads, listening for actions via take() and running effects via call()/put()/race()/takeLatest(). Power: cancellation, debounce/throttle, complex orchestration. Cost: steep learning curve, large API surface.

Selector memoisation — Reselect

createSelector(inputSelectors, resultFn) memoises derived state computation. Only recomputes when any input selector's output changes (by reference). Prevents expensive transformations (filter + sort of large arrays) from running on every dispatch, even unrelated ones. RTK uses Reselect internally via createSelector.

Redux Toolkit (RTK)

The modern way to write Redux. createSlice generates actions + reducer from one object. Uses Immer internally — lets you write mutating-looking code that actually produces immutable updates. createAsyncThunk handles pending/fulfilled/rejected action lifecycle. Eliminates 80% of Redux boilerplate.

Normalisation in Redux

RTK's createEntityAdapter provides a normalised state structure (entities by ID + allIds array) with CRUD operations built in. Prevents the N+1 update problem where denormalised state requires updating the same data in multiple places.

Code example
// RTK slice — modern Redux (no boilerplate)
import { createSlice, createAsyncThunk, createSelector } from '@reduxjs/toolkit';

export const fetchCart = createAsyncThunk('cart/fetch', async (userId) => {
  const res = await fetch(`/api/cart/${userId}`);
  return res.json();
}); // Generates cart/fetch/pending, /fulfilled, /rejected actions

const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], status: 'idle', error: null },
  reducers: {
    // Immer: write "mutating" code → Immer produces new immutable state
    addItem(state, action) { state.items.push(action.payload); },
    removeItem(state, action) {
      state.items = state.items.filter(i => i.id !== action.payload);
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchCart.pending,   (state) => { state.status = 'loading'; })
      .addCase(fetchCart.fulfilled, (state, { payload }) => {
        state.status = 'succeeded'; state.items = payload;
      })
      .addCase(fetchCart.rejected,  (state, { error }) => {
        state.status = 'failed'; state.error = error.message;
      });
  },
});

// Memoised selector — only recomputes when cart.items changes
const selectCartItems   = state => state.cart.items;
const selectCartTotal   = createSelector(
  selectCartItems,
  (items) => items.reduce((sum, i) => sum + i.price * i.qty, 0)
); // Memoised: won't recompute if unrelated state changes (e.g. auth)

// Redux Thunk vs Saga — same async logic, different model
// Thunk: async function dispatching actions
const fetchAndProcess = (userId) => async (dispatch, getState) => {
  dispatch({ type: 'LOADING' });
  try {
    const data = await fetchUser(userId);
    dispatch({ type: 'SUCCESS', payload: data });
  } catch (e) { dispatch({ type: 'ERROR', error: e }); }
};

// Saga: generator-based, cancellable, composable
function* fetchUserSaga(action) {
  try {
    const user = yield call(fetchUser, action.payload);
    yield put({ type: 'SUCCESS', payload: user });
  } catch (e) { yield put({ type: 'ERROR', error: e }); }
}
function* watchFetchUser() {
  yield takeLatest('FETCH_USER', fetchUserSaga);
  // takeLatest: cancels in-flight saga if action fires again
  // No equivalent without extra code in thunk
}
Thunk vs Saga — when to choose

Use thunk when

Simple async flows: fetch → dispatch success/error. Team knows async/await. RTK's createAsyncThunk covers 80% of real-world needs with built-in lifecycle actions. No saga learning curve needed.

Use saga when

Complex orchestration: race conditions (search autocomplete — cancel previous request on new keystroke), long-running background tasks, action-watching patterns, debounce/throttle of dispatch, transactional flows that need rollback. Power comes at the cost of generator syntax familiarity.

Common mistakes
Mutating state directly in reducers without Immer — Redux requires immutable updates. Without RTK/Immer, writing state.user.name = action.payload silently breaks Redux because the reference doesn't change, so subscribers aren't notified. Always use RTK (Immer included) or spread correctly.
Deriving state in components instead of memoised selectors — const total = useSelector(state => state.cart.items.reduce(...)) inside a component re-runs the reduce on every dispatch, even ones completely unrelated to cart. Extract into a createSelector memoised selector.
Not using takeLatest in sagas for search/autocomplete — if the user types fast, multiple saga instances run simultaneously, and a slow earlier response can overwrite a faster later one. takeLatest cancels the previous saga on each new action.
Interview perspective
"Explain Redux's unidirectional data flow."
UI dispatches an action (plain object with type). Middleware intercepts (thunk checks if it's a function, logging middleware logs it). The root reducer receives (currentState, action) and returns new state (pure function, no mutations). Store stores new state, notifies all useSelector subscribers. Subscribers re-run their selector; if the selected value changed (reference equality), component re-renders. One direction, always.
"Why must reducers be pure functions?"
Purity enables time-travel debugging (replay action history on pure functions to reproduce any state). Enables predictable testing (same inputs → same output, always). Enables React 18 concurrent rendering (render phase may call reducers multiple times — side effects in reducers would fire multiple times). Immutable updates enable efficient change detection via reference equality.
"When would you choose Redux Saga over Thunk?"
When you need: cancellation (takeLatest stops previous saga on new action), complex sequencing (take an action, call API, wait for another action, call second API), debouncing dispatches, or transactional patterns with compensation logic. Thunk is sufficient for straightforward fetch → dispatch flows.

Zustand vs Jotai vs Recoil

Store modelAtom modelRe-render granularityBundle size
Definition

Zustand, Jotai, and Recoil represent two competing mental models for global state management without Redux's verbosity. Store model (Zustand): one or more stores hold all state. Components subscribe to slices of a store via selector functions. Re-renders only when the selected slice changes. Atom model (Jotai, Recoil): state is broken into smallest possible units called atoms. Components subscribe to individual atoms. When an atom changes, only components subscribed to that atom re-render. Atoms can derive from other atoms (selectors in Recoil, derived atoms in Jotai). The key architectural difference: store model is top-down (one store, many subscribers), atom model is bottom-up (many atoms, compose upward).

Zustand

store model
  • ~1KB bundle
  • No Provider needed
  • Selector-based subscriptions
  • Middleware (devtools, persist, immer)
  • Outside-React access (non-hook API)
  • Best for: shared client UI state

Jotai

atom model
  • ~3KB bundle
  • Provider optional
  • Atom = smallest unit of state
  • Derived atoms (computed)
  • React Suspense integration
  • Best for: granular, composable state

Recoil

atom model
  • ~21KB bundle (large)
  • RecoilRoot Provider required
  • Selectors with async support
  • Atom families (parameterised)
  • Meta-maintained (less active)
  • Best for: complex graph state
How Zustand works internally

Zustand uses a vanilla JS store (not tied to React). The store is a closure containing the current state and a set of subscriber functions. useStore(selector) is a React hook that subscribes to the store and calls selector(state) — it stores the previous selected value and re-renders only when the new selected value is not strictly equal to the previous. The React integration uses useSyncExternalStore internally (React 18). Because the store is external to React, you can read and write state from anywhere — event handlers, WebSocket listeners, non-React code. This is Zustand's key advantage over Context: no Provider tree, no re-render propagation through the component tree.

How Jotai works internally

Each atom is an object with a unique key and a config (initial value or read/write functions). Jotai maintains an atom store (a WeakMap keyed by atom objects) that holds current values and subscriber sets. useAtom(atom) reads from the store and subscribes to that atom. When setAtom is called, only components subscribed to that specific atom are notified — not all components subscribed to a parent store. Derived atoms (atom(get => get(countAtom) * 2)) re-compute when their dependencies change, and only notify their own subscribers. This makes re-render granularity extremely fine — changing atom A only re-renders components that subscribed to atom A or atoms derived from A.

Code example
// ── Zustand ──
import { create } from 'zustand';
import { immer } from 'zustand/middleware/immer';
import { devtools, persist } from 'zustand/middleware';

const useCartStore = create(
  devtools(persist(immer((set) => ({
    items: [],
    count: 0,
    addItem(item) {
      set(state => { // Immer middleware: mutate freely
        state.items.push(item);
        state.count++;
      });
    },
    removeItem(id) {
      set(state => {
        state.items = state.items.filter(i => i.id !== id);
        state.count--;
      });
    },
  })), { name: 'cart-storage' })) // persist to localStorage
);

// Selector subscription — only re-renders when `count` changes
const count = useCartStore(state => state.count);
// This component is immune to `items` array changes

// Outside React — Zustand's key advantage
useCartStore.getState().addItem(newItem); // WebSocket handler, no React needed

// ── Jotai ──
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';

const countAtom = atom(0);
const doubledAtom = atom((get) => get(countAtom) * 2); // Derived
const asyncUserAtom = atom(async (get) => { // Async atom — integrates with Suspense
  return fetch('/api/user').then(r => r.json());
});

function Counter() {
  const [count, setCount] = useAtom(countAtom); // Read + write
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

function Display() {
  const doubled = useAtomValue(doubledAtom); // Read-only subscription
  return <span>{doubled}</span>;
}
// Display re-renders when doubledAtom recomputes (when countAtom changes)
// Other components not using countAtom/doubledAtom: zero re-renders
Re-render granularity comparison

Context: worst granularity

All consumers re-render when any part of the context value changes. No selector support natively. Fine for rarely-changing values (theme, locale). Terrible for frequently-changing values (counter, scroll position).

Zustand: selector granularity

Components subscribe to a slice via selector. Only re-render when that slice changes (Object.is comparison). One component can select count; another selects items — they don't affect each other. Near-optimal without atom-level precision.

Jotai/Recoil: atom granularity

Each atom is independently subscribed. Components only re-render when their specific atom(s) change. No false re-renders from unrelated state. Maximum granularity. Overhead: more atoms to manage, but composition via derived atoms is elegant.

When granularity matters

High-frequency updates (scroll position, live counters, real-time data) benefit from fine granularity. Low-frequency updates (user settings, auth) — any approach works. Over-engineering granularity for slowly-changing state adds complexity without benefit.

Common mistakes
Missing the selector in Zustand: const store = useStore() (no selector) subscribes to the entire store — every state change re-renders this component. Always pass a selector: useStore(s => s.count).
Creating too many atoms in Jotai for state that always changes together — if atomA and atomB always update together, they should be one atom with an object value. Fine-grained atoms are for state that changes independently.
Interview perspective
"How does Zustand achieve fine-grained re-renders without a Provider?"
Zustand uses useSyncExternalStore (React 18) to subscribe components to an external vanilla JS store. The selector function determines which slice of state the component cares about. React compares the old and new selected value by reference — if unchanged, no re-render. No Provider means no re-render propagation through the tree — components directly subscribe to the store, bypassing React's context mechanism entirely.
"When would you choose Jotai over Zustand?"
When state is genuinely fine-grained and independent (each form field as an atom, each list item's selection state as an atom). When you want React Suspense integration with async atoms. When you're building a library where users compose atoms rather than using a central store. For most application-level client state, Zustand's simplicity and DX are preferable.

React Query / TanStack Query internals

stale-while-revalidateCache keyBackground refetchOptimistic updates
Definition

TanStack Query (formerly React Query) is an async state manager specifically designed for server state. It implements a sophisticated cache with lifecycle management: every query is identified by a query key (array), has a configurable staleTime (how long before data is considered stale), a gcTime (formerly cacheTime — how long to keep data in cache after no subscribers), automatic background refetch on window focus, tab visibility, and network reconnect, and built-in deduplication (multiple hooks with the same key share one inflight request). It solves the core problems of server state that Redux leaves to the developer.

How it works internally

The QueryClient maintains a QueryCache — a Map keyed by serialised query keys. Each entry is a Query object containing: current state (status, data, error, dataUpdatedAt), a subscriber set, the queryFn, and scheduling state. When useQuery mounts, it registers a subscriber with the query. If no query exists for that key, one is created and the queryFn is executed. If a query exists and is fresh (data age < staleTime), the cached data is returned immediately. If stale, cached data is returned and a background refetch is triggered — this is the stale-while-revalidate pattern. When multiple components mount with the same key, they all register as subscribers to the same Query object — only one network request fires. On unmount, the subscriber is removed. When subscriber count hits 0, a gcTime timer starts — if no new subscriber arrives before it expires, the query is removed from the cache.

Subtopics

Query key design

Array-based: ['users'], ['users', userId], ['users', userId, 'posts']. Keys are serialised for comparison. Hierarchical: invalidating ['users'] also invalidates ['users', anyId]. Include all dependencies that affect the query result in the key — if it changes, the query reruns.

staleTime vs gcTime

staleTime: data is "fresh" for this duration — no refetch will trigger. After staleTime, data is "stale" — still shown but refetched in background. gcTime: how long to keep data in cache after all subscribers unmount. Default 5 minutes. After gcTime, cache entry is deleted.

Background refetch triggers

On window focus (user switches tabs and comes back). On network reconnect. On mount (if data is stale). On configurable refetchInterval (polling). On manual queryClient.invalidateQueries(). All respect staleTime — no refetch if data is fresh.

Mutations & cache invalidation

useMutation for write operations. In onSuccess: invalidate related queries via queryClient.invalidateQueries(['users']) — triggers all matching queries to refetch. This is how UI stays consistent after a POST/PUT/DELETE without manual state updates.

Prefetching

queryClient.prefetchQuery() fills the cache before the component mounts. On hover over a link, prefetch the next page's data — it's in cache when the user navigates. Used in Next.js with RSC: prefetch in Server Component, hydrate in Client Component.

Pagination & infinite queries

useInfiniteQuery for infinite scroll. Each page is a separate query entry in the cache. fetchNextPage() fetches the next page and appends. keepPreviousData option prevents UI flash when changing page parameters (search, filters).

Code example
// Core useQuery patterns
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';

function UserProfile({ userId }) {
  const { data: user, isLoading, isStale } = useQuery({
    queryKey: ['user', userId],    // Key includes all deps that affect the result
    queryFn: () => getUser(userId),
    staleTime: 5 * 60 * 1000,      // Fresh for 5min — no background refetch
    gcTime: 10 * 60 * 1000,        // Keep in cache 10min after unmount
    refetchOnWindowFocus: true,    // Refetch when user returns to tab
    retry: 3,                       // Retry failed requests 3 times
    retryDelay: (attempt) => Math.min(1000 * 2 ** attempt, 30000), // Exp backoff
  });
}

// useMutation + cache invalidation — the standard write pattern
function UpdateProfile({ userId }) {
  const queryClient = useQueryClient();

  const { mutate: updateUser, isPending } = useMutation({
    mutationFn: (data) => patchUser(userId, data),
    onSuccess() {
      // Invalidate: triggers background refetch of all matching queries
      queryClient.invalidateQueries({ queryKey: ['user', userId] });
    },
    onError(error) {
      toast.error(`Update failed: ${error.message}`);
    },
  });
}

// Dependent queries — runs only when userId is available
const { data: user } = useQuery({ queryKey: ['user'], queryFn: getUser });
const { data: orders } = useQuery({
  queryKey: ['orders', user?.id],
  queryFn: () => getOrders(user.id),
  enabled: !!user?.id,              // Don't run until user is loaded
});

// Deduplication demo — these three hooks fire ONE network request
function Header()   { const { data } = useQuery({ queryKey: ['user'], queryFn: getUser }); }
function Sidebar()  { const { data } = useQuery({ queryKey: ['user'], queryFn: getUser }); }
function Settings() { const { data } = useQuery({ queryKey: ['user'], queryFn: getUser }); }
// All three subscribe to the same Query object in QueryCache
// Only one queryFn call is made for the first mount, result shared to all three
Common mistakes
Setting staleTime: 0 (the default) and expecting no background refetches — default staleTime is 0ms, meaning data is immediately stale after fetching. Every window focus, network reconnect, and mount triggers a refetch. Set staleTime explicitly based on how frequently your data changes.
Including unstable objects in query keys — queryKey: ['users', { filter }] where filter is a new object reference on every render causes React Query to treat it as a new query and refetch constantly. Use primitive values in query keys, or stabilise objects with useMemo.
Calling invalidateQueries with a broad key after a targeted mutation — invalidateQueries(['users']) after updating one user invalidates all user queries (list, detail, infinite). Sometimes correct (list needs to update), sometimes wasteful. Use setQueryData to surgically update the cache without a refetch when you already have the new data from the mutation response.
Interview perspective
"How does React Query handle request deduplication?"
All components using the same query key share one Query object in the QueryCache. The first component to mount starts the fetch. Subsequent components mounting with the same key register as subscribers to the already-in-flight query — they don't start new requests. When the fetch resolves, all subscribers receive the result simultaneously. This is why 10 components calling useQuery with the same key = 1 network request.
"Explain the difference between staleTime and gcTime."
staleTime: how long data is considered fresh — zero refetches within this window. After staleTime, data is stale: still shown immediately (good UX) but a background refetch starts (ensuring eventual freshness). gcTime: how long after the last subscriber unmounts before the cache entry is deleted. During gcTime, if the component remounts, it gets the cached data instantly and only background-refetches if stale. After gcTime, remounting starts fresh with a loading state.

Optimistic UI

Update locally firstRollback on failureIdempotencyRace conditions
Definition

Optimistic UI is the pattern of updating the local UI state immediately on user action — before the server confirms the operation — and rolling back if the server returns an error. It makes interactions feel instantaneous. The user sees the effect of their action (message sent, item liked, todo deleted) without waiting 200-500ms for a server round trip. The tradeoff: you must be able to roll back cleanly if the server rejects the operation, and you must handle the window between the optimistic update and confirmation where the true state is uncertain. This requires idempotent operations and careful handling of concurrent mutations.

How it works with React Query's mutation lifecycle

React Query's useMutation provides three callbacks that form the optimistic update lifecycle: onMutate fires before the mutation network request starts — snapshot the current cache, apply the optimistic update to the cache. onError fires if the request fails — use the snapshot to roll back. onSettled fires after success OR error — always invalidate the query to sync with the server's true state. Between onMutate and onSettled, the UI shows the optimistic state. If the server succeeds, the eventual server response replaces the optimistic data. If the server fails, the rollback restores the snapshot.

Subtopics

Snapshot and rollback

Before the optimistic update: queryClient.getQueryData(key) captures the current cache. If the mutation fails: queryClient.setQueryData(key, snapshot) restores it. The snapshot is what makes rollback possible — without it you can't undo the optimistic change.

Idempotency requirement

Optimistic UI requires the server operation to be idempotent — running it multiple times produces the same result as running it once. If the mutation times out but actually succeeded on the server, a retry must not duplicate data. Use idempotency keys (UUID per request) on the backend.

Race conditions

User fires mutation A (optimistic: item = red). Fires mutation B before A responds (optimistic: item = blue). A responds first (success, item = red). B responds (success, item = blue). Cache is now correct. But if A responds after B: server state = red but cache shows blue from B's optimistic update. Cancel in-flight queries during onMutate to prevent stale refetches overwriting.

Temporary IDs

Adding items optimistically before server assigns an ID: generate a temporary client-side ID (e.g., temp_${uuid()}). Render with temp ID. On server success, replace with the real ID. Components using the temp ID must handle the swap gracefully — key stability matters here.

Error UX

On rollback, show an error notification explaining what failed. The user must know their action didn't take effect. "Failed to send — tap to retry" pattern. Don't silently roll back without feedback — user thinks the action worked.

Zustand optimistic pattern

Without React Query: store current state in a variable, apply optimistic update to Zustand store, on fetch failure restore from variable. Same concept, manual implementation. React Query's lifecycle callbacks formalise this pattern.

Code example
// Optimistic todo deletion with React Query
function useTodoDelete() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (todoId) => deleteTodo(todoId),

    async onMutate(todoId) {
      // 1. Cancel any in-flight refetches (prevent overwriting optimistic update)
      await queryClient.cancelQueries({ queryKey: ['todos'] });

      // 2. Snapshot current state (needed for rollback)
      const previousTodos = queryClient.getQueryData(['todos']);

      // 3. Apply optimistic update — UI responds instantly
      queryClient.setQueryData(['todos'], (old) =>
        old.filter(todo => todo.id !== todoId)
      );

      // 4. Return snapshot for potential rollback
      return { previousTodos };
    },

    onError(err, todoId, context) {
      // Server rejected — restore the snapshot
      queryClient.setQueryData(['todos'], context.previousTodos);
      toast.error('Failed to delete todo. Please try again.');
    },

    onSettled() {
      // Always sync with server (success or error) — source of truth
      queryClient.invalidateQueries({ queryKey: ['todos'] });
    },
  });
}

// Optimistic add with temporary ID
useMutation({
  mutationFn: (text) => createTodo(text),
  async onMutate(text) {
    await queryClient.cancelQueries({ queryKey: ['todos'] });
    const previous = queryClient.getQueryData(['todos']);
    queryClient.setQueryData(['todos'], (old) => [
      ...old,
      { id: `temp_${Date.now()}`, text, done: false } // Temp ID
    ]);
    return { previous };
  },
  onError(err, text, ctx) { queryClient.setQueryData(['todos'], ctx.previous); },
  onSettled() { queryClient.invalidateQueries({ queryKey: ['todos'] }); },
  // onSettled replaces temp ID with real ID via invalidation + refetch
});
Common mistakes
Forgetting to cancel in-flight queries in onMutate — if a background refetch completes after the optimistic update, it overwrites the optimistic state with stale server data. queryClient.cancelQueries() cancels in-flight requests for the affected key.
Assuming optimistic updates always succeed and skipping onError rollback — on poor network conditions or server validation errors, the rollback is exactly what separates a polished app from a data-corrupting one. Never skip the rollback path.
Using optimistic updates for non-idempotent operations without idempotency keys — a payment or order creation that times out (server succeeded, client got no response) followed by a retry creates a duplicate. Always send a UUID idempotency key for mutating operations in financial or critical flows.
Interview perspective
"Walk me through implementing an optimistic delete with React Query."
Four steps: (1) onMutate: cancel in-flight queries, snapshot current cache, apply optimistic deletion, return snapshot. (2) Network request fires. (3) onError: restore snapshot, show error to user. (4) onSettled: invalidate query regardless of outcome to sync with server truth. The snapshot is the key — without it you can't roll back. cancelQueries is critical — without it a background refetch can overwrite your optimistic state.
"What is an idempotency key and why is it needed for optimistic UI?"
A UUID sent with each mutation request that the server uses to deduplicate. If a request times out (client doesn't know if server succeeded), retrying with the same idempotency key causes the server to return the original result rather than executing again. Without it, a timed-out payment mutation retry creates a double charge. Optimistic UI creates retry scenarios — idempotency keys make them safe.

Topic 6 of 6 — Block 5: State Management

Normalised vs denormalised state

Relational dataSingle source of truthN+1 updatesComplexity tradeoff
Definition

State shape is an architectural decision with major implications for update correctness, lookup performance, and code complexity. Normalised state stores each entity once, by ID, in flat lookup tables — like a database. Relationships are represented by IDs, not embedded objects. Denormalised state embeds related data directly — a comment object contains the full author object, not just an author ID. Denormalisation simplifies reads (one object has everything you need) but creates update problems: if the author changes their name, you must find and update every comment that embeds their data. The correct choice depends entirely on data shape and mutation patterns — not a universal rule.

How normalisation works

Normalised state transforms nested API responses into flat entity stores. A response like { post: { id: 1, author: { id: 5, name: "Somesh" }, comments: [{...}] } } is split into: posts: { 1: { id: 1, authorId: 5, commentIds: [10, 11] } }, users: { 5: { id: 5, name: "Somesh" } }, comments: { 10: {...}, 11: {...} }. Now updating the user's name requires one write: users[5].name = "Somesh K". Every component that displays the user's name from users[5] automatically reflects the change. Without normalisation, you'd need to find and update the author object embedded in every post and every comment.

Normalise when

The same entity appears in multiple places (user in posts, comments, reactions). You have mutations that update an entity independently of its parent. You have relational data with many-to-many relationships. You're building a chat app, social feed, or CRM — data is deeply relational.

Don't normalise when

Data is display-only: you receive a list, render it, never update individual items. Data is simple (settings, config, user preferences). The nested structure matches exactly what you display. You're using React Query — it stores data as fetched, normalise on the server instead.

const state = {
  posts: [
    { id: 1, title: 'Post 1', author: { id: 5, name: 'Somesh' }, comments: [

Real-world use case

EngineX-style app — when normalisation pays off

Consider EngineX: resumes, users, job descriptions, and AI analysis results all reference each other. A user appears in their own profile, in resume metadata, in shared-access lists. An analysis result references both a resume ID and a job description ID. Without normalisation: updating a user's name requires finding and patching the user object in every resume's metadata. With normalisation: users[id] is the one source of truth. React Query stores raw API responses (denormalised) — normalisation is valuable here only if you're building an offline-capable app or managing complex relational state client-side. For most React Query-based apps, normalise on the server and let React Query cache per-endpoint.

Common mistakes
Normalising everything by default — normalisation adds lookup indirection (post has authorId, not author.name — you must join). For simple lists that are only displayed, never updated individually, normalisation adds complexity with no benefit. Measure the actual update problem before normalising.
Forgetting to update all related entity references when mutating normalised state — if you update the post title but forget to update the posts.ids array, or update an entity but leave stale IDs in another entity's relationship array, you get ghost references. RTK's entity adapter operations keep ids and entities in sync automatically.
Normalising state when using React Query — React Query caches per query key. If you're fetching /users/5 and /posts?authorId=5 separately, React Query handles the caching. Manual normalisation creates a parallel cache you have to synchronise. Use React Query's queryClient.setQueryData for surgical updates instead of building a normalisation layer on top.
Interview perspective
"When would you normalise client state and when would you not?"
Normalise when: the same entity is referenced in multiple places and can be updated independently (users in posts and comments; order items in both list and detail views). Don't normalise when: data is display-only, comes from one query and is only rendered once, or you're using React Query (normalise at the API/server layer instead). Normalisation trades read complexity for write consistency — worth it only when you have the write consistency problem.
"What is the N+1 update problem and how does normalisation solve it?"
Without normalisation: updating an entity requires finding and patching it everywhere it's embedded — O(N entities × M nesting levels). Miss one: inconsistent UI showing different names/statuses for the same entity. Normalisation stores each entity once by ID. Update = one write to one location. All components reading from the same ID automatically get the updated value. RTK's createEntityAdapter implements this pattern with built-in CRUD operations.
