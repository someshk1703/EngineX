Execution context & scope chain

HoistingLexical scopeTDZV8 internals
Definition

An execution context (EC) is the abstract environment V8 creates every time code is about to run — whether that's the global script, a function call, or an eval(). It holds three things: the Variable Environment (bindings for var declarations and function declarations), the Lexical Environment (bindings for let/const, the outer reference, and this), and a this binding. ECs live on the call stack. When a function is called, a new EC is pushed. When it returns, it's popped. The scope chain is the linked list of Lexical Environments from the current EC outward to the global EC — this is how identifier resolution works: V8 walks the chain upward until it finds the binding or hits the global scope.

How it works internally

V8 runs each EC in two phases. Creation phase: allocates memory for all identifiers in the current scope. var declarations are initialized to undefined (hoisted). Function declarations are fully hoisted (the entire function object is allocated). let/const are placed into the Lexical Environment but remain in the Temporal Dead Zone (TDZ) — any access before their declaration line throws a ReferenceError. Execution phase: code runs line-by-line, assignments write into the bindings. Identifier lookup during execution traverses the scope chain: current Lexical Environment → outer Lexical Environment → … → global. Each function's Lexical Environment has a hidden [[OuterEnv]] slot pointing to the environment where it was defined (not where it's called) — this is what makes lexical scoping work and closures possible.

Subtopics

Global execution context

Created once. this = window (browser) or global (Node). Houses all top-level var/function declarations. Never popped from the call stack.

Function execution context

Created per call (not per function definition). Each call gets its own EC and its own arguments object. Recursive calls = N ECs on the stack simultaneously.

Block scope (ES6)

let/const create a new Lexical Environment per block {}. The block's env's outer reference is the enclosing function/global env. var ignores blocks — scopes to the nearest function.

Temporal Dead Zone

The zone between the start of a block scope and the let/const declaration line. Binding exists (unlike undeclared) but reading it throws. Prevents "use before init" bugs silently allowed by var.

Hoisting

var + function declarations hoisted in creation phase. let/const hoisted to TDZ. Class declarations hoisted to TDZ too — a class is not accessible before its definition line.

Lexical vs dynamic scope

JS uses lexical scope — scope determined by where code is written, not where it's called. this is the only exception (it's dynamically bound unless arrow functions are used).

Code example
// Creation phase demo — hoisting
greet();                        // Works! Function fully hoisted
console.log(x);               // undefined — var hoisted, not value
// console.log(y);             // ReferenceError — let in TDZ

function greet() { return 'hi'; }
var x = 1;
let y = 2;

// Scope chain resolution
const outer = 10;            // Global Lexical Env
function parent() {
  const mid = 20;             // parent's Lexical Env
  function child() {
    const inner = 30;
    // V8 resolves `outer`: child env → parent env → global env ✓
    console.log(inner + mid + outer); // 60
  }
  child();
}

// Block scope — let creates new env per block
let i = 'outer';
{
  let i = 'inner';     // New binding in block's Lexical Env
  console.log(i);     // 'inner'
}
console.log(i);       // 'outer' — block env is gone

// Classic var-in-loop bug (scope chain explains it)
for (var j = 0; j < 3; j++) {
  setTimeout(() => console.log(j), 0); // 3,3,3 — one j in function scope
}
for (let k = 0; k < 3; k++) {
  setTimeout(() => console.log(k), 0); // 0,1,2 — new k per iteration block
}
Real-world use case

Module systems and scope isolation

ES Modules create their own Lexical Environment per module file — top-level bindings are module-scoped, not global. This is why two modules both declaring const x = 1 don't conflict. The module's env has [[OuterEnv]] = null (not global), making globals inaccessible unless explicitly imported. In Vite/Webpack, tree-shaking works by statically analyzing the scope chain of module bindings to determine what's reachable.

Trade-offs
Avoiding var entirely (let/const only) is correct practice — but TDZ errors in complex initialization order can be harder to debug than the silent undefined from var hoisting. The error is still correct; it just requires understanding TDZ.
Deeply nested functions with long scope chains add lookup overhead. V8 optimizes monomorphic access patterns, but excessively deep nesting (10+ levels) is rarely necessary and signals an architecture smell.
Common mistakes
Expecting var inside a block (if, for, while) to be block-scoped — it isn't. var hoists to the nearest enclosing function or global scope. The for-loop-closure bug is the canonical failure case.
Assuming classes are not hoisted — they are, but into TDZ. A class Foo declaration before its line throws a ReferenceError, unlike function declarations which are fully hoisted.
Confusing lexical scope (determined at write-time) with this (determined at call-time). this does NOT follow the scope chain — it has its own binding rules entirely.
Interview perspective
"What is the Temporal Dead Zone?"
The period between scope entry and the let/const declaration line where the binding exists but accessing it throws. Exists because of the creation-phase hoisting but deferred initialization. Distinguish from undeclared (which throws "is not defined").
"Why does the classic for-loop closure print the wrong value with var but not let?"
With var, one binding for j in the function scope — all closures share it. With let, a new Lexical Environment per iteration — each closure captures its own k. Interviewers want you to explain this via scope chain, not just "let fixes it".
"What happens during the creation phase of an execution context?"
Memory allocation for all identifiers. var → undefined. Function declarations → full function object. let/const → TDZ. this binding set. Outer env reference linked. Execution hasn't started yet.

Closures

Scope captureMemoryPractical patterns
Definition

A closure is a function that retains access to its enclosing Lexical Environment even after the outer function has returned and its EC has been popped from the call stack. It is not a special syntax — it's a consequence of lexical scoping. Every function in JS is a closure over the scope in which it was defined. The inner function holds a reference to the [[OuterEnv]] slot of the outer EC, preventing that environment (and its bindings) from being garbage collected as long as the inner function is reachable.

How it works internally

When V8 pops a function's EC from the call stack, it would normally be eligible for GC. But if any inner function was created within that EC, V8 allocated the outer function's Lexical Environment on the heap (not the stack), so it survives the function's return. The inner function's [[OuterEnv]] slot points to this heap-allocated environment object. As long as the inner function is reachable (held by a variable, stored in an array, passed as a callback), the entire chain of heap-allocated environments it references remains reachable and will not be GC'd. V8 performs context variable analysis at parse time — if a variable is never referenced by any inner function, it's not placed into the heap-allocated context object, staying on the stack frame instead. This means closures only "close over" variables that are actually used.

Subtopics

Module pattern

Immediately invoked function that returns a public API while keeping private state in the closure. Foundation of pre-ES6 module systems. Still relevant for plugin APIs and IIFEs.

Partial application / currying

A function that returns a function with some arguments pre-bound via closure. Enables reusable, configurable function factories without classes.

Memoization

Cache results of expensive calls in a closure-captured Map. The cache lives between calls because it's in the outer environment, not reinitialized per call.

Event handler factories

Creating handlers with captured context (e.g., which button was clicked, which user ID applies). Avoids passing extra arguments through DOM event system.

Stale closure (memory + bug risk)

React hooks stale closure: a callback captures a value from render time; by the time it runs, state has updated but the closure still holds the old value. Canonical React bug.

Memory leak via closure

A large object captured in a closure that's held indefinitely (event listener, global variable, Promise chain). V8 keeps the entire captured scope chain alive even if only one variable is needed.

Code example
// Classic closure — counter with private state
function makeCounter() {
  let count = 0;              // Heap-allocated, not stack
  return {
    inc: () => ++count,       // Closes over `count`
    get: () => count,
  };
}
const c = makeCounter();
c.inc(); c.inc();
console.log(c.get()); // 2 — count survived makeCounter's return

// Memoization via closure
function memoize(fn) {
  const cache = new Map();   // Captured in closure, persists between calls
  return (...args) => {
    const key = JSON.stringify(args);
    if (!cache.has(key)) cache.set(key, fn(...args));
    return cache.get(key);
  };
}

// Stale closure bug (React context)
function Component() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const id = setInterval(() => {
      console.log(count); // ❌ Stale! Captures count=0 from first render
    }, 1000);
    return () => clearInterval(id);
  }, []); // Empty deps — closure never refreshed
}
// Fix: add count to deps, or use useRef for the latest value

// ❌ Memory leak — large object unnecessarily captured
function setup() {
  const bigData = new Array(1e6).fill(0);
  const needed = bigData[0];  // Only this is needed
  // If the returned fn closes over `bigData`, the 1M array is kept alive
  return () => console.log(needed); // ✅ Closes over `needed`, not bigData
  // V8's context analysis only captures variables used by inner fns
}
Real-world use case

React hooks are built on closures

Every useState, useEffect, and useCallback captures state and props from the render scope. The hooks model is the closure model. React's stale closure problem (and the dependency array fix) only makes sense through the lens of: "this callback was created at render N and closes over values from that render's scope; if state updates occur, the closure doesn't see them unless recreated." useRef is the escape hatch because refs are mutable heap objects not replaced per render — the closure always sees the same ref, and can read ref.current for the latest value.

Common mistakes
Thinking closures are a "feature" you have to opt into. Every function in JS is a closure. The question is which outer environments it captures and whether that causes a problem.
Stale closures in React useEffect with empty dependency arrays — the most common production React bug. The fix (add deps or use refs) only makes sense if you understand that the effect callback is a closure over the render's scope snapshot.
Assuming V8 captures the entire outer scope — it only captures variables actually referenced by inner functions (context variable analysis). But if any inner function references a large variable, the entire binding survives.
Interview perspective
"Explain what a closure is and how it works in memory."
Define it via lexical environment: the inner function holds a reference to the heap-allocated outer env. Explain GC: as long as the inner function is reachable, the outer env chain won't be collected. Bring up V8 context analysis (only referenced vars captured).
"What is a stale closure? Give a React example."
A closure that captured a value from an older render/call and doesn't see subsequent updates because it was never recreated. Classic case: useEffect with empty deps containing state reads. Solutions: dependency array, useRef, useReducer.
"How can closures cause memory leaks?"
By keeping large objects in captured scope chains alive longer than needed, especially when the closure is stored in a long-lived reference (global, Map, event listener on a long-lived DOM node). Fix: extract only needed values, or null out references explicitly.

Prototypal inheritance

Prototype chainclass desugaringObject.create
Definition

JavaScript's inheritance model is prototypal, not classical. Every object has an internal slot [[Prototype]] (accessible via Object.getPrototypeOf(obj) or the deprecated obj.__proto__) pointing to another object — its prototype. When you access a property on an object, V8 first checks the object's own properties, then walks up the [[Prototype]] chain until it finds the property or hits null (the end). class syntax in ES6+ is pure syntactic sugar over this prototype chain mechanism — no new VM primitives, no true classical inheritance. Understanding what class desugars to is what separates candidates who know JS from candidates who know syntax.

How it works internally

When you write new Foo(), V8 does: (1) creates a plain object, (2) sets its [[Prototype]] to Foo.prototype, (3) calls Foo with this bound to the new object, (4) returns the object (unless the constructor explicitly returns another object). Foo.prototype is a plain object that has a constructor property pointing back to Foo. Methods defined on Foo.prototype are shared across all instances — they're not copied; each instance accesses them by walking the prototype chain. Object.create(proto) creates a new object with its [[Prototype]] set to proto, without calling any constructor. V8's hidden class (Shape) system tracks the prototype chain — objects sharing a shape that has the same prototype benefit from fast property lookup via inline caches without traversing the chain at runtime.

Subtopics

__proto__ vs prototype

__proto__ is an instance's link to its prototype (the [[Prototype]] slot). prototype is a property on constructor functions — it becomes the [[Prototype]] of instances created with new. Conflating these is the #1 source of confusion.

Object.create()

Creates an object with a specified prototype without a constructor call. Clean prototypal inheritance without the new / class machinery. Used in Object.create(null) for truly property-less dictionaries (no toString, no hasOwnProperty).

class desugaring

class Foo extends Bar sets up: Foo.prototype.[[Prototype]] = Bar.prototype and Foo.[[Prototype]] = Bar (for static inheritance). super() in constructor calls Bar.call(this). Methods go on Foo.prototype.

Property shadowing

Setting a property on an instance shadows the prototype's property. Does NOT modify the prototype. Own-property check: obj.hasOwnProperty(key) or Object.hasOwn(obj, key). Needed when iterating.

instanceof

Checks if Foo.prototype appears anywhere in the target's [[Prototype]] chain. Can be fooled: reassigning Foo.prototype after instances are created breaks it.

Prototype pollution

Writing to Object.prototype (e.g., obj['__proto__'] = {admin: true} via JSON merge) contaminates the prototype of all plain objects. A security vulnerability class in Node.js servers.

Code example
// What `class` actually desugars to
class Animal {
  constructor(name) { this.name = name; }
  speak() { return `${this.name} speaks`; }
}
class Dog extends Animal {
  speak() { return `${super.speak()} — woof`; }
}

// Equivalent without class syntax:
function Animal(name) { this.name = name; }
Animal.prototype.speak = function() { return `${this.name} speaks`; };

function Dog(name) { Animal.call(this, name); } // super()
Dog.prototype = Object.create(Animal.prototype); // extends
Dog.prototype.constructor = Dog;             // restore constructor
Dog.prototype.speak = function() {
  return Animal.prototype.speak.call(this) + ' — woof'; // super
};

// Prototype chain of a Dog instance:
// d → Dog.prototype → Animal.prototype → Object.prototype → null

// Object.create(null) — no prototype at all
const dict = Object.create(null);
dict.key = 'value';
// dict.hasOwnProperty → TypeError! No Object.prototype in chain
// Use: Object.hasOwn(dict, 'key') — ES2022 static method

// Property lookup order
const d = new Dog('Rex');
// d.speak(): V8 checks d own props → no speak → Dog.prototype → found ✓
// d.toString(): Dog.prototype → Animal.prototype → Object.prototype → found
Real-world use case

Prototype pollution in JSON merge utilities (security)

A known attack vector: JSON.parse('{"__proto__": {"isAdmin": true}}') when naively merged into another object via Object.assign({}, parsed) can contaminate Object.prototype.isAdmin, affecting all plain objects in the process. This hit lodash (CVE-2019-10744), query-string, and dozens of other npm packages. Defense: Object.create(null) for dictionaries, schema validation before merge, or using structuredClone instead of manual merging.

Common mistakes
Confusing __proto__ (instance property linking to prototype) with prototype (property on constructor functions). A function has prototype. An instance has [[Prototype]] (accessible as __proto__). They're different things.
Mutating Object.prototype or Array.prototype — any property added appears on all objects/arrays via chain lookup. Breaks for...in loops that don't use hasOwnProperty. The prototype pollution attack vector.
Thinking class introduces true privacy or classical inheritance — it doesn't until private class fields (#field) which use a WeakMap internally, genuinely inaccessible from outside.
Interview perspective
"How does prototypal inheritance differ from classical inheritance?"
Classical: classes are blueprints, instances are copies. Prototypal: objects directly inherit from other objects via a live link. No copying — property lookup traverses the chain at runtime. Class syntax in JS is syntactic sugar, not true classical inheritance.
"What does `new` actually do?"
Four steps: create empty object, set its [[Prototype]] to Constructor.prototype, call Constructor with this=newObj, return newObj (unless constructor returns an object explicitly).
"What is prototype pollution?"
Writing to Object.prototype (or Array/Function.prototype) contaminates all objects in the process. Security vulnerability in npm packages that do naive object merging from user input. Defense: Object.create(null), schema validation, structuredClone.

this binding

Dynamic bindingArrow functionsLost context
Definition

this in JavaScript is not determined by where a function is written (unlike scope) but by how it is called — it's a dynamic binding. V8 sets the this slot in the execution context at call time, not at definition time. There are four binding rules that apply in priority order: new binding (constructor call) → explicit binding (call/apply/bind) → implicit binding (method call) → default binding (plain function call). Arrow functions are the exception: they have no own this binding and lexically inherit this from the enclosing scope at definition time — essentially making this behave like a normal closed-over variable for arrow functions.

How it works internally

V8's EC creation sets the this value in the EC's binding object. For regular functions, the value is determined by the call site: if called as obj.method(), this = obj. If called as fn() in strict mode, this = undefined; in sloppy mode, this = globalThis. Function.prototype.call(ctx, ...args) sets this = ctx explicitly and calls immediately. Function.prototype.bind(ctx) returns a new function object that has the this pre-set — this new function ignores any later call/apply attempts to override this (though new can still override it). Arrow functions set [[ThisMode]] to "lexical" — V8 looks up the scope chain for the nearest enclosing regular function's this value, which is captured at definition time into the closure's context object.

Subtopics

Default binding

Plain function call: fn(). In strict mode: this = undefined. In sloppy mode: this = globalThis. The most common source of "undefined is not an object" bugs.

Implicit binding

Method call: obj.method(). this = obj. Lost when the method is extracted: const m = obj.method; m() → default binding applies. Classic event handler bug.

Explicit binding

fn.call(ctx, a, b) — sets this, calls immediately. fn.apply(ctx, [a,b]) — same but args as array. fn.bind(ctx) — returns new function with this locked. Cannot be overridden by call/apply.

new binding

Highest priority (except hard bind edge case). new Foo() creates a new object, sets this to it. Even if Foo was bound via .bind(), new overrides the bound this.

Arrow functions

No own this. Lexically inherits from enclosing scope at definition. Cannot be used as constructors. Cannot be rebound with call/apply/bind (the call still works, this just doesn't change).

Class methods

Regular class methods are on the prototype — this is the instance when called as a method. Class fields with arrow function syntax (onClick = () => {}) create a per-instance bound function — useful for event handlers.

Code example
// Priority: new > explicit > implicit > default
function show() { console.log(this.val); }

const obj = { val: 42, show };
obj.show();                     // 42 — implicit binding (obj.)

const detached = obj.show;
detached();                     // undefined (strict) — default binding

show.call({ val: 99 });        // 99 — explicit beats implicit

const bound = show.bind({ val: 7 });
bound.call({ val: 0 });        // 7 — hard bind ignores call/apply

// Arrow function — lexical this
class Timer {
  constructor() { this.ticks = 0; }

  // ❌ Regular method — loses this when passed as callback
  startBad() {
    setInterval(function() {
      this.ticks++; // this = undefined (strict) or globalThis
    }, 1000);
  }

  // ✅ Arrow function — captures this from constructor scope
  startGood() {
    setInterval(() => {
      this.ticks++; // this = the Timer instance ✓
    }, 1000);
  }
}

// Class field arrow — per-instance bound method
class Button {
  count = 0;
  onClick = () => {              // Own property, not on prototype
    this.count++;               // Always bound to instance
  };
}
// Cost: each Button instance gets its own onClick function object
// (unlike prototype method shared across instances)
Common mistakes
Passing a method as a callback without binding: setTimeout(obj.method, 0) — the method loses its obj binding and runs with default this. Fix: setTimeout(obj.method.bind(obj), 0) or setTimeout(() => obj.method(), 0).
Using arrow functions as object methods: const obj = { greet: () => this.name } — this is lexically bound to whatever scope surrounds the object literal (usually global/undefined), not to obj. Arrow functions should not be object methods.
Thinking bind mutates the original function — it returns a new function object. The original is unchanged. Multiple bind calls each return new functions; the outermost bind's this wins.
Interview perspective
"What are the four rules for this binding in JS? What's the priority?"
new > explicit (call/apply/bind) > implicit (obj.method()) > default (plain call). Arrow functions are exempt — they're lexical. Know all four, know the priority, know that arrow functions break the whole dynamic-binding model intentionally.
"What is the difference between call, apply, and bind?"
call/apply set this and call immediately (call = individual args, apply = array of args). bind returns a new function with this pre-set but doesn't call it. bind's result ignores subsequent call/apply rebinding.

Event delegation

BubblingPerformanceDynamic DOM
Definition

Event delegation is a pattern where a single event listener is placed on a parent element to handle events for all current and future child elements, rather than attaching individual listeners to each child. It works because of event bubbling: after a DOM event fires on a target element, it propagates ("bubbles") up through each ancestor in the DOM tree, firing on each one. The parent's listener receives the event with event.target pointing to the original element that was interacted with, allowing the handler to determine which child triggered it.

How it works internally

Blink's event dispatch algorithm follows the W3C UI Events spec: (1) Capture phase — event travels from document down to the target element (listeners registered with { capture: true } fire here). (2) Target phase — event fires on the target element itself (both capture and bubble listeners fire). (3) Bubble phase — event travels back up from target to document (default listener phase). The event object maintains a reference to the original target throughout all phases, while currentTarget refers to the element whose listener is currently executing. event.stopPropagation() halts bubbling at the current element. event.stopImmediatePropagation() also prevents other listeners on the same element from firing. Not all events bubble: focus/blur don't bubble (use focusin/focusout which do), load doesn't bubble.

Subtopics

event.target vs currentTarget

target = the element that originally triggered the event (may be a deep child). currentTarget = the element whose listener is executing (the delegating parent). currentTarget is null after async use of the event object.

event.stopPropagation

Stops bubbling from current element upward. Dangerous to use liberally — breaks parent delegated handlers silently. Prefer event.target matching in the parent handler instead.

Non-bubbling events

focus, blur, mouseenter, mouseleave, load, scroll, resize do not bubble. Use focusin/focusout or mouseover/mouseout (which do bubble) for delegation on these.

Capture phase listeners

addEventListener('click', fn, { capture: true }) fires before any bubble-phase handler. Useful for intercepting events before child components handle them (framework-level use).

Dynamic element handling

Delegated listener on a static parent automatically handles children added after the listener was attached — a huge advantage for list rendering, virtual scroll, and dynamic content.

Passive listeners

{ passive: true } on scroll/touch listeners tells the browser the handler won't call preventDefault(). Allows compositor to scroll immediately without waiting for JS. Must be used correctly on touch-scroll handlers.

Code example
// ❌ Individual listeners — O(n) listeners, breaks on dynamic adds
document.querySelectorAll('.item').forEach(el => {
  el.addEventListener('click', handleClick);
});
// New .item added later → no listener. N items = N listener objects in memory.

// ✅ Delegation — one listener, handles all children including future ones
const list = document.getElementById('list');
list.addEventListener('click', (event) => {
  const item = event.target.closest('.item'); // Walk up to matching ancestor
  if (!item) return;                           // Click was outside an item
  if (!list.contains(item)) return;           // Safety check
  handleClick(item);
});

    React's synthetic event system

    React (pre-v17) attached a single delegated listener per event type to document. Since v17, it attaches to the root React container element. All component onClick, onChange etc. are handled via this single delegated handler — no individual listeners per component. This is why React has extremely low event listener overhead regardless of component count. The tradeoff: React wraps native events in SyntheticEvent objects (pooled for performance), which become nullified after the handler returns — a source of async event access bugs in older React.

    Common mistakes
    Using event.target directly without .closest() — if the clicked element has child elements (e.g., an icon inside a button), event.target will be the icon, not the button. closest(selector) walks up to find the right ancestor.
    Assuming all events bubble — focus, blur, mouseenter, mouseleave don't. Trying to delegate these to a parent does nothing. Use their bubbling equivalents.
    Calling stopPropagation inside child components while relying on parent delegation — the parent handler never fires. This creates subtle bugs in component libraries where parent components expect to receive all events.
    Interview perspective
    "What is event delegation and why is it performant?"
    One listener on a parent instead of N listeners on N children. O(1) memory for listeners regardless of list size. Handles dynamic adds automatically. Mention bubbling as the mechanism, closest() as the pattern. Bring up React's use of it.
    "What's the difference between event.target and event.currentTarget?"
    target = originating element. currentTarget = the element whose listener is firing. In delegation, currentTarget is the parent; target is the child that was clicked. After async use (inside a Promise), currentTarget becomes null (React's SyntheticEvent pooling caveat).

    Promises & microtask queue

    Resolution orderCombinatorsMicrotask starvation
    Definition

    A Promise is an object representing the eventual completion or failure of an asynchronous operation, with three states: pending, fulfilled, or rejected. Once settled (fulfilled or rejected) it is immutable. Promises don't execute anything asynchronously by themselves — they are a queuing mechanism. When a Promise settles, its .then()/.catch() callbacks are enqueued as microtasks — the highest-priority task type in the event loop, drained completely between every macrotask. Understanding Promise combinators (all, allSettled, race, any) and their failure semantics is a core senior interview topic.

    How it works internally

    A Promise object internally holds a state machine: [[PromiseState]] (pending/fulfilled/rejected) and [[PromiseResult]]. It also holds a list of reaction records — objects that pair a fulfillment handler with a rejection handler and a promise to resolve/reject when the handler runs. When resolve(value) is called, V8 transitions the state, then for each reaction record, enqueues a PromiseReactionJob microtask. The microtask calls the appropriate handler function and resolves the chained promise with its return value. This is why .then(fn) always returns a new Promise — each .then() creates a new reaction record and a new chained Promise. Promise.resolve() with a thenable (object with a .then method) schedules two additional microtasks during the resolution process — a nuance that causes Promise.resolve(nativePromise) to be slower than returning a value directly.

    Subtopics

    Promise.all(iterable)

    Resolves when ALL promises resolve; rejects as soon as ANY rejects (fail-fast). Returns array of results in input order. Use for parallel independent operations where all must succeed.

    Promise.allSettled(iterable)

    Waits for ALL promises to settle (resolve or reject). Returns array of {status, value/reason} objects. Never rejects. Use when you need all results regardless of failures.

    Promise.race(iterable)

    Settles (resolves OR rejects) as soon as the first promise settles — regardless of outcome. Use for timeouts: race your fetch against a timeout promise.

    Promise.any(iterable)

    Resolves when the FIRST promise fulfills; rejects only if ALL reject (AggregateError). Use for redundant requests where you want the fastest success.

    Unhandled rejection

    A rejected promise with no .catch() fires unhandledrejection event on window. In Node.js, crashes the process since v15. Always add .catch() or use try/catch in async functions.

    Thenable resolution

    If you resolve a Promise with a thenable (obj with .then method), the resolution is async and takes 2 extra microtask ticks. Resolving with a plain value is synchronous within the reaction job.

    Code example
    // Microtask queue ordering
Promise.resolve('a')
  .then(v => { console.log(v); return 'b'; }) // microtask 1 → 'a'
  .then(v => console.log(v));                 // microtask 2 → 'b'
console.log('sync');
// Output: 'sync', 'a', 'b'

// Combinators — when to use which
const [user, orders, prefs] = await Promise.all([
  fetchUser(id), fetchOrders(id), fetchPrefs(id)
]); // All must succeed, all run in parallel

const results = await Promise.allSettled([
  fetchPrimary(), fetchSecondary()
]);
// [{status:'fulfilled',value:...}, {status:'rejected',reason:...}]
const succeeded = results.filter(r => r.status === 'fulfilled');

const data = await Promise.race([
  fetchData(),
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error('timeout')), 5000))
]); // First to settle wins — timeout pattern

const fastest = await Promise.any([
  fetchFromRegion('us'), fetchFromRegion('eu'), fetchFromRegion('ap')
]); // Fastest success, redundant requests for latency reduction

// Promise chaining returns new Promise each time
const p = Promise.resolve(1);
const p2 = p.then(x => x + 1); // New promise, different object
const p3 = p.then(x => x * 2); // Another branch from p (not from p2)
// p2 resolves to 2, p3 resolves to 2 — both branch from original p
    Common mistakes
    Forgetting that Promise.all fails fast — if one of 10 requests fails, the other 9's results are discarded. If you need partial results, use allSettled.
    Promise.race resolves OR rejects on first settlement — a rejected first promise rejects the race. Use Promise.any if you only care about the first success.
    Not handling unhandled rejections — in Node.js v15+, an unhandled rejection crashes the process. Always attach .catch() or wrap in try/catch. Global process.on('unhandledRejection') is for logging, not a substitute for handling.
    Creating a Promise executor with async function — errors thrown inside an async executor are swallowed because the Promise constructor only catches synchronous throws. Use new Promise((res, rej) => { async fn().then(res).catch(rej) }) pattern.
    Interview perspective
    "What is the difference between Promise.all, allSettled, race, and any?"
    all = all must succeed (fail-fast). allSettled = all settle, gives you each outcome. race = first to settle (success OR failure) wins. any = first to fulfill wins (only rejects if ALL reject). Know exact failure behavior for each.
    "Why do Promise callbacks run before setTimeout(fn, 0)?"
    Promise callbacks are microtasks. setTimeout is a macrotask. After each macrotask (or after synchronous script), the entire microtask queue is drained before the next macrotask starts. So a resolved Promise's .then fires before any setTimeout, even with delay=0.

    Async/await

    Generator desugaringError handlingSequential vs parallel
    Definition

    async/await is syntactic sugar over Promises and the generator protocol. An async function always returns a Promise. await expr suspends the async function's execution, schedules the continuation as a microtask when expr's Promise resolves, and yields control back to the event loop. The critical insight: await only suspends the async function, not the entire thread or event loop. Synchronous code elsewhere continues to run. The async function's frame is preserved on the heap (not the call stack) while it's suspended — the same mechanism as closures keeping environments alive.

    How it works internally

    V8 transforms async functions during compilation using a technique equivalent to state-machine generators. The body is split at each await into continuation segments. Each segment is wrapped as a .then() callback on the awaited Promise. When the Promise resolves, the next segment is scheduled as a microtask. Specifically: await x is lowered to roughly Promise.resolve(x).then(continuation). V8 performs an optimization — if x is already a native Promise (not a thenable wrapper), it can avoid two extra microtask ticks that would otherwise be needed for thenable resolution. This is why returning value directly in an async function is marginally faster than returning Promise.resolve(value). The async function's execution context (including all local variables) is captured in a "resumable" state object on the heap when suspended at an await.

    Subtopics

    Sequential await

    Awaiting promises one after another: each starts only after the previous resolves. Total time = sum of all durations. Correct when each operation depends on the previous result.

    Parallel await

    Start all promises before awaiting: const [a,b] = await Promise.all([fetchA(), fetchB()]). Total time = max of all durations. Use when operations are independent.

    Error handling patterns

    try/catch wraps the await. Uncaught errors in async functions reject the returned Promise. Can also use .catch() on the returned Promise from an async function call.

    Top-level await

    ES2022: await at module top level without async function. Module evaluation is async. Importing modules stall until the module's top-level await resolves. Can create import deadlocks.

    Async iteration

    for await (const item of asyncIterable). Processes each value from an async iterable (e.g., ReadableStream, paginated API). Awaits each item sequentially.

    Concurrency patterns

    Run N async tasks with max concurrency (e.g., process 100 items 5 at a time): chunk the array and await Promise.all per chunk, or use a semaphore/queue pattern.

    Code example
    // ❌ Sequential when parallel is possible — slow
async function loadDashboard(userId) {
  const user   = await fetchUser(userId);   // 300ms
  const orders = await fetchOrders(userId); // 200ms — waits for user!
  const prefs  = await fetchPrefs(userId);  // 150ms — waits for orders!
  // Total: ~650ms sequential
}

// ✅ Parallel — fire all, await completion
async function loadDashboard(userId) {
  const [user, orders, prefs] = await Promise.all([
    fetchUser(userId),
    fetchOrders(userId),
    fetchPrefs(userId),
  ]);
  // Total: ~300ms (max of the three)
}

// Error handling — try/catch in async
async function safeLoad(id) {
  try {
    const data = await fetchUser(id);
    return { data, error: null };
  } catch (err) {
    return { data: null, error: err.message };
  }
}

// Concurrency limiter — process 100 items, 5 at a time
async function processConcurrently(items, limit = 5) {
  const results = [];
  for (let i = 0; i < items.length; i += limit) {
    const chunk = items.slice(i, i + limit);
    const chunkResults = await Promise.all(chunk.map(processItem));
    results.push(...chunkResults);
  }
  return results;
}

// Async iterator — paginated API
async function* fetchAllPages(url) {
  let next = url;
  while (next) {
    const { data, nextPage } = await fetch(next).then(r => r.json());
    yield* data;
    next = nextPage;
  }

for await (const item of fetchAllPages('/api/items')) {

}
    Common mistakes
    Awaiting in a loop with forEach — arr.forEach(async item => await fn(item)) doesn't work. forEach ignores the returned Promise. Use for...of with await (sequential) or Promise.all(arr.map(...)) (parallel).
    Not knowing that await before the first await in an async function is synchronous — all code up to the first await runs synchronously in the current microtask. An async function is not deferred just because it's async.
    Using top-level await in a module imported by many others — if the awaited operation is slow, all importers stall. It's appropriate for one-time initialization but dangerous in shared utilities.
    Interview perspective
    "How does async/await work under the hood?"
    State machine compiled from generator pattern. Each await point splits the function into a continuation registered as a .then() microtask on the awaited Promise. Function frame lives on heap while suspended. async function always returns a Promise. This is the senior-level answer — don't just say "syntactic sugar".
    "What's the difference between sequential and parallel await patterns?"
    Sequential: each await waits for the previous. Total time = sum. Use when results depend on each other. Parallel: start all promises before any await, use Promise.all. Total time = max. The most common interview-detected mistake in production code.

    Web Workers

    DedicatedSharedService WorkerOff-main-thread
    Definition

    Web Workers are a browser API for running JavaScript on a separate OS thread, outside the main thread. They provide true parallelism (not just concurrency like the event loop) because they run on a different thread within the same Renderer Process. There are three types: Dedicated Workers (one-to-one with a page), Shared Workers (shared across multiple tabs of the same origin), and Service Workers (a network proxy layer, not a compute worker). Workers have their own event loop, their own microtask queue, and their own V8 isolate — but no access to the DOM.

    How it works internally

    When new Worker(url) is called, the browser creates a new OS thread within the current Renderer Process, starts a fresh V8 isolate on it, and loads the worker script. Communication between the main thread and the Worker happens via postMessage — messages are structured-clone serialized (deep copy of the data, O(n) cost) and transferred as macrotasks to the receiving thread's event loop. To avoid serialization for large binary data, Transferable objects (ArrayBuffer, MessagePort, ImageBitmap, OffscreenCanvas) can be transferred — ownership moves from sender to receiver in O(1), and the sender's reference is neutered. SharedArrayBuffer enables zero-copy shared memory between the main thread and Worker, with Atomics for synchronization. SAB requires COOP/COEP security headers to be enabled.

    Subtopics

    Dedicated Worker

    Created by one page, terminates when the page closes. One-to-one relationship. Communicates via onmessage/postMessage. Ideal for heavy per-page computation (image processing, wasm, crypto).

    Shared Worker

    Shared across all pages of the same origin. Lives as long as at least one connecting page is open. Communicates via MessagePort. Useful for shared state/cache across tabs without a server.

    Service Worker

    Network proxy, not a compute worker. Intercepts fetch events. Has its own lifecycle (install/activate/fetch). Persists independently of any tab. Used for offline caching, background sync, push notifications.

    What Workers CANNOT do

    No DOM access (document, window, Element). No localStorage/sessionStorage (can use IndexedDB). No alert/confirm/prompt. No synchronous XHR in dedicated workers. No direct canvas 2D without OffscreenCanvas.

    Transferable objects

    ArrayBuffer, MessagePort, ImageBitmap, OffscreenCanvas, ReadableStream/WritableStream. Transfer with postMessage(data, [transferList]). O(1) ownership transfer — sender's reference neutered immediately.

    SharedArrayBuffer + Atomics

    Zero-copy shared memory. Requires COOP/COEP headers. Atomics.wait() blocks a Worker thread (not main). Atomics.notify() wakes workers. Used in Wasm-compiled multithreaded C++ (Figma, AutoCAD).

    Code example
    // main.js — offload heavy computation
const worker = new Worker('/workers/compute.js');

// Transfer ArrayBuffer — zero copy, sender's ref neutered
const buffer = new ArrayBuffer(16 * 1024 * 1024); // 16MB
worker.postMessage({ buffer }, [buffer]);         // Transfer, not clone
console.log(buffer.byteLength); // 0 — neutered

worker.onmessage = ({ data }) => applyResult(data);
worker.onerror = (err) => console.error(err);

// compute.js (worker context)
self.onmessage = ({ data: { buffer } }) => {
  const arr = new Float32Array(buffer);
  for (let i = 0; i < arr.length; i++) arr[i] = Math.sqrt(arr[i]);
  self.postMessage({ result: arr }, [arr.buffer]); // Transfer back
};

// SharedArrayBuffer — zero-copy shared state
// Requires: Cross-Origin-Opener-Policy: same-origin
//           Cross-Origin-Embedder-Policy: require-corp
const sab = new SharedArrayBuffer(4);
const view = new Int32Array(sab);
worker.postMessage({ sab }); // SAB can be cloned — both sides read same memory

// In worker: Atomics for safe shared access
Atomics.add(view, 0, 1);             // Atomic increment
Atomics.wait(view, 0, 0);             // Block until view[0] !== 0
Atomics.notify(view, 0, 1);           // Wake one waiting worker
    Real-world use case

    Figma's render architecture

    Figma runs a C++ rendering engine compiled to WebAssembly inside a Dedicated Worker. The main thread handles UI events and React rendering. The Worker runs the geometry engine in Wasm, writes output into a SharedArrayBuffer (the frame buffer). The main thread reads from the SAB in a rAF callback and draws to a canvas. Zero serialization overhead for the frame data — the GPU uploads from shared memory directly. Atomics.notify() signals the main thread that a new frame is ready. This is the reference architecture for computation-heavy web apps that approach native performance.

    Common mistakes
    Trying to access document, window, or any DOM API inside a Worker — throws immediately. Workers have self (the WorkerGlobalScope), not window.
    Using structured-clone postMessage for large data repeatedly — the O(n) serialization cost exceeds the computation benefit for large payloads. Use Transferable objects or SharedArrayBuffer.
    Forgetting that Service Workers have their own lifecycle and may not be active on first page load — the first visit is always served from the network. SW activates on subsequent visits (or after skipWaiting() + clients.claim()).
    Interview perspective
    "What's the difference between a Dedicated Worker, Shared Worker, and Service Worker?"
    Dedicated: one page, compute. Shared: shared across tabs of same origin, MessagePort communication, shared state. Service: network proxy, fetch interception, offline caching, independent of tabs. Don't conflate SW with compute workers.
    "How do you share large data between a Worker and the main thread without copying?"
    Transferable objects (ArrayBuffer) — zero copy, sender neutered. SharedArrayBuffer — both sides access same memory, Atomics for sync. Requires COOP/COEP headers. This is a senior-signal answer.

    WeakMap & WeakRef

    GC implicationsMemoryCaching patterns
    Definition

    WeakMap is a Map-like collection where keys must be objects (not primitives), and the key-value pair does not prevent the key object from being garbage collected. When the key object has no other references and becomes eligible for GC, the entry is automatically removed from the WeakMap. WeakRef (ES2021) allows holding a reference to an object that also does not prevent it from being GC'd — you access the object via .deref(), which returns undefined if it has been collected. FinalizationRegistry pairs with WeakRef to run a cleanup callback after an object is collected. These APIs exist specifically for memory-conscious patterns: caches, private data storage, and metadata association without ownership semantics.

    How it works internally

    In V8, WeakMap entries use ephemeron semantics — a key-value pair is kept alive only if the key itself is reachable from other strong references. V8's GC maintains a set of "ephemeron tables" and processes them in a fixed-point loop during marking: an entry's value is marked reachable only if its key is also marked reachable. This is a special GC algorithm because the liveness of the value depends on the liveness of the key — a non-trivial dependency to track during marking. WeakRef objects go through a two-phase GC: in the first pass, the referent is tentatively collected; the FinalizationRegistry callbacks are scheduled as microtasks in a later GC cycle to avoid re-entrancy issues. WeakRef.deref() returns undefined once the referent has been collected, but the exact timing of collection is non-deterministic and implementation-dependent.

    Subtopics

    WeakMap vs Map

    Map: any key type, strong references (prevents GC), iterable. WeakMap: object keys only, weak references (allows GC), NOT iterable (GC non-determinism makes enumeration unsafe).

    WeakSet

    Like WeakMap but only stores keys (no values). Useful for tracking object membership ("has this element been visited?") without preventing GC of the elements.

    Private data pattern

    Pre-private class fields (#), WeakMap was the idiomatic way to store private data per instance: const _private = new WeakMap(). When the instance is GC'd, its private data is also GC'd automatically.

    WeakRef caching

    Cache values with WeakRef: if memory pressure causes GC, cached values are collected. Next access misses the cache and recomputes. A self-healing cache that doesn't leak memory.

    FinalizationRegistry

    Register a cleanup callback to run after an object is GC'd. Takes a held value (cannot be the object itself). Used for releasing external resources (file handles, GPU textures) when the JS object is collected.

    Non-determinism caution

    GC timing is engine-defined. Never write correctness-critical logic that depends on WeakRef deref returning undefined at a specific time. WeakRef is for optimization/cleanup, not for lifecycle control.

    Code example
    // WeakMap — private instance data (pre-# syntax pattern)
const _private = new WeakMap();
class SecureStore {
  constructor(secret) {
    _private.set(this, { secret });  // Not on the instance, not enumerable
  }
  getSecret() { return _private.get(this).secret; }
}
// When instance is GC'd, the WeakMap entry is automatically removed
// No cleanup needed — prevents leak unlike a regular Map

// DOM metadata without leaking (attaching data to elements)
const elementMeta = new WeakMap();
function attachMeta(el, data) { elementMeta.set(el, data); }
function getMeta(el) { return elementMeta.get(el); }
// When el is removed from DOM and all refs dropped → GC'd → map entry gone
// Equivalent to el.dataset but without polluting the DOM attribute namespace

// WeakRef cache — memory-safe memoization
class WeakCache {
  #cache = new Map();
  #registry = new FinalizationRegistry(key => {
    this.#cache.delete(key); // Clean up Map entry when value is GC'd
  });

  set(key, value) {
    this.#cache.set(key, new WeakRef(value));
    this.#registry.register(value, key); // cleanup on GC
  }
  get(key) {
    return this.#cache.get(key)?.deref(); // undefined if GC'd
  }
}
// If memory pressure → GC collects values → cache misses → recomputes
// No unbounded growth. FinalizationRegistry cleans the Map keys too.
    Common mistakes
    Using Map when WeakMap is appropriate for DOM-keyed metadata — the Map holds strong references to DOM elements, preventing them from being GC'd even after removal from the document. Classic memory leak source in charting/animation libraries.
    Using WeakRef for correctness logic — if (weakRef.deref()) doSomething() is a race condition with the GC. Between the deref() check and the use, GC could collect the object. Store the deref result in a local variable: const obj = ref.deref(); if (obj) use(obj).
    Trying to iterate a WeakMap — it's not iterable by design (non-deterministic GC would make iteration results unpredictable). If you need iteration, use a Map and manage lifecycle yourself.
    Interview perspective
    "Why would you use a WeakMap over a Map?"
    When the keys are objects and you don't want the Map to prevent GC of those objects. Use cases: DOM element metadata, private instance data, object-keyed caches. The WeakMap entry automatically disappears when the key is GC'd — no manual cleanup needed.
    "What is WeakRef and when is it appropriate?"
    A non-owning reference that allows the referent to be GC'd. Appropriate for caches where you want entries to be evictable under memory pressure. Not appropriate for correctness-critical logic because deref() timing is non-deterministic. Always store deref() result in a variable before using it.

    Symbols & iterators

    Iteration protocolGeneratorsLazy sequencesWell-known Symbols
    Definition

    A Symbol is a primitive value that is guaranteed to be unique and immutable. Symbol('desc') creates a value that is never equal to any other Symbol, even one with the same description. Symbols are primarily used as property keys that are non-enumerable, non-collision property names — they don't appear in for...in, Object.keys(), or JSON.stringify(). Well-known Symbols (Symbol.iterator, Symbol.toPrimitive, Symbol.hasInstance, etc.) are hooks that let objects customize built-in JS protocol behaviors. The iteration protocol defines how an object exposes its elements to for...of, spread, and destructuring — by implementing a method at [Symbol.iterator] that returns an iterator object.

    How it works internally

    V8 implements for...of by calling [Symbol.iterator]() on the target to get an iterator, then repeatedly calling iterator.next() until the result's done property is true. An object is iterable if it has [Symbol.iterator]. An object is an iterator if it has .next() returning { value, done }. Built-in iterables: Array, String, Map, Set, arguments, NodeList, HTMLCollection, TypedArray. A generator function (function*) returns a generator object that is both iterable (has [Symbol.iterator] returning itself) and an iterator (has .next()). Generators implement lazy evaluation — they only compute the next value when .next() is called, making infinite sequences possible without memory issues. Internally, V8 compiles generator functions into state machines where each yield suspends execution and returns a value, preserving the function's stack frame on the heap between calls.

    Subtopics

    Symbol.iterator

    The hook for the iteration protocol. Implement on any object to make it work with for...of, spread [...obj], destructuring, Promise.all, Array.from. The iterator factory pattern.

    Symbol.asyncIterator

    For async iteration. Implement on objects to work with for await...of. Returned iterator's .next() returns a Promise resolving to {value, done}. Used for streams, paginated APIs.

    Well-known Symbols

    Symbol.toPrimitive: customize coercion (+ operator, template literals). Symbol.hasInstance: customize instanceof. Symbol.species: what constructor to use for derived objects. Symbol.toStringTag: customize Object.prototype.toString output.

    Generator functions

    function*: returns generator object. yield pauses and emits a value. next(value) resumes and passes value back into the generator (as the result of the yield expression). yield* delegates to another iterable.

    Lazy sequences

    Generators compute values on demand. Infinite ranges, lazy filter/map pipelines — no array allocated for intermediate results. Compose generators like Unix pipes: source → transform → sink.

    Symbol.for / Symbol.keyFor

    Symbol.for('key') returns the same Symbol across realms (iframes, workers) from a global registry. Unlike Symbol('key') which creates a unique symbol each time.

    Code example
    // Custom iterable — range object
const range = {
  from: 1, to: 5,
  [Symbol.iterator]() {
    let current = this.from;
    const last = this.to;
    return {
      next() {
        return current <= last
          ? { value: current++, done: false }
          : { value: undefined, done: true };
      }
    };
  }
};
console.log([...range]); // [1,2,3,4,5] — spread uses Symbol.iterator

// Generator — lazy infinite sequence
function* naturals(start = 0) {
  while (true) yield start++; // Infinite — no array created
}

function* take(n, iterable) {
  let i = 0;
  for (const x of iterable) {
    if (i++ >= n) return;
    yield x;
  }
}

// Lazy pipeline — no intermediate arrays
function* map(fn, iter) { for (const x of iter) yield fn(x); }
function* filter(pred, iter) { for (const x of iter) if (pred(x)) yield x; }

const result = [
  ...take(5,
    map(x => x ** 2,
      filter(x => x % 2 === 0,
        naturals()
      )
    )
  )
]; // [0, 4, 16, 36, 64] — first 5 squares of even numbers
// Only generates as many values as needed — fully lazy

// Well-known Symbol usage
class Money {
  constructor(amount, currency) { this.amount = amount; this.currency = currency; }
  [Symbol.toPrimitive](hint) {
    if (hint === 'number') return this.amount;
    if (hint === 'string') return `${this.amount} ${this.currency}`;
    return this.amount;
  }
}

console.log(+price);         // 100 — numeric hint
console.log(`${price}`);    // '100 USD' — string hint
console.log(price > 50);    // true — default → number coercion
    Real-world use case

    Node.js streams as async iterables

    Node.js Readable streams implement Symbol.asyncIterator, making them directly consumable with for await...of. This is how you process large files line-by-line without buffering the entire file: const rl = createInterface({ input: fs.createReadStream('huge.txt') }); for await (const line of rl) processLine(line). The generator/iterator protocol enables true streaming — each iteration waits for the next chunk from the OS, processes it, and discards it. Memory usage stays constant regardless of file size. The TC39 iterator helpers proposal (Stage 3) adds lazy .map(), .filter(), .take() directly on iterators — bringing native lazy sequences to JS without generator composition boilerplate.

    Common mistakes
    Spreading an infinite generator without take() — [...naturals()] is an infinite loop and OOM. Always apply a terminating operation before materializing to an array.
    Confusing Symbol('desc') (always unique) with Symbol.for('desc') (global registry, same across realms). Symbol('x') === Symbol('x') is false. Symbol.for('x') === Symbol.for('x') is true.
    Not implementing the return() method on custom iterators — when a for...of loop breaks early, it calls iterator.return() for cleanup. Without it, resources held by the iterator (file handles, connections) may not be released.
    Interview perspective
    "What is the iteration protocol in JavaScript?"
    An object is iterable if it has [Symbol.iterator]() returning an iterator (object with .next() returning {value, done}). for...of, spread, destructuring all use this protocol. Generator functions implement both automatically. Custom iterables let you define how for...of traverses your data structure.
    "How do generators enable lazy evaluation?"
    Generators compute values on demand — yield suspends the function and returns a value. The consumer calls .next() to pull the next value. Infinite sequences are possible because no values are pre-computed. Compose generators like a pipeline for memory-efficient data transformation.
    "What are well-known Symbols and give an example of customizing one?"
    Built-in protocol hooks. Symbol.iterator for for...of. Symbol.toPrimitive for coercion (+, template literals). Symbol.hasInstance for instanceof. Demonstrate with a custom class implementing [Symbol.toPrimitive] to control how it behaves in arithmetic and string contexts.
