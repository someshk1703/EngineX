Cross-Site Scripting (XSS)

ReflectedStoredDOM-basedCSPDOMPurify
Definition

XSS is an attack where an adversary injects malicious JavaScript into a page viewed by other users. Once the script runs in the victim's browser, it operates with full origin privileges: it can read cookies, localStorage, DOM content, make authenticated API requests, and exfiltrate data to attacker-controlled servers. XSS is the most prevalent web vulnerability precisely because it is caused by a single class of mistake — trusting user-controlled data and interpolating it into HTML or JS without sanitisation. There are three mechanically distinct variants, each requiring a different remediation.

The three variants

Reflected XSS

Malicious payload is in the URL (query param, path). Server reflects it directly into the response HTML without sanitisation. Victim must click a crafted link. The payload is not stored — it only executes when the URL is visited. Example: https://site.com/search?q=<script>stealCookies()</script> rendered into the page as-is.

Stored XSS (persistent)

Payload is stored in the database (comment, profile bio, message). Every user who views the content executes the script. More dangerous than reflected — no crafted link needed, and it persists until the record is deleted. Example: a comment field that stores <img src=x onerror="stealCookies()"> and renders it to all page visitors.

DOM-based XSS

No server involvement — vulnerable JavaScript on the client reads attacker-controlled data (URL hash, postMessage, localStorage) and writes it unsafely to the DOM. The server response is clean; the exploit is entirely client-side. Example: document.getElementById('output').innerHTML = location.hash.slice(1) with URL #<img onerror=alert(1) src=x>. Most static site generators have this vulnerability class.

Dangerous sinks (DOM-based)

innerHTML / outerHTML

Parses the string as HTML, executing any event handlers and script content. The most common DOM XSS sink. Use textContent instead when you need to display text. Only use innerHTML with trusted, sanitised content.

document.write

Writes raw HTML into the document stream. Deprecated and dangerous — inserts unsanitised content directly into the DOM parser. Never use in modern code.

eval / Function()

Executes strings as JavaScript. If any part of the string is attacker-controlled, arbitrary code executes. Includes: setTimeout('string'), setInterval('string'), template literal tag functions that eval.

src / href / action attributes

javascript: protocol in href/src executes JS: <a href="javascript:stealData()">. Sanitise URL attributes to only allow http/https. React escapes this but only when using JSX attribute syntax — dangerouslySetInnerHTML bypasses all React escaping.

Defences

Output encoding (primary defence)

HTML-encode all user-controlled data before inserting into HTML context. < → &lt;, > → &gt;, " → &quot;. The encoding context matters: HTML body, HTML attribute, JavaScript string, CSS, and URL each have different encoding requirements. React JSX encodes by default — {userInput} is always text, never HTML.

DOMPurify (sanitisation)

When you must render HTML (rich text editors, markdown output): sanitise with DOMPurify before insertion. DOMPurify.sanitize(dirtyHTML) parses the HTML in a sandboxed context, strips script tags, event handlers, and dangerous attributes (javascript: hrefs), and returns clean HTML. Use with innerHTML only after DOMPurify.

Content Security Policy (CSP)

Defence-in-depth: even if XSS payload is injected, CSP prevents inline script execution and restricts where scripts can load from. script-src 'nonce-{random}' blocks all scripts without the correct server-generated nonce, making most XSS payloads inert even after injection.

HttpOnly cookies

Mark session cookies as HttpOnly — JavaScript cannot read them via document.cookie. XSS cannot steal session tokens stored in HttpOnly cookies. This doesn't prevent XSS from making authenticated API requests (the cookie is still sent with fetch), but it eliminates the most valuable exfiltration target.

Code example
// ❌ DOM-based XSS sink
const name = new URLSearchParams(location.search).get('name');
document.getElementById('greeting').innerHTML = `Hello, ${name}`;
// URL: ?name=<img src=x onerror="fetch('/steal?c='+document.cookie)">
// Attacker exfiltrates all non-HttpOnly cookies

// ✅ textContent — never interprets as HTML
document.getElementById('greeting').textContent = `Hello, ${name}`;

// ✅ DOMPurify — when innerHTML is genuinely needed (rich text)
import DOMPurify from 'dompurify';

function RichTextDisplay({ htmlContent }) {
  const clean = DOMPurify.sanitize(htmlContent, {
    ALLOWED_TAGS: ['p', 'b', 'i', 'ul', 'li', 'a'],
    ALLOWED_ATTR: ['href'],                  // Only href on <a>
    ALLOWED_URI_REGEXP: /^https?:\/\//i,       // No javascript: hrefs
  });
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
  // dangerouslySetInnerHTML is the only legitimate use case
}

// React JSX encoding — safe by default
const payload = '<script>alert(1)</script>';
return <div>{payload}</div>;
// Renders as text: "<script>alert(1)</script>" — NOT executed
// JSX compiles to React.createElement which uses textContent, not innerHTML

// ❌ Bypasses React encoding:
return <div dangerouslySetInnerHTML={{ __html: payload }} />; // XSS!
Common mistakes
Trusting that React prevents all XSS — React escapes JSX expressions, but dangerouslySetInnerHTML, eval(), dynamic href with user input, and document.write from JS code all bypass React's protection entirely.
Sanitising on input (before storage) instead of on output (before rendering) — sanitisation should happen at the point of rendering, not storage. Sanitisation rules change; stored-clean content may be rendered in a new context with different rules. Store raw, sanitise on output.
Using a blocklist instead of allowlist for HTML sanitisation — trying to block script tags is trivially bypassed. Allowlist: permit only known-safe tags and attributes. DOMPurify uses an allowlist by default.
Interview perspective
"What is the difference between stored, reflected, and DOM-based XSS?"
Stored: payload persists in database, executes for every viewer. Reflected: payload in URL, reflected in server response, requires victim to click a crafted link. DOM-based: entirely client-side — vulnerable JS reads attacker-controlled data (URL hash, postMessage) and writes to a dangerous sink (innerHTML, eval). Server response is clean in DOM-based XSS — it cannot be caught by output encoding on the server.
"How does React prevent XSS and what bypasses its protection?"
React JSX expressions ({value}) always produce text nodes via textContent — the value is never parsed as HTML. This prevents reflected and stored XSS in components that use JSX correctly. Bypasses: dangerouslySetInnerHTML (explicit bypass), dynamic href with javascript: URLs, eval/new Function with user input, DOM manipulation via refs that write innerHTML, or third-party libraries that use innerHTML directly.

CSRF

Cross-site forgerySameSiteCSRF tokenlocalStorage risk
Definition

Cross-Site Request Forgery (CSRF) tricks an authenticated user's browser into making an unintended request to a target site on their behalf. The attack exploits the browser's automatic inclusion of cookies with every request to a domain — if the user is logged into bank.com, a malicious script on evil.com can trigger a state-changing request to bank.com using the victim's session cookie, without the victim knowing. The attacker doesn't need to read the response — they just need the side effect (transfer money, change email, post a message). CSRF attacks are fundamentally distinct from XSS: XSS executes scripts in the victim's browser; CSRF abuses the browser's cookie inclusion behaviour.

How the attack works
User logged into bank.com
→
Visits evil.com
→
evil.com JS fires: fetch('bank.com/transfer', {method:'POST', ...})
→
Browser auto-sends bank.com session cookie
→
Transfer executes as the victim
Defences

SameSite cookie attribute

The primary modern defence. SameSite=Strict: cookie never sent on cross-site requests. SameSite=Lax: cookie sent on top-level navigation GET, but NOT on cross-origin subresource POST/fetch. SameSite=None; Secure: always sent cross-site (required for third-party auth). Lax is the default in modern browsers — this has largely eliminated CSRF for most sites.

CSRF token (synchronizer token pattern)

Server generates a unique, secret, unpredictable token per session. Embeds it in every form/AJAX request as a hidden field or custom header (X-CSRF-Token). On state-changing requests, server validates the token. Attacker cannot read the token from a cross-origin page (SOP blocks it).

Double Submit Cookie

Server sets a random token in a cookie (not HttpOnly). JS reads the cookie and sends it as a custom header. Server compares header value with cookie value. Attacker cannot set the custom header from evil.com (CORS blocks custom headers on cross-origin requests).

Origin / Referer header validation

On state-changing requests, validate Origin or Referer header matches expected origin. Simpler than tokens but can fail when headers are stripped (privacy proxies). Defence-in-depth measure, not standalone.

Why JWT in localStorage is risky for CSRF (but risky for XSS)

This is a nuanced trade-off. Cookies are vulnerable to CSRF (browser auto-includes them cross-origin). localStorage tokens must be manually added by JS — cross-origin JS cannot read localStorage (SOP enforced), so CSRF is eliminated. However, localStorage tokens are fully readable by any JavaScript running on your origin — including XSS payloads. An XSS attack can fetch('/api', { headers: { Authorization: 'Bearer ' + localStorage.getItem('token') } }) and exfiltrate data. With HttpOnly cookies, XSS cannot read the token at all. The security guidance: HttpOnly cookies + SameSite=Lax is the more secure choice. The CSRF risk with cookies is mitigated by SameSite; the XSS risk with localStorage has no mitigation.

Code example
// CSRF attack scenario (simplified)
// evil.com page contains:
<form action="https://bank.com/transfer" method="POST" id="f">
  <input name="to" value="attacker-account" />
  <input name="amount" value="10000" />
</form>
<script>document.getElementById('f').submit()</script>
// Bank.com's cookie is sent automatically — transfer executes

// Defence 1: SameSite cookies (server sets this)
Set-Cookie: session=abc123; SameSite=Lax; Secure; HttpOnly; Path=/
// Lax: cookie NOT sent on cross-site POST — CSRF blocked

// Defence 2: CSRF token (synchronizer token pattern)
const csrfToken = document.querySelector('meta[name="csrf-token"]').content;

await fetch('/api/transfer', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-CSRF-Token': csrfToken, // Custom header — cross-origin JS can't set this
  },
  body: JSON.stringify({ to: 'recipient', amount: 100 }),
  credentials: 'include',
});
// Server validates: X-CSRF-Token header must match session's stored token
// evil.com cannot read or forge this token — SOP prevents cross-origin reads

// localStorage vs cookie: the security trade-off
// localStorage: immune to CSRF (not auto-sent), vulnerable to XSS
localStorage.setItem('token', jwt); // XSS can: localStorage.getItem('token')

// HttpOnly cookie: vulnerable to CSRF (auto-sent), immune to XSS read
// Best practice: HttpOnly cookie + SameSite=Lax = both protections
Set-Cookie: token=jwt; HttpOnly; SameSite=Lax; Secure
Common mistakes
Assuming CSRF only works via HTML forms — modern CSRF attacks work via fetch/XHR with simple request methods (GET, POST with non-JSON content types). Custom content types like application/json trigger a CORS preflight for cross-origin requests, which provides some protection — but don't rely on this alone.
Storing JWTs in localStorage and considering the auth "secure" — localStorage is fully XSS-readable. A single XSS vulnerability means the attacker has the JWT and can authenticate as the victim from anywhere, until token expiry. HttpOnly cookies limit XSS damage even when a vulnerability exists.
Interview perspective
"How does CSRF work and how does SameSite=Lax prevent it?"
CSRF exploits automatic cookie inclusion — browser sends session cookies with every request to a domain, including cross-site requests from evil.com. SameSite=Lax prevents the cookie being sent on cross-site subrequest POST/PATCH/DELETE (only allows it on top-level navigation GETs). Since CSRF attacks require triggering state-changing requests, Lax blocks them while preserving normal authentication flows (e.g. OAuth redirects back to the site).
"Why is storing a JWT in localStorage a security risk?"
Any JavaScript running on your origin can read localStorage — including XSS payloads. An attacker who achieves XSS can steal the JWT, use it from any machine, and the session persists until token expiry. HttpOnly cookies prevent this: even with XSS, the cookie value is never accessible to JavaScript. The trade-off: cookies need CSRF protection (solved by SameSite=Lax), localStorage doesn't. Net recommendation: HttpOnly cookies win.

Clickjacking

iframe overlayX-Frame-Optionsframe-ancestorsUI redressing
Definition

Clickjacking (UI redressing) is an attack where an attacker renders your site inside a transparent iframe overlaid on a decoy page. The victim thinks they're clicking the attacker's page but is actually clicking on hidden UI elements of your site — confirming a purchase, approving a permissions dialog, following an account, or performing any click-triggered action. The attack requires no code execution — it exploits the browser's ability to embed any page in an iframe and control its opacity and positioning via CSS.

How it works
evil.com renders victim's site in an <iframe>
→
iframe styled: opacity:0, position:absolute, z-index:high
→
Victim sees evil.com's decoy UI (a game, a prize button)
→
Victim clicks "claim prize" → actually clicking "Delete Account" on the hidden iframe
Defences

X-Frame-Options (legacy)

X-Frame-Options: DENY — page cannot be embedded in any iframe. X-Frame-Options: SAMEORIGIN — only same-origin iframes allowed. Does not support allowlisting specific third-party origins. HTTP header, set by the server. Supported by all browsers including IE.

CSP frame-ancestors (modern)

Content-Security-Policy: frame-ancestors 'none' — equivalent to X-Frame-Options DENY. frame-ancestors 'self' — same-origin only. frame-ancestors https://trusted.com — allowlist specific origins. Supercedes X-Frame-Options. Use both for maximum compatibility.

frame-busting JS (deprecated)

Old technique: if (window !== window.top) window.top.location = window.location. Bypassed by sandboxed iframes: <iframe sandbox="allow-scripts"> blocks navigation. Never rely on JS for clickjacking protection. HTTP headers only.

SameSite cookies

If the site uses session cookies with SameSite=Strict/Lax, actions performed in the clickjacked iframe won't include session cookies (cross-site context). This limits damage to unauthenticated actions. Not a complete defence — some actions don't require authentication.

Code example
// Server-side headers (Express.js) — set on every HTML response
app.use((req, res, next) => {
  // Legacy header — all browsers
  res.setHeader('X-Frame-Options', 'SAMEORIGIN');

  // CSP frame-ancestors — modern, more flexible (overrides X-Frame-Options)
  res.setHeader(
    'Content-Security-Policy',
    "frame-ancestors 'self' https://trusted-embed-partner.com"
  );
  next();
});

// Next.js — in next.config.js headers()
module.exports = {
  async headers() {
    return [{
      source: '/(.*)',
      headers: [
        { key: 'X-Frame-Options', value: 'DENY' },
        { key: 'Content-Security-Policy', value: "frame-ancestors 'none'" },
      ],
    }];
  },
};

// ❌ Old frame-busting JS — DO NOT use
if (top !== self) { top.location = self.location; } // Bypassed by sandbox attr

// Test: DevTools → Application → Frames to see frame-ancestors enforcement
Common mistakes
Using X-Frame-Options: ALLOW-FROM to allowlist origins — this directive is not supported in modern Chrome and Firefox. Use CSP frame-ancestors instead for origin allowlisting.
Relying on JavaScript frame-busting — trivially bypassed by the attacker adding sandbox="allow-scripts allow-forms" to their iframe element. The sandbox attribute prevents the framed page's JS from navigating the top frame.
Interview perspective
"What is clickjacking and how do you prevent it?"
Clickjacking embeds the target site in a transparent iframe overlaid on a decoy page — victim clicks "land mine" buttons on the real site without seeing them. Prevention: server-side HTTP headers. X-Frame-Options: DENY or SAMEORIGIN (legacy, all browsers). CSP frame-ancestors: 'none' or 'self' (modern, supercedes XFO, allows origin allowlisting). Never rely on JavaScript frame-busting — bypassed by iframe sandbox attribute.

CSP (topic 4 — frame-ancestors is a CSP directive) iframe sandbox attribute (how attackers bypass frame-busting JS) SameSite cookies (partial defence — limits credentialed iframe actions)

Content Security Policy

DirectivesNoncesReport-onlyXSS mitigation
Definition

Content Security Policy (CSP) is an HTTP response header that instructs the browser on what resources are permitted to load and what code is permitted to execute on a page. It is a defence-in-depth layer: even if an attacker succeeds in injecting a script (via XSS), CSP prevents it from executing if it doesn't match the policy. CSP operates as an allowlist — everything not explicitly permitted is blocked. A strict CSP can render most XSS payloads inert, even those the application's output encoding failed to catch.

Key directives

script-src

Controls where scripts can load from and what inline scripts can execute. Most important directive. 'self': same-origin scripts only. 'nonce-{value}': scripts with matching nonce attribute only. 'strict-dynamic': scripts loaded by trusted scripts can load further scripts. 'unsafe-inline': allows all inline scripts — defeats XSS protection, avoid.

default-src

Fallback for all resource types not explicitly configured. default-src 'self' is a safe base: only same-origin resources for everything unless a more specific directive overrides. Always set a default-src to prevent unintended resource loading.

connect-src

Controls which origins JavaScript can connect to via fetch, XHR, WebSocket, EventSource. Restricts where data can be sent — limits exfiltration even if XSS occurs. connect-src 'self' https://api.myapp.com

style-src

Controls CSS loading. 'unsafe-inline' allows inline style attributes (common in React-generated code). Without it, inline styles break. Often needs 'unsafe-inline' for SPAs — CSS injection is less critical than script injection.

img-src

Controls image sources. img-src 'self' data: https://cdn.myapp.com. Without data:, base64-encoded images break. Important: attackers can use image requests to exfiltrate data even without JS — limit img-src to trusted origins.

frame-ancestors

Controls which origins can embed this page in an iframe. Replaces X-Frame-Options. frame-ancestors 'none': clickjacking prevention. frame-ancestors 'self': same-origin embeds allowed. (See topic 3.)

Nonces — the strict CSP approach

A nonce (number used once) is a cryptographically random value generated per HTTP response. The server includes it in the CSP header and on every legitimate <script> tag. The browser only executes scripts with a matching nonce. An attacker's injected <script> tag has no nonce and is blocked, even if it made it past output encoding. The nonce must be: cryptographically random (at least 128 bits), different on every response (never static), and never guessable. Nonce-based CSP combined with 'strict-dynamic' is the recommended approach for modern SPAs.

Report-only mode

Content-Security-Policy-Report-Only header: apply the policy without enforcing it. Violations are reported to a report-uri endpoint but resources are still loaded. Critical for rolling out CSP incrementally — you discover what breaks without breaking production. Workflow: deploy in report-only → monitor violations → fix legitimate resources → tighten policy → switch to enforcing mode.

Code example
// Strict CSP with nonces — generated per request
import crypto from 'crypto';

function generateNonce() {
  return crypto.randomBytes(16).toString('base64');
}

app.use((req, res, next) => {
  const nonce = generateNonce();
  res.locals.nonce = nonce; // Pass to template

  res.setHeader('Content-Security-Policy', [
    `script-src 'nonce-${nonce}' 'strict-dynamic'`,
    // nonce: only scripts with this nonce execute
    // strict-dynamic: scripts loaded by nonce'd scripts can load further scripts
    // (needed for bundlers that load chunks dynamically)
    "default-src 'self'",
    "connect-src 'self' https://api.myapp.com wss://ws.myapp.com",
    "img-src 'self' data: https://cdn.myapp.com",
    "style-src 'self' 'unsafe-inline'", // Needed for React inline styles
    "frame-ancestors 'none'",
    "base-uri 'self'",         // Prevent <base> tag injection
    "object-src 'none'",       // Prevent Flash/plugin injection
    "upgrade-insecure-requests", // Force HTTPS for all resources
    `report-uri /csp-report`,  // Collect violation reports
  ].join(';'));
  next();
});

// HTML template uses the nonce on every script tag:
<script nonce="<%= nonce %>" src="/bundle.js"></script>
<script nonce="<%= nonce %>">window.__INITIAL_STATE__ = {...};</script>
// Injected <script>...</script> has no nonce → BLOCKED by browser

// Report-only mode — monitor before enforcing
res.setHeader(
  'Content-Security-Policy-Report-Only',
  "script-src 'nonce-abc123'; report-uri /csp-report"
);
// Browser logs violations but doesn't block anything
// Watch /csp-report for POST requests to find legitimate resources to allowlist
Common mistakes
Using 'unsafe-inline' in script-src — negates the XSS protection entirely. Any injected inline script now executes. If you need inline scripts, use nonces instead. 'unsafe-inline' is ignored when a nonce or hash is present in the policy (nonce takes precedence).
Using a static nonce — a nonce must be cryptographically random and regenerated for every HTTP response. A static nonce embedded in your HTML template is effectively 'unsafe-inline' — an attacker who reads your source code knows the nonce and can include it in injected scripts.
Not deploying in report-only first — a CSP that blocks your own scripts breaks the site. Report-only mode reveals violations without breaking users. Always go report-only → observe → fix → enforce.
Interview perspective
"How does a CSP nonce prevent XSS even after an injection?"
The server generates a random nonce per response and includes it in both the CSP header (script-src 'nonce-{value}') and on every legitimate script tag as a nonce attribute. The browser only executes scripts with a matching nonce. An XSS-injected script tag has no nonce (the attacker can't predict it) and is blocked. The nonce must be random per response — a static nonce is equivalent to no protection.
"What is the difference between CSP report-only mode and enforcement mode?"
Report-only: the policy is evaluated and violations are sent to report-uri, but nothing is actually blocked. Used to discover what breaks before enforcement. Enforcement: violations are blocked and reported. You need report-only first to know what legitimate resources to add to the allowlist — jumping straight to enforcement breaks production.

Subresource Integrity (SRI)

Hash verificationCDN trustSupply chainintegrity attribute
Definition

Subresource Integrity (SRI) allows browsers to verify that a resource fetched from a third-party CDN or origin has not been tampered with. You provide a cryptographic hash of the expected resource content in the HTML. Before executing the script or applying the stylesheet, the browser hashes the fetched content and compares it against your provided hash. If they don't match — because the CDN was compromised, served a different version, or a supply-chain attack injected malicious code — the browser refuses to execute the resource. SRI protects against the class of attacks where a CDN (cdnjs, unpkg, jsDelivr) is compromised and begins serving malicious scripts to millions of sites.

How it works internally

You specify the integrity attribute on <script> or <link> tags with a hash in the format algorithm-base64hash. Supported algorithms: sha256, sha384, sha512 (sha384 recommended). The browser fetches the resource, computes the hash over the response body, and compares it to every hash in the integrity attribute. If any hash matches, the resource is accepted. If none match, the resource is blocked with a network error (similar to a CORS violation). The crossorigin attribute is required when using SRI for cross-origin resources — it enables CORS mode for the fetch so the response can be inspected.

Code example
<!-- SRI on CDN script — browser verifies hash before executing -->
<script
  src="https://cdnjs.cloudflare.com/ajax/libs/lodash.js/4.17.21/lodash.min.js"
  integrity="sha512-WFN04846sdKMIP5LKNphMaWzU7YpMyCU245etK3g/2ARYbPK9Ub18eG+ljU96qKRCWh+quCY7yefSmlkQw=="
  crossorigin="anonymous"
  referrerpolicy="no-referrer"
></script>
<!-- crossorigin="anonymous": browser can inspect the response for hash verification -->
<!-- If CDN serves different bytes: SHA mismatch → script blocked -->

<!-- SRI on stylesheet -->
<link
  rel="stylesheet"
  href="https://cdnjs.cloudflare.com/ajax/libs/normalize/8.0.1/normalize.min.css"
  integrity="sha512-NhSC1YmyruXifcj/KFRWoC561YpHpc5Jtzgvbuzx5VozKpWvQ+4nXholtQKqhGF1JKNT9ZkpOz8JaFOI/sw=="
  crossorigin="anonymous"
/>

// Generate SRI hash (Node.js)
import { createHash } from 'crypto';
import { readFileSync } from 'fs';

function generateSRI(filePath) {
  const content = readFileSync(filePath);
  const hash = createHash('sha384').update(content).digest('base64');
  return `sha384-${hash}`;
}
// Or use: https://www.srihash.org for one-off generation
// Or CLI: openssl dgst -sha384 -binary file.js | openssl base64 -A

// CSP + SRI: belt-and-suspenders
Content-Security-Policy: script-src 'self' https://cdnjs.cloudflare.com
// CSP limits which CDNs can serve scripts
// SRI verifies the exact content from the allowed CDN
// Together: origin allowlisted AND content verified
Subtopics

When SRI is required

Any script/stylesheet loaded from a third-party origin you don't fully control: CDNs (cdnjs, jsDelivr, unpkg), Google Fonts, third-party widget scripts. If you self-host or use content hashing (Vite output), SRI is already implicit — the hash is in the filename.

Supply chain attack context

Historical precedent: polyfill.io (June 2024) was acquired and began serving malicious JavaScript to 100k+ sites. Any site including it without SRI executed the malicious code. With SRI, the hash mismatch would have blocked execution immediately.

crossorigin attribute requirement

SRI for cross-origin resources requires crossorigin="anonymous". Without it, the browser fetches the resource without CORS, and the SRI hash cannot be computed over the response. The resource is loaded without verification. This is a silent failure mode.

Dynamic imports and SRI

SRI cannot be applied to dynamically imported scripts (import()) — the module specifier in code doesn't support the integrity attribute. Use CSP script-src with origin allowlisting as a complement. Avoid dynamically importing from untrusted CDNs.

Common mistakes
Omitting crossorigin="anonymous" on a cross-origin SRI tag — SRI silently fails to verify the hash and loads the resource without integrity checking. The attribute is mandatory for cross-origin SRI to function.
Pinning SRI hashes to a CDN URL that serves the "latest" version (without a specific version in the URL) — the CDN may update the file, changing the content and breaking the hash match. Always include an explicit version number in the URL when using SRI.
Interview perspective
"What is Subresource Integrity and what attack does it protect against?"
SRI adds an integrity attribute with a cryptographic hash to external script/link tags. The browser hashes the fetched content and compares it — mismatch blocks execution. Protects against CDN compromise (supply-chain attack): if a CDN starts serving malicious code, the hash mismatch prevents it executing. The polyfill.io attack (2024) is the canonical recent example — SRI would have blocked it entirely.

Cookie flags — every flag matters

HttpOnly

Cookie cannot be read or written by JavaScript — document.cookie returns it as empty. Protects against XSS token theft: even if XSS runs on your page, the session cookie is inaccessible. Critical for authentication cookies. No valid reason to omit this on session tokens.

Secure

Cookie is only sent over HTTPS connections — never over plain HTTP. Without this, a token sent over HTTP is transmitted in plaintext and can be intercepted by network attackers. In production, every authentication cookie must have this flag.

SameSite=Lax

Cookie is not sent on cross-site subrequest (POST, fetch). Prevents CSRF. Sent on top-level navigation GET — OAuth redirects and normal link following work. The default in modern browsers. Explicit declaration ensures consistent cross-browser behaviour.

SameSite=Strict

Cookie never sent cross-site, including top-level GET navigation. User clicking a link from Google to your site won't be logged in until after the redirect. Maximum CSRF protection but breaks OAuth flows and external links landing with session. Use for highly sensitive admin surfaces.

SameSite=None; Secure

Always sent cross-site — required for third-party auth (embedded widgets, iframes that need auth). Must be combined with Secure flag (HTTPS required). Requires explicit CSRF protection (CSRF tokens) since SameSite protection is opted out.

Path & Domain

Restrict which paths/subdomains receive the cookie. Path=/api: only sent to API paths, not the frontend. Domain=.myapp.com: sent to all subdomains. Narrowing Path and Domain limits exposure — a compromised subdomain can't steal cookies scoped to another subdomain.

Code example
// Server sets the authentication cookie (Express.js)
app.post('/auth/login', async (req, res) => {
  const user = await validateCredentials(req.body);
  const token = signJWT({ userId: user.id });

  res.cookie('session', token, {
    httpOnly: true,      // JS cannot read this cookie
    secure: true,        // HTTPS only
    sameSite: 'lax',    // CSRF protection
    maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days in ms
    path: '/',
  });
  res.json({ success: true }); // No token in response body!
});

Common mistakes
Not setting the Secure flag on auth cookies — on HTTP connections (dev without HTTPS, misconfigured production), the token is transmitted in plaintext. Any network observer (coffee shop Wi-Fi, ISP) can capture it. Always use Secure in combination with HttpOnly.
Storing long-lived refresh tokens in localStorage — if XSS occurs, the attacker has the refresh token and can generate access tokens indefinitely until the refresh token expires (days or weeks). Refresh tokens must be HttpOnly cookies. Access tokens can be in memory (lost on page refresh) or very short-lived cookies.
Thinking HttpOnly cookies make CSRF impossible — they prevent XSS token theft but CSRF still works (browser auto-includes cookies). SameSite=Lax is what prevents CSRF. Both protections must be applied together.
Interview perspective
"Where should you store a JWT and why?"
HttpOnly, Secure, SameSite=Lax cookie. Reasoning: HttpOnly prevents XSS from reading the token (localStorage doesn't). SameSite=Lax prevents CSRF (the main cookie vulnerability). Secure ensures HTTPS only. The combination gives XSS protection AND CSRF protection. localStorage gives CSRF immunity but zero XSS protection — one XSS vulnerability and the attacker has the token permanently.
"What is the difference between HttpOnly and Secure cookie flags?"
HttpOnly: prevents JavaScript access to the cookie — XSS cannot read it via document.cookie or JS APIs. Secure: cookie only transmitted over HTTPS — prevents cleartext interception over HTTP. They protect against different threats. Both must be set on authentication cookies. Omitting HttpOnly = XSS can steal it. Omitting Secure = network observer can steal it.

OAuth & PKCE in the browser

PKCE flowCode verifierImplicit flow deprecatedSPA auth
Definition

OAuth 2.0 is a delegated authorisation framework — your app (the "client") requests access to a resource owned by a user, via an authorisation server (Google, GitHub, Auth0), without handling the user's credentials directly. In browser-based SPAs, the correct OAuth flow is Authorization Code Flow with PKCE (Proof Key for Code Exchange). PKCE is a security extension that protects the authorization code against interception attacks in public clients (browsers, mobile apps) where a client secret cannot be kept confidential. The older Implicit Flow (which returned tokens directly in the URL fragment) is deprecated — it exposed tokens in browser history, referrer headers, and server logs.

Why the implicit flow is deprecated

Tokens in URL fragment

Implicit flow returned access tokens in the redirect URI's hash fragment: https://app.com/callback#access_token=xyz. Tokens leaked in: browser history, referrer headers sent to CDN and analytics, server access logs, and postMessage leaks in iframes. No way to revoke these exposures.

No refresh tokens

Implicit flow spec prohibits refresh tokens (too risky in the browser). Short-lived access tokens require re-authorisation when they expire, causing poor UX. PKCE flow allows refresh tokens — the refresh token is stored in an HttpOnly cookie and used to silently obtain new access tokens.

No way to verify token recipient

Implicit flow cannot prove the token was issued to the requesting client (no client secret). An attacker who intercepts the authorization code or token cannot be distinguished from the legitimate client. PKCE's code_challenge/code_verifier proves possession without a secret.

Authorization server fingerprinting

Tokens returned in URL fragments are accessible to all scripts on the page at the time of redirect — any script with access to location.hash could steal the token. With PKCE, the code in the URL is useless without the code_verifier held in memory.

PKCE flow — step by step
1. Generate code_verifier (random 43-128 char string)
2. code_challenge = BASE64URL(SHA256(code_verifier))
3. Redirect to AuthServer with: response_type=code, code_challenge, code_challenge_method=S256
4. User authenticates at AuthServer
5. AuthServer redirects back with: ?code=AUTH_CODE (not a token — just a code)
6. App POSTs to AuthServer token endpoint: code + code_verifier
7. AuthServer verifies: SHA256(code_verifier) == stored code_challenge → issues tokens
8. App receives access_token (short-lived) + refresh_token (long-lived)

The security comes from step 7: only the client that initiated the flow has the code_verifier. Even if the authorization code in the URL is intercepted (in browser history, referrer header), an attacker cannot exchange it for tokens — they don't have the code_verifier. The code is useless without it.

Code example
// Complete PKCE flow implementation
async function initiateOAuth() {
  // Step 1: Generate cryptographically random code verifier
  const codeVerifier = Array.from(
    crypto.getRandomValues(new Uint8Array(32))
  ).map(b => b.toString(36)).join('');

  // Step 2: Hash it → code challenge
  const encoded = new TextEncoder().encode(codeVerifier);
  const hashBuffer = await crypto.subtle.digest('SHA-256', encoded);
  const codeChallenge = btoa(
    String.fromCharCode(...new Uint8Array(hashBuffer))
  ).replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');

  // Step 3: Store verifier (sessionStorage — ephemeral, tab-scoped)
  sessionStorage.setItem('pkce_verifier', codeVerifier);
  sessionStorage.setItem('oauth_state', generateState()); // CSRF protection

  // Step 4: Redirect to authorization server
  const params = new URLSearchParams({
    response_type: 'code',          // NOT 'token' (that's implicit flow)
    client_id: 'my-spa-client',
    redirect_uri: 'https://myapp.com/callback',
    scope: 'openid profile email',
    code_challenge: codeChallenge,
    code_challenge_method: 'S256',   // SHA-256 hash
    state: sessionStorage.getItem('oauth_state'),
  });
  location.assign(`https://auth.provider.com/oauth/authorize?${params}`);
}

async function handleCallback() {
  const params = new URLSearchParams(location.search);
  const code  = params.get('code');
  const state = params.get('state');

  // Validate state (CSRF protection for the OAuth flow itself)
  if (state !== sessionStorage.getItem('oauth_state')) {
    throw new Error('State mismatch — possible CSRF on OAuth callback');
  }

  const codeVerifier = sessionStorage.getItem('pkce_verifier');
  sessionStorage.removeItem('pkce_verifier'); // One-time use
  sessionStorage.removeItem('oauth_state');

  // Exchange code + verifier for tokens
  const res = await fetch('https://auth.provider.com/oauth/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      grant_type: 'authorization_code',
      code,
      redirect_uri: 'https://myapp.com/callback',
      client_id: 'my-spa-client',
      code_verifier: codeVerifier, // The proof — attacker doesn't have this
    }),
  });
  const { access_token, refresh_token } = await res.json();

  // Store access token in memory (short-lived, ~15min)
  // Store refresh token via BFF (Backend for Frontend) as HttpOnly cookie
  // Best practice: use auth libraries (Auth0 SPA SDK, oidc-client-ts)
  // They handle PKCE, token storage, and silent refresh automatically
}
Backend for Frontend (BFF) pattern

Why BFF is the most secure SPA auth architecture

The BFF pattern routes all OAuth token exchange through a server you control. The SPA redirects to the BFF for login. The BFF handles the PKCE exchange with the auth server, receives the tokens, stores the refresh token in an HttpOnly cookie, and gives the access token only to the SPA in memory. The SPA makes API calls via the BFF, which attaches the access token server-side. The result: no tokens are ever accessible from browser JavaScript. XSS cannot steal tokens. CSRF is handled by SameSite cookies. This is the approach recommended by the OAuth 2.0 Security Best Current Practice (BCP) for SPAs.

Common mistakes
Using the Implicit Flow (response_type=token) in new applications — it is explicitly deprecated in OAuth 2.0 Security BCP (RFC 9700). All new SPA implementations must use Authorization Code + PKCE. Many auth libraries (Auth0 SDK, Google Identity) have already removed implicit flow support.
Storing access tokens or refresh tokens in localStorage after receiving them from the token endpoint — the token endpoint exchange should ideally happen server-side (BFF). If client-side, access tokens should live in memory only (lost on page refresh), and refresh tokens should be in HttpOnly cookies, not localStorage.
Not validating the state parameter on the OAuth callback — the state parameter is a CSRF token for the OAuth flow itself. Without validating it, an attacker can trick a user into completing an OAuth flow that the attacker initiated, associating the victim's account with the attacker's authorization code.
Interview perspective
"Why is the OAuth implicit flow deprecated for SPAs?"
Implicit flow returns access tokens directly in the URL fragment (#access_token=...). Tokens are exposed in browser history, referrer headers to third-party scripts, server logs, and postMessage leaks. No refresh tokens allowed. Cannot verify token recipient. PKCE fixes this: issues only an authorization code (useless without the code_verifier), exchanges it server-side (or client-side with the verifier), tokens never appear in the URL.
"How does PKCE protect against authorization code interception?"
PKCE: generate a random code_verifier, hash it to code_challenge, send the challenge to the auth server with the authorization request. Auth server stores the challenge. Client receives the code in the redirect URL. Client POSTs code + original verifier to the token endpoint. Auth server hashes the verifier and compares to stored challenge — must match. An attacker who intercepts the code cannot exchange it without the verifier, which was never transmitted over the network.
