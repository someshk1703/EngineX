Reconciliation & diffing

Virtual DOMDiffing heuristicsKeysindex-as-key
Definition

Reconciliation is React's algorithm for determining what changed between two renders and computing the minimal set of DOM mutations needed. React maintains a virtual DOM — a lightweight in-memory tree of React element descriptors ({type, props, children}). On each render, React calls component functions to produce a new element tree, then diffs it against the previous tree. The diff produces a change-set that is applied to the real DOM in the commit phase. Without reconciliation, every re-render would destroy and recreate the entire DOM — catastrophically slow. React's diffing is O(n) (not the theoretical O(n³) for general tree diffing) through two heuristics: (1) elements of different types produce different trees entirely, (2) key props let React track element identity across renders.

How it works internally

React diffs trees level-by-level (breadth). At each position it compares the old fiber node's type against the new React element's type. Same type: React keeps the DOM node and updates only changed props. Children are reconciled recursively. Different type: React unmounts the entire old subtree (running all cleanup effects, removing DOM nodes) and mounts a fresh subtree from scratch. This is why wrapping a component in a different parent element causes full remount. For lists, React uses key to match old and new elements by identity across positions. Without keys (or with index keys), React matches by position — meaning a deleted first element causes every subsequent element to appear "changed", updating N-1 elements unnecessarily. With stable keys, React correctly identifies the deletion and only removes that one node.

Subtopics

Element type comparison

React compares element.type (string for DOM elements, function/class reference for components). Two renders of the same component function = same type. A new function defined inline = different reference = React sees a different type each render.

Key reconciliation

Keys let React match elements by identity rather than position. Keys must be stable (same value across renders), unique among siblings, and ideally from the data itself (database ID, slug). Keys are not passed as props — they're consumed by the reconciler.

Index as key — the bug

When items are reordered, inserted, or deleted, index keys cause: wrong component state association (React thinks element at index 0 is still the same), unnecessary re-renders of all elements after the change, broken animations and form inputs.

Component identity

If you define a component inside another component's render, a new function reference is created each render. React sees a new type, unmounts the old subtree, mounts fresh — losing all internal state. Move component definitions outside render functions.

Bailout optimization

React.memo wraps a component to skip re-render if props haven't changed (shallow equality). PureComponent is the class equivalent. The reconciler checks memo status before even calling the component function.

Fragment keys

React.Fragment can receive a key: <Fragment key={id}>. Used when you need to key a group of elements that map from data without introducing a wrapper DOM node.

Code example
// ❌ Index as key — breaks on insert/delete/reorder
const BadList = ({ items }) => (
  <ul>
    {items.map((item, index) => (
      <li key={index}>          // React uses position as identity
        <input defaultValue={item.name} /> // State tied to position
      </li>                               // Delete item[0] → all inputs shift wrong
    ))}
  </ul>
);

// ✅ Stable ID as key
const GoodList = ({ items }) => (
  <ul>
    {items.map((item) => (
      <li key={item.id}>            // React tracks by identity across renders
        <input defaultValue={item.name} />
      </li>
    ))}
  </ul>
);

// ❌ Component defined inside render — new reference every render
function Parent() {
  const Child = () => <div>child</div>; // New function ref each render
  return <Child />;                      // React: different type! Full remount.
}

// ✅ Define outside
const Child = () => <div>child</div>;   // Stable reference
function Parent() { return <Child />; }

// Forcing remount with key — useful for resetting state
<UserProfile key={userId} userId={userId} />
// When userId changes → key changes → React unmounts old, mounts fresh
// All internal state of UserProfile is reset. Controlled remount.
Performance implications

Incorrect keys cause O(n) DOM mutations for what should be O(1). A list of 1000 items where one is deleted from the front with index keys triggers 999 prop-update patches plus a delete. With stable ID keys: 1 delete, 0 patches. This is measurable at 50+ items and catastrophic at thousands. Key-related reconciliation bugs also cause React to unmount and remount components unnecessarily, running teardown effects and losing component state — a source of flickering, lost scroll positions, and refetched data.

Common mistakes
Using array index as key whenever you have a static, never-reordered list — this is technically safe but builds a false habit. The moment the list becomes sortable or filterable, you have a bug. Default to IDs.
Using Math.random() as a key — generates a new key every render, forcing React to unmount and remount every list item on every render. Worse than index keys.
Thinking the virtual DOM eliminates DOM operations — it minimizes them. A component at the root of a deep subtree re-rendering still calls every descendant's render function (unless memoized). The vDOM diff is CPU cost, DOM mutation is GPU/layout cost.
Interview perspective
"Why is using index as a key dangerous?"
React uses keys for identity, not position. With index keys, deleting item[0] makes React think item[0] is still the same element but with item[1]'s props — it updates in-place instead of unmounting. Form inputs, animations, and component state tied to the element are now associated with the wrong data item. Stable IDs fix this by letting React identify which specific item was removed.
"How does React's diffing algorithm work and what are its heuristics?"
O(n) via two heuristics: (1) different element.type → full subtree remount, (2) key prop establishes identity across renders for list diffing. Level-by-level traversal. Emphasize what triggers full remount (type change, key change, conditional wrapping in different elements).

Fiber architecture

Work loopLanesConcurrent modeTime slicing
Definition

Fiber is React's reimplementation of its core algorithm (introduced in React 16), replacing the recursive stack-based reconciler with a linked-list-based, interruptible work scheduler. The old reconciler walked the component tree recursively — once started, it couldn't be paused. Fiber makes reconciliation incremental and preemptible: work is split into units called fiber nodes, and React can pause between units to check if higher-priority work (like a user input) has arrived. This enables Concurrent Mode: multiple renders can be in-flight simultaneously at different priorities, lower-priority renders can be interrupted and resumed, and React can discard stale work entirely if the underlying state changes before it's committed.

How it works internally

Every React element in the tree has a corresponding fiber node — a plain JS object with fields: type, stateNode (DOM node or class instance), child, sibling, return (parent), pendingProps, memoizedProps, memoizedState, updateQueue, flags (effect bitmask), and lanes. React maintains two trees: the current tree (what's on screen) and the work-in-progress (WIP) tree (being built). Fibers in these trees are linked — each WIP fiber has an alternate pointer to its current counterpart. The work loop (workLoopSync or workLoopConcurrent) iterates the fiber tree using a depth-first traversal: performUnitOfWork on each fiber, which calls beginWork (process fiber, return child or null) and completeWork (bottom-up, collect effects). In concurrent mode, the loop checks shouldYield() after each fiber — if the deadline is exceeded (5ms slices), it yields to the event loop via scheduler. Lanes are a priority bitmask system (31 lanes) — each update is assigned a lane (SyncLane, InputContinuousLane, DefaultLane, TransitionLane, IdleLane). The scheduler only processes fibers whose lanes match the current render priority.

Subtopics

Fiber node structure

Linked list: child → first child, sibling → next sibling, return → parent. This replaces the call stack — fiber nodes live on the heap, traversal is iterative (loop), not recursive. Stack can be "rewound" by abandoning WIP.

Double buffering

Current tree (on screen) and WIP tree (being built) exist simultaneously. On commit, the WIP tree becomes current (pointer swap). Fibers are recycled via alternate pointers — no GC pressure per render.

Lanes system

31 priority lanes via bitmask. React batches updates at the same lane. Higher-priority lanes interrupt lower-priority renders. Transition updates (startTransition) use TransitionLane — interruptible. Sync updates use SyncLane — must complete.

Work loop & scheduler

React Scheduler (separate package) uses MessageChannel macrotasks for 5ms work slices. shouldYield() checks elapsed time. If exceeded, React yields, pending work is saved in the WIP tree, resumes on next scheduler message.

beginWork / completeWork

beginWork: determine what changed for this fiber (bailout if no updates in lane). Call component function/render method. Return child fiber or null. completeWork: propagate flags up, attach DOM node references, build effect list.

Concurrent features

Enabled by Concurrent Mode (React 18 default). startTransition marks low-priority. useDeferredValue defers a value to lower-priority render. Suspense boundaries catch Promise throws and render fallback while awaiting resolution.

Code example
// Conceptual fiber node structure (simplified)
const fiberNode = {
  tag: 0,                  // FunctionComponent, ClassComponent, HostComponent...
  type: MyComponent,       // The function/class/string
  stateNode: domElement,    // Actual DOM node (for host fibers)
  child: childFiber,        // First child
  sibling: siblingFiber,    // Next sibling
  return: parentFiber,      // Parent fiber
  alternate: currentFiber,  // Current tree counterpart
  pendingProps: {},
  memoizedProps: {},
  memoizedState: hookList,  // Head of hooks linked list
  updateQueue: updates,     // Pending state updates
  flags: 0b00000100,        // Effect bitmask (Update, Placement, Deletion...)
  lanes: 0b0001,            // Which priority lanes this fiber has work in
};

// Simplified work loop (concurrent mode)
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress); // beginWork → completeWork
  }
  // If shouldYield() fired: WIP tree partially built, saved in memory
  // Scheduler resumes via MessageChannel macrotask
}

// Lane priority check — why startTransition works
// SyncLane   = 0b0000001 — synchronous, cannot be interrupted
// InputLane  = 0b0000100 — user input, high priority
// TransLane  = 0b0100000 — transition, low priority, interruptible
// If user types (InputLane) while transition render is in-flight:
// → React abandons WIP transition tree
// → Processes input update synchronously
// → Restarts transition render from scratch with new state
Common mistakes
Conflating "Fiber" (the data structure) with "Concurrent Mode" (the feature). Fiber has existed since React 16. Concurrent Mode (interruptible rendering) shipped in React 18. Fiber made concurrent mode possible but didn't enable it immediately.
Thinking React.StrictMode invokes renders twice due to a bug — it's intentional. StrictMode double-invokes render functions and effect setup/teardown in development to surface side effects in what should be pure functions. This is a direct consequence of the Fiber architecture needing render to be re-playable.
Interview perspective
"What is React Fiber and why was it introduced?"
Fiber replaced the recursive stack reconciler with an iterative, heap-allocated linked list traversal, making reconciliation interruptible. Enabled concurrent features: prioritized updates, time slicing, startTransition, Suspense. Explain the old stack reconciler's problem: started = committed, no way to pause.
"How does React's lanes system work?"
31-bit priority bitmask. Each update tagged with a lane. Scheduler processes lanes from highest to lowest priority. Multiple lanes can batch. TransitionLane = interruptible. SyncLane = must flush synchronously. This is what allows input events to interrupt transition renders.

Render phase vs commit phase

Pure renderSide effectsDOM ops in render
Definition

React separates a re-render into two distinct, strictly ordered phases with different rules. The render phase (also called reconciliation) calls component functions and computes what changed — it must be pure and side-effect-free because React may call it multiple times (StrictMode double-invoke, concurrent interruption and restart). The commit phase applies the computed changes to the real DOM and runs effects — it runs synchronously to completion, cannot be interrupted, and is the only place where DOM mutations, refs, and side effects are safe.

How it works internally

React splits the commit phase into three sub-phases to respect the order of effect execution. Before mutation (commitBeforeMutationEffects): reads the DOM before any mutations, runs getSnapshotBeforeUpdate in class components. This is where React captures scroll positions before DOM changes. Mutation (commitMutationEffects): applies all DOM mutations — insertions, updates, deletions. Runs useLayoutEffect cleanup from the previous render. Refs are detached (set to null). Layout (commitLayoutEffects): runs useLayoutEffect setup synchronously (DOM is updated, but browser hasn't painted yet). Attaches refs. After the commit phase finishes and the browser paints, React schedules useEffect setup via the Scheduler — asynchronous, after paint. This ordering is critical: useLayoutEffect runs before paint (safe for DOM measurements and synchronous mutations), useEffect runs after paint (safe for async operations, subscriptions).

Subtopics

Render phase purity requirement

Component functions must be pure: same props/state → same output, no side effects. React 18 + StrictMode double-invokes them in dev to catch violations. In concurrent mode, React may discard and re-run render multiple times before committing.

useLayoutEffect

Runs synchronously after DOM mutations but before browser paint. Use for: reading layout (getBoundingClientRect), synchronous DOM mutations (tooltip positioning), synchronizing third-party DOM libraries. Blocks paint if slow.

useEffect

Runs after paint. Use for: data fetching, subscriptions, event listeners, analytics. Never blocks paint. Runs in the order effects appear, matching their cleanup from previous render first.

Why no DOM ops in render

Render phase may be called multiple times (concurrent interruption). DOM mutations in render would be applied multiple times incorrectly. React doesn't know which render "wins" until commit. DOM is a side effect — belongs in commit.

ref attachment timing

Refs are set to null during mutation phase (before DOM is updated), then set to the new DOM node during layout phase. This means in useLayoutEffect, ref.current is the up-to-date DOM node. In useEffect, it's also up-to-date, but the DOM has already been painted.

Passive effects queue

useEffect callbacks are collected into a "passive effects" queue, flushed by the Scheduler after the browser has had a chance to paint. This is what makes useEffect non-blocking — it's a macrotask, not a synchronous call.

Code example
// ❌ Side effect in render phase — breaks in concurrent mode
function BadComponent({ id }) {
  analytics.track('view', id); // Fires on every render, may fire 2x in StrictMode
  return <div>{id}</div>;
}

// ✅ Side effect in useEffect — fires once after commit
function GoodComponent({ id }) {
  useEffect(() => {
    analytics.track('view', id); // After paint, once per committed render
  }, [id]);
  return <div>{id}</div>;
}

// useLayoutEffect — measure before paint (tooltip positioning)
function Tooltip({ target }) {
  const ref = useRef();
  useLayoutEffect(() => {
    // DOM is mutated, but browser hasn't painted yet
    // Safe to read layout and synchronously reposition
    const rect = target.getBoundingClientRect();
    ref.current.style.top = rect.bottom + 'px'; // No visual flash
  });
  return <div ref={ref} className="tooltip" />;
}

// useEffect would cause visible flash: paint happens → layout → reposition
// User sees element jump. useLayoutEffect prevents the flash.

// StrictMode double-invoke — effect setup+cleanup runs twice in dev
useEffect(() => {
  const sub = subscribe(channel); // Runs in dev: setup → cleanup → setup
  return () => sub.unsubscribe(); // Must be idempotent
}, [channel]);
Common mistakes
Using useLayoutEffect for data fetching — it blocks paint while awaiting. Only use it for synchronous DOM reads/writes. Network calls belong in useEffect or in a data-fetching library (SWR, React Query, RSC).
Not providing cleanup in useEffect for subscriptions/listeners — in StrictMode, React runs setup→cleanup→setup to verify cleanup is correct. Without cleanup, you'll accumulate duplicate listeners. In production, navigation away without cleanup leaks the subscription.
Thinking StrictMode double-invoke is a bug — it's intentional. It surfaces non-idempotent effects. If your effect breaks on double-invoke, your cleanup is incorrect.
Interview perspective
"What is the difference between useEffect and useLayoutEffect?"
useLayoutEffect: synchronous, after DOM mutation, before paint. useEffect: async, after paint. Use useLayoutEffect for DOM measurements and synchronous mutations to avoid visual flash. Use useEffect for everything else. useLayoutEffect blocks paint — must be fast.
"Why can't you perform side effects in the render phase?"
Render may execute multiple times — React can discard and re-execute in concurrent mode (StrictMode deliberately does this). Side effects (DOM mutations, network calls, subscriptions) applied multiple times or to discarded renders would be incorrect. Only committed renders are real — commit phase is the safe zone.

useState & useReducer internals

Hook linked listDispatcherConditional hooks
Definition

useState and useReducer are the primary state primitives in React function components. Internally they are nearly identical — useState(initialValue) is useReducer(basicStateReducer, initialValue) where basicStateReducer is (state, action) => typeof action === 'function' ? action(state) : action. Both store state in the fiber's memoizedState field as a linked list of hook objects. This linked list is the reason hooks cannot be called conditionally, inside loops, or after early returns — React identifies each hook by its call order position in the list, not by name.

How it works internally

During the first render (mount), each hook call creates a new hook object { memoizedState, queue, next } and appends it to the fiber's hook list. During subsequent renders (update), React reads hook objects from the existing list in order — hook call 1 reads hook node 1, hook call 2 reads hook node 2, etc. The dispatcher is the object that React swaps between mount and update phases to differentiate behavior: during mount, ReactCurrentDispatcher.current = HooksDispatcherOnMount; during update, it's HooksDispatcherOnUpdate. This is what makes hooks stateful across renders without explicit identifiers. When you call setState(newValue), React creates an update object ({ action, next, lane }) and enqueues it in the hook's queue (a circular linked list of updates). React then schedules a re-render at the appropriate lane priority. During the re-render, React processes the update queue to compute the new state value via the reducer.

Subtopics

Hook linked list

Each hook call during render appends to / reads from a singly linked list stored in fiber.memoizedState. The list is ordered by call sequence. This is the concrete mechanism behind "hooks must be called in the same order every render".

Dispatcher swap

React uses a global dispatcher object that switches between mount/update/re-render implementations. In dev, there's also a special dispatcher that throws "cannot call hooks conditionally" errors by checking the expected vs actual hook call count.

Update queue (circular)

setState calls enqueue update objects in a circular singly linked list on the hook's queue. During render, React applies all pending updates in order by running them through the reducer, yielding the new state value.

Batching

Multiple setState calls within one event handler create multiple update objects in the queue but trigger only one re-render. React 18 automatic batching extends this to all async contexts (setTimeout, Promise callbacks, etc.).

useState vs useReducer

useState: reducer is built-in (identity or function call). useReducer: explicit reducer function, explicit action objects. Prefer useReducer when next state depends on previous in complex ways, or when multiple sub-values need atomic updates.

Lazy initialization

useState(fn) and useReducer(reducer, arg, initFn) — if a function is passed as initial state, it's called once on mount and its return value used. Avoids re-running expensive initial computation on every render.

Code example
// Hook linked list visualization
function Component() {
  const [count, setCount] = useState(0);   // Hook node 0: {state:0, next:→}
  const [name, setName]   = useState('');    // Hook node 1: {state:'', next:→}
  useEffect(() => {}, []);                   // Hook node 2: {deps:[], next:null}
}
// fiber.memoizedState → hookNode0 → hookNode1 → hookNode2 → null

// ❌ Conditional hook — breaks the linked list order
function Bad({ show }) {
  if (show) useState(0); // Sometimes hook 0 exists, sometimes not
  useState('');         // React reads hook 0 here when show=false → wrong state!
}

// useState is useReducer with a built-in reducer
function basicStateReducer(state, action) {
  return typeof action === 'function' ? action(state) : action;
}
// setState(5)       → action = 5          → state = 5
// setState(n => n+1)→ action = fn         → state = fn(currentState)

// useReducer — when to prefer over useState
const formReducer = (state, action) => {
  switch (action.type) {
    case 'SET_FIELD': return { ...state, [action.field]: action.value };
    case 'RESET':     return initialFormState;
    case 'SUBMIT':    return { ...state, submitting: true };
  }
};
// Multiple related fields updated atomically in one dispatch
// Logic is extractable, testable, and replayable

// Lazy initializer — runs once, not every render
const [data, setData] = useState(() => expensiveParse(rawInput));
Common mistakes
Using object spread for all state updates in useState — setState({...state, field: value}) stales when multiple state updates batch. Prefer functional update setState(prev => ({...prev, field: value})) or useReducer for multi-field state.
Calling hooks inside callbacks, conditions, or loops. The React rule "hooks must be called at the top level" exists because the linked list must have the same node count and order every render. Any conditional call shifts all subsequent hook nodes to wrong positions.
Passing an initial value to useState that involves computation: useState(computeExpensive()) runs the computation on every render. Use lazy initializer: useState(() => computeExpensive()).
Interview perspective
"Why can't hooks be called conditionally?"
Hooks are identified by position in a linked list stored on the fiber, not by name. React reads hook N from list position N on every render. If a hook is conditionally skipped, all subsequent hooks shift by one position — React reads the wrong state for every hook after the skipped one. The rule enforces stable list order.
"What is the internal relationship between useState and useReducer?"
useState is useReducer with a built-in reducer: identity for values, function call for functional updates. The implementation is literally the same code path. This means everything you can do with useState, you can do with useReducer — the difference is ergonomics for complex state transitions.

useEffect internals

Dependency arrayCleanup timingStrictModePassive effects
Definition

useEffect is React's mechanism for synchronizing a component with external systems after render. It schedules a passive effect — a callback that runs asynchronously after the browser has painted. The dependency array controls when the effect re-runs: it's compared using Object.is (reference equality for objects/functions) between the previous and current render's deps array. React stores the effect as a hook node on the fiber's hook list, with the previous deps cached in the hook node's memoizedState. The cleanup function (returned from the effect) is called before re-running the effect and on unmount — this is not optional, it's the mechanism React uses to prevent resource accumulation.

How it works internally

During the render phase, useEffect creates an effect object and attaches it to the fiber's updateQueue. It does NOT run during render. After commit, React collects all passive effects from the committed fiber tree and schedules them via the Scheduler as a single flushPassiveEffects task. This task runs cleanup for effects whose deps changed (calling the previous render's cleanup), then runs new setup callbacks. The ordering: all cleanups first (across the component tree), then all setups. In StrictMode (development only), React intentionally runs setup→cleanup→setup to verify that cleanup correctly reverses setup — simulating what happens during fast refresh or concurrent mode interruption.

Subtopics

No dependency array

Effect runs after every render. Equivalent to componentDidMount + componentDidUpdate. Rarely the correct choice — causes excessive effect runs including on renders where nothing relevant changed.

Empty dependency array

Effect runs once on mount, cleanup on unmount. Equivalent to componentDidMount + componentWillUnmount. Common for subscriptions, event listeners with stable references.

Dependency comparison

React uses Object.is for each dep. Primitives compare by value. Objects/arrays/functions compare by reference. A new object literal {a:1} in props causes useEffect to re-run every render even if content is identical — referential instability.

Stale closure in effects

If an effect captures a value but that value is not in the deps array, it reads the stale value from the closure at the time the effect was created. The ESLint exhaustive-deps rule catches this — treat it as an error, not a warning.

Cleanup function

Returned function runs before the next effect invocation and on unmount. Must reverse all setup: remove listeners, cancel subscriptions, abort fetches, clear timers. Without cleanup, effects accumulate on each render cycle.

AbortController pattern

Standard pattern for cancellable fetch in useEffect: create AbortController in setup, pass signal to fetch, abort in cleanup. Prevents race conditions where a slow response from an old render overwrites a fast response from a newer render.

Code example
// Effect hook node in fiber (conceptual)
const effectHookNode = {
  memoizedState: {
    create: setupFn,        // The effect callback
    destroy: cleanupFn,     // Returned cleanup fn (from previous run)
    deps: [userId],         // Previous render's deps (Object.is compared)
    next: nextEffectNode,
    tag: HookPassive,       // vs HookLayout for useLayoutEffect
  }
};

// Race condition without AbortController
useEffect(() => {
  // ❌ User changes userId quickly: two fetches in flight
  // Slower old fetch may resolve AFTER newer fetch
  // Overwrites correct data with stale data
  fetchUser(userId).then(setUser);
}, [userId]);

// ✅ AbortController pattern — cancels stale requests
useEffect(() => {
  const controller = new AbortController();
  fetch(`/api/users/${userId}`, { signal: controller.signal })
    .then(r => r.json())
    .then(setUser)
    .catch(err => {
      if (err.name !== 'AbortError') setError(err);
    });
  return () => controller.abort(); // Cleanup: abort previous fetch on re-run
}, [userId]);

// Stale closure bug — missing dep
useEffect(() => {
  const id = setInterval(() => {
    console.log(count); // ❌ Always prints initial count — stale closure
  }, 1000);
  return () => clearInterval(id);
}, []); // ← Missing [count] in deps

// StrictMode lifecycle in dev:
// 1. Mount: setup() runs
// 2. React unmounts (simulated): cleanup() runs
// 3. React remounts: setup() runs again
// If your app breaks at step 2, cleanup is wrong
Common mistakes
Suppressing the react-hooks/exhaustive-deps eslint rule with a comment instead of fixing the actual stale closure. The rule exists to prevent the most common class of React bugs. Every suppression is technical debt.
Putting async functions directly as the effect callback: useEffect(async () => {...}) — the async function returns a Promise, but React expects the cleanup function (or undefined). The cleanup is never called correctly. Wrap with an immediately-invoked async inside the effect.
Not handling the race condition in data-fetching effects — this is so common that React's docs now recommend using a data-fetching library (React Query, SWR) instead of raw useEffect for fetching. The AbortController pattern is the manual solution.
Interview perspective
"Explain the race condition that can occur in useEffect data fetching and how to fix it."
If userId changes quickly, two fetches are in-flight. The slower response from the first (stale) fetch may resolve after the second (current) fetch, overwriting correct data. Fix: AbortController in setup, abort in cleanup. Or ignore stale responses via a boolean flag. Libraries like React Query handle this automatically.
"Why does StrictMode invoke effects twice in development?"
To simulate component unmount/remount (fast refresh, concurrent mode interruptions). Verifies that cleanup correctly reverses setup. If double-invocation breaks your app, it means your effect is not idempotent — you have a bug that will also manifest in production during navigation and HMR.

useMemo & useCallback

Referential equalityMemoizationPremature optimisation
Definition

useMemo(fn, deps) memoizes the return value of a function — it returns a cached value from the previous render unless a dep changed. useCallback(fn, deps) memoizes the function itself — it's literally useMemo(() => fn, deps). Both exist to stabilize values across renders for two specific purposes: (1) avoiding expensive recomputations, and (2) maintaining referential equality so that downstream consumers (React.memo, useEffect deps, child components) don't see a "new" value every render. The critical insight most developers miss: memoization has a cost — it's only beneficial when the cost of recomputing exceeds the cost of memoization overhead plus the benefit of skipping downstream re-renders.

How it works internally

Both hooks store their value in a hook node on the fiber's hook list. On each render, React compares the new deps array against the cached deps using Object.is for each element. If all deps pass (unchanged), the cached value is returned without calling the function. If any dep changed, the function is called, the new value stored, and new deps cached. The comparison happens every render regardless — you always pay the comparison cost. For useCallback, the function passed to it is created fresh every render (JS creates a new closure object on each render call), but React discards it and returns the previous function reference if deps haven't changed.

Subtopics

When useMemo helps

Expensive pure computations (sorting/filtering large arrays, complex derived data) that would run on every render but whose inputs (deps) change infrequently. Benchmark first — function call overhead is ~μs, useMemo comparison is also ~μs.

When useCallback helps

Stabilizing function references passed to: (1) memoized child components (React.memo), (2) useEffect/useCallback/useMemo dependency arrays. Without stable reference, the consumer sees a "new" function every render and bypasses memoization.

When they're premature optimization

Wrapping every value and function "just in case". Components that don't receive the value as a dep or that aren't wrapped in React.memo. Cheap computations (object property access, simple arithmetic). Creates cognitive overhead without measurable benefit.

Referential equality trap

Object literals and arrow functions in JSX create new references every render: <Child style={{color:'red'}} onClick={() => doThing()} />. Even if Child is memoized, new object/function references cause re-render. useMemo/useCallback stabilize these.

React Compiler (React 19)

Auto-inserts useMemo/useCallback at the right places via static analysis. Eliminates most manual memoization needs. Rules: pure components, no mutation of props/state. The compiler tracks "reactive values" and generates optimal memo wrapping.

Memoization cache invalidation

React may evict memoized values from cache (future feature for low-memory conditions). Don't use useMemo for values with side effects or that must persist — it's a performance hint, not a guarantee.

Code example
// useMemo — expensive computation
function ProductList({ products, searchTerm, sortBy }) {
  // ❌ Runs on EVERY render, including unrelated state changes
  const filtered = products
    .filter(p => p.name.includes(searchTerm))
    .sort((a, b) => a[sortBy] > b[sortBy] ? 1 : -1);

  // ✅ Only recomputes when products/searchTerm/sortBy change
  const filtered = useMemo(() =>
    products
      .filter(p => p.name.includes(searchTerm))
      .sort((a, b) => a[sortBy] > b[sortBy] ? 1 : -1),
    [products, searchTerm, sortBy]
  );
}

// useCallback — stabilizing for memoized child
const ExpensiveChild = React.memo(({ onSubmit }) => {
  console.log('child rendered'); // Should not run on parent re-render
  return <button onClick={onSubmit}>Submit</button>;
});

function Parent({ userId }) {
  const [unrelated, setUnrelated] = useState(0);

  // ❌ New function reference every render → React.memo bypassed
  const handleSubmit = () => submitForm(userId);

  // ✅ Stable reference when userId doesn't change
  const handleSubmit = useCallback(
    () => submitForm(userId),
    [userId]
  );

  return <>
    <button onClick={() => setUnrelated(n => n+1)}>Unrelated</button>
    <ExpensiveChild onSubmit={handleSubmit} /> // Only re-renders when userId changes
  </>;
}

// useCallback is useMemo returning a function:
useCallback(fn, deps) === useMemo(() => fn, deps)
Common mistakes
Memoizing a child component with React.memo but forgetting to stabilize prop functions with useCallback — React.memo does a shallow props comparison. A new function reference fails the comparison, re-rendering the child every time, negating the memo entirely.
Wrapping every function in useCallback "for safety" — adds cognitive load, increases bundle size marginally, and provides no benefit if the function is used in JSX without memo'd children or effect deps.
Using useMemo for primitive values — useMemo(() => count * 2, [count]) is pure overhead. useMemo is for object/function references and expensive computations. Primitive math is never the bottleneck.
Interview perspective
"When would you NOT use useMemo or useCallback?"
When the computation is cheap (primitives, simple operations). When the child isn't wrapped in React.memo. When the value isn't used in effect deps. The cost of memoization (comparison on every render, cognitive overhead, closure allocation) must be justified by measurable improvement. Profile first.
"What is referential equality and why does it matter for React optimization?"
JS objects/functions created in render have new references each time, even if content is identical. React.memo, useEffect, and downstream hooks compare by reference (Object.is). Without stable references from useMemo/useCallback, every render looks like a change to these consumers, bypassing all optimization.

React 18 features

Auto batchingstartTransitionuseDeferredValueSuspense
Definition

React 18 shipped three major categories of improvements: (1) Automatic batching — extending state update batching from synchronous event handlers to all async contexts; (2) Concurrent features — startTransition and useDeferredValue for marking updates as interruptible low-priority work; (3) Suspense improvements — Suspense for data fetching (not just code splitting) and server-side streaming SSR support. All concurrent features required the Fiber architecture (React 16+) to be possible, and are enabled by default in React 18 via the new createRoot API.

Subtopics

Automatic batching

Pre-18: multiple setStates in async callbacks (setTimeout, Promises) triggered separate renders. React 18: all setState calls are batched into one render regardless of context. Reduces renders for async event handlers from N to 1.

startTransition

Marks an update as non-urgent (TransitionLane). React can interrupt the transition render when higher-priority work (user input) arrives. The transition is discarded and restarted with the new state. UI stays responsive during expensive renders.

useDeferredValue

Defers a value to a lower-priority render. Like startTransition but for values rather than state updates — useful when you don't control the state setter (e.g., prop from parent). React renders with the old value first, then deferred value when idle.

useId

Generates a stable unique ID that matches between server and client renders (avoiding hydration mismatches from incrementing counters). Solves the SSR accessibility ID problem for form inputs and ARIA attributes.

Suspense for data

Components can throw a Promise; Suspense catches it and shows fallback until the Promise resolves. Previously only usable with React.lazy (code splitting). React 18 enables data-fetching frameworks (Next.js, Relay) to integrate with Suspense boundaries.

Streaming SSR

renderToPipeableStream allows the server to stream HTML progressively. Suspense boundaries that aren't ready are sent as fallbacks; when data resolves, the resolved HTML is streamed and replaces the fallback via a script tag. True streaming, not just blocking HTML generation.

Code example
// Automatic batching — React 18
async function handleSubmit() {
  const result = await submitForm(data);
  // React 17: two separate re-renders below
  // React 18: batched into ONE re-render ✓
  setSubmitting(false);
  setResult(result);
}

// startTransition — keep input responsive during slow render
function SearchInput() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);

  function handleChange(e) {
    setQuery(e.target.value);               // Urgent: update input immediately
    startTransition(() => {
      setResults(filterResults(e.target.value)); // Non-urgent: can be interrupted
    });
  }
  // If user types fast: input update runs, transition renders discard + restart
  // Input field stays responsive; results may lag but never block typing
}

// useDeferredValue — defer a prop you don't control
function ResultList({ query }) {
  const deferredQuery = useDeferredValue(query);
  // React renders ResultList twice:
  // 1st: synchronous render with old deferredQuery (fast, no flicker)
  // 2nd: deferred render with new query when idle
  const items = useMemo(() => expensiveFilter(deferredQuery), [deferredQuery]);
  // Show stale=dimmed while deferred render is in progress
  const isStale = query !== deferredQuery;
  return <div style={{ opacity: isStale ? 0.5 : 1 }>{items}</div>;
}

// Suspense for data (framework integration)
function UserProfile({ userId }) {
  // use() (React 19) or Relay/Next.js data integrations throw Promises
  const user = use(fetchUser(userId)); // Suspends until resolved
  return <h1>{user.name}</h1>;
}
// Parent wraps with Suspense boundary:
<Suspense fallback={<Skeleton />}><UserProfile userId={id} /></Suspense>
Common mistakes
Using startTransition for urgent updates — only wrap updates that produce a heavy, interruptible render. Navigation, form submissions, and input changes should remain urgent (SyncLane). Only filter/sort/search result rendering is a good transition candidate.
Not migrating to createRoot from ReactDOM.render — concurrent features are entirely disabled in legacy render mode. The migration is one line change but is a prerequisite for React 18 features.
Using useDeferredValue without useMemo — useDeferredValue gives you a stale value, but if the component re-renders with the deferred value and the computation isn't memoized, you lose the performance benefit. Always pair useDeferredValue with useMemo.
Interview perspective
"What is the difference between startTransition and useDeferredValue?"
startTransition wraps a state setter call — use when you control the state. useDeferredValue wraps a value — use when you receive a prop you don't control. Both mark work as low-priority and interruptible. useDeferredValue also gives you the stale value for use during the pending render, allowing you to show the old result dimmed.
"How does automatic batching work in React 18 and what changed from 17?"
React 17: batching only in synchronous event handlers. Inside setTimeout, Promises, async handlers — each setState triggered a separate render. React 18: all setState calls are batched regardless of where they occur. Achieved by React 18 tracking "execution context" more broadly via the Scheduler. Use flushSync() to opt out of batching when needed.

Context API performance

Consumer re-rendersContext splittingZustand tradeoff
Definition

React Context provides a way to pass values through the component tree without prop drilling. However, Context has a critical performance characteristic: when a context value changes, every consumer of that context re-renders, regardless of whether the part of the value they use actually changed. This is not a bug — it's a consequence of how Context compares values (single Object.is on the context value object). A single context containing both frequently-changing and rarely-changing data will cause unnecessary re-renders of all consumers on every change.

How it works internally

React propagates context changes by walking the fiber tree from the Provider downward. When it finds a fiber that consumes the context (has useContext or Context.Consumer), it marks that fiber as having pending work and schedules a re-render. The comparison is Object.is(prevContextValue, newContextValue) — a single reference comparison on the whole context value. This means if you do value={{ user, settings }} in JSX, a new object is created every render — all consumers re-render every time the Provider's parent renders, even if user and settings haven't changed. React does skip subtrees that are between the Provider and consumer if they're bailed out via memo — but the consumer itself always re-renders when the context value reference changes.

Subtopics

The re-render problem

All consumers of a context re-render when the context value changes. There's no selector mechanism built into React Context (unlike Redux useSelector). Every consumer gets the full value and must re-render to "see" it, even if the specific field it reads hasn't changed.

Context splitting

Separate frequently-changing values (e.g. current user's notification count) from rarely-changing values (e.g. theme, locale) into different contexts. Consumers subscribe only to the context they need — theme consumers don't re-render when notification count updates.

Stable value reference

Memoize the context value with useMemo to avoid re-creating the object on every parent render. Memoize setter functions with useCallback. Without this, every render of the Provider's parent creates a new context value object, triggering all consumers.

State/dispatch split pattern

Put mutable state in one context, dispatch (setter) in another. Components that only dispatch never subscribe to state changes — they always have the stable dispatch context value. Pattern popularized by Kent C. Dodds.

Zustand vs Context

Zustand uses an external store with selector-based subscriptions. Components subscribe to slices of state; they only re-render when their specific slice changes. No Provider needed. Better for frequently-changing global state. Context better for static/near-static config (theme, locale, auth user).

use(Context) in React 19

The new use() API allows reading Context inside conditional logic (unlike useContext). Doesn't change the re-render behavior — all consumers still re-render on value change. Adds ergonomic flexibility, not performance improvement.

Code example
// ❌ One large context — all consumers re-render on ANY change
const AppContext = createContext();
function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');
  const [notifications, setNotifications] = useState([]);
  // New object every render → ALL consumers re-render when notifications update
  return <AppContext.Provider value={{ user, theme, notifications, setUser }}>
    {children}
  </AppContext.Provider>;
}

// ✅ Context splitting + stable refs
const ThemeContext = createContext();    // Changes rarely
const UserContext = createContext();     // Changes on login/logout
const NotifContext = createContext();    // Changes frequently

function NotifProvider({ children }) {
  const [notifs, setNotifs] = useState([]);
  // useMemo prevents new value object on unrelated parent re-renders
  const value = useMemo(() => ({ notifs, setNotifs }), [notifs]);
  return <NotifContext.Provider value={value}>{children}</NotifContext.Provider>;
}
// Theme consumers never re-render when notifs change — different contexts

// Zustand — selector-based subscriptions
const useStore = create((set) => ({
  user: null, count: 0,
  increment: () => set(s => ({ count: s.count + 1 })),
}));
// Only re-renders when `count` changes, not when `user` changes
const count = useStore(state => state.count);
// Shallow selector: only re-renders when selected slice changes by Object.is
Common mistakes
Passing an inline object as context value: value={{ user, setUser }} — creates a new object reference on every Provider render, causing all consumers to re-render. Always memoize the context value object.
Using Context as a replacement for all state management. Context is a propagation mechanism, not a state management system. It has no batching, no selectors, no devtools, no derived state. Use for genuinely global, infrequently-changing values.
Wrapping the entire app in a single Provider that holds all state — the "God context" anti-pattern. Every state change re-renders every consumer in the entire app.
Interview perspective
"Why does React Context cause performance issues and how do you fix them?"
Context uses a single Object.is comparison on the value — any reference change triggers all consumers to re-render. Fix: (1) split into multiple contexts by change frequency, (2) memoize the value object, (3) split state and dispatch into separate contexts, (4) consider Zustand for frequently-changing shared state that needs subscriptions.
"When would you choose Zustand over React Context?"
When state changes frequently (counters, real-time data, filters), when you need selector subscriptions (only re-render when specific slice changes), when you want devtools integration, or when Context re-render propagation is causing measurable performance issues. Context remains appropriate for theme, locale, auth user, and feature flags.

Server components vs client components

RSCSerialization boundaryZero bundleComposition
Definition

React Server Components (RSC) is an architecture where components are divided by where they execute: server components run only on the server (or at build time), have direct access to databases, file systems, and secrets, and their output is serialized as a special RSC payload — not HTML, not JSON, but a React-specific format that includes rendered subtrees and references to client component placeholders. Client components are traditional React components that run in the browser, can use hooks, event handlers, and browser APIs. The boundary between them is explicit: a "use client" directive at the top of a file marks it as a client component. The architectural shift is from "component as a JavaScript unit" to "component as a rendering location" — some components live exclusively on the server and ship zero JavaScript to the client.

How it works internally

The RSC protocol defines a serialized wire format (the "RSC payload") that the server sends to the client. This is not HTML — it's a streaming JSON-like format that describes the React tree, distinguishing server-rendered subtrees (sent as pre-rendered markup) from client component boundaries (sent as references to client component modules + their props). The client React runtime receives this payload and hydrates only the client components — server component subtrees are never re-executed on the client. Props passed across the server-client boundary must be serializable: JSON-compatible values, Promises, and special objects like Blobs. Functions (closures) and class instances cannot cross the boundary — they're not serializable. The "use client" directive triggers the bundler (webpack/turbopack) to mark that module as a client boundary: the bundler ships the module to the client bundle and replaces the import in the server bundle with a reference placeholder.

Subtopics

Server component capabilities

Direct database/ORM queries, file system access, environment secrets, async/await at the component level, access to server-only modules. No hooks, no event handlers, no browser APIs, no component state.

Client component capabilities

useState, useEffect, event handlers (onClick, onChange), browser APIs (window, localStorage, geolocation), real-time data (WebSocket, EventSource). Cannot directly access database or secrets. Marked with "use client".

Serialization boundary

Props crossing server→client boundary must be serializable: strings, numbers, arrays, plain objects, Dates, Promises. Functions (including event handlers) cannot be serialized. Server actions ("use server") are the mechanism for server-side mutations triggered by client events.

Server actions

Functions marked "use server" can be called from client components — they're serialized as a POST request to a server endpoint. Enables form actions, mutations, and RPC-style calls without manually writing API routes.

Composition pattern

Server components can pass client components as children (via props.children). This allows interleaving: <ServerWrapper><ClientInteractive /></ServerWrapper>. The server component renders the wrapper; the client component handles interactivity. Critical pattern for performance.

Zero JS for server components

Server components ship no JavaScript to the client browser. A complex data-fetching component that renders a table from a DB query contributes 0 bytes to the client bundle. Only "use client" modules are bundled and shipped.

// Server component — no "use client", runs only on server

  // Direct DB access — no API route needed, no exposed secret
  const metrics = await db.query(`SELECT * FROM metrics WHERE date > NOW() - INTERVAL '7d'`);
  // Ships ZERO JS to client — this component doesn't exist in the browser bundle
  return (
    <main>
      <MetricsTable data={metrics} />   {/* Also a server component */}
      <FilterControls />               {/* "use client" — client component */}
    </main>
  );
}

  const [filter, setFilter] = useState('all'); // ✓ Hooks allowed
  return <select onChange={e => setFilter(e.target.value)}>...</select>;

Common mistakes
Trying to pass non-serializable props across the server/client boundary — functions, class instances, Map/Set objects, React component references. The bundler and runtime will throw. Design the boundary around serializable data shapes.
Marking too many components as "use client" — entire component trees become client-side, losing the benefits of RSC. The "use client" directive marks a boundary; everything imported from that file is also a client component. Push client boundaries as far down the tree as possible.
Confusing RSC with SSR — SSR renders HTML on the server for the initial load, then hydrates to make it interactive. RSC is a persistent server-rendered component model where components never ship JS to the client. Both can be used together (and are in Next.js App Router).
Interview perspective
"What is the difference between a React Server Component and SSR?"
SSR renders components to HTML on the server for every request, then ships the same component's JS to the client for hydration. RSC components run only on the server, ship no JS to the client, and output a special RSC payload rather than HTML. They can be composed with client components. RSC reduces client bundle size; SSR improves initial HTML delivery. Next.js App Router uses both.
"What can and cannot cross the server/client boundary?"
Can cross: strings, numbers, arrays, plain objects, Dates, Promises (for Suspense), JSX (children prop). Cannot cross: functions, class instances, Map/Set, non-serializable objects, React hooks. Server actions (use server) are the escape hatch for calling server logic from client events.

React rendering patterns

CSRSSRSSGISR
Definition

Rendering pattern describes where and when a React application's HTML is generated. This fundamentally determines performance characteristics (TTFB, LCP, TTI), data freshness, infrastructure requirements, and SEO behavior. The four canonical patterns form a spectrum from fully dynamic to fully static: CSR (client-side rendering — browser builds HTML from JS), SSR (server-side rendering — server generates HTML per request), SSG (static site generation — HTML generated at build time), and ISR (incremental static regeneration — SSG with background revalidation). In Next.js App Router, these are no longer page-level choices — they're per-fetch decisions, enabling individual components to use different data strategies.

Subtopics

CSR (Client-Side Rendering)

Server sends empty HTML + JS bundle. Browser downloads, parses, executes JS, fetches data, renders HTML. TTFB is fast (tiny HTML), but LCP is slow (waits for JS + data). No SEO for crawlers that don't execute JS. Suits SPAs, dashboards, auth-gated pages where SEO doesn't matter.

SSR (Server-Side Rendering)

Server generates full HTML per request, sends to browser. Fast FCP and LCP. TTFB depends on server render time. JS hydration still needed for interactivity (TTI may lag behind LCP = "uncanny valley"). Suits highly dynamic, user-personalized pages.

SSG (Static Site Generation)

HTML generated at build time, served from CDN. Near-zero TTFB (CDN edge). Best possible LCP. Zero server compute per request. Data can be stale (as old as last build). Suits content that changes infrequently: blogs, docs, marketing pages.

ISR (Incremental Static Regeneration)

SSG pages are revalidated in the background after a TTL expires. First request after TTL: serves stale HTML, triggers background regeneration. Next request: gets fresh HTML. stale-while-revalidate at the page level. Suits e-commerce product pages, news articles, prices.

Partial prerendering (PPR)

Next.js 14+ experimental. A page renders a static shell at build time (fast, cacheable) with Suspense boundaries as dynamic holes. Dynamic holes are streamed in per-request. Best of SSG + SSR in one page.

Hydration & TTI gap

SSR/SSG send pre-rendered HTML (good FCP/LCP) but the page is not interactive until hydration completes (JS downloaded, parsed, React tree reconciled). During this gap the page looks ready but doesn't respond to clicks. Large JS bundles worsen this gap.

Performance implications per pattern

TTFB

SSG/ISR: ~10ms (CDN edge). CSR: ~50ms (tiny HTML, fast network). SSR: 200ms–2s (server render time + network). PPR: ~10ms (static shell from CDN).

LCP

SSG best (pre-rendered, CDN). SSR good (full HTML on arrival). CSR worst (JS must load and run first). ISR = SSG after first revalidation cycle.

TTI

CSR: worst (all JS upfront). SSR/SSG with small bundles: best. Code splitting + streaming SSR reduces the hydration cost. RSC reduces bundle size (zero JS for server components).

Data freshness

SSR: always fresh (per-request). ISR: stale up to TTL. SSG: stale until rebuild. CSR: fresh on each client-side fetch. ISR + manual revalidation (on-demand ISR) = fresh on content change.

Code example
// Next.js App Router — per-fetch rendering strategy

// SSG: no-store = never cache, cache: 'force-cache' = build-time cache
const staticData = await fetch('/api/static', {
  cache: 'force-cache'     // SSG — cached at build time, served from CDN
});

// ISR: revalidate after N seconds (stale-while-revalidate)

  next: { revalidate: 60 }  // Revalidate every 60s in background
});

// SSR: no-store = always fresh, bypass all caches
const dynamicData = await fetch('/api/user-cart', {
  cache: 'no-store'         // SSR — fresh on every request
});

// On-demand ISR revalidation — trigger rebuild from webhook
// app/api/revalidate/route.ts
import { revalidatePath, revalidateTag } from 'next/cache';
export async function POST(req) {
  revalidateTag('products'); // Invalidate all fetches tagged 'products'
  return Response.json({ revalidated: true });
}

function Dashboard() {
  const { data } = useSWR('/api/realtime-metrics', fetcher, {

Trade-offs decision matrix

Use SSG/ISR when

Content is the same for all users. Can tolerate some staleness. Need maximum performance and low cost. Examples: product catalog, documentation, marketing pages, blog posts.

Use SSR when

Page is personalized per user (session data, cart, permissions). Data must be real-time fresh. SEO matters. Examples: user dashboards, checkout, search results with personalization.

Use CSR when

Auth-gated content (no SEO needed). Highly interactive UIs where data changes in real-time (trading apps, live collaboration). Content that's meaningless without JS anyway.

Use PPR when

Page has a cacheable shell (header, layout, non-personalized sections) with dynamic holes (cart count, user name, personalized content). Best of both worlds — static speed with dynamic content.

Common mistakes
Using SSR for content that's the same for all users — wastes server compute on every request. Static generation + CDN is always faster and cheaper for shared content.
Using SSG for user-personalized content — SSG can't vary per user at CDN level (without edge functions). Using SSG for cart pages or dashboards causes incorrect/empty content at CDN edges.
Ignoring the hydration TTI gap with SSR — SSR gives fast FCP but if the JS bundle is large, the page looks interactive but isn't for several seconds. This is worse UX than a slow-loading CSR app that only shows content when it's actually interactive.
Interview perspective
"When would you choose SSR over SSG for a Next.js page?"
SSR when content is user-specific (session-dependent), requires real-time data, or when the page changes faster than builds can keep up. SSG when content is shared across users and can tolerate some staleness. ISR bridges the gap: static speed + periodic freshness. In App Router, the decision is per-fetch, not per-page.
"What is the hydration gap and how does it affect UX?"
SSR/SSG delivers pre-rendered HTML quickly (fast FCP/LCP), but the page is not interactive until React hydrates it (JS bundle downloaded, parsed, component tree reconciled). During this gap, the page appears ready but ignores clicks/inputs. Reduce it with: smaller JS bundles, streaming SSR, selective hydration (Suspense), and RSC (removes server component JS from bundle).
"Design the rendering strategy for an e-commerce product page."
Product details (shared across users): ISR with 60s revalidation + on-demand revalidation from CMS webhook. Product images: CDN-cached SSG. Personalized section (cart count, recommended): client-side hydrated or PPR dynamic hole. Inventory/pricing: could be ISR or SSR depending on how quickly it changes. Reviews: ISR. Checkout flow: SSR (session data required).
