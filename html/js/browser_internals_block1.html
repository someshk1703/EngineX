Browser Architecture

LCPTTFBINPSecurity
Definition

Modern browsers (Chrome, Edge) use a multi-process architecture. Rather than running the entire browser in one process (which meant a single rogue tab could crash the browser), Chrome's engineers split functionality across isolated OS processes: a Browser Process (the coordinator), one or more Renderer Processes (one per site, where your JS and DOM live), a GPU Process (handles compositing and drawing), a Network Process (fetches resources, independent of tabs), and Utility Processes (audio, extensions). Each process has its own memory space. Communication happens over IPC (inter-process communication) channels. This is Chromium's "Site Isolation" model, introduced as a full default post-Spectre (2018).

How it works internally

When you open a tab, the Browser Process spawns a Renderer Process for that site origin. The Renderer Process contains the Blink rendering engine (HTML/CSS parsing, DOM), the V8 JS engine, and the Compositor thread. These run the main thread and compositor thread within that process. The Browser Process intercepts navigation requests and hands them to the Network Process, which performs DNS lookup, TLS, HTTP — entirely outside the renderer. When a response arrives, the Browser Process decides whether it needs a new Renderer Process (different origin = new process under Site Isolation). The GPU Process receives "draw quads" (layer tiles) from the compositor and calls into OpenGL/Vulkan/Metal to push pixels. The main thread and compositor thread are the two most important threads you'll be profiling. Long tasks on the main thread block rendering. Work on the compositor thread (CSS transforms, opacity) is fast because it never touches the main thread.

Subtopics

Browser Process

Owns the UI chrome, address bar, tab management. Coordinates all other processes. Runs as a privileged OS process.

Renderer Process

One per site origin (Site Isolation). Contains Blink + V8. Sandboxed — can't access disk or network directly. This is where your code runs.

GPU Process

Single shared process. Receives rasterized tiles and composites them. Handles hardware-accelerated layers. GPU crashes don't take down tabs.

Network Process

Owns all HTTP, WebSocket, fetch. Applies caching policies. Handles cookies, CORS, HSTS. Isolated from renderer for security.

Sandboxing

Renderer processes run in OS sandbox (no syscalls to disk/network). Exploiting a JS bug can't escalate without a separate sandbox-escape exploit. This is why browser vulns chain two CVEs.

Site Isolation

Enabled post-Spectre. Cross-origin iframes get their own Renderer Process. Prevents one origin from timing-attacking another's memory via speculative execution.

Performance implications

Process isolation adds IPC overhead — postMessage across origins, structured-clone serialization cost. Memory cost: each Renderer Process has its own V8 heap, which inflates RAM per tab. TTFB: Network Process has dedicated resources — parallel DNS preresolution is possible because the Network Process isn't blocked by JS. INP: Long tasks on the main thread (Renderer Process) directly tank INP — the OS can't preempt it. LCP: Resource fetching in the Network Process overlaps with parsing in the Renderer, enabling the "preload scanner" to fetch ahead of parser blockage.

Real-world architecture use case

Micro-frontends & iframes at scale

In micro-frontend architectures (e.g., Spotify's Web Player, Google Docs plugin system), different features load in different origins inside iframes. Post-Site Isolation, each iframe origin gets its own Renderer Process. This means one sluggish micro-frontend can't block another's main thread. Cross-MFE communication via postMessage pays an IPC + structured-clone cost — smart teams serialize with MessagePack instead of JSON and use SharedArrayBuffer (requires COOP/COEP headers) for zero-copy data sharing between Renderer Processes.

Trade-offs
Process isolation increases RAM usage significantly. A browser with 20 tabs can consume 2-4GB because each renderer has its own V8 heap (~50-100MB baseline). Chrome's "Tab Freezing" and "Memory Saver" features are direct mitigations.
IPC between Browser/Renderer/GPU adds latency. Synchronous IPC calls (Mojo) block the calling thread. This is why Chrome moved rendering to be async-first — the compositor thread can draw without waiting on the main thread.
Site Isolation breaks some legitimate uses of document.domain for same-site cross-origin access. Legacy single-sign-on systems that depended on parent-frame DOM access are broken under full Site Isolation.
Common mistakes
Conflating "thread" and "process" — Web Workers are threads within the Renderer Process. Service Workers run in their own separate Renderer-like context (distinct process). Shared Workers can be shared across tabs of the same origin.
Assuming postMessage is free — structured clone is O(n) in data size. For large ArrayBuffers, use transfer: worker.postMessage(buf, [buf]). After transfer, the sender's reference is neutered.
Placing CPU-heavy work in a Service Worker for "performance" — SW runs on the network fetch path, blocking it. Heavy CPU work in SW delays resource delivery.
Interview perspective
"Why does Chrome use multiple processes instead of multiple threads?"
Expected depth: OS sandbox isolation, crash isolation, Spectre mitigation via Site Isolation. Mention Mojo IPC. Differentiate from multi-threaded design (no memory isolation between threads).
"What happens in the browser when you type a URL and press Enter?"
Classic. Most candidates stop at DNS → TCP → HTTP. Senior answer adds: Browser Process intercepts, Network Process DNS+TLS, Site Isolation check, new Renderer Process spawned, preload scanner fires, CRP begins. Connect to TTFB.
"How does browser architecture affect micro-frontend isolation?"
Practical system design bridge. Same-origin = same process, different origin = separate process. postMessage IPC cost. COOP/COEP for SharedArrayBuffer. Connects to micro-frontend architecture decisions.

Critical Rendering Path

LCPFCPCLSTTITTFB
Definition

The Critical Rendering Path (CRP) is the sequence of steps the browser must complete before it can paint the first pixel to screen: parse HTML → build DOM → parse CSS → build CSSOM → merge into Render Tree → Layout (geometry calculation) → Paint (pixels into layers) → Composite (layers merged onto screen). The length of this pipeline — measured in bytes, round trips, and compute time — directly determines First Contentful Paint (FCP) and Largest Contentful Paint (LCP). Optimizing the CRP means either reducing resources that block it, deferring work that's off the critical path, or shortening the network round-trip chain.

How it works internally

The HTML parser (in Blink) is incremental — it emits DOM nodes as bytes arrive. When it encounters a <script> without async/defer, it stops parsing and waits for the script to download and execute (because JS can call document.write or modify the DOM). CSS is "render-blocking" but not "parser-blocking" — the parser keeps running, but the browser will not build the Render Tree until CSSOM is complete (because JS may query computed styles). The preload scanner is a second lightweight parser that runs ahead of the main parser, speculatively fetching <link>, <script>, and <img> tags while the main parser is blocked. Layout computes the exact position and size of every box (box model, floats, flexbox, grid) in a tree traversal. Paint records drawing commands into a display list. Compositing splits the page into GPU-uploaded layers and the GPU blends them — this step can happen on the compositor thread, off the main thread.

Subtopics

DOM Construction

Bytes → tokens → nodes → DOM tree. Incremental. Parser halts on synchronous <script>.

CSSOM Construction

CSS is not incremental for rendering — partial CSSOM is unusable. Render tree waits for full CSSOM. All stylesheets (not media="print") are render-blocking.

Render Tree

Merges DOM + CSSOM. Excludes display:none elements (visibility:hidden stays). Maps visible nodes to computed styles.

Layout (Reflow)

Box model geometry for every visible node. Block formatting context. Percentage → absolute px resolution. A full layout traversal is O(n) but partial dirty-region relayouts are targeted.

Paint

Converts Render Tree to display list commands (fill, stroke, shadow). Happens per layer. Not actual pixel rasterization yet.

Compositing

Layers are rasterized (on raster threads or GPU), then the GPU compositor blends them. transform, opacity animations skip layout+paint entirely.

Render-blocking resources

Synchronous <script> in <head>, non-media CSS. Use async/defer/rel=preload to get off the critical path.

Preload Scanner

Lookahead parser. Fires <link rel=preload> equivalent for fonts, images, scripts before main parser reaches them. Critical for waterfall optimization.

Performance implications

LCP is the primary victim: every render-blocking resource adds one RTT minimum before any paint. CLS: layout shifts happen when images without dimensions push content, or when late-loading fonts cause FOUT (Flash of Unstyled Text), triggering relayout. TTI: large JS bundles parsed synchronously delay interactivity. FCP: inlining critical CSS removes a round trip. The CRP length is roughly: RTTs_for_critical_resources × RTT_latency + parse_time + layout_time. On mobile (150ms RTT), every additional blocking resource costs 300ms (request + response).

Code example

<head>
  <link rel="stylesheet" href="/all-styles.css">  <!-- render-blocking -->
  <script src="/app.js"></script>              <!-- parser-blocking -->
</head>


<head>
  <!-- 1. Inline critical CSS (above-fold only) -->
  <style>/* critical path styles only */</style>

  <!-- 2. Preload LCP image — tells preload scanner early -->

  <!-- 4. Lazy-load non-critical CSS via JS trick -->
  <link rel="stylesheet" href="/rest.css" media="print"
    onload="this.media='all'">

Real-world architecture use case

Next.js streaming SSR + React Suspense

Next.js 13+ uses HTTP streaming (chunked transfer encoding) to send HTML shell → flush critical above-fold HTML → stream in Suspense boundaries as they resolve. This maps directly to CRP optimization: the shell contains inlined critical CSS and sets up the render tree immediately. LCP element (e.g. hero image) arrives in the first flush with fetchpriority="high". Non-critical content arrives in later chunks, avoiding blocking the initial paint. The network process receives chunks, the renderer parses and paints progressively — true parallel CRP.

Trade-offs
Inlining critical CSS blooms HTML payload size and breaks long-term caching of CSS files. Requires build-time extraction (Critters, penthouse) and must be regenerated when above-fold styles change.
rel=preload on the wrong resource wastes bandwidth and can compete with the actual LCP resource. Preloading everything is worse than preloading nothing — priority scheduler gets confused.
Streaming SSR with Suspense requires careful data-fetching architecture. Waterfalls in server-side fetches delay chunk streaming and negate the CRP benefit.
Common mistakes
Treating all CSS as render-blocking — media="print" CSS is not render-blocking. Adding media attributes to non-applicable CSS files removes them from the critical path.
Using async on scripts that depend on the DOM or on other scripts — async executes immediately on download, which may be before DOMContentLoaded. defer respects document order and waits for parsing to complete.
Not setting explicit width/height on images — causes CLS because the browser can't reserve layout space. The browser computes the aspect ratio box from HTML attributes before the image loads.
Forgetting that rel=preload without as= is useless — the browser doesn't know the content type and uses the wrong priority queue.
Interview perspective
"Walk me through everything that happens from the moment an HTML byte arrives to pixels on screen."
Don't skip steps. Bytes → tokenize → DOM nodes → CSSOM parallel build → Render Tree merge → Layout (box model) → Paint (display list) → Composite (GPU layers). Call out what blocks what.
"Why is CSS render-blocking but not parser-blocking?"
Key insight: the parser keeps going (preload scanner also fires), but Render Tree construction halts. Reason: JS can query computed styles (getComputedStyle), so CSSOM must be ready before any JS runs that follows a stylesheet.
"How would you optimize LCP for a Next.js e-commerce product page?"
System design answer: SSR with streaming, inline critical CSS, fetchpriority=high on hero image, preconnect to image CDN, aggressive image compression (AVIF/WebP), edge caching for HTML (low TTFB), remove render-blocking third-party scripts.

Reflow vs Repaint vs Compositing

INPCLSFrame budgetJank
Definition

After the initial CRP, subsequent changes to the DOM or styles trigger one of three update pipelines, each with dramatically different costs. Reflow (Layout): geometry must be recalculated — the browser traverses the DOM subtree recomputing positions, widths, heights. Repaint: geometry is unchanged but visual appearance changes (color, background, visibility) — the browser re-executes paint commands for affected layers without touching layout. Compositing: only layer transform or opacity changes — the GPU repositions or blends pre-rasterized layer tiles without touching the CPU-side pipeline at all. These three form a strict cost hierarchy: Reflow > Repaint > Compositing. The goal is to keep animations and interactions on the compositor thread, off the main thread.

How it works internally

Blink maintains a dirty bit system. When a style or attribute changes, affected nodes are marked dirty. Before the next paint frame, Blink resolves dirty nodes bottom-up. If any geometry property changed (width, height, top, margin, padding, font metrics), it triggers a Layout pass — potentially reflowing the entire document if the change affects block formatting contexts. After Layout, Blink compares what changed against cached paint records; only dirty layers are repainted into their display lists. These display lists are then rasterized (on raster threads, via SkiaRenderer or Vulkan). The Compositor thread then takes the rasterized tiles and, for compositor-only properties (transform, opacity, filter on promoted layers), it can apply changes directly — the GPU repositions or alpha-blends tiles without involving the main thread at all. This is why transform: translateX(100px) is 60fps even when the main thread is busy.

Subtopics

Layout Thrashing

Reading a layout property (offsetWidth) after a style write forces synchronous reflow mid-JS. Blink must flush the dirty queue immediately to return an accurate value.

Layer Promotion

Elements get their own GPU layer (promoted) when: will-change: transform, position: fixed, 3D transforms, <video>/<canvas>, or overlapping elements with opacity. More layers = more GPU memory, not always faster.

Compositor-only properties

transform and opacity on promoted layers skip Layout and Paint entirely. Ideal for animations. filter can be compositor-only if layer is promoted.

Invalidation regions

Blink tracks which parts of which layers are dirty. Only invalid regions are repainted. Changing a deeply nested element's color only repaints that layer's dirty rect.

requestAnimationFrame

rAF callbacks fire before the browser's next paint, after all event handlers. Batching DOM reads and writes inside rAF prevents layout thrashing.

FastDOM / Batching

Pattern for preventing layout thrashing: batch all DOM reads together (layout flushed once), then batch all writes. Never interleave read-write-read-write.

Performance implications

INP: Layout thrashing during event handlers directly bloats interaction latency. A single el.offsetWidth read after a style write can force a synchronous 5-20ms reflow. CLS: Reflows that shift content (late image loads, dynamically inserted banners) cause layout shifts measured by CLS. Frame budget: At 60fps, each frame has 16.7ms. Layout + Paint + Composite must fit. A reflow of a complex DOM can easily exceed 10ms alone. Compositor-thread animations are immune to main-thread jank — they have their own 60fps budget independent of the main thread. This is why GSAP, Framer Motion, and CSS transitions prefer transform.

Code example
// ❌ Layout thrashing — forces synchronous reflow 3 times
function badUpdate(elements) {
  elements.forEach(el => {
    const w = el.offsetWidth;     // READ: forces reflow #1
    el.style.width = w + 10 + 'px'; // WRITE
    const h = el.offsetHeight;    // READ after write: reflow #2
    el.style.height = h + 10 + 'px';
  });
}

// ✅ Batch reads then writes
function goodUpdate(elements) {
  const dims = elements.map(el => ({     // All reads first
    w: el.offsetWidth, h: el.offsetHeight
  }));
  elements.forEach((el, i) => {           // Then all writes
    el.style.width  = dims[i].w + 10 + 'px';
    el.style.height = dims[i].h + 10 + 'px';
  });
}

// ✅ Compositor-only animation — zero layout/paint cost
// CSS: .slide { transition: transform 300ms ease; will-change: transform; }
// JS:
el.style.transform = `translateX(${offset}px)`;
// Animator runs on compositor thread, main thread untouched

// ❌ Equivalent but causes reflow every frame:
el.style.left = offset + 'px'; // triggers layout → paint → composite
Real-world architecture use case

Virtual scroll in large data tables (e.g. AG Grid, TanStack Virtual)

Rendering 100k rows in a DOM would trigger catastrophic reflows on every scroll event. Virtual list libraries maintain a fixed DOM window (~30 rows) and update only which rows are visible. Critically: they use transform: translateY to position the content container — this is a compositor-only operation. The scroll event calculates new visible range (JS, main thread), updates DOM nodes (write batch), and sets the container's translateY. The translation itself happens on the GPU compositor — the visible repositioning is smooth even if the JS data-swapping takes 5ms. This is the canonical "hybrid" performance pattern: main thread for logic, compositor for motion.

Trade-offs
Layer promotion (will-change: transform) consumes GPU memory per layer. Promoting every element is a memory leak pattern — GPU memory is limited (especially mobile). Only promote elements that actually animate.
Over-use of will-change hints causes the browser to pre-allocate layer backing before any animation, increasing memory pressure and potentially hurting performance on low-end devices more than the animation would have.
CSS contain: strict creates layout isolation but prevents elements inside from contributing to the document's scroll height — a footgun in dynamic content containers.
Common mistakes
Using position: left/top for animations instead of transform: translate — left/top trigger full Layout + Paint on every frame. This is the single most common cause of animation jank in production apps.
Reading scrollTop, getBoundingClientRect(), or any computed geometry inside a rAF write phase — these force a synchronous layout flush mid-frame, wasting the entire frame budget.
Animating box-shadow or border-radius on a large element — these trigger repaint on every frame (not compositor-only). Fade a pseudo-element with opacity instead.
Forgetting that filter only avoids repaint when the element is on its own compositor layer. Without promotion, filter changes force repaint.
Interview perspective
"What CSS properties can you animate without triggering layout or paint?"
Answer: transform and opacity (on promoted layers). Everything else causes at minimum repaint. The follow-up is always "why" — compositor thread, GPU tile blending, no main thread involvement.
"What is layout thrashing and how do you fix it?"
Interleaved DOM reads and writes that each force synchronous reflow. Fix: batch reads, then writes. Use rAF. FastDOM library pattern. Interview often follows up with "show me code" so be ready.
"How does will-change work, and when should you NOT use it?"
Creates a new stacking context and compositor layer. Don't use it on static elements, don't use it on too many elements simultaneously, don't leave it on after animation ends. Add/remove dynamically around animation lifecycle.

JavaScript Engine — V8 Internals

INPTTIParse timeJIT
Definition

V8 is Google's open-source JavaScript (and WebAssembly) engine, used in Chrome and Node.js. It compiles JS to machine code using a two-tier pipeline: Ignition (an interpreter that generates and executes bytecode) and TurboFan (an optimizing JIT compiler that compiles "hot" functions to highly optimized machine code). V8 also uses hidden classes and inline caches to give dynamically-typed JS the performance characteristics of statically-typed languages — when your objects have consistent shapes. V8 is why "write predictable, shape-consistent objects" is a performance rule, not just a style preference.

How it works internally

Source JS → Scanner + Parser → AST → Ignition generates bytecode and begins interpreting. V8 profiles which functions are called frequently ("hot functions"). Hot functions are passed to TurboFan, which uses speculative optimization: it assumes the types it's seen so far are stable and generates type-specialized machine code (e.g., assuming x is always a Smi — small integer). This code runs at near-native speed. If the type assumption breaks (x suddenly becomes a string), TurboFan deoptimizes — throws away the compiled code and falls back to Ignition. Hidden classes (Shapes): V8 assigns a "shape" to each object based on its property layout. Objects with the same properties in the same order share a shape, allowing V8 to store properties at fixed offsets (like a C struct) rather than doing hash lookups. Inline caches (ICs): call sites remember the shapes they've seen. A monomorphic IC (one shape) is fastest; polymorphic (2-4 shapes) is slower; megamorphic (5+ shapes) falls back to hash lookup.

Subtopics

Ignition (Interpreter)

Generates compact bytecode. Starts executing immediately. Collects type feedback. Lower startup overhead than compiling everything upfront.

TurboFan (JIT)

Sea-of-nodes IR compiler. Speculative optimization based on Ignition's type feedback. Inlines functions, eliminates bounds checks, vectorizes loops.

Hidden Classes (Shapes/Maps)

Objects with the same property order share a shape. Shape transitions occur when properties are added out of order. Shapes enable fixed-offset property access — critical for JIT perf.

Inline Caching

Property access sites cache the last shape(s) seen. Monomorphic → Polymorphic → Megamorphic. Megamorphic = generic hash map lookup. Shape consistency = monomorphic ICs = fast.

Deoptimization

When a type assumption is violated, TurboFan discards compiled code, re-executes in Ignition, relearns types, and may recompile. Deopts appear in Chrome's "Optimize JavaScript" DevTools panel.

Heap Zones

V8 heap: Young generation (nursery + intermediate — short-lived objects), Old generation (survived GC cycles), Large object space (>256KB), Code space (compiled bytecode/machine code).

Performance implications

TTI: Script parse and compile time is often ignored. V8 must parse all JS before executing it (unless code splitting is used). A 500KB bundle takes ~300-500ms to parse on mobile. INP: Functions that repeatedly deoptimize add latency on every call. JIT-compiled code is 10-100x faster than interpreted bytecode. Memory: Shape explosion (many object shapes) inflates V8's internal metadata structures. Each shape has a transition tree — pathological code can generate thousands of shapes for what looks like simple object creation.

Code example
// ❌ Shape inconsistency — breaks hidden class optimization
function makePoint(x, y, hasZ) {
  const p = { x, y };
  if (hasZ) p.z = 0; // Added conditionally → two different shapes
  return p;            // ICs see 2 shapes → polymorphic
}

// ✅ Shape consistent — always same property order
function makePoint(x, y, z = 0) {
  return { x, y, z }; // Always same shape → monomorphic ICs
}

// ❌ Type confusion — forces deoptimization
function add(a, b) { return a + b; }
add(1, 2);       // Smi + Smi → TurboFan optimizes for integers
add('a', 'b');   // String + String → DEOPT → recompile

// ✅ Monomorphic call site — stable types
function addInt(a, b) { return (a | 0) + (b | 0); } // Force int32

// Check deopt reasons in Node.js:
// node --trace-deopt --trace-opt yourScript.js
// In Chrome: DevTools → Performance → "Optimize JavaScript"
Real-world architecture use case

React's reconciler and V8 shape optimization

React's Fiber node objects are created with consistent property layouts — the team explicitly designed Fiber to have stable shapes so V8's hidden class system gives monomorphic ICs on Fiber property accesses. In the React source, you'll see properties initialized to null even when not used — this ensures every Fiber has the same shape from construction. This is not cargo-culted OOP: it's a deliberate V8 optimization. Similarly, Redux's reducers returning plain objects with the same shape every time ensure the ICs on state.someKey stay monomorphic across renders.

Trade-offs
V8's speculative compilation trades correctness for speed, then corrects when wrong. In type-diverse code (third-party integrations, dynamic APIs), deopts are unavoidable — don't over-optimize for V8 at the cost of code clarity.
Hidden class optimization only helps for object property access. Array access uses a separate element kind system (Smi array, Double array, generic array). Mixing types in arrays causes the array to become "holey" and lose fast-element optimizations.
Common mistakes
Deleting object properties with delete obj.key — this creates a "dictionary mode" object that abandons the hidden class system entirely. Set to undefined or null instead.
Mixing numbers and objects in arrays — [1, 2, 'three'] forces V8 to use a generic array element kind, losing Smi or Double array fast paths. Keep arrays homogeneous.
Assuming "modern JS = slow" or "classes are slow" — V8 heavily optimizes class syntax since shapes are more predictable with class than dynamic prototypal patterns.
Interview perspective
"How does V8 compile JavaScript to fast code?"
Walk: Ignition bytecode → type feedback → TurboFan speculation → deopt path. Hidden classes = shape stability = monomorphic ICs. This is a senior/staff-level question — most candidates stop at "JIT compiles it".
"Why is `delete` bad for performance?"
Dictionary mode — V8 demotes the object from its hidden class to a hash-map-backed slow object. Object ceases to benefit from shape optimization. All subsequent property accesses on that object are hash lookups.
"What does 'deoptimization' mean in V8 and what causes it?"
Type-specialized machine code gets invalidated when a type assumption is violated. V8 bails out to Ignition, re-profiles, may re-JIT. Common causes: type-polymorphic functions, mixed arrays, argument count mismatches.

Event Loop

INPTTIConcurrency model
Definition

The event loop is the mechanism by which JavaScript achieves concurrency despite running on a single thread. It continuously checks whether the call stack is empty and, if so, pulls the next task from its task queues and pushes it onto the stack. The critical architectural insight: JS itself is single-threaded, but the environment (browser, Node.js) is multi-threaded. Web APIs execute asynchronous work on separate threads and deposit callbacks into the event loop's queues. The event loop's job is sequencing these callbacks onto the JS thread in a defined priority order: microtask queue (Promises, MutationObserver) is drained completely before the next macrotask (setTimeout, setInterval, I/O, UI events). requestAnimationFrame sits between macrotask execution and the browser's rendering step.

How it works internally

One iteration of the event loop ("tick"): (1) Pick one macrotask from the task queue and execute it to completion on the call stack. (2) After the macrotask completes, drain the entire microtask queue — run every microtask, including any microtasks queued by microtasks, until the queue is empty. (3) If rendering is due (frame budget), the browser runs requestAnimationFrame callbacks, then performs style recalculation, layout, paint, composite. (4) Return to step 1. The HTML spec mandates that the microtask checkpoint runs after every task and after every script evaluation — which means Promise chains execute entirely before any setTimeout callback, even a setTimeout(fn, 0). In V8, microtasks are processed by v8::MicrotasksScope — a scope that flushes the microtask queue on scope exit.

Subtopics

Call Stack

LIFO stack of execution contexts. Functions push frames; return pops them. Stack overflow = exceeding maximum depth (~10k frames in V8). JS executes synchronously within a single stack frame.

Macrotask Queue (Task Queue)

setTimeout, setInterval, setImmediate (Node), I/O callbacks, UI events (click, input), MessageChannel. One macrotask per event loop iteration.

Microtask Queue

Promise callbacks (.then/.catch/.finally), queueMicrotask(), MutationObserver callbacks, async/await continuations. Entire queue drained between macrotasks. Starvation risk: infinite microtask loops block rendering forever.

requestAnimationFrame

Fires after macrotask + microtask drain, before the rendering pipeline. Guaranteed to run once per frame (not per event loop tick). The right place for DOM writes before paint.

requestIdleCallback

Runs when the browser has idle time after a frame. No frame-budget pressure. Correct for analytics, prefetching, non-urgent work. Can be cancelled if frame overruns.

async/await desugaring

await x suspends the async function, schedules the continuation as a microtask when x resolves. The function is paused but the event loop continues. Continuation is a Promise .then callback.

Performance implications

INP: Event handlers are macrotasks. A long-running event handler (>50ms) delays the next task and all rendering. The browser cannot paint mid-task — so a 200ms synchronous computation in a click handler will freeze the UI for 200ms regardless of how many Promises you use inside it. TTI: Heavy JS initialization in macrotasks (parsing, hydration) blocks the event loop, delaying the point where the browser can respond to user input. Microtask starvation: A recursive Promise chain that always creates a new Promise can monopolize the event loop — nothing gets rendered while microtasks run.

Code example

setTimeout(() => console.log('2'), 0);  // macrotask

  console.log('3');                       // microtask
  queueMicrotask(() => console.log('4')); // microtask nested in microtask
});

console.log('5');                         // sync
// Output: 1, 5, 3, 4, 2
// Sync → microtask drain (3,4) → macrotask (2)

// ❌ Blocking the event loop — kills INP

});

// ✅ Yield to event loop with scheduler API

  await scheduler.yield();   // Yield: flush pending tasks/rendering
  const second = processChunk(data, 1000, 2000);
  updateUI([...first, ...second]);
});
Real-world architecture use case

React's concurrent mode scheduler

React's concurrent mode (via its own Scheduler package) implements cooperative multitasking on the event loop. It slices React's reconciliation work into 5ms chunks, yielding to the event loop between chunks using MessageChannel (a macrotask — chosen because it has lower overhead than setTimeout(0) in most browsers and fires before requestAnimationFrame). This means user input (another macrotask) can interrupt reconciliation. The scheduler tracks elapsed time with performance.now() and breaks out of the work loop when a deadline is exceeded. startTransition marks updates as low-priority — they're processed in background chunks, interruptible by high-priority input events.

Trade-offs
Splitting work into async chunks (chunked processing via yielding) adds overhead: each chunk resumption is a new macrotask, meaning rendering can occur between chunks. This is usually desirable but can cause visual tearing if the intermediate state is visible.
Web Workers move computation off the main thread entirely — but they can't touch the DOM. The serialization cost of postMessage for large data can exceed the computation savings for small work.
Common mistakes
Thinking async/await makes code non-blocking — await only yields at the await point. Synchronous code before the first await in an async function still runs synchronously and blocks the call stack.
Using setTimeout(fn, 0) to "defer to next tick" — this schedules a macrotask, not the next microtask. If you need the next microtask checkpoint, use queueMicrotask(fn) or Promise.resolve().then(fn).
Infinite Promise chains as "event loops" — mutually recursive async functions with no base case drain the microtask queue indefinitely, starving rendering and input handling.
Interview perspective
"What is the output of [code with setTimeout, Promise, sync]? Explain why."
Order: sync code → microtask drain → macrotask. Be precise about nested microtasks (queued microtasks run before the next macrotask but after the outer microtask returns). Interviewers test this at FAANG consistently.
"How does async/await work under the hood?"
Syntax sugar over Promise + generator pattern. Each await suspends the function, registers .then on the awaited promise, and the continuation is a microtask. The async function frame is preserved on the heap (not the stack) while suspended.
"How does React's concurrent mode relate to the event loop?"
Senior-level. React scheduler uses MessageChannel macrotasks for work slicing. Concurrent mode makes React's reconciliation preemptible by yielding the event loop between chunks. startTransition marks low-priority updates.

Web APIs vs JS Engine

ConcurrencyThreading modelINP
Definition

The JavaScript engine (V8) knows nothing about the DOM, network, timers, or file system. These are provided by the host environment (browser or Node.js) via Web APIs — C++ implementations that live outside V8. When your JS code calls setTimeout, fetch, or addEventListener, it's calling into the browser's C++ layer, not V8. V8 only handles pure ECMAScript. The browser's Web APIs execute asynchronous work on their own threads (the Network Process for fetch, the OS timer thread for setTimeout) and when complete, deposit the callback into the event loop's task queue for V8 to eventually execute. This separation is the architectural reason JS can be non-blocking despite being single-threaded.

How it works internally

setTimeout(fn, 100): V8 passes fn and the delay to the browser's timer Web API (implemented in the Browser Process, backed by OS high-resolution timers). V8 continues executing. After 100ms, the timer Web API pushes a macrotask containing fn into the event loop's task queue. V8 pulls it at the next available iteration. fetch(url): V8 passes the request to the Network Process via IPC. The Network Process performs DNS, TLS, HTTP independently of V8. On response, it posts a task back to the Renderer Process's event loop with the response data, which resolves the Promise. addEventListener('click', fn): The browser registers the listener in a C++ event dispatch table. When the OS reports a mouse event, the browser routes it to the correct Renderer Process and posts a task for fn onto the event loop. The key: V8 never polls, never blocks waiting, never owns a timer — it only processes callbacks when the event loop hands them over.

Subtopics

Timer APIs (setTimeout, setInterval)

Minimum delay is ~1ms (4ms in nested/background tabs per spec). Delay is minimum, not guarantee — if the call stack is busy, firing is delayed. setInterval can skip fires if callback exceeds interval.

fetch / XHR

Handled by the Network Process. Returns a Promise. Response streaming (ReadableStream) allows processing chunks before full download. fetch does NOT fire XHR progress events — use ReadableStream for progress.

DOM Events

OS input → Browser Process → Renderer IPC → event dispatch. addEventListener is a C++ binding. Event bubbling and capture happen in C++ (Blink's event system), callbacks are JS. Touch events have special coalescing behavior to avoid flooding the event loop.

Web Workers

Separate thread with its own JS context (separate V8 isolate). No shared memory by default — only postMessage. Can use SharedArrayBuffer + Atomics for shared memory with explicit synchronization.

Service Worker

A special type of Worker that intercepts network requests. Lives in its own Renderer-like context. Acts as a client-side proxy — can cache, modify, or fabricate responses. Critical for PWA offline behavior.

WebSocket

Persistent TCP connection managed by the browser (Network Process). onmessage fires a macrotask per message. High-frequency messages can flood the event loop — throttle in the handler.

Performance implications

INP: DOM events land as macrotasks, so a long-running prior macrotask will delay event handling. Passive event listeners (touch/scroll) — declaring { passive: true } tells the browser the handler won't call preventDefault(), allowing the compositor to scroll immediately without waiting for the JS handler to run. This is critical for scroll performance. fetch vs XHR: Both use the Network Process, but fetch is Promise-based (microtask continuations), giving it slightly different scheduling than XHR's onload macrotask. Timers in background tabs: Chrome throttles setTimeout to ≥1000ms in background tabs to save CPU — don't rely on precise timers for background work.

document.addEventListener('click', cb); // → C++ event table

Real-world architecture use case

Real-time collaborative apps (Figma-style)

Figma uses a WebSocket (Network Process managed) for operation sync, a Web Worker running a WebAssembly C++ engine for geometry computation, and the main thread for rendering. The Web Worker uses SharedArrayBuffer to share state with the main thread (zero-copy, no serialization). WebSocket onmessage fires on the main thread, dispatches to the Worker via postMessage. The Worker does all heavy vector math, writes results into SharedArrayBuffer, signals the main thread via Atomics.notify(). The main thread reads from the buffer in a rAF callback. This is the canonical architecture for computation-heavy web apps: Web API (WebSocket) → main thread coordination → Worker computation → SharedArrayBuffer → main thread render.

Trade-offs
Web Workers avoid main thread blocking but add postMessage serialization cost and programming complexity. For small tasks, the overhead exceeds the benefit. Rule of thumb: use Workers when work exceeds ~10ms and runs repeatedly.
Service Workers add network request interception overhead. Every fetch() goes through the SW scope check. On first load, the SW isn't active yet (requires an activate event). Cache-first strategies can serve stale content — requires careful versioning.
Common mistakes
Using setInterval for animation — it's not synchronized with the frame budget. Callbacks can fire mid-frame or be skipped. Use requestAnimationFrame for any visual update.
Forgetting that fetch doesn't timeout by default — a hung request will hold the connection indefinitely. Always wrap with AbortController and a timeout.
Calling DOM APIs inside a Web Worker — Workers have no access to the DOM. They can use fetch, WebSocket, IndexedDB, and some Canvas APIs (OffscreenCanvas), but never document or window.
Interview perspective
"Who handles `setTimeout` — is it V8 or the browser?"
The browser's timer Web API (C++ in the Browser Process). V8 just calls into it and moves on. The callback lands back in V8 via the event loop when the timer fires. Interviewers check if you know the JS engine / host environment boundary.
"What are passive event listeners and why do they matter?"
Allows the compositor to scroll without waiting for the JS handler. Matters for touch scroll jank on mobile. Chrome warns in DevTools about missing passive declarations on touchstart/wheel handlers.

Topic 7 of 8 — Block 1: Browser Internals

Memory Management in Browser

TTIHeap memoryGC pausesMemory leaks
Definition

V8 manages the JavaScript heap using a generational garbage collector — based on the empirical observation that most objects die young. Memory is divided into zones with different GC strategies. Unlike languages with manual memory management, V8 automatically frees unreachable objects via tracing GC. In browser applications, memory management failures manifest as: (1) memory leaks (heap grows without bound, eventually causing OOM crashes), (2) GC pauses (stop-the-world collection freezes the main thread, causing jank), and (3) heap bloat (excessive live objects keeping GC pressure high). Understanding V8's GC is essential for debugging production memory issues in long-lived SPAs.

How it works internally

V8's heap has zones: Young Generation (new space, 1-8MB) — most objects are allocated here. Young GC is called Scavenger (minor GC): a copying collector that copies live objects from "from-space" to "to-space", discarding dead ones. Fast (~1ms) because it only touches the small young gen. Objects that survive multiple Scavenges are promoted to the Old Generation (old space, 128MB-2GB+). Old gen GC is Mark-Compact (major GC): mark all reachable objects (tracing from roots: stack, globals, DOM references), compact the heap to remove fragmentation. Major GC is expensive (can be 50-200ms on large heaps). V8 uses incremental marking (breaks marking into small steps interleaved with JS execution), lazy sweeping, and concurrent marking (GC worker threads mark while JS runs) to reduce pause times. The DOM is part of Blink's C++ heap, not V8's. V8 and Blink share ownership of DOM nodes via a cross-heap tracing mechanism — a JS object holding a DOM node reference keeps the C++ node alive.

Subtopics

Young Generation (Scavenger)

Copy collector. Fast. Short-lived objects. "From-space" to "to-space" flip. Most allocations die here. Promotes survivors to old gen after 2 collections.

Old Generation (Mark-Compact)

Tri-color incremental marking. Sweeps dead objects. Compacts live objects. Can run concurrently on helper threads. Full GC here is the source of production jank.

Detached DOM nodes

A DOM node removed from the document but still referenced by a JS variable. Blink frees the C++ node, but the JS wrapper remains alive, preventing GC. Classic leak pattern in SPAs.

Closure leaks

Closures capture the entire scope chain. A large object captured in a closure that's attached to a long-lived reference (event listener, Promise) will never be GC'd even if you only needed a small part of it.

Event listener leaks

addEventListener without removeEventListener keeps the callback (and its closure scope) alive as long as the element exists in the DOM (or longer if the element is detached but referenced).

WeakRef / FinalizationRegistry

WeakRef allows a reference that doesn't prevent GC. FinalizationRegistry fires a callback when a weakly-referenced object is collected. Used for cache cleanup, not lifecycle management.

Performance implications

TTI: Heavy allocation at startup (hydration, data parsing) triggers Scavenger GCs which, while fast, add up. Long-session jank: As SPAs run for hours, heap grows. Major GCs (Mark-Compact) become more frequent and longer. A 500MB heap can trigger 200ms GC pauses — visible as periodic freezes. Memory pressure: Mobile devices have 1-2GB RAM for the entire system. A tab consuming 300MB competes with the OS and other apps, triggering aggressive GC and potentially tab suspension. Memory leaks in SPAs are particularly insidious — they accumulate slowly and are hard to spot in development (short sessions).

  cache.set(key, new WeakRef(value)); // GC can collect if memory pressure

Real-world architecture use case

Long-running SPA memory profiling workflow

Production SPAs at companies like Notion or Linear run for hours without page reload. Teams use Chrome DevTools Memory tab → "Allocation timeline" to catch leaks. The workflow: take a heap snapshot, perform a user action (navigate to a route and back), force GC, take another snapshot. Objects that appear in "Objects allocated between snapshots" but shouldn't exist are leaks. A key pattern at scale: React component trees that hold references to WebSocket instances, EventSource objects, or interval handles without cleanup in useEffect destructors. Each route mount adds another listener; after 50 route navigations, there are 50 resize listeners all firing simultaneously.

Trade-offs
WeakMap/WeakSet are useful for private data associated with objects (the key doesn't prevent GC), but they're not iterable — you can't enumerate all keys. This makes debugging harder.
Aggressive use of object pooling (reusing objects instead of allocating new ones) reduces GC pressure but adds complexity and can introduce subtle bugs if objects are not properly reset before reuse.
Common mistakes
Global variables as caches without expiration — any object reachable from a global is a GC root, i.e., it will never be collected. Unbounded Maps/Sets on window are slow memory leaks.
Assuming null-ing a reference immediately frees memory — GC is non-deterministic. Setting a reference to null makes the object eligible for collection, but collection happens at V8's discretion (during idle time or under pressure).
Using setInterval without clearInterval in component teardown — the callback closure keeps the component's state and props alive, preventing GC of the entire component closure scope.

Browser Caching

LCPTTFBTTINetwork
Definition

The browser maintains a hierarchy of caches, each with different scope, eviction behavior, and lookup priority. In order from fastest to slowest: Memory cache (in-process, cleared on tab close), Service Worker cache (programmatic, persistent, JS-controlled), HTTP cache (disk, governed by HTTP response headers), Push cache (HTTP/2 server push, session-scoped, very short-lived). The browser consults them in a defined order for every resource request. Understanding this hierarchy is fundamental to building zero-TTFB repeat visits, progressive loading strategies, and avoiding stale-content bugs in production deployments.

How it works internally

When the Network Process receives a resource request, it first checks the memory cache — if the resource was fetched during this page session and is still in-process memory, it returns instantly (0ms). If not, it checks the Service Worker (if one is registered for the origin) — the SW intercepts the fetch event and can return from caches API (Cache Storage), fabricate a response, or forward to network. If SW passes through, the Network Process checks the HTTP cache (disk cache): if the response has a valid Cache-Control: max-age that hasn't expired, it returns without a network request (cache hit, "200 from cache"). If max-age is expired but an ETag or Last-Modified is present, the browser sends a conditional request (If-None-Match / If-Modified-Since) — if the server says 304 Not Modified, the browser uses the cached body without re-downloading it. If no cache applies, a full network request is made.

Subtopics

Memory Cache

In-memory, per-tab, cleared on close. Duplicate requests in the same page session served from here. preload puts resources here. No headers needed — implicit.

HTTP Cache (Disk Cache)

Governed by Cache-Control, Expires, ETag, Last-Modified. Persistent across sessions. Shared across tabs. Primary cache for static assets.

Cache-Control directives

max-age=N: fresh for N seconds. no-cache: always revalidate (not "no caching"). no-store: truly no caching. immutable: never revalidate during max-age. stale-while-revalidate: serve stale, refresh in background.

ETag / Conditional Requests

ETag is a server-generated hash of the resource content. Browser sends If-None-Match: <etag>. Server returns 304 (no body, saves bandwidth) or 200 with new content. Saves re-download, still costs an RTT.

Service Worker Cache

Explicit JS-controlled cache via Cache Storage API. Survives tab close. Can implement cache-first, network-first, stale-while-revalidate strategies programmatically. Intercepts all fetch events for the origin.

Cache busting

Changing the filename/URL (content hash) forces the browser to treat it as a new resource, bypassing HTTP cache. Vite/webpack append [contenthash] to JS/CSS filenames. The HTML document itself should have no-cache so it always revalidates.

Vary header

Tells the cache that the same URL may have different responses based on request headers (e.g., Vary: Accept-Encoding caches gzip and brotli separately). Misuse causes cache fragmentation.

stale-while-revalidate

Serve the stale cached response immediately (0 TTFB), then fetch fresh content in the background for the next request. Best of both worlds for content that can tolerate one revision of staleness.

Performance implications

TTFB: HTTP cache hits serve resources with 0 network time — the dominant TTFB optimization for repeat visits. LCP: The LCP image must be either in memory cache (preloaded earlier) or HTTP-cached locally, or it needs a full network fetch. The difference is 100-500ms on typical connections. TTI: JS bundles cached with max-age=31536000, immutable (after content-hash naming) load from disk with zero network overhead, dramatically shortening parse time on repeat visits since even disk read is faster than a round-trip.

Code example
// HTTP headers — what the server should send
// Static assets (JS/CSS with content hash in filename)
Cache-Control: public, max-age=31536000, immutable
// "Never check back for a year, asset name won't change unless content changes"

// HTML document — always revalidate
Cache-Control: no-cache
// "Before using cached copy, check if server has a newer version"
// (NOT the same as no-store — the browser still caches it, just always revalidates)

// API responses with stale-while-revalidate

// "Fresh for 60s, up to 300s serve stale while refreshing in background"

// Service Worker — cache-first strategy
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      // Return cached immediately, fetch fresh in background
      const fetchPromise = fetch(event.request).then((res) => {
        const clone = res.clone();
        caches.open('v1').then(c => c.put(event.request, clone));
        return res;
      });
      return cached || fetchPromise; // Cache-first with background refresh
    })
  );
});

Real-world architecture use case

Next.js ISR + CDN caching strategy

Next.js Incremental Static Regeneration combines server-side stale-while-revalidate with CDN edge caching. A product page is statically rendered, served from the CDN with s-maxage=60, stale-while-revalidate headers. The CDN serves cached HTML instantly (near-zero TTFB), then in the background revalidates with the Next.js origin server. The browser receives the HTML which references content-hashed JS/CSS bundles (immutable in browser HTTP cache). The result: sub-100ms TTFB, sub-200ms LCP on repeat visits, and pages that auto-update within 60 seconds of content changes — without a full CDN purge. The Service Worker layer (via next-pwa) adds offline support by caching the SW cache separately from the HTTP cache.

Trade-offs
no-cache on the HTML document means every navigation costs one conditional request RTT to verify freshness — on mobile (150ms RTT), this is 150ms added to every navigation. Acceptable for content freshness but measurable in TTFB.
Service Worker cache is powerful but adds complexity: stale SW can serve outdated content after a deployment if the SW update flow isn't handled correctly. skipWaiting() + clients.claim() + versioned cache names are required for safe updates.
Vary: Accept-Encoding correctly separates brotli/gzip caches, but Vary: Cookie or Vary: Authorization effectively disables CDN caching since every user's request varies — a common misconfiguration.
Common mistakes
Confusing no-cache and no-store — no-cache means "cache but always revalidate". no-store means "never write to cache". Using no-store on assets that only change on deployment wastes all the performance benefits of caching.
Not using content hashes in asset filenames — deploying new JS with the same filename (app.js) with long max-age means users get cached old code. Content hashing is non-negotiable for static assets with long cache TTLs.
Caching API responses without understanding Vary — a JSON API response that differs based on Accept-Language or Authorization must include these in Vary, or the cache will return the wrong variant to users.
Service Worker intercepting requests without proper cache versioning — on SW update, old cached assets continue to serve until the cache name changes. Teams burn production incidents on this monthly.
Interview perspective
"What's the difference between no-cache and no-store?"
no-cache = cache it, but always validate with server before using (304 path). no-store = never write to any cache, ever. Interviewers test this constantly because it's counterintuitively named.
"How would you design the caching strategy for a high-traffic e-commerce site?"
System design answer: CDN for HTML (stale-while-revalidate, purge on publish), immutable + content-hash for JS/CSS/images, no-cache for HTML document itself, API responses vary by user auth (no CDN cache, browser max-age=30), Service Worker for offline shell. Mention cache invalidation as the hard problem — always purge by deploying new content-hash URLs.
"What is the browser cache lookup order?"
Memory cache → Service Worker cache → HTTP disk cache → Push cache → Network. Know which layer each resource type typically lives in. Static assets with long TTL: HTTP disk cache on repeat. Preloaded resources: memory cache. SW-controlled: SW cache first.
