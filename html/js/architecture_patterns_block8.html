Topic 1 of 8 — Block 8: Architecture

Micro-frontends

Module federationiframe isolationWeb componentsTrade-offs
Definition

Micro-frontends extend the microservices concept to the frontend: decompose a large SPA into independently deployable, loosely coupled frontend applications, each owned by a separate team. Each MFE owns a vertical slice of the product — its own repository, build pipeline, deployment, and often its own React version. Teams can ship independently without coordinating releases. The architectural problem they solve: a monolithic frontend becomes a deployment bottleneck as team count grows. The cost: significant operational complexity, runtime integration challenges, and potential for duplicated dependencies inflating bundle size.

Integration approaches

Module Federation (Webpack 5 / Vite)

Runtime JS sharing via dynamic remote imports. A "host" app loads "remote" modules from separately deployed URLs at runtime. Dependencies can be shared (one React instance across all MFEs). No iframe — full DOM sharing. Teams can use different versions of non-shared libraries.

iframe isolation

Each MFE rendered in its own iframe. Maximum isolation — separate browsing context, separate DOM, separate JS heap. Cross-MFE communication via postMessage only. Cost: no shared state, no shared cookies by default, layout challenges, full-page iframe scrolling issues. Use for untrusted third-party embeds.

Web Components

Framework-agnostic custom elements as the MFE boundary. A React MFE wraps itself in a Custom Element — any host page can <my-team-widget> it regardless of the host's framework. Shadow DOM provides style isolation. Natural interop layer between frameworks.

Module Federation internals

Module Federation gives each build a "container" — a runtime module registry. The host app, at bootstrap, fetches the remote's remoteEntry.js (the container manifest). This manifest describes what modules the remote exposes and what shared dependencies it uses. When the host imports a remote module (import('remote/Button')), Module Federation checks: does the host already have a compatible version of React in the shared scope? If yes, the remote uses the host's React instance. If no, the remote loads its own. This singleton-or-independent decision is made at runtime per dependency per version range. Shared modules are loaded only once, negotiated via the shared scope object.

MFE vs monorepo — the real trade-off

When MFE makes sense

Large org (10+ frontend teams). Independent deployment is genuinely required. Teams have different release cadences. Different parts of the app have radically different tech requirements. Legacy systems that can't be refactored but need new UI on top.

When monorepo is better

Shared component library needs consistent versioning. Teams need to refactor across the boundary frequently. Bundle optimization (deduplication) matters. Sub-10-team org — the operational overhead of MFE exceeds the coordination cost it solves. Monorepo with feature flags gives most benefits at lower cost.

Hidden MFE costs

Cross-MFE routing becomes complex. Shared auth state requires synchronization. Each MFE has its own devops pipeline. Performance: multiple React instances, duplicated polyfills, multiple network waterfall points on page load. Error boundaries between MFEs need explicit design.

Communication patterns

URL (navigate with state in params). Custom events (window.dispatchEvent). Shared event bus (window-level pub/sub). Props/callbacks (for iframe: postMessage). Shared server state (React Query with same cache keys across MFEs using same query client).

    // Version mismatch: federation falls back to separate instance

Common mistakes
Running multiple React instances due to mismatched version ranges in shared config — components from different React trees can't share context, portals don't work across boundaries, and hooks that depend on React internals (useContext) silently fail. Always use singleton: true for React/ReactDOM.
Decomposing too early — teams of fewer than 5-6 dedicated frontend engineers rarely benefit from MFE's independent deployment advantage over a well-organized monorepo with clear module boundaries. The coordination overhead of shared types, design systems, and auth integration often exceeds the release-speed benefit.
Interview perspective
"How does Module Federation work under the hood?"
Each build outputs a remoteEntry.js manifest describing exposed modules and shared dependencies. The host loads this at runtime, negotiates shared dependency versions via a shared scope object, and dynamically imports remote modules on demand. Singleton React ensures hooks work across the boundary. The remote's chunk files live on a different origin and are loaded via dynamic import at runtime — independent of the host's deployment.
"When would you choose micro-frontends over a monorepo?"
MFE when: multiple large teams need truly independent deployments, teams have fundamentally different release cadences or tech stacks, or you're integrating legacy systems. Monorepo when: teams need to refactor across boundaries, bundle deduplication matters significantly, or the org is small enough that release coordination is cheap. MFE adds significant operational complexity — it must solve a real deployment problem to justify the cost.

Monorepo frontend

TurborepoNxBuild cachingShared packages
Definition

A monorepo is a single version-controlled repository containing multiple related packages — multiple apps, shared UI libraries, utility packages, and configuration packages — that are developed and versioned together. The monorepo model enables: atomic changes across package boundaries (change a component and its consumers in one commit), consistent tooling configuration, easy cross-package refactoring, and shared CI infrastructure. Turborepo and Nx are the two dominant build orchestration tools that make large frontend monorepos tractable by adding caching, parallelism, and dependency-graph-aware task scheduling.

How Turborepo works internally

Turborepo builds a task dependency graph from your turbo.json configuration. Each task (build, test, lint) has declared dependencies: "dependsOn": ["^build"] means "run after upstream packages' build tasks complete." Turborepo hashes the inputs to each task (source files, env vars, package.json, lockfile) and stores task outputs (build artifacts, test results) in a content-addressed cache. On subsequent runs, if the hash matches, the cached output is restored rather than re-running. This cache can be remote (Vercel Remote Cache, self-hosted S3) — teammates share the same cache, so CI never re-runs a task whose inputs haven't changed since any previous run, anywhere.

Subtopics

Workspace structure

Packages are symlinked together via npm/yarn/pnpm workspaces. apps/: deployable applications. packages/: shared code (UI library, config, utilities, types). Each package has its own package.json. Internal cross-references use workspace protocol: "@myco/ui": "workspace:*".

Build caching

Turborepo: hash-based, file-output cache (dist/ directories, test results). Local + remote (Vercel). Nx: computation cache with sophisticated task graph. Key insight: unchanged package = zero build time. A 10-minute CI pipeline becomes seconds when only one of 20 packages changed.

Affected-only runs

nx affected --target=test or turbo run test --filter=[HEAD^1]: only run tasks for packages that have changed (based on git diff against a base ref). PRs that touch packages/ui rebuild all apps that depend on it; PRs that only touch one app skip unrelated packages entirely.

Shared configuration

ESLint configs, TypeScript configs, Vite configs as packages: @myco/eslint-config, @myco/tsconfig. Apps extend from these. One change to the shared ESLint config propagates everywhere. Turborepo treats config package changes as affecting all dependents.

Nx vs Turborepo

Turborepo: simpler, file-output caching, great DX, Vercel Remote Cache. Nx: more powerful dependency graph, code generators, sophisticated plugin system, affected detection via TypeScript analysis (not just file hash). Nx scales better for very large codebases; Turborepo is easier to adopt.

Internal package publishing

Packages can be consumed internally via workspace symlinks (no publish needed) or published to npm/private registry for external use. Changesets manages versioning across packages in a monorepo — tracks which packages changed and bumps versions appropriately on publish.

// Repository structure
├── apps/
│   ├── web/          # Next.js app
│   └── admin/        # Vite SPA
├── packages/
│   ├── ui/           # Shared component library
│   ├── config/       # ESLint, TS, Vite configs
│   └── utils/        # Shared utilities
└── turbo.json

// turbo.json — task pipeline
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],   // Wait for upstream packages to build first
      "outputs": ["dist/**", ".next/**"], // Cached outputs
      "inputs": ["src/**", "package.json"] // Cache key inputs
    },
    "test": {
      "dependsOn": ["^build"],
      "outputs": ["coverage/**"]
    },
    "lint": { "outputs": [] },  // No file outputs — cached by exit code
    "dev": { "cache": false, "persistent": true } // Never cache dev server
  }
}

// packages/ui/package.json
{
  "name": "@myco/ui",
  "main": "./dist/index.js",
  "exports": { ".": { "import": "./dist/index.mjs" } }
}

// apps/web/package.json
{
  "dependencies": {
    "@myco/ui": "workspace:*"  // Resolved to local symlink, not npm
  }
}

// CI — only build/test what changed
$ turbo run build test --filter=[origin/main]
# Compares current HEAD against origin/main
# Builds only changed packages + their dependents
# Cache hit: restored from Vercel Remote Cache in ~1s
# Cache miss: runs the task, uploads output to cache
Common mistakes
Not declaring all cache inputs in turbo.json — if an environment variable affects the build output but isn't listed in globalEnv, Turborepo will serve a stale cache hit with the wrong env var baked in. Always declare all variables that affect build output in globalEnv or task-level env.
Building shared packages with TypeScript source directly imported (skipping the build step) — works in dev with ts-node/tsx but production builds need the compiled output. Inconsistency between dev and prod environments. Always build packages before apps: dependsOn: ["^build"].
Interview perspective
"How does Turborepo's caching work?"
Turborepo hashes all inputs to a task (source files, env vars, package.json, lockfile). The hash is the cache key. If a matching entry exists in local or remote cache, the task is skipped and cached outputs (dist/ files, test coverage) are restored. Cache misses run the task and upload outputs. Remote cache shares hits across all team members and CI runs — so a build that passed on one engineer's machine is never re-run for the same input hash.

Design systems

Component APICompound componentsPolymorphicHeadless
Definition

A design system is the single source of truth for UI — a shared library of components, design tokens (colours, spacing, typography), patterns, and guidelines that ensures consistent UI across a product. At the engineering level, a design system is a versioned package that teams import rather than reimplementing UI primitives. The quality of a design system is determined by the quality of its component APIs: how easy it is to compose components for new use cases without forking them, how flexible they are without being un-opinionated, and how well they handle accessibility without requiring consumers to think about it.

Component API design principles

Controlled vs uncontrolled

Uncontrolled: component manages its own state, optionally calls back via onChange. Controlled: state is owned by the consumer (value + onChange). Design system components should support both — use the "controlled when value provided, uncontrolled otherwise" pattern. Radix UI primitives model this well.

Composition over configuration

Avoid boolean prop explosion: hasIcon hasAvatar isRounded isCompact. Instead, compose: <Button><Icon /> Label</Button>. The children API handles infinite variation without new props. The compound component pattern formalises this.

Escape hatches

No design system handles every case. Provide: className for CSS overrides, style for inline overrides, as prop for polymorphism, ref forwarding for DOM access, and data-* passthrough. Components that block these force consumers to fork.

Accessibility by default

Keyboard navigation, ARIA attributes, focus management — bake them in. Consumers shouldn't need to know ARIA to use your Select. Radix UI's primitive layer handles the ARIA contract; your design system layer handles visual styling on top.

Compound components pattern

Compound components are a set of related components that share implicit state via React Context. Instead of configuring a complex <Select options={[...]} renderOption={...}>, you compose: <Select.Root><Select.Trigger>...<Select.Content><Select.Item>. Each sub-component reads shared state from the context set up by the Root. This gives consumers full control over layout and composition while the Root handles the stateful logic. React's Context + forwardRef enable this pattern cleanly.

Polymorphic components

A polymorphic component renders as different HTML elements based on an as prop while maintaining its styling contract: <Text as="h1">, <Button as="a" href="/link">, <Box as="section">. The challenge is TypeScript: the component's prop types must merge the design system's own props with the HTML attributes of the target element. The correct type: ComponentProps<T> & OwnProps where T is the as prop value. Libraries like Radix UI and Chakra implement this with the asChild prop + Slot pattern (avoids type complexity while achieving polymorphism).

Code example
// Compound component pattern — Select
const SelectContext = createContext<{ value: string; onSelect: (v: string) => void }>(null!);

function SelectRoot({ children, defaultValue = '', onChange }: SelectRootProps) {
  const [value, setValue] = useState(defaultValue);
  const onSelect = (v: string) => { setValue(v); onChange?.(v); };
  return <SelectContext.Provider value={{ value, onSelect }}>{children}</SelectContext.Provider>;
}

function SelectItem({ value, children }: { value: string; children: React.ReactNode }) {
  const { value: selected, onSelect } = useContext(SelectContext);
  return (
    <div
      role="option"
      aria-selected={selected === value}
      onClick={() => onSelect(value)}
      onKeyDown={e => e.key === 'Enter' && onSelect(value)}
      tabIndex={0}
    >{children}</div>
  );
}

// Namespace export
export const Select = { Root: SelectRoot, Item: SelectItem, Trigger: SelectTrigger };

// Consumer — full control, no prop explosion
<Select.Root onChange={setCountry}>
  <Select.Trigger>Choose country</Select.Trigger>
  <Select.Content>
    <Select.Item value="in">India</Select.Item>
    <Select.Item value="us">United States</Select.Item>
  </Select.Content>
</Select.Root>

// Polymorphic component — type-safe `as` prop
type BoxProps<T extends ElementType = 'div'> = {
  as?: T;
  children: ReactNode;
} & ComponentPropsWithoutRef<T>;

function Box<T extends ElementType = 'div'>({ as, children, ...rest }: BoxProps<T>) {
  const Component = as ?? 'div';
  return <Component {...rest}>{children}</Component>;
}

<Box as="section" aria-label="main content">...</Box> // Renders <section>
<Box as="a" href="/home">Go home</Box>            // Renders <a href>
Common mistakes
Prop explosion for variation — adding hasIcon, iconPosition, iconSize, hasAvatar, avatarVariant as boolean/string props. The API becomes unmaintainable and undiscoverable. Use children composition instead. The test: if you need more than 4-5 props to configure a single component's appearance, the API is wrong.
Not forwarding refs — design system components that don't forward refs force consumers to wrap them in DOM-accessing code. Always use React.forwardRef on any component that renders a DOM element. Without it, you can't use the component with Floating UI, form libraries, or any code that needs a DOM reference.
Interview perspective
"What is the compound component pattern and why is it better than prop-based configuration?"
Compound components share implicit state via Context between a Root and its sub-components. Consumers compose the layout themselves using structured sub-components rather than passing all configuration as props. Benefits: consumers control layout, infinite variation without new props, each part is independently styleable, and the API is self-documenting through the component hierarchy. The alternative — a single component with 20 props — is rigid and can't anticipate every layout need.

Accessibility (a11y)

ARIA rolesFocus managementKeyboard navScreen reader
Definition

Accessibility (a11y) ensures that web applications are usable by people with disabilities — visual, motor, auditory, cognitive. In practice for frontend engineers, this means: the page has correct semantic HTML structure, interactive elements are keyboard-operable, screen readers receive meaningful descriptions of UI state via ARIA, focus is managed correctly during dynamic content changes (modals, routes, notifications), and colour contrast meets WCAG standards. Accessibility is not a separate concern — it is a quality attribute of the code, like performance. Inaccessible code is incorrect code.

Semantic HTML first

The first rule of ARIA: don't use ARIA if native HTML does it correctly. <button> is keyboard-focusable, fires on Enter/Space, has role=button, and has a name from its text content — for free. <div onClick> requires: role="button", tabIndex={0}, keyboard handler for Enter/Space, and an aria-label if the content is an icon. Use the right element and get accessibility for free.

ARIA fundamentals

Roles

Define what an element is: role="dialog", role="alert", role="combobox", role="listbox". Roles communicate element semantics to assistive technology when HTML semantics are insufficient. Never override a native element's role unless the element is purely presentational.

Properties

Describe element characteristics: aria-label="Close dialog", aria-labelledby="heading-id", aria-describedby="desc-id", aria-required, aria-invalid. Static attributes that don't change based on state.

States

Describe element condition: aria-expanded, aria-checked, aria-selected, aria-disabled, aria-hidden. Must be updated dynamically as UI state changes. A collapsed accordion without aria-expanded="false" is invisible to screen readers.

Live regions

aria-live="polite": screen reader announces content updates when idle. aria-live="assertive": announces immediately, interrupts current reading. Use assertive sparingly (errors only). role="status" is live=polite. role="alert" is live=assertive. Dynamic notifications need live regions.

Focus management patterns

Modal/dialog focus

When a modal opens: move focus to the first interactive element inside (or the dialog element itself). Trap focus: Tab cycles only within the modal. When modal closes: return focus to the element that triggered it. Without return: keyboard user loses their place in the document.

Route change focus

SPAs don't reload the page — screen readers don't know the content changed. On route change: move focus to the main heading of the new page, or to the main landmark. React Router + a live region announcing the page title is the common pattern.

Skip links

<a href="#main" class="sr-only focus-visible:not-sr-only">Skip to main content</a> as the first focusable element. Keyboard users skip the navbar on every page. Required for WCAG compliance on sites with repetitive navigation.

Focus visible

Never outline: none without replacement. Focus must be visible. :focus-visible (keyboard focus only, not mouse) is the modern selector. Browsers now respect this by default — don't suppress the focus ring for mouse users and then wonder why keyboard users have no focus indicator.

Code example
// ❌ Inaccessible custom button
<div onClick={handleClick} style={{ cursor: 'pointer' }}>Submit</div>
// No role, no tabIndex, no keyboard handler, not focusable

// ✅ Accessible: use <button>
<button type="button" onClick={handleClick}>Submit</button>
// Free: role=button, tabIndex=0, Enter/Space activation, focus management

// Accessible modal with focus trap
function Modal({ open, onClose, children }) {
  const dialogRef = useRef<HTMLDivElement>(null);
  const triggerRef = useRef<HTMLElement>(null);

  useEffect(() => {
    if (open) {
      triggerRef.current = document.activeElement as HTMLElement; // Remember trigger
      dialogRef.current?.focus();  // Move focus into dialog
    } else {
      triggerRef.current?.focus(); // Return focus on close
    }
  }, [open]);

  const trapFocus = (e: KeyboardEvent) => {
    if (e.key !== 'Tab') return;
    const focusable = dialogRef.current?.querySelectorAll(
      'button, [href], input, [tabindex]:not([tabindex="-1"])'
    );
    const [first, last] = [focusable?.[0], focusable?.[focusable.length - 1]] as HTMLElement[];
    if (e.shiftKey && document.activeElement === first) { last.focus(); e.preventDefault(); }
    else if (!e.shiftKey && document.activeElement === last) { first.focus(); e.preventDefault(); }
  };

  if (!open) return null;
  return (
    <div role="dialog" aria-modal="true" aria-labelledby="modal-title"
      ref={dialogRef} tabIndex={-1} onKeyDown={trapFocus}>
      <h2 id="modal-title">Confirm deletion</h2>
      {children}
      <button onClick={onClose}>Cancel</button>
    </div>
  );
}

// Live region for async notifications
function StatusAnnouncer({ message }) {
  return (
    <div
      role="status"     // aria-live="polite" equivalent
      aria-live="polite"
      aria-atomic="true"
      className="sr-only" // Visually hidden, screen reader visible
    >{message}</div>
  );
}

Common mistakes
Using aria-label on elements that already have visible text — <button aria-label="Close">Close</button> is redundant. Worse: <button aria-label="Close">✕</button> where the label and content differ confuses screen readers. Use aria-label only on elements without visible text, or where the visual text is meaningless.
Setting aria-hidden="true" on elements with focusable children — screen readers skip aria-hidden elements, but keyboard focus can still reach children. A focused element inside an aria-hidden container creates a ghost focus state the user can't perceive or understand. Remove focusable children or use inert attribute instead.
Interview perspective
"How do you manage focus when a modal opens and closes?"
On open: save a reference to the triggering element (document.activeElement), move focus to the dialog element or first interactive child. Trap focus: Tab cycles within the dialog (check first/last focusable, loop around). On close: restore focus to the triggering element. Without restoration, keyboard users lose their document position — their cursor is now at the top of the page with no indication of where they were.
"What is an aria-live region and when do you use it?"
A container with aria-live="polite" (announces when screen reader is idle) or "assertive" (interrupts immediately). Use for dynamic content that updates without a page reload: form validation messages, async operation results, real-time notifications, cart updates. Without a live region, screen readers have no way to know the content changed and users miss the update.

Internationalisation (i18n)

Locale detectionRTL layoutIntl APITranslation loading
Definition

Internationalisation (i18n) is the architecture for making an application adaptable to multiple languages, regions, and cultural conventions without code changes per locale. This is distinct from localisation (l10n) — the actual work of translating content for a specific locale. A well-architected i18n system handles: language detection and switching, string externalisation and namespace loading, plural forms and gender agreement, date/time/number/currency formatting per locale, right-to-left (RTL) layout for Arabic/Hebrew/Persian, and content negotiation with the server.

Subtopics

Locale detection order

URL path prefix (/en/, /ar/) → subdomain (ar.site.com) → query param (?lang=ar) → Accept-Language header (server) → navigator.language (browser). URL-based is SEO-friendly and shareable. Persist user preference in a cookie/localStorage override.

Plural forms

English: singular/plural. Arabic: 6 forms (zero, one, two, few, many, other). Russian: 3 forms by last digit. Intl.PluralRules returns the category for a number. i18next/react-intl handle this automatically when translations include plural keys (item_one, item_other, item_zero).

Intl API

Built-in browser/Node.js API for locale-sensitive formatting. Intl.DateTimeFormat, Intl.NumberFormat, Intl.RelativeTimeFormat, Intl.Collator (locale-aware sorting). No library needed for formatting — the Intl API is comprehensive and supported everywhere.

RTL layout

Arabic/Hebrew/Persian are right-to-left. CSS logical properties (margin-inline-start instead of margin-left) automatically flip in RTL. Set dir="rtl" on the html element. Tailwind CSS supports RTL via the rtl: variant. Text alignment, flexbox direction, absolute positioning, and icon placement all need audit.

Translation loading strategy

Load only the active locale's translations (not all 20 languages). Code-split by namespace: common.json always loaded, checkout.json loaded on checkout route. React Suspense for translation loading: component suspends until translations are ready (no flash of untranslated strings).

ICU message format

Standardised syntax for complex translations: {count, plural, =0{No items} one{# item} other{# items}}. Handles plurals, gender, select, and number formatting within a single translation key. Used by FormatJS (react-intl), i18next, and LinguiJS.

Code example
// Intl API — no library, built into the browser
const locale = 'hi-IN'; // Hindi India

// Numbers
new Intl.NumberFormat(locale, { style: 'currency', currency: 'INR' })
  .format(1234567.89); // "₹12,34,567.89" — Indian lakh format

// Dates
new Intl.DateTimeFormat(locale, { dateStyle: 'long' })
  .format(new Date()); // "10 जनवरी 2025" — Hindi month name

// Relative time
new Intl.RelativeTimeFormat('ar', { numeric: 'auto' })
  .format(-2, 'day'); // "أول أمس" (day before yesterday in Arabic)

// Plural rules
const pr = new Intl.PluralRules('ar');
pr.select(0);  // 'zero'
pr.select(1);  // 'one'
pr.select(2);  // 'two'
pr.select(5);  // 'few'
pr.select(11); // 'many'
pr.select(100);// 'other'
// Arabic has 6 plural forms — handle all in translations

// RTL layout — CSS logical properties
/* ❌ Physical — wrong in RTL */
.card { margin-left: 1rem; padding-right: 16px; text-align: left; }

/* ✅ Logical — correct in LTR and RTL */
.card {
  margin-inline-start: 1rem;    /* left in LTR, right in RTL */
  padding-inline-end: 16px;     /* right in LTR, left in RTL */
  text-align: start;             /* left in LTR, right in RTL */
}

Common mistakes
Concatenating translated strings: "Your order " + t('status') + " was confirmed" — word order differs between languages. Use ICU message format with interpolation: t('orderConfirmed', { status }) → "Your {status} order was confirmed". Some languages need the status at the beginning or end of the sentence.
Using text-align: left and margin-left for layout — breaks completely in RTL. Audit all physical CSS properties and replace with logical equivalents. Tailwind's ms- (margin-start) and me- (margin-end) utilities handle this cleanly.
Interview perspective
"How would you add Arabic language support to an existing app?"
Multiple layers: (1) Extract all strings to translation keys (audit hardcoded text). (2) Add Arabic translation file with ICU plural forms (Arabic has 6). (3) Set html dir="rtl" for Arabic locale. (4) Audit CSS — replace physical properties (margin-left, text-align: left) with logical equivalents. (5) Test icon orientation — some icons are directional (arrows) and should mirror in RTL. (6) Test number/date formatting with Intl API for locale. (7) Test font — Arabic requires a different font, often heavier weight for readability.
Connects to
Intl API (built-in, no library needed for formatting) CSS logical properties (RTL-safe layout) Next.js i18n routing (URL-based locale) HTTP Accept-Language header (server locale detection)
Topic 6 of 8 — Block 8: Architecture

Progressive enhancement vs graceful degradation

Bottom-up approachTop-down approachCore experienceFeature detection
Definition — they are NOT the same thing

Progressive enhancement (PE) is a bottom-up design philosophy: start with the simplest possible working experience (semantic HTML, no JS), then layer enhancements (CSS styling, JS interactivity, advanced APIs) for capable environments. Every user gets a functional baseline; capable environments get more. PE is the correct approach for the web's inherently uncertain runtime environment.

Graceful degradation (GD) is a top-down approach: build the full-featured experience first, then add fallbacks so that when features are unavailable, the experience degrades rather than breaks completely. You start at the top and plan for failure. GD is more common in native app development and is what most developers actually practice when they say "progressive enhancement."

The practical differences

Progressive enhancement in practice

A form works without JavaScript (native HTML form submission). Then JS enhances it with client-side validation, async submission, and better UX. Core use case: the form submits to the server, a user gets a confirmation. Enhanced use case: no page reload, instant feedback. Even if JS fails to load, the form works.

Graceful degradation in practice

A complex data visualisation is built with WebGL. A fallback SVG chart is shown when WebGL isn't available. The primary experience is the full-featured one; the degraded experience is acceptable but less capable. GD acknowledges the gap between primary and fallback.

Feature detection (not user-agent sniffing)

Correct approach: detect the capability, not the browser. if ('IntersectionObserver' in window), if (CSS.supports('display', 'grid')). Never if (navigator.userAgent.includes('Chrome')) — browser features don't track user agent strings reliably. Use Modernizr, or write feature tests directly.

When each applies

PE: content-driven sites, forms, navigation, server-rendered pages. GD: complex interactive applications where the core value IS the rich experience (video editors, CAD tools, games). Most modern SPAs practice GD (assume JS), then add offline fallbacks (service worker) — not true PE.

Code example
// Progressive enhancement — form that works without JS
<form action="/api/subscribe" method="POST">  {/* Works without JS */}
  <input type="email" name="email" required />
  <button type="submit">Subscribe</button>
</form>

// JS layer enhances it — only if JS is available
if (typeof window !== 'undefined') {
  document.querySelector('form')?.addEventListener('submit', async (e) => {
    e.preventDefault();               // Override native submission
    const email = e.target.email.value;
    const result = await subscribeUser(email);
    showInlineSuccess(result);         // Enhanced UX
  });
}

// Feature detection — not browser detection
// ❌ Never do this:
if (navigator.userAgent.includes('Safari')) { useFallback(); }

// ✅ Test the capability itself:
const supportsContainerQueries = CSS.supports('container-type: inline-size');
const supportsWebP = await testWebP(); // createImageBitmap test
const supportsIntersectionObserver = 'IntersectionObserver' in window;

// CSS progressive enhancement
.grid {
  display: flex;               /* Baseline: all browsers */
  flex-wrap: wrap;
}

@supports (display: grid) {
  .grid {
    display: grid;             /* Enhancement: grid-capable browsers */
    grid-template-columns: repeat(auto-fit, minmax(200px, 1fr));
  }
}
Common mistakes
Confusing the two terms — interviewers notice. PE starts from the bottom with a working baseline; GD starts from the top with the rich experience and degrades. Most teams practice GD (build for modern browsers, add fallbacks). Saying "we use progressive enhancement" when you mean GD is a credibility issue.
Using @supports not for progressive enhancement — the common mistake is writing fallback CSS after the enhanced CSS. CSS cascade order is the natural progressive enhancement mechanism: write the baseline rule first, then override with @supports. The unsupporting browser ignores the @supports block and keeps the baseline.
Interview perspective
"What is the difference between progressive enhancement and graceful degradation?"
PE is bottom-up: build a working baseline (semantic HTML, no JS), add enhancements for capable environments. Every user gets something functional. GD is top-down: build the full experience, add fallbacks for incapable environments. PE is philosophically the web's native model; GD is closer to native app development. Modern SPAs are mostly GD — they require JS and then add offline fallbacks. PE is most natural for content sites and server-rendered apps.

Edge rendering

Edge FunctionsCloudflare WorkersStreaming SSRPartial hydration
Definition

Edge rendering moves the rendering compute from a centralised origin server to geographically distributed edge nodes — the same infrastructure that CDNs use to serve static assets. Instead of a user in Mumbai waiting 200ms for a response from a US-East server, their request is handled by an edge node that is <50ms away. Edge Functions (Vercel) and Cloudflare Workers run JavaScript at the network edge, enabling dynamic personalised pages, A/B testing, authentication, and full SSR at CDN-level latency. The constraint: edge runtimes are stripped-down environments — no Node.js native modules, no filesystem access, limited runtime APIs. Code must run in a V8 isolate, not a full Node.js process.

How edge runtimes work

Cloudflare Workers use V8 isolates — not Node.js processes. Each request gets its own V8 isolate (extremely lightweight, ~5ms startup vs ~100ms for a container). Isolates share the V8 engine process but have separate heaps. Workers run globally: code is distributed to 300+ PoPs (Points of Presence), and the nearest one handles each request. The programming model uses the Service Worker API: a fetch event handler. The Fetch API, Cache API, KV storage, and Durable Objects are available. Node.js-specific modules (fs, path, crypto, http) are not — or have polyfills. Vercel Edge Functions wrap this in a Next.js-compatible API: export const runtime = 'edge' on a route converts it from a Node.js serverless function to a Cloudflare Worker-like deployment.

Subtopics

Use cases for edge

Personalisation without latency: choose locale, A/B variant, or auth-gated content at the edge before sending HTML. Middleware: auth checks, redirects, bot filtering. Streaming SSR for globally distributed users. Edge-side includes (ESI): compose fragments from different origins.

Streaming SSR at edge

React's renderToPipeableStream (Node) / renderToReadableStream (edge) starts sending HTML before all async data resolves. Suspense boundaries flush as their data arrives. First byte reaches the user in ~50ms from edge; page streams in progressively. TTFB is edge-latency, not origin-latency.

Partial hydration

Ship less JS by only hydrating interactive components. Static components (headers, footers, text content) never get JS. Interactive islands get their JS. Astro pioneered this; Next.js RSC implements it via server vs client component split. Reduces TTI for content-heavy pages.

Edge KV and Durable Objects

Cloudflare KV: globally replicated key-value store, eventually consistent, for session data, feature flags, config. Durable Objects: single-location stateful objects with consistent storage — for real-time collaboration, game state, WebSocket coordination. Session state without a central database.

Limitations

No Node.js native modules. Cold start constraints (isolate startup faster than container but still exists). No persistent filesystem. Some libraries don't work in edge runtimes (Prisma, bcrypt, sharp). Execution time limits (50ms CPU time on Cloudflare free tier). Must check library edge compatibility.

Vercel Edge Config

Ultra-low latency key-value reads at edge nodes (sub-millisecond). For feature flags, A/B test config, allowlists — data that must be read on every request without a database round trip. Replicated to all PoPs; reads are local to the edge node handling the request.

Code example
// Next.js Edge Middleware — runs at edge before any page
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export const config = { matcher: ['/((?!_next|favicon).*)'] };
export const runtime = 'edge'; // Runs in V8 isolate at 300+ PoPs

export function middleware(req: NextRequest) {
  const country = req.geo?.country ?? 'US';          // Edge-detected, no DB
  const locale = req.headers.get('accept-language')?.split(',')[0] ?? 'en';

  // A/B test — split traffic at edge, zero origin hit for the decision
  const bucket = Math.random() > 0.5 ? 'control' : 'variant';
  const res = NextResponse.next();
  res.cookies.set('ab-bucket', bucket, { maxAge: 60 * 60 * 24 });

  // Personalised redirect without origin round trip
  if (country === 'IN' && req.nextUrl.pathname === '/') {
    return NextResponse.redirect(new URL('/in', req.url));
  }
  return res;
}

Common mistakes
Using Node.js-only packages in edge routes — Prisma (uses native binary), bcrypt (C++ binding), and sharp (libvips) don't work in V8 isolates. Use edge-compatible alternatives: Kysely + Neon HTTP for Postgres at edge, Web Crypto API instead of bcrypt, Cloudflare Images instead of sharp. Always test with export const runtime = 'edge' locally before deploying.
Treating edge functions as always-on servers — V8 isolates terminate after the request. No in-memory caching persists between requests (use KV or Cache API instead). Writing to a global variable and expecting it to be there on the next request is a silent bug.
Interview perspective
"What is the difference between a Cloudflare Worker and a traditional serverless function?"
Traditional serverless (Lambda): Node.js process, full runtime, ~100ms cold start, runs in one region. Cloudflare Workers: V8 isolate, stripped runtime (no Node.js native modules), ~5ms cold start, runs at 300+ global PoPs nearest to the user. Workers trade capability for global low-latency. The tradeoff: many libraries that work in Node.js don't work at the edge — you must use edge-compatible alternatives.

Islands architecture

Astro modelSelective hydrationZero JS defaultBundle reduction
Definition

Islands architecture (coined by Jason Miller, Preact co-creator) is a rendering approach where a page is predominantly static HTML generated at build time or on the server, with discrete "islands" of interactivity that are selectively hydrated with JavaScript. The key inversion from SPAs: the default is no JavaScript — each interactive component "opts in" to hydration. A product page with a static header, static product description, and an interactive "Add to Cart" button ships zero JS for the header and description, and only the Add-to-Cart JS for the button island. Astro is the primary implementation; React's RSC + Server Components is a different implementation of the same architectural principle.

How Astro implements islands

In Astro, .astro components render to pure HTML at build time (or on the server) with zero client-side JavaScript. To make a component interactive, you import it from any framework (React, Vue, Svelte, Solid) and use a client directive: <ReactButton client:load />, <Counter client:visible />. The directive controls when the island's JavaScript is loaded and the component is hydrated. Non-interactive components ship only their HTML output — no framework JS, no component JS, no hydration. The result: a page that is 90% static HTML with islands of interactivity loads far less JavaScript than an equivalent SPA.

Client directives (hydration strategies)

client:load

Hydrate immediately on page load. Use for above-fold interactive components that the user needs immediately: navigation menus, hero CTAs. Equivalent to standard React hydration.

client:idle

Hydrate when the browser is idle (requestIdleCallback). Use for below-fold components that aren't immediately critical: comment sections, related articles. Doesn't block the main thread during initial load.

client:visible

Hydrate when the component enters the viewport (IntersectionObserver). Use for far-below-fold islands: footer widgets, deeply nested interactive components. JS loads only when the user scrolls to it.

client:media

Hydrate only when a CSS media query matches: client:media="(max-width: 768px)". Use for mobile-only interactive components. Desktop users never download the JS; mobile users get it on load.

Islands vs SPA vs RSC comparison

SPA (React default)

All JS downloaded on first visit. Full hydration of every component. JS is the runtime. SSR is optional add-on. Rich interactivity everywhere. Best for: app-like UIs, dashboards, highly interactive tools.

Islands (Astro)

HTML-first. JS only for interactive islands. Directives control hydration timing. Mix frameworks per island. Best for: content sites with pockets of interactivity — blogs, documentation, marketing, e-commerce PDPs.

RSC (React)

Component-level server/client split. Server components ship no JS. Client components hydrate. Fine-grained control within React. Best for: React apps needing server-state colocation and granular bundle control with React's DX.

    <p>{product.description}</p>

    <AddToCartButton
      client:load
      productId={product.id}
      price={product.price}
    />

    <!-- ISLAND: Svelte component, hydrates when scrolled into view -->
    <ReviewSlider

  </body>

      <p>{product.description}</p> {/* Server: 0 JS */}
      <AddToCartButton productId={product.id} /> {/* Client: has JS */}

    </>

}

Performance implications

A typical Astro e-commerce page ships ~50-100KB of JavaScript (only the interactive islands' bundles). An equivalent Next.js pages-router SPA ships ~150-300KB (React runtime + all component hydration). The difference is dramatic on mobile. TTI improvement: the main thread isn't busy parsing and executing JS for components that are purely visual. LCP improvement: HTML arrives pre-rendered, no JS needed before the LCP element renders. INP improvement: less JS competing for the main thread during user interactions.

Common mistakes
Making too many islands — if 80% of your page components are client:load, you've recreated an SPA with extra steps. Islands architecture's benefit comes from restraint: the majority of the page should be static HTML. Audit each interactive use case — can it be handled with CSS or native HTML instead of JS?
Islands sharing state — Astro islands are isolated by default. An AddToCart island and a CartCount island in the header can't share React state (separate React trees). Use a shared store (nanostores, for framework-agnostic atoms) or URL/custom events for cross-island communication.
Interview perspective
"What is islands architecture and how does it differ from an SPA?"
Islands: HTML-first rendering, with selective JavaScript hydration only for interactive components. The default is zero JS; each island opts in. SPAs: JavaScript-first, full hydration of every component. Islands ship dramatically less JS — only the interactive islands' bundles. Best for content-heavy pages with pockets of interactivity. The RSC model is React's implementation of the same principle: server components ship no JS, client components are the islands.
"How does Astro's client:visible directive work?"
Uses IntersectionObserver to watch the island's placeholder element. When the element enters the viewport, Astro dynamically loads the island's JavaScript bundle and hydrates the component in place. Until then, the server-rendered HTML shows a static version (or nothing, depending on the component). The JS never loads if the user never scrolls to the island — pure bandwidth savings.
