fetch API internals

Request/ResponseReadableStreamAbortControllerBody consumption
Definition

The Fetch API is the browser's modern interface for making HTTP requests. Unlike XHR, it is Promise-based, composable, and built around the WHATWG Streams specification — response bodies are ReadableStreams, not synchronously available strings. fetch() returns a Promise that resolves to a Response object when the response headers arrive — not when the body is complete. The body is a separate async operation. This separation enables streaming responses: you can begin processing a response before it finishes downloading, which is how server-sent HTML streaming and large file downloads work efficiently in the browser.

How it works internally

When you call fetch(url, options), the browser constructs a Request object and hands it to the Network Process (separate OS process from the renderer). The Network Process handles DNS, TCP, TLS, and HTTP. When response headers are received, a Response object is created in the Renderer Process and the Promise resolves. The response body remains as a ReadableStream — a chain of chunks delivered from the Network Process as bytes arrive. Calling response.json() internally reads the stream to completion, decodes the bytes as UTF-8, then parses as JSON. It returns a new Promise. Calling any body reader (json(), text(), arrayBuffer(), blob()) consumes the stream — you cannot call it twice. The body has a bodyUsed flag that becomes true after consumption. response.clone() creates a tee of the underlying stream, enabling reading the body twice.

Subtopics

Request object

Constructed from (url, init). Init: method, headers, body, mode (cors/no-cors/same-origin), credentials (omit/same-origin/include), cache, redirect, signal (AbortController). Immutable after construction. Body can be: string, URLSearchParams, FormData, Blob, BufferSource, ReadableStream.

Response object

Properties: ok (status 200-299), status, statusText, headers, body (ReadableStream), bodyUsed, url, redirected, type. Methods: json(), text(), arrayBuffer(), blob(), formData(), clone(). The Promise from fetch() resolves here — even for 4xx/5xx. Only network failure rejects.

ReadableStream body

A pull-based stream. Call response.body.getReader() for a ReadableStreamDefaultReader. reader.read() returns a Promise resolving to {value: Uint8Array, done: boolean}. Process chunks as they arrive — critical for large payloads and server-sent partial HTML.

AbortController

Create before fetch: const controller = new AbortController(). Pass signal: fetch(url, { signal: controller.signal }). Cancel: controller.abort(reason). The fetch Promise rejects with AbortError. The underlying network request is cancelled in the Network Process. One controller can abort multiple fetches.

Credentials & cookies

Default credentials: 'same-origin' — cookies sent only to same origin. 'include': send cookies cross-origin (requires CORS Access-Control-Allow-Credentials: true and non-wildcard origin). 'omit': never send cookies. XHR default was equivalent to 'same-origin'; fetch changed this, catching developers off guard.

No timeout built-in

fetch has no native timeout. Pattern: AbortController with setTimeout, or AbortSignal.timeout(5000) (modern — creates a signal that auto-aborts after ms). Without a timeout, hung requests hold connections indefinitely, exhausting connection pools.

Code example
// The Promise resolves on headers — body is still streaming
const response = await fetch('/api/large-export');
console.log(response.ok);         // true — headers arrived, body not yet
console.log(response.status);     // 200
console.log(response.bodyUsed);   // false — not consumed yet

// Stream response body chunk by chunk
const reader = response.body.getReader();
const decoder = new TextDecoder();
while (true) {
  const { done, value } = await reader.read(); // Uint8Array chunk
  if (done) break;
  processChunk(decoder.decode(value, { stream: true }));
  // Process before full body arrives — critical for server-sent HTML streaming
}

// ❌ Body consumed twice — throws
const r = await fetch('/api/data');
const text = await r.text();
const json = await r.json(); // TypeError: body already consumed

// ✅ Clone before consuming (tees the underlying ReadableStream)
const r = await fetch('/api/data');
const clone = r.clone();                   // Tee the stream
const [text, json] = await Promise.all([r.text(), clone.json()]);

// AbortController — timeout pattern
async function fetchWithTimeout(url, ms = 5000) {
  const signal = AbortSignal.timeout(ms); // Modern: auto-aborts after ms
  try {
    const res = await fetch(url, { signal });
    return res.json();
  } catch (err) {
    if (err.name === 'AbortError' || err.name === 'TimeoutError') {
      throw new Error(`Request timed out after ${ms}ms`);
    }
    throw err;
  }
}

// ❌ Common mistake: assuming 4xx rejects the Promise
const r = await fetch('/api/not-found');
// r.status === 404, but Promise did NOT reject!
// Only network errors (offline, DNS failure) reject fetch

// ✅ Check ok / status explicitly
if (!r.ok) throw new Error(`HTTP ${r.status}: ${r.statusText}`);
Real-world use case

Streaming LLM responses (EngineX AI)

LLM APIs (OpenAI, Anthropic) use Server-Sent Events or raw streaming responses. With fetch + ReadableStream, you can render tokens to the UI as they arrive: fetch('/api/generate') → response headers arrive (200 OK) → stream response.body chunk by chunk → decode each Uint8Array as UTF-8 → parse SSE lines → append tokens to the displayed text. Without streaming, the user would wait the full generation time (10-30s) before seeing any output. With streaming, first token appears in ~200ms. This is the exact pattern powering EngineX's AI resume parser.

Common mistakes
Assuming fetch rejects on HTTP errors (4xx, 5xx) — it only rejects on network failure. A 500 response resolves the Promise. Always check response.ok or response.status before calling response.json(), or you'll silently try to parse an error HTML page as JSON and get a confusing parse error.
Not providing a timeout — fetch has no default timeout. On poor mobile connections, requests can hang indefinitely. Always use AbortSignal.timeout(ms) or a manual AbortController + setTimeout pattern. This is the number one cause of user-perceived hangs in production apps.
Calling response.json() on non-JSON responses (HTML error pages, binary files) — always check response.headers.get('Content-Type') or guard with response.ok before parsing. A 502 Bad Gateway from a load balancer returns HTML, and response.json() will throw a JSON parse error that obscures the real HTTP error.
Interview perspective
"When does fetch() reject its Promise?"
Only on network-level failures: DNS resolution failure, connection refused, CORS violation (browser blocks the response), offline. HTTP errors (4xx, 5xx) resolve the Promise with a Response where ok=false. This is a deliberate design choice — the network succeeded, the server just returned an error. Interviewers test this to distinguish developers who've actually debugged fetch from those who only read tutorials.
"What is a ReadableStream and how does fetch use it?"
A WHATWG Streams specification primitive representing a sequential source of data consumed asynchronously. fetch() resolves on response headers; the body is a ReadableStream of Uint8Array chunks from the Network Process. You can process chunks as they arrive (streaming) or consume all at once via convenience methods (json(), text()). Body consumption is one-time — bodyUsed flag prevents double-reading. clone() tees the stream for dual consumption.

CORS

PreflightSimple vs complexResponse headersCommon mistakes
Definition

Cross-Origin Resource Sharing (CORS) is a browser security mechanism that controls which cross-origin HTTP requests are permitted. It is enforced exclusively by the browser — servers receive every request, but the browser may block the response from the page's JavaScript based on CORS headers. A "cross-origin" request is any request where the scheme, host, or port differs from the page origin. CORS exists to prevent malicious scripts from silently reading data from authenticated APIs on behalf of logged-in users (the CSRF-adjacent attack). The key insight: CORS is not about security of the server's data itself — it's about what the page's JavaScript is allowed to read from responses.

Simple vs complex requests — the most critical distinction

Not all cross-origin requests trigger a preflight. Simple requests are sent directly (one round trip). Complex requests trigger a preflight OPTIONS request first — the browser asks the server "will you accept this?" before sending the real request. The preflight costs one extra RTT. Whether a request is simple depends on the method, headers, and content type.

What the server must send

For all CORS responses

Access-Control-Allow-Origin: https://yourapp.com (or * for public APIs). Wildcard * cannot be used when credentials are included — must be explicit origin. Without this header, the browser blocks the response even though it arrived at the JS layer.

For preflight response (OPTIONS)

Access-Control-Allow-Methods: GET, POST, PUT, DELETE — which methods are permitted. Access-Control-Allow-Headers: Authorization, Content-Type, X-Custom — which request headers are permitted. Access-Control-Max-Age: 86400 — browser caches the preflight result for this many seconds, avoiding a preflight on every subsequent request.

For credentialed requests

Access-Control-Allow-Credentials: true — required when the request sends cookies or Authorization headers AND the page needs to read the response. Must be paired with a non-wildcard Access-Control-Allow-Origin. Without this, fetch with credentials: 'include' will receive the response but JavaScript cannot read it.

Exposing response headers

By default, JS can only read basic response headers (Content-Type, Cache-Control, Expires, Last-Modified). To expose others (X-Request-Id, X-RateLimit-Remaining): Access-Control-Expose-Headers: X-Request-Id, X-RateLimit-Remaining. Without this, response.headers.get('X-Request-Id') returns null even though the header was sent.

Code example
// What the browser sends as a preflight:
OPTIONS /api/users HTTP/1.1
Origin: https://myapp.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: content-type, authorization

// What the server must respond with (Express.js):
app.use((req, res, next) => {
  const allowedOrigins = ['https://myapp.com', 'https://staging.myapp.com'];
  const origin = req.headers.origin;

  if (allowedOrigins.includes(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);    // Not * if credentials
    res.setHeader('Access-Control-Allow-Credentials', 'true');
  }

  if (req.method === 'OPTIONS') { // Preflight
    res.setHeader('Access-Control-Allow-Methods', 'GET,POST,PUT,PATCH,DELETE');
    res.setHeader('Access-Control-Allow-Headers', 'Content-Type,Authorization,X-Request-Id');
    res.setHeader('Access-Control-Max-Age', '86400'); // Cache preflight for 1 day
    res.setHeader('Access-Control-Expose-Headers', 'X-Request-Id,X-RateLimit-Remaining');
    return res.status(204).end();
  }
  next();
});

// Browser-side: what triggers a preflight
// Simple (no preflight):
fetch('https://api.other.com/data');

// Complex (preflight fired):
fetch('https://api.other.com/data', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', 'Authorization': 'Bearer ...' },
  body: JSON.stringify(data),
  credentials: 'include',  // Sends cookies cross-origin
});
// Browser fires OPTIONS first, checks CORS headers, then sends the real POST
Common mistakes
Using Access-Control-Allow-Origin: * with credentials: 'include' — the browser rejects this combination. When credentials are involved, the allowed origin must be an explicit domain, not wildcard. This is the most common CORS configuration error in production APIs.
Thinking CORS protects your API from server-side — it doesn't. CORS only controls what JavaScript in a browser page can read from the response. A curl command, Postman, or server-to-server call ignores CORS entirely. CORS is purely a browser security feature.
Not setting Access-Control-Max-Age — without it, the browser sends a preflight OPTIONS request before every single complex request. On a dashboard that makes 20 API calls per page, this adds 20 extra OPTIONS round trips. Setting max-age to 86400 caches the preflight result for a day, eliminating this overhead.
Forgetting that CORS errors surface as a generic "Network Error" in JS — the actual CORS failure details are only visible in the browser DevTools console, not in the catch(err) handler. err.message will be "Failed to fetch", not "CORS: missing Access-Control-Allow-Origin". This trips up debugging.
Interview perspective
"What triggers a CORS preflight request?"
Any request that doesn't meet all three simple-request criteria: method must be GET/POST/HEAD; Content-Type must be one of the three allowed types; only safe headers allowed. Using application/json or Authorization header — which virtually every real API requires — makes it a complex request. The browser sends OPTIONS first to verify the server accepts the real request.
"Why can't you use Access-Control-Allow-Origin: * with cookies?"
The wildcard means "any origin can read this response." Allowing any origin to read a credentialed (cookie-bearing) response would let any malicious site read your authenticated API data on behalf of logged-in users. The browser enforces this: credentials + wildcard = error. Exact origin required to establish explicit trust.
"Does CORS protect your API from being called by scripts?"
No. CORS is entirely browser-enforced. A server-side script, curl, Postman, or any non-browser client ignores CORS headers entirely. CORS only prevents browser-side JavaScript from reading cross-origin responses. API authentication (tokens, API keys) is the actual protection against unauthorized server-side calls.

WebSocket in the browser

Connection lifecycleReconnectionHeartbeatvs SSE
Definition

WebSocket is a persistent, full-duplex communication protocol over a single TCP connection. Unlike HTTP, it is bidirectional — the server can push data to the client at any time without a client request. The connection starts as an HTTP/1.1 upgrade handshake, then switches to the WebSocket protocol (ws:// or wss://). Once established, data is exchanged as frames — lightweight binary or text messages with no HTTP overhead per message. WebSocket is the right choice when you need server→client push with sub-100ms latency or bidirectional communication (chat, collaborative editing, live trading data, multiplayer gaming).

WebSocket states

CONNECTING (0)

Handshake in progress. ws.send() throws if called here. Wait for onopen before sending. Buffer messages if you need to send during connection.

OPEN (1)

Connection established. ws.send(string|ArrayBuffer|Blob) is available. Messages arrive via ws.onmessage = ({data}) => {}. data is string or Blob/ArrayBuffer depending on binaryType setting.

CLOSING (2)

Close handshake initiated (close frame sent, waiting for server close frame). After both sides exchange close frames, transitions to CLOSED.

CLOSED (3)

Connection terminated. onclose fires with code and reason. Code 1000 = normal closure. Code 1006 = abnormal (network drop, no close frame). Code 1008 = policy violation. Reconnect logic triggers here.

Subtopics

Reconnection strategy

The browser does NOT auto-reconnect after close. You must implement it: on onclose, wait then create a new WebSocket. Use exponential backoff to avoid thundering herd: attempt 1 → 1s, attempt 2 → 2s, attempt 3 → 4s, max ~30s. Add jitter (random offset) so all clients don't reconnect simultaneously after a server restart.

Heartbeat / ping-pong

TCP connections can drop silently (NAT timeout, firewall idle timeout, mobile radio sleep). Without a heartbeat, neither side knows the connection is dead. Server sends ping frames (WebSocket protocol level). Client's browser automatically responds with pong. If no pong arrives within a timeout window, server closes. Client-side: send a custom application-level heartbeat if the server doesn't ping.

Message buffering

ws.bufferedAmount: bytes queued but not yet sent. If the client is sending faster than the connection can drain, the buffer grows. Monitor it before sending large messages. Backpressure: if bufferedAmount > threshold, pause sending until it drains.

Binary messages

Set ws.binaryType = 'arraybuffer' (default) or 'blob'. Use for binary protocols, compressed data, game state. Binary frames are more efficient than JSON strings for numeric/structured data. Pair with MessagePack or protobuf for structured binary serialisation.

WebSocket vs Server-Sent Events (SSE)

WebSocket

Full-duplex (client sends AND server pushes). Custom protocol over upgraded TCP. Binary and text. No built-in reconnect. Requires WebSocket server support. Use for: chat, collaborative editing, gaming, real-time bidding, live collaboration (Zombie Hunt).

Server-Sent Events (SSE)

Server-only push over regular HTTP. Text only (JSON events). Built-in reconnect with Last-Event-ID. Works through HTTP/2 multiplexing. No special server needed — just a long-lived HTTP response. Use for: live feeds, notifications, streaming LLM output, dashboard updates where client never sends data.

Code example
// Production WebSocket client with reconnection + heartbeat
class ReliableWebSocket {
  #ws = null;
  #reconnectAttempts = 0;
  #maxDelay = 30000;
  #heartbeatTimer = null;
  #messageQueue = [];

  constructor(private url: string, private onMessage: (data: any) => void) {
    this.connect();
  }

  connect() {
    this.#ws = new WebSocket(this.url);

    this.#ws.onopen = () => {
      this.#reconnectAttempts = 0;
      this.#startHeartbeat();
      this.#messageQueue.forEach(msg => this.#ws.send(msg)); // Drain queued
      this.#messageQueue = [];
    };

    this.#ws.onmessage = ({ data }) => {
      if (data === 'pong') return; // Heartbeat response, not user data
      this.onMessage(JSON.parse(data));
    };

    this.#ws.onclose = (event) => {
      this.#stopHeartbeat();
      if (event.code === 1000) return; // Normal closure — don't reconnect
      this.#scheduleReconnect();
    };

    this.#ws.onerror = () => { this.#ws.close(); }; // Close triggers onclose → reconnect
  }

  #startHeartbeat() {
    this.#heartbeatTimer = setInterval(() => {
      if (this.#ws.readyState === WebSocket.OPEN) this.#ws.send('ping');
    }, 30000); // Every 30s — before NAT timeout
  }

  #scheduleReconnect() {
    const delay = Math.min(
      1000 * 2 ** this.#reconnectAttempts,  // Exponential backoff
      this.#maxDelay
    ) + Math.random() * 1000;              // Jitter: prevent thundering herd
    this.#reconnectAttempts++;
    setTimeout(() => this.connect(), delay);
  }

  send(data: any) {
    const msg = JSON.stringify(data);
    if (this.#ws?.readyState === WebSocket.OPEN) {
      this.#ws.send(msg);
    } else {
      this.#messageQueue.push(msg); // Buffer while reconnecting
    }
  }

  close() { this.#ws?.close(1000); } // Code 1000 = normal, won't reconnect
}
Common mistakes
Not implementing reconnection logic — browsers do not auto-reconnect WebSockets. A network hiccup, server restart, or proxy timeout will silently close the connection. Without reconnection code, the user appears to be connected but receives no updates until they reload.
Sending messages before the connection is OPEN — calling ws.send() in CONNECTING state throws an error. Buffer messages in a queue and drain them in onopen. This is especially common when components are initialized before WebSocket connects.
Reconnecting immediately without backoff — if the server restarts, all clients reconnect simultaneously (thundering herd). This can overwhelm the server. Exponential backoff + jitter distributes reconnection across a time window, giving the server a chance to recover.
Interview perspective
"How does a WebSocket connection differ from HTTP?"
WebSocket starts as HTTP (GET with Upgrade: websocket header). Server responds 101 Switching Protocols. The TCP connection is then repurposed — HTTP protocol is abandoned and the WebSocket framing protocol takes over. Full-duplex: both sides can send at any time with no per-message HTTP overhead. HTTP is half-duplex (request/response pattern). Persistent TCP connection vs HTTP connection-per-request.
"When would you choose SSE over WebSocket?"
SSE when: server-only push (no client→server messages), text-only data, you want built-in reconnect with event ID replay, or you're on HTTP/2 (SSE multiplexes efficiently). WebSocket when: bidirectional (client also sends), binary data, sub-100ms latency requirements, or custom protocol. For streaming LLM output or live dashboards: SSE. For chat, gaming, or collaborative editing: WebSocket.

GraphQL from the frontend

Query vs mutation vs subscriptionFragment colocationN+1 on clientOver-fetching
Definition

GraphQL is a query language and runtime for APIs where the client specifies exactly what data it needs. Unlike REST (fixed URL = fixed response shape), a GraphQL request contains a typed document describing the fields to return. The server resolves only those fields. A single GraphQL endpoint handles all operations: queries (reads), mutations (writes), and subscriptions (real-time streams over WebSocket). From the frontend perspective, GraphQL eliminates over-fetching (receiving more fields than needed) and under-fetching (needing multiple REST endpoints for one screen), at the cost of query complexity, caching challenges, and a learning curve.

Operations

Query

Read operation. Cacheable. Can be parallelised — multiple queries can run simultaneously. Sent as GET (simple queries) or POST with JSON body. Response: { data: {...}, errors: [...] }. Errors can coexist with partial data.

Mutation

Write operation. NOT cached. Sequential by convention (mutations run one at a time to avoid conflicts). Always POST. Response includes affected data — use to update local cache (Apollo: writeFragment, React Query: setQueryData). Side effects belong in mutations.

Subscription

Real-time stream. Implemented over WebSocket (typically graphql-ws protocol). Client sends subscription document; server pushes updates when underlying data changes. Client-side: useSubscription hook (Apollo/URQL). Each push is a GraphQL result with the subscription's data shape.

Fragment colocation

Fragment colocation is the pattern of defining the data requirements of each component as a GraphQL fragment next to the component itself, then composing those fragments into a parent query. Instead of the parent knowing what every child needs (violating component encapsulation), each child declares its own data contract with a fragment. The parent query spreads all child fragments: query Page { user { ...UserCard ...UserStats } }. This enables: components to be changed independently (add a field to the fragment → automatically included in all queries using it), Relay-style type-safe co-location, and automatic data masking (a component only sees its own fragment's data, not the full query result). Apollo Client's useFragment and Relay's fragment system implement this.

N+1 on the client

The N+1 problem in GraphQL occurs on the client when rendering a list where each list item needs data that requires a separate request. Example: render 50 posts, and each post component independently calls useQuery to fetch its author details. Result: 1 query for posts + 50 queries for authors = 51 requests. Solutions: (1) Request author data in the parent posts query (include author fields in the list query). (2) Use DataLoader-style batching on the server. (3) Use fragment colocation — the post query fragment includes author { name avatar }, so one query fetches everything. The client-side N+1 is often worse than the server-side equivalent because network latency compounds.

Code example
// Fragment colocation — UserCard owns its data requirements
const USER_CARD_FRAGMENT = gql`
  fragment UserCardFields on User {
    id
    name
    avatarUrl
    role
  }
`;

function UserCard({ user }) { /* uses id, name, avatarUrl, role */ }

    team(id: $teamId) {
      name
      members {
        ...UserCardFields        # Spread child's fragment
      }
    }
  }

// N+1 on client — DO NOT do this
function PostList({ posts }) {

// Subscription with apollo client
const NEW_MESSAGE_SUB = gql`

    newMessage(roomId: $roomId) {

Common mistakes
Treating GraphQL errors like HTTP errors — a GraphQL response can have HTTP 200 with errors: [...] in the body. Apollo and URQL surface this as a separate error state. Always check both data (may be partial) and errors.
Over-using subscriptions when polling or cache invalidation suffices — subscriptions open and hold WebSocket connections, adding server complexity. For data that changes every few minutes, a pollInterval on a query is simpler. Subscriptions are for genuinely real-time data (<1s latency requirement).
Not colocating fragments — putting all field requirements in the page-level query means changing a child component requires editing a distant parent query. Fragment colocation makes components self-contained data contracts.
Interview perspective
"What is the N+1 problem in GraphQL and how do you prevent it on the client?"
N+1: a list query fetches N items, then each item triggers an additional request for related data = N+1 total requests. Client-side prevention: include related data in the list query (author data in the posts query). Fragment colocation enforces this — the list query fragment must include all fields any list item component uses. Server-side: DataLoader batches and deduplicates resolver calls.
"What is fragment colocation and why does it matter?"
Each component declares its own data requirements as a GraphQL fragment, co-located in the component file. Parent queries compose child fragments. Benefits: components are self-contained data contracts (change a field, only edit the component); no parent needs to know what children need; enables Relay-style data masking (component only accesses its own fragment's data). The alternative — page-level queries knowing all child requirements — breaks component encapsulation.

HTTP caching headers

Cache-ControlETagVaryRevalidation
Definition

HTTP caching headers are the negotiation layer between the browser (and intermediate CDN caches) and the server, controlling whether a cached response can be used and for how long. The browser maintains an HTTP cache (disk cache) keyed by URL. Before making a network request, it checks whether a cached response exists and whether it's still valid. "Fresh" responses are served instantly from cache. "Stale" responses require revalidation with the server (conditional GET). Understanding this layer precisely is critical for building fast repeat-visit experiences, and for debugging the class of production bugs where users see stale content after a deployment.

Cache-Control directives

max-age=N

Response is fresh for N seconds from the time it was generated. Browser serves from cache without any network request during this window. After N seconds, becomes stale — revalidation needed.

no-cache

Misleadingly named. The response IS cached, but must be revalidated with the server before each use (via conditional GET with ETag / Last-Modified). If server says 304 Not Modified, cached copy is used — no body re-download.

no-store

Never cache this response anywhere. Every request goes to the network. Use for sensitive user data (banking pages, personal data). Never cached even transiently.

immutable

Tells browser: the content at this URL will never change while max-age is valid. Never revalidate during max-age, even on force-refresh. Use with content-hashed URLs (JS/CSS bundles). max-age + immutable = perfect caching.

stale-while-revalidate=N

After max-age expires: serve stale content immediately, then fetch fresh content in the background. The background response is used for the next request. Trades one stale response for instant load speed. CDN-level commonly used.

s-maxage=N

Like max-age but for shared caches (CDN, proxy). CDN caches for N seconds. Browser ignores s-maxage and uses max-age. Enables different TTLs for CDN edge (long) vs browser (short) for the same resource.

Validation headers — conditional GET

ETag / If-None-Match

Server sends ETag: "abc123" (hash of the response content). Browser stores it. On revalidation: If-None-Match: "abc123". If content unchanged: server responds 304 Not Modified (no body, saves bandwidth). If changed: 200 with new body and new ETag.

Last-Modified / If-Modified-Since

Server sends Last-Modified: Wed, 21 Oct 2024 07:28:00 GMT. Browser sends If-Modified-Since: Wed, 21 Oct 2024 07:28:00 GMT on revalidation. Less precise than ETag (1-second granularity, can't detect same-second changes). ETag preferred.

304 Not Modified

The optimal revalidation outcome: server confirms content is unchanged, browser uses cached body. Network cost: headers only (~200 bytes), not the full response body. Saves bandwidth but still costs one RTT (vs cache hit which costs zero RTT).

Vary header

Vary: Accept-Encoding tells caches to store separate entries per Accept-Encoding value (gzip, br, identity). Vary: Accept-Language per language. Vary: Cookie or Vary: Authorization effectively disables CDN caching (every user is different). Misuse of Vary is a common cause of cache fragmentation.

Common mistakes
Confusing no-cache with no-store — no-cache means "cache but always validate." no-store means "never cache." Using no-store on static assets wastes the cache entirely. Using no-cache where no-store is needed can leave sensitive data in browser disk cache. The naming is confusing by historical accident.
Not versioning static assets (content hashing) while using long max-age — without content hashing, deploying new JS with the same filename (app.js) leaves users serving stale code for up to max-age seconds. Content hashing (app-B3k2mX.js) is mandatory for long-TTL caching.
Setting Vary: Cookie on API responses — effectively prevents CDN caching because every user has a different cookie. The CDN creates a separate cache entry per cookie value, exhausting cache storage. Use Cache-Control: private for user-specific responses instead.
Interview perspective
"Explain the difference between Cache-Control: no-cache and no-store."
no-cache: browser caches the response but sends a conditional GET (If-None-Match with ETag) before reusing it. Server says 304 → use cache (saves bandwidth). Server says 200 → update cache. Still faster than no-store for unchanged content. no-store: never write to any cache. Every request is a full round trip. Use for genuinely sensitive content. Most HTML documents should use no-cache, not no-store.
"What is an ETag and how does it enable efficient caching?"
An ETag is a server-generated token representing the version of a resource (usually a hash of its content). On first response, server sends ETag header. Browser caches it. On revalidation (after max-age), browser sends If-None-Match: etag-value. If content hasn't changed, server responds 304 with no body — saves re-downloading the content while still confirming freshness. One RTT instead of full round trip + body download.

Request waterfall

Sequential vs parallelParallel fetchingResource hintsLCP impact
Definition

A request waterfall is a sequence of HTTP requests where each request can only start after the previous one completes. Every waterfall step adds at minimum one full RTT to the critical path. On a 150ms RTT mobile connection, a 3-step waterfall adds 450ms of pure waiting before any meaningful content renders — even if each individual request takes only 50ms. Waterfalls are the single most impactful performance issue in JavaScript-heavy applications, particularly SPAs and component-driven data fetching. Eliminating waterfalls is the difference between a 2-second LCP and a 5-second LCP on real mobile networks.

Root causes of waterfalls

Component-level data fetching

Parent component fetches, renders children, children fetch their own data. Each level adds one RTT. A 4-level deep component tree = 4 sequential round trips before the deepest content renders. The "render-then-fetch" pattern.

Lazy-loaded JS discovering data needs

Lazy code-split chunk loads (RTT 1), executes and discovers it needs to fetch data (RTT 2). The data dependency is hidden behind the JS download — the browser couldn't prefetch it. Two RTTs that look like one.

Auth gating

App fetches user session first, then conditionally fetches feature data based on user role. Even if the feature data is the same for all users, it waits behind the auth check. Can be parallelised or eliminated with SSR.

Dependent queries

await fetchUser() → use user.teamId → await fetchTeam(teamId). Genuinely sequential when data from request A is required to construct request B. Eliminate by: including team data in the user response, or using parallel SSR.

Elimination strategies

Parallel fetching with Promise.all

If requests are independent: start all simultaneously, await all. Promise.all([fetchUser(), fetchOrders(), fetchPrefs()]). Total time = max of all, not sum. The single most impactful fix for fetch-heavy components.

Server-side parallel fetching (RSC/SSR)

Move data fetching to the server. Next.js Server Components can fire parallel fetches without RTT to the data layer (co-located server). First HTML byte includes all data. Client receives fully-formed content, no client-side waterfall.

Hoist fetches to route level

React Router loaders / Next.js layout data fetching: begin fetching at the route level before child components mount. Children receive data as props — no child-level fetch, no waterfall. Data is already in flight during navigation.

Resource hints as waterfall shortcut

rel=preconnect to API origin before JS loads. rel=prefetch for next-page data. link rel=preload for data that the HTML can discover early but JS would fetch late. Shifts discovery from JS execution time to HTML parse time.

Code example
// ❌ Sequential waterfall — 3 RTTs on a component mount
async function DashboardPage() {
  const user    = await fetchUser();          // RTT 1: wait 150ms
  const orders  = await fetchOrders(user.id); // RTT 2: wait 150ms
  const metrics = await fetchMetrics();        // RTT 3: wait 150ms
  // Total: 450ms. User sees blank screen.
}

// ✅ Parallel — same data, one RTT
async function DashboardPage({ userId }) {
  const [user, metrics] = await Promise.all([
    fetchUser(userId),   // Fires simultaneously
    fetchMetrics(),       // Fires simultaneously
  ]);
  const orders = await fetchOrders(user.id); // Genuinely sequential — needs user.id
  // Total: max(user, metrics) + orders = ~300ms (2 RTTs instead of 3)
}

// ❌ Component waterfall — each layer adds one RTT
function Page() {
  const { data: user } = useQuery(GET_USER);   // RTT 1
  if (!user) return null;
  return <UserDashboard userId={user.id} />;   // Mounts only after RTT 1
}
function UserDashboard({ userId }) {
  const { data: orders } = useQuery(GET_ORDERS, { variables: { userId } }); // RTT 2
  if (!orders) return null;
  return <OrderList orders={orders} />;         // Mounts only after RTT 2
}

// ✅ Parallel in React Query via prefetch or hoisted query
function Page({ userId }) {
  // Both start simultaneously — no sequential dependency here
  const { data: user }   = useQuery({ queryKey: ['user', userId], queryFn: getUser });
  const { data: orders } = useQuery({
    queryKey: ['orders', userId],
    queryFn: () => getOrders(userId),
    enabled: !!userId,  // userId is known from props, not from user fetch
  });
  // Both queries fire in parallel on mount
}

// Next.js App Router — data fetching hoisted to server, parallel
export default async function DashboardPage({ params }) {
  const userPromise    = getUser(params.id);    // Start — don't await yet
  const metricsPromise = getMetrics(params.id); // Start simultaneously
  const [user, metrics] = await Promise.all([userPromise, metricsPromise]);
  // Server fetches both before sending any HTML — client sees no waterfall
  return <Dashboard user={user} metrics={metrics} />;
}
Real-world use case

EngineX dashboard — eliminating the auth waterfall

A common SPA pattern: app loads → fetch /auth/session → if authenticated, fetch /user/profile → render dashboard → component fetches /resumes. Three sequential requests before any resume data appears. Fix: (1) The session token is in a cookie — no auth request needed if the first API call includes credentials. (2) Fetch user profile and resumes in parallel once authenticated. (3) Move to Next.js RSC: the server component has direct database access, fetches session + resumes in one parallel server-side operation, and sends pre-rendered HTML to the client. Zero client-side waterfall. First meaningful paint includes actual resume data.

Common mistakes
Awaiting independent requests sequentially — the most common and impactful mistake. const a = await fetchA(); const b = await fetchB() when A and B have no dependency on each other. Total time = A + B instead of max(A, B). Always identify whether requests are genuinely dependent before awaiting sequentially.
Rendering before fetching — mounting a component, showing a loading state, then starting the fetch when the component mounts. The fetch could have been started at route-entry time (React Router loaders, Next.js page data), before the component tree is even rendered. Render-then-fetch creates a guaranteed waterfall.
Over-splitting components where each level has its own useQuery — creates a component-level waterfall. Either hoist the fetch to the parent, pass data as props, or use React Query's prefetch to start the query before the component mounts.
Interview perspective
"What is a request waterfall and how do you eliminate it?"
Sequential fetches where each starts only when the previous completes. Each step adds one RTT. Elimination: (1) Identify which requests are genuinely dependent vs independent. (2) Parallelize independent requests with Promise.all. (3) Hoist fetches to route level (before component mount). (4) Move to SSR/RSC where server can fetch in parallel before first HTML byte. (5) Use resource hints to start connections before JS executes.
"Design the data fetching strategy for a dashboard with user info, recent orders, and metrics."
userId is known from URL params — no auth waterfall needed. Start user, orders, and metrics fetches simultaneously with Promise.all. If SSR: fire all three as server-side promises before awaiting any, stream HTML as soon as shell is ready. If CSR: use React Query with all three useQuery hooks mounted simultaneously — no sequential dependency. Prefetch on route hover for instant perceived load.
