What each metric measures & root causes

LCP — what it measures

Time from navigation start to when the largest visible content element (hero image, H1, video poster) finishes rendering. Not total page load — just the dominant above-fold element. Reported element: largest by rendered area (img, video, background-image via CSS, block text).

LCP — what makes it poor

Slow TTFB (server not responding fast enough). Render-blocking CSS/JS in <head> (delays first paint). LCP resource not discovered early by preload scanner. No CDN edge caching. Large unoptimized images. Lazy-loading the LCP image (critical mistake — should be eager).

LCP — how to fix

CDN with edge caching for HTML (sub-200ms TTFB). Inline critical CSS. fetchpriority="high" on LCP image. <link rel="preload" as="image">. Never lazy-load LCP element. AVIF/WebP + correct sizing. Remove render-blocking scripts above LCP element.

INP — what it measures

Responsiveness to user input across the entire page lifetime (not just first interaction like old FID). Measures the worst-case input latency: time from user gesture (click, key, tap) to the next visual update (next paint after the handler). 75th percentile of all interactions.

INP — what makes it poor

Long JS tasks on the main thread blocking the event loop (>50ms). Heavy synchronous event handlers. Layout thrashing triggered by handlers. Third-party scripts monopolizing the main thread. Large React re-renders triggered by interactions without startTransition.

INP — how to fix

Break long tasks with scheduler.yield(). Move computation to Web Workers. Wrap non-urgent state updates in startTransition. Debounce high-frequency handlers. Remove/defer third-party scripts. Measure with PerformanceObserver event type for long event handlers.

CLS — what it measures

Unexpected layout shifts throughout page lifetime. Each shift scored by: impact fraction (how much of the viewport shifted) × distance fraction (how far elements moved). Cumulative score across all unexpected shifts (excluding shifts within 500ms of user interaction).

CLS — what makes it poor

Images/video without explicit width+height (browser can't reserve space). Late-loading ads or banners injected above content. Web fonts causing FOUT (text reflow). Dynamically injected content above existing content. Animations using top/left instead of transform.

CLS — how to fix

Always set width+height on img/video elements. CSS aspect-ratio for responsive images. font-display: optional eliminates FOUT entirely. Reserve space for ads with min-height. Use transform for animations (compositor-only, no layout). Avoid inserting DOM nodes above existing content.

TTFB — what it measures

Time from navigation request to first byte of the response body. Encompasses: DNS lookup + TCP handshake + TLS negotiation + server processing time + first byte of response. Not a Core Web Vital but a diagnostic metric that bottlenecks LCP.

Code example — measuring all vitals
// web-vitals library — report to analytics
import { onLCP, onINP, onCLS, onTTFB, onFCP } from 'web-vitals';

function sendToAnalytics({ name, value, rating, id }) {
  fetch('/analytics', {
    method: 'POST',
    body: JSON.stringify({ name, value: Math.round(value), rating, id }),
    keepalive: true,  // Survives page unload
  });
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);   // INP replaced FID in March 2024
onCLS(sendToAnalytics);
onTTFB(sendToAnalytics);

// Manual INP investigation — find long event handlers
const obs = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 200) {
      console.warn(`Slow interaction: ${entry.name} took ${Math.round(entry.duration)}ms`);
      console.warn(`Processing time: ${Math.round(entry.processingEnd - entry.processingStart)}ms`);
    }
  }
});
obs.observe({ type: 'event', durationThreshold: 16, buffered: true });

// CLS investigation — which elements shift
const clsObs = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (!entry.hadRecentInput) { // Exclude user-initiated shifts
      entry.sources?.forEach(source => {
        console.warn('CLS source:', source.node, 'score:', entry.value.toFixed(3));
      });
    }
  }
});
clsObs.observe({ type: 'layout-shift', buffered: true });
Performance implications per vital

LCP is the primary SEO and UX signal for perceived load speed — Google uses CWVs as a ranking signal. INP replaced FID in March 2024 because FID only measured the first interaction; INP measures the worst interaction across the session lifetime, making it far harder to game. CLS directly causes users to mis-click (tap target shifts while they're tapping) — a conversion killer in e-commerce. TTFB is a prerequisite for everything: no amount of client-side optimization compensates for a 3-second server response time.

Common mistakes
Adding loading="lazy" to the LCP image — lazy loading defers the request until the image is near the viewport. For the LCP element (which is usually already in the viewport), this delays the most important image on the page. Never lazy-load the LCP element. Use loading="eager" (default) or add fetchpriority="high".
Measuring CWVs only in Lighthouse (lab data) and not in Real User Monitoring. Lighthouse runs on a controlled throttled connection — it won't surface INP issues from complex interaction patterns or CLS issues from ad networks that load differently in the lab.
Ignoring INP because "FID was fine" — FID measured input delay only (time before JS starts handling), while INP measures the full interaction latency including processing time and next paint delay. A "good" FID score app can have terrible INP.
Interview perspective
"INP replaced FID — why? What does INP measure that FID didn't?"
FID measured only the delay before the first interaction's handler started — it excluded processing time and paint delay. It also only measured the first interaction. INP measures the full latency (input → next paint) across all interactions during the session, taking the 75th percentile. Harder to game, more representative of real user experience under load.
"Walk me through diagnosing a poor LCP score in production."
Check TTFB first (server/CDN response speed). Check if LCP element is render-blocked (CSS/JS in head). Check if LCP image has fetchpriority=high or rel=preload. Check image format and size (should be AVIF/WebP, correctly sized for viewport). Check if lazy loading is accidentally applied. Use CrUX data + web-vitals library for field data vs Lighthouse for lab diagnosis.

Code splitting

dynamic importReact.lazyRoute splittingTTI
Definition

Code splitting is the technique of dividing a JavaScript bundle into smaller chunks that are loaded on demand rather than all at once. Without splitting, every user downloads the entire application JS on the first visit — including code for routes they may never visit, features hidden behind permissions, and third-party libraries used only in specific flows. Code splitting reduces the initial parse and execute cost (directly improving TTI and INP), downloads only what's needed for the current view, and allows parallel chunk loading via HTTP/2.

How it works internally

Dynamic import() is the native ECMAScript mechanism. When a bundler (Vite, webpack) encounters import('./Module'), it creates a new chunk boundary — the referenced module and its entire dependency subtree are extracted into a separate file. The bundler outputs a chunk manifest (in Vite: manifest.json) mapping module IDs to chunk filenames. At runtime, import('./Module') returns a Promise. The browser fetches the chunk file (a network request), parses it, executes it, and resolves the Promise with the module exports. React.lazy() wraps this in React's Suspense protocol — it calls the dynamic import, and if the Promise is pending, throws it (triggering the nearest Suspense boundary to show its fallback). When the chunk loads and the Promise resolves, React re-renders the Suspense boundary with the actual component.

Subtopics

Route-based splitting

Split by page/route — the most impactful split. Each route's component tree becomes its own chunk. Load only the current route's JS. On navigation, load the next route's chunk. In Next.js App Router, this is automatic — each page.tsx is a separate chunk.

Component-based splitting

Split heavy components that are conditionally shown: modals, rich text editors, data visualization libraries, admin panels. Only load the chunk when the user triggers the component (opens modal, navigates to admin page).

Vendor splitting

Extract stable third-party libraries (React, lodash, charting libs) into a separate vendor chunk. The vendor chunk changes rarely — cached long-term with content-hash. App code changes frequently — its chunk is small and invalidates often. Separate caching profiles.

Suspense boundary placement

The fallback UI shows while the chunk loads. Place Suspense boundaries strategically: too high = blank page during load; too low = awkward partial renders. Route-level Suspense is standard. Nested Suspense enables progressive streaming of page sections.

Preloading split chunks

After initial load, preload likely-needed chunks using import(/* webpackPrefetch: true */ './Heavy') or <link rel="prefetch">. On hover over a nav link, preload that route's chunk — it'll be in browser cache by the time the user clicks.

Chunk naming & debugging

Use magic comments for readable chunk names: import(/* webpackChunkName: "editor" */ './Editor'). Names appear in bundle analyzers and DevTools network tab. Vite uses module path as chunk name by default.

Code example
// Route-based splitting with React.lazy + Suspense
import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

// Each route becomes a separate chunk — not in initial bundle
const Dashboard  = lazy(() => import('./pages/Dashboard'));
const Settings   = lazy(() => import('./pages/Settings'));
const Analytics  = lazy(() => import('./pages/Analytics'));

export function App() {
  return (
    <Suspense fallback={<PageSkeleton />}>
      <Routes>
        <Route path="/"          element={<Dashboard  />} />
        <Route path="/settings"  element={<Settings   />} />
        <Route path="/analytics" element={<Analytics  />} />
      </Routes>
    </Suspense>
  );
}

// Component-based splitting — heavy modal loaded on demand
const RichTextEditor = lazy(() =>
  import(/* webpackChunkName: "editor" */ './components/RichTextEditor')
);

function EditPostModal({ open }) {
  if (!open) return null; // Don't render, don't load chunk
  return (
    <Suspense fallback={<EditorSkeleton />}>
      <RichTextEditor /> // Chunk fetched only when modal opens
    </Suspense>
  );
}

// Preload on hover — chunk ready before user clicks
const preloadDashboard = () => import('./pages/Dashboard'); // Warm network

<NavLink
  to="/dashboard"
  onMouseEnter={preloadDashboard}
  onFocus={preloadDashboard}
>Dashboard</NavLink>

// Vite config — explicit chunk splitting
// vite.config.ts
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['react', 'react-dom'],
          charts:  ['recharts', 'd3'],
          editor:  ['@tiptap/react', '@tiptap/starter-kit'],
        },
      },
    },
  },
});
Common mistakes
Over-splitting — splitting every tiny component creates too many HTTP requests and chunk overhead. The waterfall of small chunk fetches can be slower than one medium bundle. Meaningful split points: routes, heavy libraries (>50KB), features with distinct user flows.
Not placing Suspense correctly — forgetting a Suspense boundary causes React to throw an unhandled error when the lazy import is encountered. The boundary must be an ancestor of the lazy component, not a sibling.
Splitting eagerly-needed code — don't split components that are required for the above-the-fold render. Splitting the header, hero, or navigation only adds a network roundtrip to the initial load.
Interview perspective
"How does React.lazy work under the hood?"
It calls the dynamic import function (which triggers a network fetch of the chunk). If the returned Promise is pending, the component throws it — Suspense catches the thrown Promise and shows the fallback. When the Promise resolves, Suspense re-renders with the loaded component. This is React's "throw a Promise" protocol that also powers Suspense for data fetching.
"What's the difference between route-based and component-based code splitting?"
Route-based: split at page boundaries — most impactful, each route is a separate chunk, automatic in Next.js. Component-based: split heavy conditional components (modals, admin panels, rich editors) that aren't needed on every page render. Route-based is always the starting point; component-based is the next level of optimization for specific heavy features.

Tree shaking

Dead code eliminationES modulessideEffectsStatic analysis
Definition

Tree shaking is the bundler process of statically analyzing module import/export graphs and eliminating code that is never imported (dead code). The metaphor: shake a tree and the dead leaves (unused exports) fall away. It depends entirely on ES module static syntax (import/export) — not CommonJS (require/module.exports), which is dynamic and cannot be analyzed at build time. When tree shaking works correctly, importing one function from a 200-function utility library only bundles that one function, not the entire library.

How it works internally

Rollup (and Vite, which uses Rollup for production builds) builds a module graph — a directed graph of all imports starting from the entry point. It marks which exports are actually consumed by the entry point's reachable code. Exports that are not reachable are marked as dead. The "shaking" pass removes these dead exports. The key requirement: ESM static imports let bundlers know at parse time (before execution) exactly which symbols are imported. import { debounce } from 'lodash-es' — bundler statically knows only debounce is used. Compare: const { debounce } = require('lodash') — bundler cannot know what require('lodash') returns at parse time, so the entire library must be included. The sideEffects field in package.json tells bundlers whether any module in the package has side effects on import (e.g., adds to a global registry, patches prototypes). If "sideEffects": false, bundlers can safely remove entire files that export nothing used. If specific files have side effects, list them: "sideEffects": ["./src/polyfill.js"].

Subtopics

ES modules requirement

Static import/export syntax only. CommonJS modules (require) cannot be tree-shaken — the entire module is included. This is why lodash-es (ESM version) tree-shakes perfectly while lodash (CJS) doesn't. Always check if a library ships an ESM build.

sideEffects field

"sideEffects": false in package.json: all files can be safely removed if their exports aren't used. Incorrect value causes live code to be dropped. Incorrect false causes CSS imports in JS files to be dropped silently. CSS-in-JS and global styles are common side effects.

Named vs default exports

Named exports tree-shake more reliably: export const fn = .... Default exports export a single value — if that value is an object with methods, bundlers can't selectively include methods (all-or-nothing). Prefer named exports in utility libraries.

Dynamic imports exception

import(variable) cannot be tree-shaken — the bundler doesn't know what module is being imported at parse time. Only static import './specific-file' is analyzable. Dynamic imports create chunk boundaries but don't enable tree shaking within the chunk.

Barrel files anti-pattern

index.ts that re-exports everything: export * from './ComponentA'; export * from './ComponentB'. Even if you only import ComponentA, bundlers must include the barrel file which pulls in all re-exports if sideEffects is not false. Barrel files can defeat tree shaking.

Class methods

Class method tree shaking is hard — a class is typically treated as one unit. If you import a class, all methods come with it even if you only call two. Functional programming patterns (exported functions) tree-shake better than large classes with many methods.

Code example
// ❌ CJS — entire lodash bundled (70KB+)
const { debounce } = require('lodash');

// ❌ ESM default import with CJS package underneath — no tree shaking
import _ from 'lodash';
const debounced = _.debounce(fn, 300);

// ✅ ESM named import from ESM-published package
import { debounce } from 'lodash-es'; // Only debounce included (~2KB)

// package.json — sideEffects field
{
  "name": "my-ui-lib",
  "sideEffects": [
    "**/*.css",                // CSS imports have side effects (injected into DOM)
    "./src/polyfills.js"       // Polyfills mutate globals — keep them
  ]
  // All other .js files: no side effects → can be dropped if unused
}

// ❌ Barrel file anti-pattern
// src/components/index.ts
export * from './Button';      // 5KB
export * from './DataTable';   // 80KB — chart library dep
export * from './RichEditor';  // 120KB — prosemirror
// Importing just Button: import { Button } from '@/components'
// Result: bundler pulls in entire barrel → potentially all 205KB
// if sideEffects is not configured correctly

// ✅ Direct import — guaranteed tree shaking
import { Button } from '@/components/Button'; // Only 5KB

// Verify tree shaking in Vite — check bundle output
// vite build --report → opens rollup-plugin-visualizer
// Look for unexpectedly large chunks from utility libs
Common mistakes
Setting "sideEffects": false in a library that imports CSS files — Vite/webpack will drop CSS imports during tree shaking, resulting in unstyled components. Always list CSS files in the sideEffects array: "sideEffects": ["**/*.css"].
Importing from a library's CJS build assuming tree shaking works — check the "module" or "exports" field in the library's package.json to confirm an ESM build exists. If only "main" is specified, it's CJS.
Using large barrel files in your own codebase and relying on bundler tree shaking to compensate — barrel files slow down TypeScript type checking, HMR, and may cause tree shaking to fail if sideEffects is misconfigured. Prefer direct imports in large monorepos.
Interview perspective
"Why can't CommonJS modules be tree-shaken?"
CJS uses dynamic require() — the module specifier can be a variable, making it impossible to determine statically at parse time which exports are needed. ESM import statements are syntactically static — the bundler knows exactly which symbols are imported before executing any code. Static analysis = tree shaking. Dynamic resolution = bundler must include everything.
"What is the sideEffects field in package.json?"
A hint to bundlers about which files can be safely removed if their exports aren't consumed. false = all files are side-effect-free. Array of glob patterns = only listed files have side effects. Without it, bundlers default to "everything might have side effects" and skip dead-code elimination on the package.

Bundle optimisation

Chunk strategyVendor splitVisualizerCache busting
Definition

Bundle optimisation is the practice of structuring your JavaScript output files (chunks) to minimize the amount of code downloaded on first visit, maximize browser cache utilization across deployments, and eliminate code that was never needed. It combines tree shaking, code splitting, chunk grouping strategy, compression, and content-hash-based caching into a coherent build pipeline. The goal: users download exactly what they need, as fast as possible, and re-download as little as possible on subsequent visits.

How it works internally — chunk splitting strategy

Bundlers produce multiple output files. The optimal chunk structure depends on how often each piece of code changes. Stable code (React, lodash, charting libs) should be in a separate vendor chunk — it changes rarely, so the browser cache hits on subsequent visits and after deployments. Application code changes frequently — keep it in small, route-scoped chunks so that changing one page only invalidates that page's chunk. Each chunk gets a content hash in its filename (dashboard-B3k2mX9P.js) — if the content hasn't changed, the hash hasn't changed, and the browser serves it from cache with Cache-Control: immutable (no re-validation needed). The HTML file references these chunks and uses Cache-Control: no-cache (always revalidate) — so users always get the latest HTML pointing to the correct content-hashed assets.

Subtopics

rollup-plugin-visualizer

Generates an interactive treemap/sunburst of bundle contents — which modules are in which chunks, and their sizes. Run after build: npx vite build outputs stats.html. Look for: unexpectedly large modules, duplicate dependencies in multiple chunks, entire library imports where one function was needed.

Vendor chunk

Group stable third-party dependencies into a separate chunk via manualChunks. The vendor chunk rarely changes → long-term cache (max-age=1y, immutable). Application chunks change per-deployment → also content-hashed but expected to miss cache more often.

Chunk size targets

No hard rule, but common guidelines: initial bundle <170KB gzipped (Lighthouse budget). Each route chunk <100KB parsed. Chunks <30KB may not benefit from splitting (HTTP/2 overhead). Vite warns when chunks exceed 500KB uncompressed.

Compression

Brotli (br) compression achieves ~15-20% better ratios than gzip on text/JS. Serve pre-compressed .br files from CDN — avoids on-the-fly compression cost. Vite plugin: vite-plugin-compression generates .br and .gz alongside each chunk.

Module federation

Webpack Module Federation (Vite with vite-plugin-federation) allows multiple independently deployed apps to share runtime modules. A host app loads a remote module from a different deployment — dynamic chunk loading across deployment boundaries. Used in micro-frontend architectures.

Analyzing duplicate dependencies

Multiple chunks including the same library at different versions (e.g., React 17 in vendor chunk, React 17.0.1 in a widget bundle) bloat the total download. Use bundler deduplication config or peer dependencies to enforce single instance.

Code example
// vite.config.ts — production bundle optimisation
import { visualizer } from 'rollup-plugin-visualizer';
import viteCompression from 'vite-plugin-compression';

export default defineConfig({
  plugins: [
    react(),
    visualizer({ open: true, gzipSize: true, brotliSize: true }),
    viteCompression({ algorithm: 'brotliCompress' }),
  ],
  build: {
    rollupOptions: {
      output: {
        manualChunks(id) {
          // Stable third-party libs → vendor chunk (long cache)
          if (id.includes('node_modules/react') ||
              id.includes('node_modules/react-dom')) {
            return 'vendor-react';
          }
          if (id.includes('node_modules/recharts') ||
              id.includes('node_modules/d3')) {
            return 'vendor-charts'; // Charts only loaded on analytics page
          }
        },
        // Content-hash filenames — change = new URL = cache bust
        chunkFileNames: 'assets/[name]-[hash].js',
        assetFileNames: 'assets/[name]-[hash][extname]',
      },
    },
    // Warn on large chunks before they reach production
    chunkSizeWarningLimit: 500, // KB
  },
});

// Nginx: serve pre-compressed brotli + long cache for hashed assets
// location ~* \.(js|css|woff2)$ {
//   add_header Cache-Control "public, max-age=31536000, immutable";
//   brotli_static on;  # Serve .br files if available
// }
// location / {
//   add_header Cache-Control "no-cache";  # HTML always revalidates
// }
Common mistakes
Putting everything in one vendor chunk — if any vendor dependency updates, the entire vendor chunk cache is busted. Split by update frequency: vendor-react (changes almost never), vendor-ui (changes sometimes), vendor-charts (changes when you add features).
Forgetting that content hashing requires the HTML to always revalidate — if the HTML is also cached with a long TTL, users won't get the new chunk filenames after a deployment, loading stale chunks. HTML must use Cache-Control: no-cache.
Interview perspective
"Describe your strategy for splitting chunks in a large SPA."
Start with route-based splitting (biggest win). Extract a vendor chunk for stable third-party deps (React, router). Group heavy libraries used on specific routes into their own chunks (chart libs, rich text editors). Use content hashing for cache busting. Serve vendor chunks with max-age=1y immutable. Run the visualizer after every major dependency change to catch bundle regressions.

Image optimisation

WebP / AVIFlazy loadingsrcsetLCP impact
Definition

Images are typically the largest assets on a web page and the dominant cause of poor LCP scores. Image optimisation addresses three dimensions: format (choose the most efficient encoding), size (serve dimensions matching the display size, not the original camera resolution), and loading strategy (load eagerly for above-fold, lazily for below-fold). A 4MB JPEG hero image converted to AVIF at correct dimensions can become 80KB — a 50x size reduction with identical perceptual quality, directly translating to 2-3 second LCP improvement on mobile.

Subtopics

AVIF

Best compression (30-50% smaller than WebP at same quality). Slow encoding. Excellent browser support (95%+). Use for hero images, product photos. Encodes at build time, not on-the-fly.

WebP

~25-35% smaller than JPEG/PNG. Near-universal support. Fast encoding. Good fallback for AVIF. Use when encoding speed matters (user-uploaded images, dynamic transforms).

JPEG/PNG

JPEG: photos, no transparency. PNG: lossless, transparency. Both significantly larger than modern formats. Only use when AVIF/WebP unavailable or when tooling doesn't support conversion.

srcset & sizes

Serve different resolution images per viewport: srcset lists available sizes, sizes tells browser which to pick at which viewport width. Browser chooses smallest image that covers the display size × device pixel ratio.

lazy loading

loading="lazy" defers image fetch until image approaches viewport (intersection observer internally). Native browser API — no JS needed. Never apply to LCP element. Apply to all below-fold images.

fetchpriority

fetchpriority="high" on LCP image moves it to the network's high-priority queue, fetched before lower-priority resources competing on the same connection. Replaces the pattern of rel=preload for images.

Code example
<!-- LCP image — eager, high priority, responsive -->
<picture>
  <source
    type="image/avif"
    srcset="hero-400.avif 400w, hero-800.avif 800w, hero-1200.avif 1200w"
    sizes="(max-width: 600px) 100vw, (max-width: 1200px) 50vw, 1200px"
  />
  <source
    type="image/webp"
    srcset="hero-400.webp 400w, hero-800.webp 800w, hero-1200.webp 1200w"
    sizes="(max-width: 600px) 100vw, (max-width: 1200px) 50vw, 1200px"
  />
  <img
    src="hero-1200.jpg"             <!-- fallback -->
    alt="Hero image"
    width="1200" height="600"       <!-- explicit dims → no CLS -->
    loading="eager"                <!-- never lazy on LCP element -->
    fetchpriority="high"           <!-- high network priority -->
    decoding="async"               <!-- decode off main thread -->
  />
</picture>

<!-- Below-fold images — lazy loaded -->
<img
  src="product-thumb.webp"
  loading="lazy"                  <!-- deferred until near viewport -->
  width="300" height="300"          <!-- prevents CLS -->
  alt="Product thumbnail"
/>

import Image from 'next/image';

<Image
  src="/hero.jpg"
  alt="Hero"
  width={1200} height={600}
  priority          // = fetchpriority=high + eager + preload link
  sizes="(max-width: 768px) 100vw, 1200px"
/>

// on-demand resizing via /api/image route, blur placeholder
Common mistakes
Lazy loading the hero/LCP image — the most common and impactful image optimisation mistake. Adds a Largest Contentful Paint penalty of 300-500ms on fast connections, seconds on slow ones. Use loading="eager" (the default) or add fetchpriority="high" for the LCP element.
Serving desktop-sized images on mobile — a 1200×600px image displayed at 375×188px (standard mobile) wastes 10x the bytes. Use srcset and sizes to serve appropriately sized images per device. On a 2x DPR mobile, 750px wide is sufficient, not 1200px.
Not setting explicit width and height attributes — causes CLS because the browser allocates no space before the image loads. Always set width and height that match the image's intrinsic aspect ratio.
Interview perspective
"How would you optimize a page's LCP when the LCP element is an image?"
Check fetchpriority=high is set (and loading is not lazy). Verify image format: AVIF or WebP. Check if the image is correctly sized for the display (srcset + sizes). Preconnect to the image CDN domain. Ensure the image URL is in the initial HTML (not injected by JS). Check TTFB for the image resource itself. Measure with both Lighthouse and field data.

Font optimisation

font-displayFOIT / FOUTvariable fontsCLS risk
Definition

Web fonts are render-blocking for the text they style. The browser downloads font files (which can be 100-500KB each) and must decide what to display while they load. Without optimization, users see either invisible text (FOIT — Flash of Invisible Text) or unstyled fallback text (FOUT — Flash of Unstyled Text). Both hurt Core Web Vitals: FOIT harms LCP (largest text element invisible), FOUT causes CLS when the font swap shifts layout. Font optimization covers: choosing the right font-display strategy, preloading critical fonts, subsetting font files, and using variable fonts to reduce the number of font files needed.

Subtopics

font-display: swap

Show fallback immediately, swap to web font when loaded. Prevents FOIT. Causes FOUT (flash when font swaps in) which contributes to CLS. Suitable for body text where invisible text is unacceptable but a brief font flash is acceptable.

font-display: optional

Give font a very short window (100ms). If not loaded, use fallback for this page — don't swap. Zero FOUT, zero CLS from font swap. Font loads in background for subsequent pages. Best for CLS optimization. Bad for brand consistency on first load.

font-display: block

Invisible text for up to 3 seconds (block period) then swap. Worst of both worlds: user sees nothing (FOIT) then a shift (FOUT). Only valid for icon fonts where fallback characters would be meaningless.

Font preloading

<link rel="preload" as="font" crossorigin>. Moves font fetch into the preload scanner's early fetch queue, alongside HTML parsing. Reduces time-to-font from sequential (HTML parse → CSS → @font-face → font fetch) to parallel with initial HTML load.

Font subsetting

Only include characters actually used: unicode-range in @font-face allows browsers to download only the character subsets used on the page. A full Latin font file: 80KB. Latin subset (basic English): 20KB. Chinese font subset: even more critical.

Variable fonts

A single font file containing an entire type family (all weights, widths, italics) via OpenType variable axes. Replace 6 separate files (Regular, Bold, Italic, Bold Italic, Light, Light Italic) with one ~50-100KB file. Reduces font requests from 6 to 1.

Code example
<!-- In <head>: preload critical fonts early -->
<link
  rel="preload"
  as="font"
  type="font/woff2"
  href="/fonts/inter-var.woff2"
  crossorigin               <!-- Required for fonts, even same-origin -->
/>

/* CSS — self-hosted variable font */
@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter-var.woff2') format('woff2-variations');
  font-weight: 100 900;          /* Variable weight range */
  font-style: normal italic;      /* Both normal and italic from one file */
  font-display: swap;             /* FOUT over FOIT */
  unicode-range: U+0000-024F;    /* Basic Latin + Latin Extended */
}

/* font-display: optional — zero CLS, best for body text at scale */
@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter-var.woff2') format('woff2-variations');
  font-display: optional;         /* No swap → no layout shift */
}

/* Size-adjust: reduce CLS from font swap by matching metric */
@font-face {
  font-family: 'FallbackForInter';
  src: local('Arial');
  size-adjust: 107%;              /* Adjust fallback to match Inter's metrics */
  ascent-override: 90%;
  descent-override: 22%;
  /* Fallback now visually matches Inter → swap causes minimal shift */
}
body { font-family: 'Inter', 'FallbackForInter', sans-serif; }

import { Inter } from 'next/font/google';
const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
  variable: '--font-inter',

Common mistakes
Using font-display: block for text fonts — creates invisible text for up to 3 seconds (full FOIT block period). Only appropriate for icon fonts where the fallback character is meaningless. Text should use swap or optional.
Preloading fonts without crossorigin attribute — font preloads require crossorigin="anonymous" even for same-origin fonts (CORS mode for font fetches). Without it, the preload hint is ignored and the font is fetched again by the @font-face rule — double download.
Preloading every font variant — preloading 6 font files removes the benefit of conditional loading. Preload only the font files that are immediately rendered above-the-fold. Others load on demand via @font-face rules.
Interview perspective
"What is the difference between FOIT and FOUT, and how does font-display control this?"
FOIT (Flash of Invisible Text): text is invisible while font loads (font-display: block behavior). FOUT (Flash of Unstyled Text): fallback font shows immediately, then swaps to web font when loaded (font-display: swap). FOIT harms LCP and UX. FOUT harms CLS. font-display: optional eliminates both by using the web font only if it's cached, with no swap on first load. size-adjust reduces FOUT-induced CLS by making the fallback font metrically similar to the web font.

Prefetching & preloading

rel=preloadrel=prefetchpreconnectDNS-prefetch
Definition

Resource hints are HTML link tags that instruct the browser to perform network operations ahead of when they're actually needed — shifting resource discovery from "the parser found this" to "the developer told us about this". The four hints are: preload (fetch this resource now, I need it for this page), prefetch (fetch this resource speculatively, I might need it on the next page), preconnect (establish a connection to this origin, I'll be fetching from it soon), dns-prefetch (just do DNS resolution for this origin). Each operates at a different network layer and has different priority, cost, and appropriate use cases.

Subtopics

rel=preload

High-priority fetch for resources needed in the current page. Fires immediately on HTML parse. Use for: LCP image, critical fonts, critical scripts that the preload scanner would miss (dynamically constructed URLs, CSS background images). Must specify as= attribute or the hint is ignored.

rel=prefetch

Low-priority speculative fetch for resources needed on future navigations. Fires during browser idle time. Use for: next-page's JS chunks, likely-next resources based on user navigation patterns. Stored in HTTP cache, used on next navigation.

rel=preconnect

Establishes TCP connection + TLS handshake to an origin in advance. Saves 100-500ms per origin at request time (depending on RTT + TLS version). Use for: CDN origin, Google Fonts, analytics domains, API servers on different origins. Limit to 4-6 preconnects — each maintains an open connection with overhead.

rel=dns-prefetch

Performs only DNS resolution (no TCP/TLS). Cheaper than preconnect. Use for origins you might connect to but aren't certain about (A/B test CDNs, optional third parties). Fallback for browsers that don't support preconnect.

modulepreload

<link rel="modulepreload"> preloads ES modules (and their imports recursively). Use in Vite apps to preload critical route modules without waiting for the JS to parse and discover them. Parses and compiles the module immediately, not just downloads it.

Priority ordering

Browser priority hierarchy: preload (Highest) → prefetch (Lowest) → normal fetch (varies by resource type). Don't preload low-priority resources — elevating their priority competes with the critical path. Preload should be reserved for truly render-critical or LCP-critical resources.

Code example
<head>
  <!-- 1. Preconnect to critical third-party origins -->
  <link rel="preconnect" href="https://fonts.googleapis.com" />
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
  <link rel="preconnect" href="https://cdn.myapp.com" />

  <!-- DNS-prefetch for less-certain third parties -->
  <link rel="dns-prefetch" href="https://analytics.third-party.com" />

  <!-- 2. Preload LCP image (discovered before CSS is parsed) -->
  <link
    rel="preload"
    as="image"
    href="/hero-800.avif"
    type="image/avif"
    fetchpriority="high"
  />

  <!-- 3. Preload critical font -->
  <link
    rel="preload"
    as="font"
    type="font/woff2"
    href="/fonts/inter-var.woff2"
    crossorigin
  />

  <!-- 4. Preload critical JS (if not already in <script> in head) -->
  <link rel="modulepreload" href="/assets/vendor-react-B3k2mX.js" />
</head>

    const hint = document.createElement('link');
    hint.rel = 'prefetch';
    hint.href = link.dataset.chunkUrl;  // Next route's JS chunk
    hint.as = 'script';
    document.head.appendChild(hint);

Common mistakes
Preloading too many resources — every preload competes with other critical resources for bandwidth. Preloading 10 assets elevates all of them to high priority, defeating the priority system. The browser's automatic prioritization is often better than indiscriminate preloading. Limit to 1-3 critical above-fold resources.
Using rel=preload without the as= attribute — the browser fetches the resource with a generic priority instead of the correct content-type priority, and may fetch it again when the actual resource type is needed. Always specify as="image", as="font", as="script", etc.
Preloading resources that appear conditionally — if the preloaded resource isn't actually used on this page, the browser reports an "unused preload" warning in DevTools. This wastes bandwidth. Only preload resources that are definitely needed on the current page.
Interview perspective
"What is the difference between rel=preload and rel=prefetch?"
preload: high-priority fetch for resources needed on the current page — fires immediately, competing with critical resources. prefetch: low-priority speculative fetch for resources likely needed on future pages — fires during idle time, doesn't compete with current page. Using prefetch for current-page resources wastes time; using preload for next-page resources wastes bandwidth and priority.
"When would you use preconnect vs dns-prefetch?"
preconnect when you're certain you'll fetch from that origin soon (CDN, fonts, API) — it does DNS + TCP + TLS, saving all three round trips. dns-prefetch when you're not sure or want a lightweight hint for many origins — only DNS, cheaper. Don't preconnect to more than 4-6 origins; unused connections are wasted resources.

Service workers & offline

Cache-firstNetwork-firstSWR strategyWorkbox
Definition

A Service Worker is a JavaScript file that runs in a separate browser context (its own thread, separate from the page), acting as a programmable network proxy. It intercepts all fetch events from its scope (origin + path prefix) and can respond with: a cached response, a network response, a fabricated response, or any combination. This enables offline functionality, instant repeat-visit loads, and background sync. Service Workers have a strict lifecycle: install → activate → fetch. They persist independently of any page and are essential for Progressive Web Apps (PWAs).

Cache strategies

Cache-first

Try cache first. Fall back to network if miss. Cache network response for future. Use for: static assets (JS, CSS, images) with content hashing. Speed: instant on cache hit. Freshness: stale until cache eviction.

Network-first

Try network first. Fall back to cache on network failure. Use for: API data, frequently-updated content, HTML documents. Speed: network latency on success. Freshness: always current when online. Offline: last cached response.

Stale-while-revalidate

Serve stale immediately, fetch fresh in background for next time. Use for: non-critical content, news feeds, social data. Speed: instant. Freshness: one response behind. Best balance for most content.

Subtopics

SW lifecycle

Install: SW script downloaded, evaluated, caches pre-cached assets (precaching). Activate: old SW removed, new SW takes control. By default, new SW waits until all tabs using old SW close. skipWaiting() + clients.claim() take control immediately on activate.

Precaching vs runtime caching

Precaching: during install, explicitly cache a list of assets (app shell, critical routes) — guaranteed available offline. Runtime caching: cache responses on first fetch using a strategy — builds cache over user's browsing session, not guaranteed on first offline use.

Cache Storage API

Separate from HTTP cache — explicitly controlled by JS. caches.open(name) creates a named cache bucket. cache.put(request, response), cache.match(request). Persist independently of HTTP cache eviction. Must be manually versioned and cleaned up.

Workbox

Google's library for Service Worker patterns. Abstracts cache strategies into one-liner: registerRoute(pattern, new CacheFirst()). Handles precaching manifest generation, cache versioning, cache cleanup, background sync, and push notifications. Used by Next.js (next-pwa), Create React App, and Vite PWA plugin.

Background sync

Queue network requests that fail (offline). When connectivity restores, SW replays the queue. Use for: form submissions, analytics events, data mutations while offline. Requires the Background Sync API (supported in Chrome/Edge, not Safari).

Cache versioning

On SW update: delete old named caches in the activate event. Serve stale cache with old SW, new cache with new SW. Critically: if cache names aren't changed between deployments, old and new SW can read each other's stale assets.

Code example
// Manual Service Worker — demonstrates all three strategies
// sw.js

const CACHE_VERSION = 'v3';
const STATIC_ASSETS = ['/', '/offline.html', '/assets/vendor-react-B3k.js'];

// Install: precache static shell
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(`static-${CACHE_VERSION}`)
      .then(cache => cache.addAll(STATIC_ASSETS))
      .then(() => self.skipWaiting()) // Don't wait for old SW to close
  );
});

// Activate: clean old caches
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then(keys =>
      Promise.all(keys.filter(k => !k.includes(CACHE_VERSION)).map(k => caches.delete(k)))
    ).then(() => clients.claim())
  );
});

// Fetch: route to different strategies
self.addEventListener('fetch', (event) => {
  const { request } = event;

  // Cache-first: hashed static assets
  if (request.url.includes('/assets/')) {
    event.respondWith(cacheFirst(request, `static-${CACHE_VERSION}`));
    return;
  }
  // Network-first: API calls
  if (request.url.includes('/api/')) {
    event.respondWith(networkFirst(request, `api-${CACHE_VERSION}`));
    return;
  }
  // Stale-while-revalidate: HTML pages
  event.respondWith(staleWhileRevalidate(request, `pages-${CACHE_VERSION}`));
});

// Workbox equivalent (much simpler):
import { registerRoute } from 'workbox-routing';
import { CacheFirst, NetworkFirst, StaleWhileRevalidate } from 'workbox-strategies';
import { precacheAndRoute } from 'workbox-precaching';

precacheAndRoute(self.__WB_MANIFEST);  // Injected by Workbox build plugin
registerRoute(({ url }) => url.pathname.startsWith('/api/'), new NetworkFirst());
registerRoute(({ request }) => request.destination === 'image', new CacheFirst());
Common mistakes
Not incrementing the cache version name on SW update — the new SW serves assets from the old cache, potentially mixing old JS chunks with new HTML. Cache names must change with each deployment containing precached asset changes.
Applying cache-first strategy to HTML documents — users receive stale HTML (old chunk references) for up to the cache TTL. HTML should use network-first (serve fresh, fall back to cache if offline) or stale-while-revalidate.
Not testing offline behavior explicitly — SW offline strategies only matter when the network fails. Use DevTools → Application → Service Workers → Offline checkbox to test. Many teams ship SWs that break in offline mode because they've never tested the fallback path.
Interview perspective
"Explain the three main service worker caching strategies and when to use each."
Cache-first: speed over freshness. Use for content-hashed static assets (JS, CSS, fonts, images) that never change without a URL change. Network-first: freshness over speed. Use for API responses and HTML pages where stale content is incorrect. Stale-while-revalidate: instant load + background freshness. Use for content that benefits from speed but can tolerate being one version behind (news, avatars, non-critical API data).
"What is the Service Worker lifecycle and what does skipWaiting() do?"
Install (fetch+cache precache list) → waiting (old SW still active) → activate (old SW removed, new takes control). Without skipWaiting(), a new SW waits for all open tabs using the old SW to close — can take hours in production. skipWaiting() activates immediately; clients.claim() takes control of open pages without reload. Required for fast SW updates but can cause issues if old and new SW have incompatible cache formats.

Performance measurement

LighthouseChrome DevToolsRUM vs syntheticweb-vitals
Definition

Performance measurement divides into two categories: synthetic (lab) testing — controlled, reproducible measurements from a scripted environment (Lighthouse, WebPageTest); and Real User Monitoring (RUM) — measurements collected from actual users in the wild, aggregated and reported to analytics. Synthetic tests help identify issues in development and CI; RUM reveals what users actually experience across device types, network conditions, and geographic regions that no lab can simulate. The gap between them is meaningful: a "100 Lighthouse score" site can have terrible field CWV scores due to third-party scripts, slow user devices, or bad network conditions that the lab doesn't replicate.

Subtopics

Lighthouse

Google's automated auditing tool. Simulates a throttled mid-tier Android device (Moto G4) on a slow 4G connection. Scores 0-100 across Performance, Accessibility, SEO, Best Practices. Performance score is a weighted average of FCP, LCP, TBT (Total Blocking Time), CLS, Speed Index. Run in DevTools, CLI (npx lighthouse url), or CI via Lighthouse CI.

Chrome DevTools — Performance tab

Frame-by-frame recording of the page lifecycle. Shows: call stack, long tasks (red blocks), layout events, paint events, network requests, JS execution, GC pauses. Use for: debugging specific INP interactions, identifying which JS function causes a long task, tracing CLS back to a specific layout event.

Chrome DevTools — Network tab

Waterfall view of all resource fetches. Shows: request priority, connection reuse, TTFB per resource, time to download. Use for: identifying render-blocking resources, visualizing prefetch/preload behavior, measuring per-resource load times.

web-vitals library

Google's library for measuring Core Web Vitals in production. Callbacks fire with actual metric values when the browser finalizes them (LCP fires when user scrolls/clicks; CLS accumulates throughout session). Essential for RUM infrastructure. 1KB gzipped.

RUM infrastructure

web-vitals → Beacon API / fetch with keepalive → backend ingest → analytics DB → dashboards. Use Beacon API (navigator.sendBeacon) for unload-time events (session-end CLS, final INP). Segment by: URL pattern, device type, network type, geography.

CrUX (Chrome UX Report)

Google's field data aggregated from real Chrome users (opt-in). Available in: PageSpeed Insights, Search Console, BigQuery. 28-day rolling window. The source of truth for what Google sees when evaluating your CWV for search ranking. Shows p75 values for LCP, INP, CLS per URL and site.

Lighthouse CI

Run Lighthouse against a staging URL on every pull request. Assert that performance scores and specific metrics don't regress below thresholds. Block merges on performance regressions. Prevents the common pattern of introducing a 200KB dependency without noticing.

WebPageTest

More granular than Lighthouse. Real devices on real networks. Filmstrip view (frame-by-frame visual progress). Waterfall analysis. Comparison between multiple page versions. Tests from multiple global locations. Use for deep CWV diagnosis and competitor benchmarking.

Code example
// RUM setup — collect field data from real users
import { onLCP, onINP, onCLS, onTTFB, onFCP } from 'web-vitals';

const queue = new Set();

function addToQueue(metric) {
  queue.add({ ...metric, url: location.href, timestamp: Date.now() });
}

function flush() {
  if (!queue.size) return;
  // Beacon API: fires even during page unload
  navigator.sendBeacon('/metrics', JSON.stringify([...queue]));
  queue.clear();
}

const opts = { reportAllChanges: false }; // Only final values
onLCP(addToQueue, opts);
onINP(addToQueue, opts);    // Reports 75th percentile of all interactions
onCLS(addToQueue, opts);    // Cumulative value at session end
onTTFB(addToQueue, opts);
onFCP(addToQueue, opts);

// Flush on visibility change (page hide / tab close)
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') flush();
});

// Lighthouse CI config — lighthouse-ci.js
module.exports = {
  ci: {
    collect: {
      url: ['http://localhost:3000', 'http://localhost:3000/dashboard'],

RUM vs synthetic — the key distinctions

Synthetic (Lighthouse, WebPageTest)

Controlled environment. Reproducible. Identifies issues before production. Fixed device + network = not representative. Good for development, CI regression prevention, A/B comparison between code changes.

RUM (web-vitals + CrUX)

Real users, real devices, real networks. Reveals geographic distribution, device-class breakdown, real percentiles. Delayed (aggregated over time). Reveals what Lighthouse can't: slow user devices, third-party script impact, real long session INP.

Common mistakes
Using only Lighthouse scores as the performance KPI in production dashboards. A Lighthouse 100 can coexist with a poor CrUX field score if Lighthouse's throttling doesn't match your actual users' conditions. Always compare lab scores against field data (CrUX / web-vitals RUM).
Running Lighthouse once on a single page and calling it done — Lighthouse results vary by ~10 points between runs due to CPU contention. Run 3-5 times and median the result. Lighthouse CI automates this with numberOfRuns config.
Using performance.now() inside requestAnimationFrame to measure "render time" — rAF fires before paint, not after. The actual paint time is measured by PerformanceObserver with 'paint' entries. Custom metrics should use the PerformanceObserver API, not rAF timestamps.
Interview perspective
"What is the difference between lab data and field data for web performance?"
Lab (synthetic): controlled device/network in a tool like Lighthouse. Reproducible, useful in CI. Misses: real user devices (often slower), third-party scripts, geographic latency, long session patterns. Field (RUM): actual user experience from web-vitals library or CrUX. Shows 75th percentile across real conditions. Lab gives you the problem; field confirms you fixed it for users.
"How would you set up a performance monitoring pipeline for a production app?"
Three layers: (1) Lighthouse CI in PR pipeline — catch regressions before merge, assert on LCP/TBT/CLS thresholds. (2) web-vitals library in production — beacon real user metrics to analytics, segment by URL/device/geography. (3) CrUX monitoring in Search Console — track 28-day field data for Google ranking signal. Alert when p75 LCP > 2.5s or INP > 200ms in field data.
