ArrayList vs LinkedList
Internal array · dynamic resizing · doubly-linked nodes · when to choose each
Definition

Both implement List. ArrayList stores elements in a contiguous Object[] — random access is O(1) but insertions may trigger a resize. LinkedList stores elements in doubly-linked Node objects — insertions at known positions are O(1) but random access is O(n) and cache performance is poor.

ArrayList — choose when
Mostly reads / random access. Appending to tail. Iterating sequentially. Memory-sensitive (no node overhead).
LinkedList — choose when
Frequent insert/delete at head or in middle via iterator. Using as Deque (addFirst/removeLast). Rarely: very large elements where move cost dominates.
Internals
1
ArrayList backing array — default initial capacity 10. When size exceeds capacity, a new array of capacity × 1.5 is allocated and Arrays.copyOf runs — O(n) amortised cost, but rare. ensureCapacity(n) lets you pre-size to avoid this.
2
ArrayList remove(i) — elements after index i shift left via System.arraycopy — O(n). That's a native memory-copy, very fast, but still linear.
3
LinkedList node — each element is a heap-allocated Node<E>{E item; Node prev; Node next}. Three pointer fields = 24+ bytes overhead per element on 64-bit JVM (vs 4 bytes reference in ArrayList). Terrible cache locality — every node may be a TLB/cache miss.
4
LinkedList get(i) — O(n/2) average — starts from head or tail (whichever is closer) and walks pointers. Never use indexed get in a loop on LinkedList — it's O(n²).
5
Iterator-based removal — iterator.remove() on LinkedList is O(1) because the iterator holds a direct node reference. Same on ArrayList is O(n) (shift). This is the one case LinkedList wins in practice.

Common mistakes
!
Indexed get in a loop on LinkedList — O(n²). Always iterate with for-each or iterator on LinkedList. This is a production performance killer.
!
Choosing LinkedList "for faster inserts" without benchmarking — ArrayList's System.arraycopy is a native memory move that beats pointer-chasing in practice. LinkedList is almost never faster in modern JVM benchmarks due to cache effects.
!
Not pre-sizing ArrayList — if you know you'll add 10k elements, new ArrayList<>(10000) eliminates all resize copies. Cheap and impactful.
!
Storing ArrayList as List field in a class but casting it — breaks encapsulation. Program to the List interface. Changing implementation later requires zero caller changes.
Interview questions
Classic
When would you use LinkedList over ArrayList?
Almost never in modern Java. LinkedList wins only when: (1) you're doing frequent insert/delete via iterator in the middle and NOT random access, (2) you need it as a Deque. In practice, ArrayList's cache locality beats LinkedList even for mid-list inserts in benchmarks. The correct answer acknowledges cache effects — not just Big-O.
Depth
How does ArrayList resize and what is the amortised complexity?
When size == capacity, a new array of capacity * 1.5 is allocated and all elements are copied (Arrays.copyOf). This is O(n) for that single operation. But a sequence of n appends triggers at most O(log n) resizes, each doubling (roughly) the work. Total work across n appends = n + n/1.5 + n/1.5² + ... converges to O(n), so each append is O(1) amortised.
Follow-up
What is CopyOnWriteArrayList and when do you use it?
A thread-safe variant where every write (add, remove, set) creates a complete new copy of the underlying array. Reads are always lock-free and see a snapshot. Use when reads dominate writes — e.g., listener/subscriber lists. Never use if writes are frequent — O(n) per write.

HashMap internals
Hashing · buckets · chaining · treeification · load factor · rehashing
Definition

HashMap<K,V> is a hash table that maps keys to values with O(1) average-case get/put. It computes a bucket index from key.hashCode(), stores entries in an array of linked lists (or trees for long chains), and automatically doubles when the load threshold is exceeded.

Step-by-step internals
1
Hash spreading — raw key.hashCode() is spread with h ^ (h >>> 16) to mix high bits into low bits. This reduces collisions when table size is a power of 2.
2
Bucket index — index = (n-1) & hash where n is always a power of 2. Equivalent to hash % n but via bitmasking — much faster.
3
Collision resolution — bucket holds a singly-linked list of Node<K,V>. On get, walk the list comparing via == then .equals().
4
Treeification (Java 8+) — when a bucket chain reaches 8 nodes AND table capacity ≥ 64, the linked list is converted to a Red-Black tree. Tree lookup = O(log n) per bucket. Reverts to linked list when nodes drop to 6.
5
Load factor & rehash — default load factor 0.75. When size > capacity × 0.75, a new array of 2× size is allocated and every entry is re-hashed. This is O(n) but amortised O(1) per insertion.
6
Null key — HashMap allows one null key (always stored in bucket 0). HashTable and ConcurrentHashMap do not allow null keys.

Map<String, Integer> map = new HashMap<>(32, 0.75f); // pre-size map.put("key", 1); // hash spread → bucket index → put node map.get("key"); // same hash → same bucket → walk & equals map.getOrDefault("x", 0); // safe null-avoiding get // Merge pattern — count word frequencies map.merge(word, 1, Integer::sum); // Compute pattern — safer than get+put map.computeIfAbsent("grp", k -> new ArrayList<>()).add("item"); // Iteration — entrySet is fastest for (Map.Entry<String,Integer> e : map.entrySet()) { System.out.println(e.getKey() + "=" + e.getValue()); } // DANGER: using mutable key List<String> key = new ArrayList<>(); map.put(key, 1); key.add("oops"); // hashCode changed → entry now lost!
Common mistakes
!
Mutable keys — if you mutate a key after insertion, its hashCode changes and the entry becomes unreachable. Always use immutable keys (String, Integer, record).
!
Breaking the hashCode/equals contract — if a.equals(b) is true, a.hashCode() MUST equal b.hashCode(). Violation causes duplicate entries or unfindable entries. Override both or neither.
!
Not pre-sizing for known capacity — default capacity 16, load 0.75 → first resize at 12 entries. If you're adding 1000 entries: new HashMap<>(1334) (1000/0.75 + 1) avoids all rehashes.
!
Using HashMap in multithreaded code without sync — concurrent modifications cause infinite loops (Java 7) or data loss (Java 8+). Use ConcurrentHashMap.
Interview questions
Classic
What happens when two keys have the same hashCode?
They land in the same bucket. The bucket holds a linked list; each new node is appended. On lookup, the list is walked using equals() to find the right key. Java 8+: if the chain length hits 8 and table capacity >= 64, the chain converts to a Red-Black tree for O(log n) bucket lookup instead of O(n).
Depth
Why does HashMap always use a power-of-2 capacity?
Because then bucket_index = hash & (capacity-1) is equivalent to hash % capacity, but via a single bitwise AND — much faster than modulo. Every resize doubles capacity (stays power of 2). The hash spreading (h ^ h>>>16) also assumes power-of-2 so high bits contribute to bucket selection.
Concurrency
How is ConcurrentHashMap different from HashMap?
Java 8+: ConcurrentHashMap uses CAS (compare-and-swap) on individual bucket heads and synchronized blocks only when modifying a bucket. No global lock. Reads are always lock-free. Null keys/values are forbidden (can't distinguish missing vs null in concurrent scenarios). Offers atomic compute, merge, forEach operations.

LinkedHashMap vs TreeMap
Insertion-order doubly-linked list · Red-Black BST for sorted keys
Definition

LinkedHashMap extends HashMap and adds a doubly-linked list threading through all entries — preserving insertion (or access) order. TreeMap uses a Red-Black BST to keep keys in sorted order at all times, at the cost of O(log n) per operation instead of O(1).

LinkedHashMap — choose when
You need HashMap speed AND predictable iteration order. LRU cache (accessOrder=true). Maintaining the order items were inserted (API responses, ordered configs).
TreeMap — choose when
Keys must stay sorted. You need range queries: subMap, headMap, tailMap, floorKey, ceilingKey. Leaderboards, interval trees, ordered event logs.
Internals
1
LinkedHashMap — each Entry extends HashMap.Node with two extra pointers: Entry before, after. These form a doubly-linked list in insertion order (or access order if accessOrder=true). The hash table still provides O(1) lookup; the linked list provides ordered iteration.
2
LRU cache trick — new LinkedHashMap<>(cap, 0.75f, true) uses access order. Override removeEldestEntry to return size() > maxSize and you have a complete O(1) LRU cache with zero extra code.
3
TreeMap — backed by a Red-Black tree (self-balancing BST). Guarantees height ≤ 2 log(n+1). All operations — get, put, remove, floor, ceiling, subMap — are O(log n). Requires keys to be Comparable or a Comparator provided at construction.
4
Red-Black tree rules — every node is Red or Black. Root is Black. Red node's children are Black. Every path from root to null has same number of Black nodes. These constraints keep the tree balanced after rotations.

// LRU cache with LinkedHashMap class LRUCache<K,V> extends LinkedHashMap<K,V> { private final int max; LRUCache(int max) { super(max, 0.75f, true); // accessOrder=true this.max = max; } protected boolean removeEldestEntry(Map.Entry e) { return size() > max; // evict LRU when over capacity } } // TreeMap — range queries TreeMap<Integer,String> tm = new TreeMap<>(); tm.put(10,"ten"); tm.put(20,"twenty"); tm.put(30,"thirty"); tm.floorKey(25); // 20 — largest key ≤ 25 tm.ceilingKey(25); // 30 — smallest key ≥ 25 tm.subMap(10, 25); // {10=ten, 20=twenty} tm.headMap(20); // {10=ten} tm.firstKey(); // 10 — minimum key, O(log n)
Common mistakes
!
Using TreeMap for O(1) lookups — every get/put is O(log n). If you don't need sorted order, use HashMap. Common in interview implementations where a plain HashMap would suffice.
!
Keys not implementing Comparable in TreeMap — throws ClassCastException at runtime, not compile time. Provide a Comparator at construction or ensure keys implement Comparable.
!
Forgetting accessOrder in LRU LinkedHashMap — default is insertion order. For LRU semantics you must pass true as the third constructor argument. Easy to forget, hard to debug.
Interview questions
Design
Implement an LRU cache in Java.
Two approaches: (1) LinkedHashMap with accessOrder=true and removeEldestEntry — 5 lines. (2) Manual: HashMap + doubly-linked list for O(1) get and move-to-front. Interviewers often want approach 2 to test your pointer manipulation. Know both. LinkedHashMap approach is production code; manual approach demonstrates you understand the mechanism.
Compare
HashMap vs TreeMap vs LinkedHashMap — when do you pick each?
HashMap: default choice — O(1), unordered. LinkedHashMap: when you need HashMap speed plus predictable iteration order (insertion or access). TreeMap: when keys must stay sorted or you need range/floor/ceiling operations. The ordering guarantee in TreeMap costs O(log n) everywhere.
Depth
What are Red-Black tree properties and why does Java use it?
5 properties: (1) nodes are Red or Black, (2) root is Black, (3) leaves (nulls) are Black, (4) red node has only black children, (5) all paths from a node to leaves have same number of black nodes. These guarantee height ≤ 2log(n+1) → O(log n) worst case. Java chose RB-tree over AVL because RB-tree requires fewer rotations on insert/delete — better write performance.

HashSet · LinkedHashSet · TreeSet
All backed by their Map counterparts — add null differences
Definition

Sets hold unique elements — no duplicates, no key-value pairs. Java's three main Set implementations are thin wrappers around their Map counterparts: each stores elements as keys in the backing map with a shared dummy PRESENT object as the value.

HashSet
Backed by HashMap. O(1) add/remove/contains. No order. Allows one null.
LinkedHashSet
Backed by LinkedHashMap. O(1) operations + insertion-order iteration. One null allowed.
TreeSet
Backed by TreeMap. O(log n). Always sorted. Null not allowed (no comparison possible).
Internals — the backing map trick
1
Shared PRESENT sentinel — private static final Object PRESENT = new Object(). Every element added becomes map.put(element, PRESENT). The value slot is wasted but the key set is exactly the elements.
2
contains() — delegates directly to map.containsKey() — same O(1)/O(log n) as the backing map.
3
Ordering contract — HashSet: no guarantee. LinkedHashSet: insertion order (same guarantees as LinkedHashMap). TreeSet: natural ordering or Comparator — same as TreeMap. TreeSet also implements NavigableSet giving floor, ceiling, headSet, tailSet.
Code
Set<String> hs = new HashSet<>(Arrays.asList("b","a","c")); // iteration order undefined — could be a, b, c or any order Set<String> lhs = new LinkedHashSet<>(Arrays.asList("b","a","c")); // iteration: b, a, c — insertion order preserved TreeSet<String> ts = new TreeSet<>(Arrays.asList("b","a","c")); // iteration: a, b, c — always sorted ts.floor("bb"); // "b" ts.ceiling("ab"); // "b" ts.headSet("b"); // [a] // Dedup a list while preserving order List<Integer> deduped = new ArrayList<>(new LinkedHashSet<>(list));
Common mistakes
!
Mutable elements in HashSet — if you modify an element after adding it, its hashCode changes and it becomes "lost" in the set (same issue as HashMap mutable keys).
!
Adding null to TreeSet — TreeSet calls compareTo on every add. null.compareTo() throws NullPointerException. The exception occurs on the second add, not the null add itself, which makes it confusing to debug.
!
Expecting consistent order from HashSet — iteration order can change between JVM versions, hash seeds, and even runs. Never depend on it. If order matters, use LinkedHashSet or TreeSet explicitly.
Interview questions
Internals
How does HashSet ensure uniqueness?
It delegates to HashMap.put(element, PRESENT). HashMap's put returns the old value if the key existed. HashSet.add() returns false if put returned non-null (meaning the element was already present). Uniqueness is enforced by the map's key uniqueness — no special logic needed.
Choose
How would you deduplicate a list while preserving insertion order?
new ArrayList<>(new LinkedHashSet<>(list)). LinkedHashSet preserves insertion order while removing duplicates. Wrapping back in ArrayList gives you a List. Time: O(n). Interviewers sometimes expect you to know this one-liner.

PriorityQueue (min-heap)
Binary heap in an array · heapify · sift-up · sift-down · comparator
Definition

Java's PriorityQueue is a min-heap — the element with the lowest priority value (natural order or comparator) is always at the head. Internally it's stored as a complete binary tree flattened into an Object[]. Peek is O(1); poll and offer are O(log n).

Internals — array-as-tree math
1
Array mapping — for element at index i: left child = 2i+1, right child = 2i+2, parent = (i-1)/2. No pointers needed — pure arithmetic navigation.
2
offer(e) — sift-up — add element at the next leaf position (end of array). Compare with parent; if smaller, swap. Repeat upward. O(log n).
3
poll() — sift-down — swap root with last element, remove last, sift-down from root: compare with both children, swap with smaller child, repeat. O(log n).
4
heapify (bulk build) — when constructing from a collection, sift-down from the last non-leaf backwards. O(n) — faster than n individual inserts (O(n log n)).
5
No O(1) contains / remove(Object) — PriorityQueue is a heap, not a sorted list. contains and remove(Object) are O(n) linear scans. If you need fast removal by value, pair it with a HashMap tracking positions.

// Default: min-heap (natural order) PriorityQueue<Integer> minPQ = new PriorityQueue<>(); minPQ.offer(5); minPQ.offer(1); minPQ.offer(3); minPQ.peek(); // 1 — O(1), doesn't remove minPQ.poll(); // 1 — O(log n), removes min // Max-heap via reversed comparator PriorityQueue<Integer> maxPQ = new PriorityQueue<>(Comparator.reverseOrder()); // Custom comparator — task scheduling by priority PriorityQueue<Task> taskQ = new PriorityQueue<>(Comparator.comparingInt(Task::getPriority)); // K-th largest element — classic interview pattern PriorityQueue<Integer> kth = new PriorityQueue<>(k); for (int n : nums) { kth.offer(n); if (kth.size() > k) kth.poll(); // evict smallest } kth.peek(); // k-th largest // WARNING: iteration is NOT sorted order for (int x : minPQ) System.out.print(x); // arbitrary heap order
Common mistakes
!
Iterating PriorityQueue expecting sorted output — the for-each iterator returns elements in heap array order, NOT sorted order. Use repeated poll() to drain in sorted order.
!
Calling remove(Object) for efficiency — it's O(n). If you need to update priorities (Dijkstra's with lazy deletion), don't remove old entries — mark them stale and skip when polled.
!
Forgetting to negate for max-heap on primitives — new PriorityQueue<>((a,b) -> b-a) works for small integers but overflows for large values. Use Comparator.reverseOrder() or Integer.compare(b,a) instead.
Interview questions
Classic
Explain sift-up and sift-down in a binary heap.
Sift-up (insert): place new element at end, then repeatedly swap with parent while element < parent. O(log n). Sift-down (delete-min): move last element to root, then repeatedly swap with the smaller child while element > any child. O(log n). These two operations are the entire heap algorithm.
DSA Pattern
How do you find the K-th largest element efficiently?
Min-heap of size K. Iterate all elements: offer each, then if heap size > K poll (remove the min). At the end the heap's min (peek()) is the K-th largest. Time O(n log k), space O(k). Common patterns: K closest points to origin (max-heap by distance), merge K sorted arrays (min-heap of heads), median from stream (two heaps).
Depth
Why is building a heap from n elements O(n) and not O(n log n)?
Heapify sifts-down from the last non-leaf backwards. Leaves need 0 swaps. Nodes at height h need at most h swaps. Summing h * (n/2^h) for all heights gives a geometric series that converges to 2n = O(n). The insight: most nodes are near the bottom and need few swaps.

ArrayDeque
Circular array · head/tail pointers · preferred over Stack and LinkedList
Definition

ArrayDeque is a double-ended queue backed by a resizable circular array. It supports O(1) add/remove at both head and tail. It is the recommended replacement for both Stack (legacy, synchronised) and LinkedList (poor cache performance) when you need stack or queue semantics.

Use as Stack
push = addFirst, pop = removeFirst, peek = peekFirst. LIFO. No null elements.
Use as Queue
offer = addLast, poll = removeFirst, peek = peekFirst. FIFO. Preferred over LinkedList.
Internals — circular array
1
Backing structure — Object[] elements with int head, tail indices. Initial capacity 16 (always power of 2). Elements wrap around — it's a ring buffer.
2
addFirst — head = (head - 1) & (elements.length - 1); store at new head. The bitwise AND with (length-1) handles the wrap-around without a branch — power-of-2 trick again.
3
addLast — store at tail, then tail = (tail + 1) & (elements.length - 1). Both operations are O(1).
4
Resize — when head == tail (full), double the capacity, copy elements to a new array in linearised order starting at index 0. O(n) amortised once.
5
Why no nulls — ArrayDeque uses null as a sentinel to detect the empty condition. Storing null would break this check, hence NullPointerException on null input.

Common mistakes
!
Storing null in ArrayDeque — throws NullPointerException immediately. If you need nullable elements, use LinkedList (though that's rarely the right choice).
!
Using Stack class — java.util.Stack extends Vector, so every operation is synchronised — unnecessary overhead in single-threaded use. Always prefer ArrayDeque for stack semantics.
!
Mixing Stack and Queue API names — ArrayDeque implements both Deque and Queue. push/pop are stack (LIFO); offer/poll are queue (FIFO). Mixing them on the same Deque produces confusing ordering bugs. Be consistent.
Interview questions
Why
Why is ArrayDeque preferred over Stack and LinkedList?
Over Stack: Stack is synchronised (inherits from Vector) — unnecessary overhead, and inheritance is a design mistake (Stack IS-A Vector? No.). Over LinkedList: ArrayDeque uses a contiguous circular array — much better cache locality. Both O(1) at head and tail, but ArrayDeque is consistently faster in benchmarks. No pointer overhead per element.
Pattern
What is a monotonic stack and name 2 problems it solves.
A stack that maintains elements in strictly increasing or decreasing order. When an element violates the monotonicity, pop until it doesn't, then push. Problems: (1) Next Greater Element — for each element, find the first element to its right that's larger. (2) Largest Rectangle in Histogram. (3) Sliding Window Maximum uses a monotonic deque. Pattern: O(n) solutions to problems that seem O(n²).

Collections utility class
sort · binarySearch · reverse · frequency · unmodifiableList · synchronizedList
Definition

java.util.Collections is a utility class of static methods that operate on or return collections. It provides algorithm implementations (sort, shuffle, binarySearch), transformation wrappers (unmodifiable, synchronized, singleton), and min/max/frequency helpers.

Key methods — what they do internally
1
Collections.sort(list) — delegates to list.sort(null) which calls Arrays.sort on the backing array. Uses TimSort — a hybrid merge+insertion sort. O(n log n) worst case, O(n) best case (already sorted). Stable sort.
2
Collections.binarySearch(list, key) — requires sorted list. Uses index-based binary search for RandomAccess lists (ArrayList), pointer-walking for others (O(n) — use an array instead). Returns insertion point if not found: -(index+1).
3
Collections.reverse(list) — swaps first with last, then second with second-to-last etc. O(n). Works in-place on the list.
4
Collections.unmodifiableList(list) — returns a wrapper that throws UnsupportedOperationException on any mutating method. The underlying list is still mutable — callers who hold a reference to it can still modify it. This is a view, not a copy. For a truly immutable list use List.copyOf() (Java 10+) or List.of() (Java 9+).
5
Collections.synchronizedList(list) — wraps every method with synchronized(mutex). Safe for concurrent access but iteration must be manually synchronised: synchronized(list) { for(x : list)... }. Prefer CopyOnWriteArrayList for read-heavy, ConcurrentLinkedQueue for queues.
6
Collections.frequency(coll, o) — O(n) linear scan counting elements equal to o. nCopies(n, o) — returns an immutable list of n copies of o in O(1) (backed by a view, not n actual copies). disjoint(c1,c2) — true if no common elements.
Code
List<Integer> nums = new ArrayList<>(Arrays.asList(3,1,4,1,5,9)); Collections.sort(nums); // [1,1,3,4,5,9] — TimSort Collections.sort(nums, Comparator.reverseOrder()); // [9,5,4,3,1,1] Collections.reverse(nums); // [1,1,3,4,5,9] Collections.shuffle(nums); // random order Collections.frequency(nums, 1); // 2 Collections.min(nums); // 1 Collections.max(nums); // 9 int idx = Collections.binarySearch(nums, 4); // list MUST be sorted first // Immutable views vs true immutability List<String> orig = new ArrayList<>(Arrays.asList("a","b")); List<String> view = Collections.unmodifiableList(orig); orig.add("c"); // mutates orig AND view sees it! List<String> safe = List.copyOf(orig); // true defensive copy, immutable // Efficient n-copy List<String> zeros = Collections.nCopies(1000, "0"); // O(1) memory
Common mistakes
!
binarySearch on unsorted list — returns undefined/wrong results. No exception thrown. You will get a random answer and spend hours debugging. Always sort first, or better yet use a TreeSet if you need sorted search.
!
Thinking unmodifiableList makes the data immutable — it's a view. The original list can still be modified by whoever holds it. Use List.of() or List.copyOf() when you need a truly immutable snapshot.
!
Iterating synchronizedList without external lock — individual operations are thread-safe but iteration is not atomic. Two threads can interleave during iteration → ConcurrentModificationException or missed elements. Wrap iteration in synchronized(list) { }.
Interview questions
Classic
What is the difference between Collections.unmodifiableList and List.of?
unmodifiableList is a view around a mutable list — the underlying list can change and the view reflects it; throws UnsupportedOperationException on writes. List.of() (Java 9+) creates a brand new immutable list from the provided elements — no backing list to mutate, structurally immutable. List.copyOf() creates an immutable copy of an existing collection — safe from original-list mutations.
Algo
What sorting algorithm does Collections.sort use and why?
TimSort — a hybrid of merge sort and insertion sort. Advantages: (1) stable sort — equal elements maintain relative order, (2) O(n) best case on already-sorted or nearly-sorted input (common in real data), (3) O(n log n) worst case. Insertion sort is used for small sub-arrays (< 32 elements) because it has low overhead and good cache behaviour for small n.
