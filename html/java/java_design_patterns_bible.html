Singleton
Creational
Ensures one instance exists · lazy vs eager · thread-safe DCL · enum singleton
Definition

Singleton ensures a class has exactly one instance and provides a global access point to it. The class itself controls instantiation — the constructor is private, and a static method returns the single shared instance. Used for shared resources where multiple instances would cause incorrect behaviour (config, connection pool, registry).

Eager initialisation
Instance created at class-load time. Thread-safe by the JVM's class-loading guarantee. Wastes memory if the instance is never used.
Lazy initialisation
Instance created on first request. Saves memory. Requires thread-safety mechanism — naive implementation has a race condition.
4 implementations — ranked from worst to best
1
Naive lazy (broken) — if(instance == null) instance = new Singleton() with no synchronisation. Two threads can both pass the null check simultaneously and create two instances. Never use in multithreaded code.
2
Eager (JVM-guaranteed) — private static final Singleton INSTANCE = new Singleton(). JVM class loading is thread-safe by spec — the class is loaded once. Safe. Downside: created even if never used, can't handle exceptions cleanly in constructor.
3
Double-checked locking (DCL) — check → sync → check again. Requires volatile on the field to prevent partial construction visibility (see Block 4 JMM). Best when: lazy + thread-safe + performance matters (avoids lock on every read after first init).
4
Enum singleton — the gold standard — Java's enum guarantees: single instance per JVM, thread-safe construction by the JVM, serialisation-safe (enums survive deserialization as the same instance), reflection-proof (can't call enum constructor via reflection). The only approach that handles all edge cases. Joshua Bloch's Effective Java: "single-element enum type is often the best way to implement a singleton."
Code — all four implementations
// 1. Eager — simple, JVM-thread-safe public class Config { private static final Config INSTANCE = new Config(); private Config() {} public static Config getInstance() { return INSTANCE; } } // 2. DCL — lazy + thread-safe public class Cache { private static volatile Cache instance; // volatile: mandatory for DCL private Cache() {} public static Cache getInstance() { if (instance == null) { // 1st check — avoid lock on every call synchronized(Cache.class) { if (instance == null) // 2nd check — win the race instance = new Cache(); } } return instance; } } // 3. Enum singleton — best practice (Effective Java Item 3) public enum AppRegistry { INSTANCE; private final Map<String, Object> registry = new HashMap<>(); public void register(String key, Object val) { registry.put(key, val); } public Object lookup(String key) { return registry.get(key); } } // Usage: AppRegistry.INSTANCE.register("svc", myService); // 4. Holder idiom — lazy without synchronized public class MetricsService { private MetricsService() {} private static class Holder { // loaded only when referenced static final MetricsService INSTANCE = new MetricsService(); } public static MetricsService get() { return Holder.INSTANCE; } }
Real-world use cases
🔧
Spring ApplicationContext — Spring manages beans as singletons by default (@Scope("singleton")). The bean is created once, stored in the context, and injected everywhere. Spring's container IS the singleton manager — you don't write Singleton pattern yourself for beans.
📊
Connection pool (HikariCP) — the HikariDataSource is a singleton — one pool per application, shared by all threads. Creating multiple pools per request would exhaust DB connections.
⚙️
Configuration registry — application config loaded once from environment variables, exposed as a singleton. All services read from the same immutable config object.
📝
Logger — LogManager.getLogger() returns the same logger instance per name. Creating a new logger per call would lose context and waste memory.
Common mistakes
!
Omitting volatile in DCL — without volatile, the JIT can reorder object construction: assign the reference before the constructor finishes. Thread B reads a non-null but partially-constructed object. This is a real bug that surfaces under load.
!
Singleton broken by serialisation — deserialising a singleton creates a new instance via readObject. Fix: implement readResolve() { return INSTANCE; } to return the existing instance. Enum singletons handle this automatically.
!
Singleton broken by reflection — Constructor.newInstance() can bypass the private constructor. Fix: throw an exception in the constructor if already initialised. Enum constructors cannot be called via reflection.
!
Overusing singleton — Singleton is a global variable in disguise. It creates hidden dependencies, makes testing hard (can't swap with a mock), and causes coupling. In modern Java, use dependency injection (Spring) — the container manages singleton scope without the pattern's drawbacks.
Interview questions
Classic
Write a thread-safe Singleton in Java.
Three acceptable answers: (1) Enum singleton — the best answer. enum AppSingleton { INSTANCE; } — thread-safe by JVM, serialisation-safe, reflection-proof. (2) Eager initialisation — private static final instance at field declaration. (3) DCL with volatile — two null checks, synchronized on the class, volatile field. If they ask WHY volatile is required: the JVM can reorder the assignment before constructor completion; volatile adds a store fence preventing this. Mention you'd use Spring @Scope("singleton") in production rather than hand-rolling this pattern.
Why volatile
Why must the instance field be volatile in double-checked locking?
Object creation has three steps: (1) allocate memory, (2) run constructor, (3) assign reference. The JIT can reorder to 1→3→2. Thread A assigns the reference before the constructor runs. Thread B reads a non-null reference and uses a partially constructed object. volatile adds a store fence between steps 2 and 3 (via the JMM's happens-before for volatile writes), preventing the reordering. This bug was only formally documented and fixed in Java 5 with the updated JMM specification.
Enum
Why is the enum singleton considered the best approach?
It gets thread safety, lazy class loading safety, serialisation safety, and reflection safety for free — handled by the JVM, not by application code. No boilerplate. No edge cases. The spec explicitly guarantees enum values are created once per JVM. The only downside: can't extend another class (enums implicitly extend Enum). Joshua Bloch, Effective Java Item 3: "a single-element enum type is often the best way to implement a singleton."

Factory & Abstract Factory
Creational
Decouple object creation · Factory Method · Abstract Factory · when to use each
Definition

Factory patterns decouple the caller from the concrete class being created. The caller asks for a "shape" or a "payment processor" — the factory decides which concrete implementation to instantiate. This enables the Open/Closed Principle: add new product types without changing existing client code.

Factory Method
One factory method per product type. Subclasses override the method to decide which concrete class to create. Creates one product family member at a time.
Abstract Factory
A factory of factories. One interface creates entire families of related products. Ensures the created products are compatible with each other (all "Dark theme" or all "MySQL" components).
Structure — UML in code form
«interface» PaymentProcessor
+ process(amount): Receipt
+ refund(id): boolean
→
StripeProcessor
+ process(amount): Receipt
+ refund(id): boolean
PaymentFactory (abstract)
+ createProcessor(): PaymentProcessor
+ createWebhook(): WebhookHandler
→
StripeFactory
+ createProcessor(): StripeProcessor
+ createWebhook(): StripeWebhook

Abstract Factory ensures the processor and webhook are always from the same payment provider — you can't accidentally mix Stripe's processor with Razorpay's webhook.

Code — Factory Method + Abstract Factory
// FACTORY METHOD — one product, subclasses decide which interface NotificationSender { void send(String msg); } class EmailSender implements NotificationSender { public void send(String m) { /*smtp*/ } } class SmsSender implements NotificationSender { public void send(String m) { /*twilio*/ } } class PushSender implements NotificationSender { public void send(String m) { /*fcm*/ } } class SenderFactory { public static NotificationSender create(String type) { return switch(type) { case "EMAIL" -> new EmailSender(); case "SMS" -> new SmsSender(); case "PUSH" -> new PushSender(); default -> throw new IllegalArgumentException("Unknown: " + type); }; } } // ABSTRACT FACTORY — family of related objects interface UIFactory { Button createButton(); TextField createTextField(); Dialog createDialog(); } class DarkThemeFactory implements UIFactory { public Button createButton() { return new DarkButton(); } public TextField createTextField() { return new DarkTextField(); } public Dialog createDialog() { return new DarkDialog(); } } // Caller only knows UIFactory — can swap theme without changing UI code UIFactory factory = getThemeFactory(userPref); // dark or light Button btn = factory.createButton(); // always consistent
Real-world use cases
🏭
JDBC DriverManager — DriverManager.getConnection(url) is a factory method. The URL prefix determines which Driver implementation to instantiate (MySQL, PostgreSQL, H2). Your code never references a concrete driver class.
🌿
Spring BeanFactory / ApplicationContext — Spring's IoC container IS an Abstract Factory. getBean("userService") creates (or returns) the configured bean. The caller never calls new. Changing the implementation requires only config changes, not code changes.
📱
Notification routing — a NotificationService uses a factory to get the right sender (Email/SMS/Push) based on user preference or notification urgency. Adding WhatsApp support = add a new sender + register it in the factory, zero change to routing logic.
Common mistakes
!
Using a factory when a simple constructor or static method is enough — factories add indirection. If you have one concrete type and no variation anticipated, just use new. Don't pattern-match prematurely.
!
Confusing Factory Method with Abstract Factory — Factory Method = one method, one product. Abstract Factory = one factory interface, creates a family of related products. The distinction matters in interviews and design discussions.
Interview questions
Distinguish
What is the difference between Factory Method and Abstract Factory?
Factory Method: defines one factory method in a superclass, subclasses override it to decide which concrete class to instantiate. One method, one product type. Abstract Factory: an interface with multiple factory methods, each creating a different product in a family. Guarantees the created objects are compatible with each other. Use Factory Method when you need to vary one product type. Use Abstract Factory when you need to create families of related objects and ensure they stay compatible.
SOLID
How do factory patterns support the Open/Closed and Dependency Inversion principles?
Open/Closed: adding a new payment processor (e.g., RazorpayProcessor) requires creating a new class and registering it in the factory — zero changes to existing client code or routing logic. The factory is the only change point. Dependency Inversion: the caller depends on the PaymentProcessor interface (abstraction), not on StripeProcessor (concretion). The factory resolves the abstraction to a concrete type at runtime, keeping the caller isolated from implementation details.

Builder
Creational
Fluent API · complex object construction · telescoping constructor problem
Definition

Builder separates the construction of a complex object from its representation, allowing the same construction process to create different representations. It solves the telescoping constructor problem: when a class has many optional parameters, constructors multiply (all combinations) or one mega-constructor with many nulls is needed. Builder provides a readable, fluent alternative.

The problem builder solves

Without builder — which null means what?

new User("Somesh", null, null, true, false, 25, null)

With builder — self-documenting

User.builder().name("Somesh").age(25).active(true).build()
How builder works internally
1
Static inner Builder class — the Builder mirrors all the fields of the target class. Each setter on the Builder returns this (the Builder) — enabling method chaining. The target class's constructor is private, forcing callers to use the Builder.
2
build() method — performs validation (mandatory fields, constraints) and passes all accumulated values to the private constructor of the target class. This is the single place where construction correctness is enforced.
3
Immutability — Builder pattern pairs naturally with immutable classes. The target object has final fields set once in the private constructor. The Builder accumulates the mutable state, then creates one final immutable object. This is how record's compact constructor and Lombok's @Builder work.
4
Director (optional) — a Director class knows HOW to use a builder to construct specific configurations: director.buildMinimalUser(builder) vs director.buildAdminUser(builder). Used when you have multiple fixed "recipes" for the same product.
Code — hand-rolled + Lombok + real HTTP client example
// Hand-rolled Builder public class HttpRequest { private final String url, method, body; private final Map<String,String> headers; private final int timeoutMs; private HttpRequest(Builder b) { // private — only Builder creates it this.url = b.url; this.method = b.method; this.body = b.body; this.headers = Map.copyOf(b.headers); this.timeoutMs = b.timeoutMs; } public static class Builder { private String url, method = "GET", body; private Map<String,String> headers = new HashMap<>(); private int timeoutMs = 5000; public Builder url(String u) { this.url=u; return this; } public Builder method(String m) { this.method=m; return this; } public Builder body(String b) { this.body=b; return this; } public Builder header(String k,String v) { headers.put(k,v); return this; } public Builder timeout(int ms) { this.timeoutMs=ms; return this; } public HttpRequest build() { Objects.requireNonNull(url, "url is required"); // validation return new HttpRequest(this); } } } // Usage — fluent, self-documenting HttpRequest req = new HttpRequest.Builder() .url("https://api.example.com/users") .method("POST") .header("Authorization", "Bearer " + token) .body(jsonPayload) .timeout(3000) .build(); // Lombok @Builder — generates all this automatically @Builder @Getter public class OrderRequest { private final String userId, productId; private final int quantity; @Builder.Default private final String currency = "INR"; }
Real-world use cases
🌐
OkHttp / RestTemplate — new Request.Builder().url(...).addHeader(...).build(). HTTP requests have dozens of optional parameters — builder is the natural fit.
🔎
JPA Criteria API / QueryDSL — building complex queries fluently: criteriaBuilder.and(...).or(...). The query object is only constructed when build() is called after all predicates are accumulated.
☁️
AWS SDK v2 — S3Client.builder().region(Region.AP_SOUTH_1).credentialsProvider(...).build(). All AWS service clients use Builder.
Interview questions
Classic
What problem does the Builder pattern solve and how?
It solves the telescoping constructor problem: when a class has many optional parameters, you'd need constructors for every combination (N parameters = 2^N constructors) or one big constructor with nulls that's unreadable. Builder provides a fluent, named-parameter API where callers set only what they need, and build() performs validation at construction time. Additional benefits: natural fit for immutable objects (Builder holds mutable state during construction, product is final/immutable), self-documenting call sites, and the ability to reuse a builder to create similar objects.
Lombok
How does Lombok's @Builder work and what are its limitations?
@Builder generates: a static inner Builder class with all fields, setter methods returning 'this', a static builder() method, and a build() method calling the all-args constructor. Limitations: (1) @Builder on a class with inheritance is complex — use @SuperBuilder for parent-child hierarchies. (2) @Builder.Default for field defaults (non-null defaults aren't auto-carried). (3) Generated builders are not validated — you must add validation manually in a custom build() method or use @Builder with toBuilder=true for copy-modification patterns. In production Spring Boot: @Builder + @Getter + @Value (Lombok) covers most DTO use cases.

Prototype
Creational
Cloning objects · shallow vs deep copy · Cloneable vs copy constructors
Definition

Prototype creates new objects by copying an existing object (the prototype) instead of creating from scratch. Useful when object creation is expensive (network call, complex computation) and the new object differs only slightly from an existing one. In Java, implemented via Object.clone() or copy constructors.

Shallow vs deep copy — the critical distinction
S
Shallow copy — copies the object's fields. Primitives are duplicated. Object references are SHARED — the copy and the original point to the same nested objects. Mutating a nested list in the copy also mutates the original. Object.clone() does a shallow copy by default.
D
Deep copy — recursively copies all nested objects. The copy is fully independent. Methods: (1) manual: clone each nested object in clone(). (2) serialisation round-trip: serialise to bytes, deserialise to a new object (slow but complete). (3) copy constructor: new Order(original) — manually copy each field including nested collections.
!
Cloneable pitfalls — Cloneable is a marker interface with no clone() method — the method is on Object. Implementing Cloneable but not overriding clone() gives a protected shallow copy. clone() throws CloneNotSupportedException if Cloneable not implemented. Most modern Java: prefer copy constructors over Cloneable. Effective Java Item 13: "Avoid Cloneable."
Code — Cloneable vs copy constructor vs serialisation
// 1. Cloneable — avoid in modern Java class ShallowOrder implements Cloneable { String id; List<Item> items; // shared reference in shallow copy! public ShallowOrder clone() throws CloneNotSupportedException { return (ShallowOrder) super.clone(); // items list is SHARED } } // 2. Copy constructor — preferred (explicit, safe) class Order { final String id; final List<Item> items; final Address shippingAddr; Order(Order src) { // copy constructor — deep copy this.id = src.id; this.items = src.items.stream() .map(Item::new) // Item also has a copy constructor .collect(Collectors.toList()); this.shippingAddr = new Address(src.shippingAddr); } } // 3. Serialisation deep clone — convenient but slow @SuppressWarnings("unchecked") public static <T extends Serializable> T deepClone(T obj) { try (var bos = new ByteArrayOutputStream(); var oos = new ObjectOutputStream(bos)) { oos.writeObject(obj); try (var ois = new ObjectInputStream( new ByteArrayInputStream(bos.toByteArray()))) { return (T) ois.readObject(); } } catch (Exception e) { throw new RuntimeException(e); } }
Interview questions
Classic
What is the difference between shallow and deep copy in Java?
Shallow copy: duplicates the object's field values. Primitive fields get new copies. Reference fields get copied references — both the original and the copy point to the same nested objects. Mutating a shared nested object affects both. Default Object.clone() is shallow. Deep copy: recursively copies all nested objects. The result is fully independent — modifying the copy doesn't affect the original. Implement via copy constructors, manual clone() overrides, or serialisation round-trip.
Why
Why does Effective Java say to avoid Cloneable?
Multiple issues: (1) Cloneable has no clone() method — the contract is on Object, not the interface you implement. (2) You must call super.clone() up the hierarchy correctly to get the right type back. (3) Final fields can't be set by clone() — they'd require using reflection. (4) Exception handling is awkward. (5) Inheritance complicates it further. Preferred alternatives: copy constructor new MyClass(original) — explicit, refactoring-friendly, no exception contract. Or static factory method MyClass.copyOf(original). Records get a free canonical constructor for reconstruction.

Adapter
Structural
Bridging incompatible interfaces · class adapter vs object adapter · real integrations
Definition

Adapter converts the interface of a class into another interface the client expects. It wraps an existing object (the adaptee) and exposes a different interface. The client talks to the adapter in the language it knows; the adapter translates and delegates to the adaptee. Like a plug adapter — same power, different socket shape.

Object vs class adapter
Object Adapter (preferred)
Wraps the adaptee via composition. Works even when adaptee is final. More flexible — can adapt a class and its subclasses. This is the Go4 recommended approach for Java.
Class Adapter (rare in Java)
Extends the adaptee and implements the target interface. Only possible when adaptee is not final. Java's single inheritance limits usefulness. Prefer composition.
Code — payment gateway integration + Java I/O adapter
// Target interface — what your system expects interface PaymentGateway { PaymentResult charge(String customerId, long amountPaisa); } // Adaptee — Stripe's actual SDK (you can't change this) class StripeSDK { public StripeCharge createCharge(Map<String,Object> params) { /* ... */ return null; } } // Adapter — translates PaymentGateway to StripeSDK class StripeAdapter implements PaymentGateway { private final StripeSDK stripe; // composition — object adapter StripeAdapter(StripeSDK sdk) { this.stripe = sdk; } public PaymentResult charge(String customerId, long amountPaisa) { Map<String,Object> params = Map.of( "customer", customerId, "amount", amountPaisa, // Stripe expects paise natively "currency", "inr" ); StripeCharge charge = stripe.createCharge(params); return new PaymentResult(charge.getId(), charge.getStatus()); } } // Client never knows it's talking to Stripe specifically PaymentGateway gateway = new StripeAdapter(new StripeSDK()); PaymentResult result = gateway.charge(userId, 50000L); // Java I/O — classic adapter usage // InputStreamReader adapts InputStream (byte stream) to Reader (char stream) Reader reader = new InputStreamReader(socket.getInputStream(), StandardCharsets.UTF_8);
Real-world use cases
🔌
Java I/O — InputStreamReader adapts InputStream (bytes) to Reader (chars). OutputStreamWriter adapts OutputStream to Writer. Classic adapter pattern in the JDK.
💳
Payment gateway aggregators — a PaymentGateway interface in your system. StripeAdapter, RazorpayAdapter, UPIAdapter wrap each SDK. Switching providers = swap the adapter, zero change to business logic.
📚
Arrays.asList() / List.of() — wraps an array in a List interface. The array is the adaptee; the List is the target interface your code expects.
Interview questions
Distinguish
What is the difference between Adapter and Decorator patterns?
Adapter changes the interface — it wraps a class with a different interface so clients that expect interface A can use an object that implements interface B. The purpose is compatibility. Decorator keeps the same interface — it wraps a class with the same interface to add behaviour. The purpose is extension. Memory device: Adapter = translator (different language). Decorator = wrapper (same language, added meaning). Both use composition/wrapping but for opposite purposes.

Decorator
Structural
Same interface · adds behaviour by wrapping · Java I/O is a decorator stack
Definition

Decorator attaches additional responsibilities to an object dynamically by wrapping it with another object that implements the same interface. It's an alternative to subclassing for extending behaviour. You can stack multiple decorators — each adds one concern — producing composed behaviour without class explosion.

Java I/O — a decorator stack you use every day
new BufferedReader(
  new InputStreamReader(
    new GZIPInputStream(
      socket.getInputStream()
    ) ← adds decompression
  )     ← adds char encoding
)        ← adds buffering
How decorator works
1
Same interface — the decorator implements the same interface as the component it wraps. This makes it transparent to the client — the client doesn't know whether it's talking to the raw component or a stack of decorators.
2
Delegates to wrapped component — the decorator holds a reference to the component (or another decorator). It calls the wrapped component's method and adds behaviour before, after, or instead of the delegation.
3
Composable at runtime — unlike inheritance (fixed at compile time), decorator composition is dynamic. You can choose to add logging, caching, and rate-limiting to one service instance and only logging to another, using the same decorator classes.
!
vs inheritance — 3 base classes with 3 optional features = 8 subclasses (2³) with inheritance. With decorator: 3 base classes + 3 decorator classes = 9 total classes, but unlimited combinations at runtime. This is why the GoF says "favour composition over inheritance" — decorator is the composition solution to feature explosion.
Code — service decorator with logging + caching
interface UserService { User getUser(String id); } class UserServiceImpl implements UserService { public User getUser(String id) { return db.findById(id); } // real work } // Logging decorator class LoggingUserService implements UserService { private final UserService delegate; // wraps any UserService public User getUser(String id) { log.info("getUser({})", id); User u = delegate.getUser(id); // delegate, then add logging log.info("returned: {}", u); return u; } } // Caching decorator class CachingUserService implements UserService { private final UserService delegate; private final Cache<String,User> cache; public User getUser(String id) { return cache.computeIfAbsent(id, delegate::getUser); } } // Compose at runtime — order matters (outer decorator runs first) UserService svc = new LoggingUserService( new CachingUserService( new UserServiceImpl(), cache ) ); // Request: log → check cache → (if miss) hit DB → log result // Spring does this for you: @Transactional, @Cacheable, @Async // all wrap beans with proxy decorators at startup
Real-world use cases
📦
Java I/O streams — every InputStream, OutputStream, Reader, Writer wrapper is a decorator. BufferedInputStream adds buffering. GZIPInputStream adds decompression. Stack them arbitrarily.
🌿
Spring AOP proxies — @Transactional, @Cacheable, @Async, @Retryable all wrap your beans with JDK dynamic proxies or CGLIB proxies that implement the same interface. The Decorator pattern at framework scale.
🔒
Security middleware — authentication filter → authorisation filter → rate-limiting filter → your endpoint. Each filter is a decorator on the request handler chain.
Interview questions
Classic
Explain the Decorator pattern with a real Java example.
Decorator wraps an object with the same interface to add behaviour. Classic example: Java I/O. BufferedReader(new InputStreamReader(new FileInputStream(...))). Each layer adds a concern — buffering, character encoding, file reading — without modifying the others. In application code: @Transactional in Spring works this way — Spring wraps your @Service bean with a proxy that begins a transaction before your method runs and commits/rolls back after. Your code is unaware of the wrapping. This is the decorator at framework scale.
vs Inheritance
Why use Decorator instead of inheritance to add behaviour?
Inheritance is static (fixed at compile time) and causes class explosion with multiple optional features (3 features × 2 states = 8 subclasses). Decorator is dynamic — you compose any combination of decorators at runtime. You can add caching to one instance, logging to another, and both to a third, using the same decorator classes. The Open/Closed Principle: you add new decorators without modifying existing code. Decorator also supports single responsibility — each decorator does exactly one thing.

Observer
Behavioural
Event-driven · publish-subscribe · Spring events · decoupling producers from consumers
Definition

Observer defines a one-to-many dependency so that when one object (the subject/publisher) changes state, all its dependents (the observers/subscribers) are notified automatically. It decouples the publisher from the subscribers — the publisher doesn't know who's listening or what they do with the notification.

Push model
Publisher sends event data to observers. Observers receive what the publisher decided to send. Simpler, tighter coupling on the event shape.
Pull model
Publisher sends only a notification. Observers pull the data they need from the publisher. More flexible — observers choose their data — but requires a reference to the subject.
How it works
1
Registration — observers call subject.subscribe(this) to register. The subject maintains a list of observers (a List<Observer>). Observers can also unsubscribe — crucial to prevent memory leaks (a registered observer is a GC root via the subject's list).
2
Notification — when the subject's state changes, it iterates its observer list and calls observer.update(event) on each. Synchronous by default — observers run in the publisher's thread. For decoupled async notification: post to a queue, use Spring's @Async @EventListener, or Guava EventBus.
3
Memory leak risk — if observers register but never deregister, the subject holds strong references to them indefinitely. They're never garbage-collected even if the observer is logically "done". Fix: use WeakReference<Observer> in the subject's list, or always deregister in lifecycle callbacks (Spring's @PreDestroy, Android's onPause).
Code — hand-rolled + Spring ApplicationEvent
// Hand-rolled observer interface OrderObserver { void onOrderPlaced(Order order); } class OrderService { private final List<OrderObserver> observers = new CopyOnWriteArrayList<>(); public void subscribe(OrderObserver o) { observers.add(o); } public void unsubscribe(OrderObserver o) { observers.remove(o); } public void placeOrder(Order order) { persist(order); observers.forEach(o -> o.onOrderPlaced(order)); // notify all } } class InventoryService implements OrderObserver { public void onOrderPlaced(Order o) { reserveStock(o); } } class NotificationService implements OrderObserver { public void onOrderPlaced(Order o) { sendConfirmation(o); } } // Spring ApplicationEvent — production approach record OrderPlacedEvent(Order order) extends ApplicationEvent { OrderPlacedEvent(Order o) { super(o); this.order = o; } } @Service class OrderSvc { ApplicationEventPublisher pub; void place(Order o) { persist(o); pub.publishEvent(new OrderPlacedEvent(o)); } } @Component class InvSvc { @EventListener @Async // async: different thread void handle(OrderPlacedEvent e) { reserveStock(e.order()); } }
Real-world use cases
🎯
Spring ApplicationEvents — @EventListener is the Observer pattern at framework scale. Order placed → inventory reserved, email sent, analytics tracked. Services don't know about each other — they both listen to the same event.
📡
Kafka / RabbitMQ — the message queue IS the Observer pattern at infrastructure scale. Producers publish events. Multiple consumer groups subscribe. Adding a new consumer (observer) requires zero change to the producer.
🌊
Reactive streams (RxJava, Reactor) — Observable/Flux is a publisher; Observer/Subscriber is the consumer. The entire reactive paradigm is observer pattern + backpressure + async composition.
Interview questions
Classic
Explain the Observer pattern with a real production example.
Observer: subject maintains a list of observers and notifies them on state changes. Observers don't know about each other — only the subject. Production example: order processing. OrderService publishes an OrderPlacedEvent (Spring ApplicationEvent). InventoryService, NotificationService, and AnalyticsService each have an @EventListener. When an order is placed, all three are notified. Adding a new RewardService requires zero change to OrderService — just add an @EventListener. This is the Open/Closed Principle applied to event handling.
Memory
How can the Observer pattern cause memory leaks and how do you prevent them?
When an observer registers with a long-lived subject, the subject holds a strong reference to it. If the observer's lifecycle ends but it never unregisters, the GC can't collect it. The subject's list is a GC root. Classic case: Android Activity registering with a singleton EventBus and not unregistering in onDestroy — the Activity leaks. Prevention: (1) always unsubscribe in lifecycle callbacks (Spring @PreDestroy, Android onPause). (2) Use WeakReference in the observer list — the observer can be GC'd even if not explicitly unregistered. (3) Use Spring ApplicationEvents — Spring manages listener lifecycles with bean lifecycle; when a bean is destroyed, its listeners are deregistered.

Strategy
Behavioural
Swappable algorithms · lambdas as strategies · Comparator is a strategy
Definition

Strategy defines a family of algorithms, encapsulates each one, and makes them interchangeable. The client holds a reference to a strategy interface and delegates the algorithm to whatever concrete strategy is injected. This lets the algorithm vary independently of the clients that use it.

Modern Java: lambdas replaced boilerplate Strategy classes

Before Java 8: Strategy = interface + concrete class per algorithm. After Java 8: a functional interface IS the Strategy interface; a lambda IS a concrete strategy. Comparator, Predicate, Function are all built-in strategy interfaces in the JDK.

Real-world use cases
🔀
Payment routing — PaymentStrategy interface, implementations for Stripe/Razorpay/UPI. Selected based on user country, amount, or preference. Switching providers for A/B testing = swap the strategy bean in config.
📊
Sorting / Comparator — Comparator is a built-in Strategy in the JDK. Collections.sort(list, comparator) — the sorting algorithm is fixed, but the comparison strategy is injected.
🗃️
Spring Security authentication — AuthenticationProvider is a Strategy interface. Multiple providers can be chained: username/password, OAuth, JWT, SAML. Spring tries each strategy in order.
Interview questions
Modern Java
How do lambdas change the way you implement the Strategy pattern in Java 8+?
Before Java 8: Strategy required a dedicated interface AND a concrete class per algorithm. After Java 8: any @FunctionalInterface IS a Strategy interface; a lambda IS the concrete strategy. Instead of new SeasonalDiscountStrategy(), you write price -> price * 0.8. This eliminates class files for simple strategies — the algorithm is an inline value. The pattern's intent is unchanged: inject behaviour, vary the algorithm independently. Lambdas just eliminate the boilerplate. Comparator, Predicate, Function — all GoF Strategy pattern instances already in the JDK.
vs Template Method
How does Strategy differ from Template Method?
Both allow varying part of an algorithm. Template Method: the skeleton is fixed in the superclass, variable parts are abstract methods that subclasses override. Uses inheritance. Strategy: the entire algorithm is encapsulated in a separate strategy object injected into the context. Uses composition. Key difference: Template Method uses inheritance (IS-A), Strategy uses composition (HAS-A). Strategy is more flexible — you can change the algorithm at runtime. Template Method is set at compile time by subclassing. Prefer Strategy; it aligns with "favour composition over inheritance."

Template Method
Behavioural
Fixed skeleton · variable steps · Hollywood Principle · Spring's JdbcTemplate
Definition

Template Method defines the skeleton of an algorithm in a base class, deferring some steps to subclasses. Subclasses can override specific steps without changing the algorithm's overall structure. Implements the Hollywood Principle: "Don't call us, we'll call you" — the framework calls your hooks, not the other way around.

Algorithm skeleton — what's fixed vs what's variable
final processOrder() { ← fixed: can't override
  validateOrder(); ← fixed step
  applyDiscount(); ← hook — subclass overrides
  processPayment(); ← abstract — subclass must implement
  sendConfirmation(); ← fixed step
}
Two types of variable steps
Abstract steps
No default implementation. Subclass MUST override. Used for steps that are mandatory and fundamentally different per variant. Compile error if not implemented.
Hook methods
Have a default implementation (often empty). Subclass MAY override for optional extensions. The template calls them at defined points — subclasses opt in to customise.
Code — data export pipeline + Spring JdbcTemplate analogy
// Template Method — data export pipeline abstract class DataExporter { // TEMPLATE METHOD — final: algorithm skeleton is fixed public final void export(String destination) { List<?> data = fetchData(); // abstract — subclass provides data List<?> transformed = transform(data); // hook — default = no-op validate(transformed); // fixed — always validates write(transformed, destination); // abstract — subclass writes format onExportComplete(destination); // hook — optional post-action } protected abstract List<?> fetchData(); protected abstract void write(List<?> data, String dest); protected List<?> transform(List<?> data) { return data; } // hook: default no-op protected void onExportComplete(String d) {} // hook: default empty private void validate(List<?> d) { /* always validates */ } } class CsvExporter extends DataExporter { protected List<Order> fetchData() { return orderRepo.findAll(); } protected void write(List<?> d, String p) { /* write CSV */ } protected void onExportComplete(String p) { notify(p); } // opt-in hook } // Spring JdbcTemplate IS Template Method // You provide the SQL and RowMapper; Spring handles connection, exception, close List<User> users = jdbcTemplate.query( "SELECT * FROM users WHERE active=?", (rs, row) -> new User(rs.getString("id"), rs.getString("name")), true ); // YOUR hooks: sql + mapper. Spring's skeleton: get conn → execute → map → close
Real-world use cases
🌿
Spring JdbcTemplate — the skeleton: acquire connection, prepare statement, execute, map results, close connection, handle exceptions. Your hook: the SQL string and RowMapper. You never write the connection lifecycle code.
🌐
HttpServlet — service() is the template method. It dispatches to doGet(), doPost(), etc. — which are hooks you override. The request parsing, response flushing, error handling are all in the fixed skeleton.
🧪
JUnit — @BeforeEach, @Test, @AfterEach are hooks the JUnit framework calls at fixed points in the test lifecycle. You fill in the hooks; the test runner manages the skeleton.
Interview questions
Classic
What is the Hollywood Principle and how does Template Method implement it?
"Don't call us, we'll call you." The framework or base class controls the flow. Your code (the subclass) doesn't call the base class — the base class calls you at specific extension points (abstract methods and hooks). Template Method: the base class's final method drives execution and calls abstract/hook methods that subclasses implement. You never explicitly invoke the skeleton — you just provide the hooks and the framework invokes them. This inverts control: you extend the framework's behaviour without controlling when it runs. Same principle as Spring's lifecycle callbacks — you implement @PostConstruct; Spring calls it at the right time.
vs Strategy
When do you choose Template Method over Strategy?
Template Method: fixed algorithm structure, vary specific steps using inheritance. Best when: the steps are tightly coupled to the skeleton, you want to reuse large chunks of the base implementation, and you control the class hierarchy. Strategy: encapsulate the entire algorithm in a separate object, inject it. Best when: you need to change the whole algorithm at runtime, you want to compose behaviours without inheritance, or you want to test strategies independently. Modern preference: Strategy (via lambdas) over Template Method because inheritance locks you in. Template Method shines in frameworks (Spring, Servlet, JUnit) where the framework owns the skeleton and you extend it.

Command
Behavioural
Encapsulate requests · undo/redo · queuing commands · macros
Definition

Command encapsulates a request as an object, thereby allowing you to parameterise clients with different requests, queue or log requests, and support undoable operations. It decouples the object that invokes the operation (invoker) from the object that knows how to perform it (receiver). The command object bridges them.

Client
Creates command, sets receiver, passes to invoker
Command
Interface with execute() and optionally undo()
Invoker
Calls execute() at the right time. Doesn't know what it does
Receiver
Knows HOW to perform the actual work
What Command enables
1
Undo/Redo — push executed commands onto a history stack. undo() pops and calls the reverse operation. Text editors, drawing apps, IDEs all use this. The command stores enough state to reverse itself.
2
Queuing — commands are serialisable objects. Put them in a BlockingQueue. Worker threads drain the queue and execute. Rate limiting, job scheduling, retry — all natural fits.
3
Macro commands — a MacroCommand implements Command and holds a list of sub-commands. execute() runs them in order. Transactions, batch operations, workflow steps.
4
Audit logging — every executed command is logged (who, what, when, with what params). The command object carries all context needed for the audit trail.
Code — text editor undo/redo + async command queue
interface Command { void execute(); void undo(); } class TextEditor { // Receiver StringBuilder content = new StringBuilder(); void insert(int pos, String text) { content.insert(pos, text); } void delete(int pos, int len) { content.delete(pos, pos+len); } } class InsertCommand implements Command { private final TextEditor editor; private final int position; private final String text; public void execute() { editor.insert(position, text); } public void undo() { editor.delete(position, text.length()); } } class CommandHistory { // Invoker with undo/redo Deque<Command> history = new ArrayDeque<>(); void execute(Command cmd) { cmd.execute(); history.push(cmd); } void undo() { if(!history.isEmpty()) history.pop().undo(); } } // Modern Java: Runnable/lambda IS a Command (execute only, no undo) BlockingQueue<Runnable> jobQueue = new LinkedBlockingQueue<>(); jobQueue.put(() -> processOrder(order)); // command as lambda jobQueue.put(() -> sendEmail(user)); // Worker thread drains and executes — invoker never knows what the command does // Spring @Async — Command queued and executed by framework @Async void sendWelcomeEmail(User user) { emailService.send(user.email()); } // Caller: just calls the method. Spring wraps it as a Runnable, submits to executor
Real-world use cases
↩️
Text editor undo/redo — IntelliJ, VS Code, Google Docs all maintain a command history. Ctrl+Z pops the last command and calls undo(). Each keystroke creates a command object with enough state to reverse it.
📋
ExecutorService / thread pools — executor.submit(runnable) — the Runnable is a Command. The pool is the Invoker. It executes commands on whatever thread is available. You can queue, cancel (Future), and track completion.
💳
Database transactions as commands — a TransactionalCommand wraps multiple DB operations. execute() runs them in a transaction. undo() runs a compensating transaction. This is the Saga pattern at the service level.
🤖
EngineX use case — interview question generation, feedback scoring, resume parsing — each is a Command. They can be queued, retried on failure, prioritised, and tracked independently. The AI invocation is the Receiver; the pipeline orchestrator is the Invoker.
Interview questions
Classic
What does the Command pattern enable that plain method calls don't?
Command encapsulates a request as an object, which enables: (1) Undo/redo — the command stores state for reversal; a history stack of executed commands can be popped and undone. (2) Queuing — commands are objects; they can be put in queues, delayed, retried, and rate-limited independent of the caller. (3) Logging/audit — the command object carries all context; execution can be logged with full parameters. (4) Macro commands — a command can contain other commands and execute them in sequence (batch operations, transactions). None of these are possible with plain method calls because the call is ephemeral — it doesn't exist as a first-class object.
Modern Java
Is Runnable the Command pattern in Java?
Runnable implements the execute() part of Command. It's a degenerate Command — execute only, no undo. ExecutorService.submit(runnable) is textbook Command pattern: you (the client) create the command (runnable), hand it to the invoker (executor), and the receiver (pool thread) executes it. You can cancel (Future.cancel()), which is a form of preventing execution. For full Command with undo, you implement a custom interface with execute() and undo(). In practice: Runnable/Callable covers 90% of Command use cases in Java; undo is only needed in specific domains (editors, drawing tools).
