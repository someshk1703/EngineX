Lambda expressions
Syntax · effectively final · target typing · invokedynamic under the hood
Definition

A lambda is an anonymous function — a block of code that can be passed around as data. In Java, a lambda is always an instance of a functional interface (any interface with exactly one abstract method). Lambdas enable functional-style programming: passing behaviour as arguments, returning behaviour from methods, and composing pipelines.

Concise
Replace anonymous inner classes with a single expression. (x) -> x * 2 vs 5 lines of boilerplate.
Target-typed
The compiler infers the lambda's type from context — the functional interface it's being assigned to or passed into.
Captures context
Can close over local variables (must be effectively final) and instance fields (via implicit this).
How lambdas work under the hood — invokedynamic
1
Not anonymous inner classes — before Java 8, anonymous inner classes were the pattern. They compile to separate ClassName$1.class files. Lambdas do NOT compile to inner class files — they're fundamentally different at the bytecode level.
2
invokedynamic — each lambda call site compiles to an invokedynamic bytecode instruction (JDK 7 feature, Java 8 first consumer). This is a "bootstrap" mechanism — on first call, LambdaMetafactory.metafactory() is invoked which spins up an implementation class at runtime. Subsequent calls are direct — the bootstrap only runs once.
3
LambdaMetafactory — generates a class that implements the functional interface and delegates to the lambda body, which the compiler extracted as a private synthetic method in the enclosing class. The JVM caches this generated class.
4
Effectively final — local variables captured by a lambda must be final or effectively final (never reassigned after assignment). Why? The lambda may outlive the method stack frame. Effectively final guarantees the captured value is immutable — safe to copy into the lambda's closure. Mutable locals would require heap allocation and thread-safety guarantees the JVM doesn't provide here.
5
Instance fields vs locals — instance fields captured by a lambda are accessed via an implicit this reference (always effectively final as a reference, even if the field's value changes). This means a lambda can mutate an object's fields — just not reassign a local variable.
Syntax — every form in one place
// Zero params Runnable r = () -> System.out.println("hello"); // One param — parens optional Consumer<String> c = s -> System.out.println(s); Consumer<String> c2 = (s) -> System.out.println(s); // also valid // Two params BiFunction<Integer, Integer, Integer> add = (a, b) -> a + b; // Block body — needs explicit return Function<String, Integer> parse = s -> { int result = Integer.parseInt(s.trim()); return result * 2; }; // With explicit types (rarely needed — usually inferred) BiFunction<String, Integer, Boolean> fn = (String s, Integer n) -> s.length() > n; // Capturing local — MUST be effectively final int threshold = 10; // effectively final — never reassigned Predicate<Integer> above = n -> n > threshold; // threshold = 20; // would cause compile error above // Capturing instance field — via implicit this (fine) class Processor { private int multiplier = 3; Function<Integer, Integer> scale = n -> n * multiplier; // ok — uses this }

Common mistakes
!
Mutating a captured local via an array trick — int[] count = {0}; list.forEach(x -> count[0]++); This is a code smell — the array is the workaround for effectively-final, but the mutation is not thread-safe. Use AtomicInteger or collect and count with streams.
!
Checking instanceof on a lambda — myLambda instanceof Runnable is true but instanceof Comparator is false even if the signature matches. Lambdas implement exactly one functional interface — the one they were targeted to. Don't rely on multi-interface casting.
!
Serialising lambdas — lambdas are not serialisable by default. If you need to serialise a functional interface value, either use an anonymous class or cast the lambda to Serializable (a hack). Better: don't serialise lambdas — redesign.
Interview questions
Internals
How does the JVM implement lambda expressions? Are they anonymous inner classes?
No — lambdas are NOT compiled to anonymous inner class files. Each lambda call site compiles to an invokedynamic bytecode instruction. On first execution, LambdaMetafactory.metafactory() is called: it generates (at runtime) a class that implements the target functional interface by delegating to the lambda body, which the compiler extracted as a private synthetic static method. This generated class is cached. Benefits over inner classes: (1) no .class file per lambda, (2) JVM can choose implementation strategy at runtime (e.g. direct method handle), (3) faster class loading.
Effectively final
Why must captured local variables be effectively final in lambdas?
A lambda may outlive the stack frame of the method that created it. Local variables live on the stack; when the method returns, the stack frame is gone. Java's solution: copy the local variable's value into the lambda's closure (a heap-allocated object). For this copy to be safe, the variable must never change after the copy is made — hence effectively final. If it could change, the lambda would have a stale copy and the original would be gone — an undetectable bug. Instance fields don't have this problem because they're on the heap and accessed via a stable 'this' reference.

Stream API
Lazy evaluation · intermediate vs terminal · map/filter/reduce/collect/flatMap
Definition

A Stream is a lazily-evaluated pipeline of operations on a sequence of elements. It doesn't store data — it pulls elements from a source, passes each through a chain of intermediate operations, and drives execution with a terminal operation. Streams support both sequential and parallel execution without changing the pipeline code.

→
map
→
sorted
→
collect
Intermediate operations
Return a new Stream. Lazy — nothing runs until a terminal op is called. Examples: filter, map, flatMap, distinct, sorted, peek, limit, skip.
Terminal operations
Consume the stream, produce a result or side-effect. Trigger all lazy intermediate ops. Examples: collect, forEach, reduce, count, findFirst, anyMatch, toList (Java 16+).
Lazy evaluation — how the pipeline actually runs
1
Nothing runs at declaration — calling .filter(), .map() builds a chain of StatelessOp/StatefulOp objects. No elements are processed. This is key: a stream pipeline with no terminal op does nothing.
2
Pull model — the terminal op pulls one element at a time through the entire pipeline. Element 1 goes through filter → map → sorted before element 2 is pulled. Exception: stateful ops like sorted and distinct must buffer the entire stream before emitting — they're "pipeline breaks".
3
Short-circuit operations — findFirst, anyMatch, limit(n) stop pulling as soon as they have their answer. With limit(1) after a heavy computation, only ONE element is computed — not all. This is the power of lazy evaluation.
4
Streams are single-use — once a terminal op is called, the stream is consumed. Calling any method on a used stream throws IllegalStateException. Unlike Iterable, you can't iterate a stream twice — recreate from the source.
5
Parallel streams — list.parallelStream() or stream.parallel(). Uses the ForkJoinPool common pool. Each chunk is processed on a separate thread and merged. Use for CPU-bound stateless operations on large datasets (>10k elements). Avoid for I/O, stateful ops, or small data — overhead dominates.
Key operations — what each does internally
map
map(Function) — 1-to-1 transform. Returns Stream of same size. stream.map(String::toUpperCase). Type can change: Stream<String> → Stream<Integer> via .map(String::length).
fM
flatMap(Function) — 1-to-many then flatten. Function returns a Stream per element; flatMap concatenates them. Essential for nested collections. orders.flatMap(o -> o.getItems().stream()) — gives a flat stream of all items across all orders.
fi
filter(Predicate) — keeps elements matching the predicate, drops the rest. Stateless, O(1) per element. Always push filters early in the pipeline to reduce elements for downstream ops.
re
reduce(identity, BinaryOperator) — fold the stream into a single value. stream.reduce(0, Integer::sum) = sum. Associative operator required for parallel correctness. The identity element must be the neutral element for the operation (0 for sum, 1 for product).
co
collect(Collector) — mutable reduction. Collects into a container (List, Map, Set, String). More efficient than reduce for building collections. Key collectors: toList(), toMap(), groupingBy(), joining(), partitioningBy().
di
distinct/sorted/limit/skip — stateful ops. distinct() buffers seen elements in a LinkedHashSet. sorted() buffers entire stream into an array then sorts. Both break lazy pull. limit(n)/skip(n) are stateful but don't require full buffering — just a counter.
Code — real-world pipeline patterns
// GROUP and aggregate — sales by category Map<String, Double> totalByCategory = orders.stream() .filter(Order::isCompleted) .collect(Collectors.groupingBy( Order::getCategory, Collectors.summingDouble(Order::getTotal))); // FLAT the nested — all products across all orders List<Product> allProducts = orders.stream() .flatMap(o -> o.getProducts().stream()) .distinct() .sorted(Comparator.comparing(Product::getName)) .collect(Collectors.toList()); // PARTITION — premium vs standard users Map<Boolean, List<User>> partitioned = users.stream() .collect(Collectors.partitioningBy(User::isPremium)); // JOIN strings String csv = items.stream() .map(Item::getName) .collect(Collectors.joining(", ", "[", "]")); // [a, b, c] // REDUCE — custom accumulator Optional<BigDecimal> total = transactions.stream() .map(Transaction::getAmount) .reduce(BigDecimal::add); // returns Optional — stream could be empty // PARALLEL — use for CPU-bound, stateless, large data long count = hugeList.parallelStream() .filter(x -> expensiveCheck(x)) .count();
Common mistakes
!
Reusing a stream — streams are single-use. After a terminal op, the stream is closed. Calling any op on it throws IllegalStateException: stream has already been operated upon or closed. Always recreate from the source.
!
Side effects in map/filter — using map() to mutate external state (a list, a counter) is a code smell and breaks parallel stream safety. Use forEach for side-effects and keep map/filter pure.
!
Parallel stream on small data or I/O-bound tasks — parallel uses ForkJoinPool threads. I/O blocks threads, starving the pool. Parallelism overhead (splitting, merging) beats the gain for small lists. Benchmark before parallelising. The common pool is shared — blocking it hurts CompletableFuture too.
!
forEach vs collect for building results — using forEach to add to an external list is not thread-safe in parallel and is a side-effect anti-pattern. Use collect(toList()) — it's the correct, thread-safe, functional way to build a result list.
Interview questions
Classic
What is the difference between map and flatMap?
map applies a 1-to-1 function — each element becomes exactly one output element. flatMap applies a 1-to-many function — each element becomes a Stream of outputs, and flatMap concatenates all those streams into one. Example: if each Order has a List<Item>, stream.map(o -> o.getItems()) gives Stream<List<Item>> (nested). stream.flatMap(o -> o.getItems().stream()) gives Stream<Item> — flat. Rule: use flatMap whenever your mapping function returns a collection/stream that you want flattened.
Lazy
Why do streams use lazy evaluation and why does it matter?
Lazy evaluation means intermediate operations don't run until a terminal operation is called, and they run one element at a time through the pipeline (pull model). It matters because: (1) short-circuit ops (findFirst, limit) can stop processing after finding their answer — without processing the whole stream. (2) You can build infinite streams (Stream.generate, Stream.iterate) because only the elements you actually consume are computed. (3) Memory-efficient — elements aren't buffered between stages (except for stateful ops like sorted). A stream pipeline with no terminal op literally does nothing — this trips up beginners who don't see output.
Design
When would you use parallelStream() and when would you avoid it?
Use parallelStream when: (1) data is large (>10k elements as a rough guide), (2) operations are CPU-bound and stateless (no shared mutable state), (3) each operation is expensive enough to justify parallelism overhead. Avoid when: (1) tasks involve I/O (blocks ForkJoinPool threads, starves other tasks), (2) data is small (overhead of splitting/merging exceeds gain), (3) operations are ordered or have side effects (parallelism changes encounter order), (4) you're inside a web framework that manages its own thread pool. Always benchmark — premature parallelism is often slower than sequential.

Optional
Why it exists · correct usage · anti-patterns that make things worse
Definition

Optional<T> is a container that may or may not hold a non-null value. Its purpose: make the possibility of absence explicit in the type system, forcing callers to handle the "no value" case instead of silently receiving null and potentially triggering a NullPointerException later.

Designed for
Return types of methods that legitimately may not have a result. Stream terminal ops (findFirst, min, max, reduce). Repository methods (findById). Never for parameters, fields, or collections.
Not designed for
Method parameters (just use overloading or null). Fields in classes (serialisation overhead, null is fine in private fields). Collections (use empty collection instead of Optional<List>).
API — every method, when to use it
get
orElse(default) — always evaluates the default. Use when the default is cheap (a literal, a constant). orElseGet(Supplier) — lazy default: only calls the supplier if empty. Use when the default is expensive (DB call, object construction). This is the most missed distinction in interviews.
map
map(Function) — transform the value if present, return empty Optional if not. Returns Optional<U>. flatMap(Function) — when the function itself returns an Optional (avoids nested Optional<Optional<T>>). Same as Stream flatMap — unwraps one level.
fil
filter(Predicate) — if present and predicate is true, return this Optional; else return empty. Useful for conditional extraction in a pipeline.
!
get() — throws NoSuchElementException if empty. The worst way to use Optional — defeats its purpose (you'd have the same risk as null). Only safe after isPresent() check, but even then: use orElseThrow() with a meaningful exception message.
if
ifPresent(Consumer) — run a side-effect if value exists. ifPresentOrElse(Consumer, Runnable) (Java 9+) — handle both branches. or(Supplier<Optional>) (Java 9+) — chain Optional fallbacks.
Correct usage patterns
// GOOD — return type signals "may not exist" public Optional<User> findUser(String id) { return userRepo.findById(id); // Spring Data returns Optional } // GOOD — pipeline without null checks String city = findUser(id) .map(User::getAddress) .map(Address::getCity) .orElse("Unknown"); // GOOD — orElseGet for lazy expensive default User user = findUser(id) .orElseGet(() -> createDefaultUser(id)); // only called if empty // GOOD — orElseThrow with meaningful exception User user = findUser(id) .orElseThrow(() -> new UserNotFoundException("User not found: " + id)); // GOOD — flatMap when inner call also returns Optional Optional<String> email = findUser(id) .flatMap(User::getOptionalEmail); // avoids Optional<Optional<String>> // GOOD — filter + ifPresent findUser(id) .filter(User::isPremium) .ifPresent(u -> sendPremiumOffer(u));
Anti-patterns — what NOT to do
!
Using Optional as a method parameter — void process(Optional<String> name). Forces callers to wrap values in Optional.of(). Just use overloading or @Nullable. Optional is a return type container, not a parameter type.
!
Calling get() without checking — optional.get() throws if empty — you're back to the same risk as a null dereference. Use orElse, orElseGet, orElseThrow, or ifPresent.
!
Optional.of(null) — throws NullPointerException immediately. Use Optional.ofNullable(value) when the value may be null. Optional.of() is only for values you're certain aren't null.
!
Storing Optional in a field — Optional is not Serializable (it's deliberately not). Storing it in an entity/DTO field breaks serialisation. Use nullable fields in domain objects; wrap in Optional only in return types.
!
Optional<List> — use an empty list to represent "no results". Optional.empty() means "there was no list at all" — a different semantic. return Collections.emptyList() is clearer and avoids wrapping.
Interview questions
Classic
What is the difference between orElse and orElseGet?
orElse(T) always evaluates its argument — even if the Optional is present. orElseGet(Supplier<T>) only calls the supplier if the Optional is empty (lazy). Use orElse for cheap defaults like literals. Use orElseGet when the default requires a computation, DB call, or object construction — to avoid unnecessary work. This is a frequent senior interview question and the answer signals whether you understand eager vs lazy evaluation.
Design
When should you NOT use Optional?
Don't use Optional for: (1) method parameters — forces callers to wrap, use overloading instead, (2) fields in entities/DTOs — not Serializable, adds overhead, (3) collections — return empty collection instead of Optional<List>, (4) primitives — use OptionalInt/OptionalLong instead of Optional<Integer> to avoid boxing. Optional's single correct use: as the return type of a method that legitimately may not produce a result. It's a documentation tool that makes absence explicit at the type level.

Method references
Four types — when each applies and what the compiler does
Definition

A method reference is a shorthand lambda that refers to an existing method instead of writing a new body. Syntax: ClassName::methodName or instance::methodName. The compiler converts it into a lambda that calls the referenced method. They're cleaner than equivalent lambdas when the lambda does nothing but call a single method.

All four types — what they mean and when to use
1
Static method reference — ClassName::staticMethod. Lambda equivalent: (args) -> ClassName.staticMethod(args). Example: Integer::parseInt = s -> Integer.parseInt(s). Use when the method takes all the parameters the functional interface needs.
2
Instance method on arbitrary instance (unbound) — ClassName::instanceMethod. Lambda equivalent: (instance, args) -> instance.instanceMethod(args). The first lambda parameter becomes the target instance. Example: String::toUpperCase = s -> s.toUpperCase(). Most used in stream pipelines.
3
Instance method on specific instance (bound) — object::instanceMethod. Lambda equivalent: (args) -> object.instanceMethod(args). The target instance is fixed at reference creation time. Example: System.out::println = s -> System.out.println(s).
4
Constructor reference — ClassName::new. Lambda equivalent: (args) -> new ClassName(args). Matches Supplier, Function, BiFunction, etc. depending on how many params the constructor takes. Example: ArrayList::new as Supplier<ArrayList>.
Code — all four in context
// 1. Static method reference List<String> strs = List.of("1","2","3"); List<Integer> ints = strs.stream() .map(Integer::parseInt) // same as s -> Integer.parseInt(s) .collect(Collectors.toList()); // 2. Unbound instance method (ClassName::method) List<String> upper = strs.stream() .map(String::toUpperCase) // same as s -> s.toUpperCase() .collect(Collectors.toList()); strs.stream() .sorted(String::compareToIgnoreCase) // BiFunction: (s1,s2) -> s1.compareToIgnoreCase(s2) .forEach(System.out::println); // 3. Bound instance method (instance::method) String prefix = "hello"; Predicate<String> startsWith = prefix::startsWith; // fixed target strs.stream().filter(startsWith).forEach(System.out::println); // 4. Constructor reference Function<String, User> factory = User::new; // String constructor List<User> users = names.stream() .map(User::new) .collect(Collectors.toList()); // Stream toList() with constructor ref — very common pattern List<Thread> threads = tasks.stream() .map(Thread::new) // Runnable -> Thread constructor .collect(Collectors.toList());
Common mistakes
!
Using a method reference when a lambda is clearer — if you're doing more than calling one method, or you need to adapt parameters, write a lambda. Method references shine for single-method delegation. x -> someMethod(x, extraParam) cannot be a simple method reference — it needs a lambda.
!
Confusing unbound and bound instance references — String::toUpperCase (unbound) takes a String as first arg (the instance). myString::toUpperCase (bound) takes no args — the instance is fixed. They satisfy different functional interfaces.
Interview questions
Four types
List the four types of method references with examples.
(1) Static: Integer::parseInt — equivalent to s -> Integer.parseInt(s). (2) Unbound instance: String::toUpperCase — s -> s.toUpperCase(). The lambda parameter becomes the method target. (3) Bound instance: System.out::println — s -> System.out.println(s). The target is a specific fixed instance. (4) Constructor: ArrayList::new — () -> new ArrayList() or (n) -> new ArrayList(n) depending on functional interface arity. The key distinction between 2 and 3: unbound takes the instance as the first lambda arg; bound has the instance baked in.

Functional interfaces
Predicate · Function · Consumer · Supplier · BiFunction — the core six
Definition

A functional interface has exactly one abstract method (SAM — Single Abstract Method). This makes it the target type for lambdas and method references. @FunctionalInterface is an optional annotation that tells the compiler to enforce this rule. The java.util.function package provides the standard set for all common function shapes.

The six core functional interfaces
Predicate<T>
boolean test(T t)
Test a condition. Returns boolean. Composable: .and(), .or(), .negate(). Used in: stream filter, removeIf, anyMatch/allMatch.
Function<T, R>
R apply(T t)
Transform T into R. Composable: .andThen(), .compose(). Used in: stream map, computeIfAbsent, Optional.map.
Consumer<T>
void accept(T t)
Consume T with side effect, return nothing. Composable: .andThen(). Used in: stream forEach, ifPresent, ThreadLocal.
Supplier<T>
T get()
Produce a T with no input. Used for lazy evaluation, default factories. Used in: orElseGet, computeIfAbsent, orElseThrow.
BiFunction<T, U, R>
R apply(T t, U u)
Two inputs, one output. BinaryOperator<T> is a specialised BiFunction where both inputs and output are same type T. Used in: Map.merge, reduce.
UnaryOperator<T>
T apply(T t)
Specialised Function where input and output are same type. Used in: List.replaceAll, stream map when type doesn't change.
Code — composition and real usage
// Predicate composition Predicate<String> notBlank = s -> !s.isBlank(); Predicate<String> maxLength = s -> s.length() <= 50; Predicate<String> valid = notBlank.and(maxLength); Predicate<String> invalid = valid.negate(); // Function composition — andThen vs compose Function<String, Integer> parse = Integer::parseInt; Function<Integer, Integer> doubl = x -> x * 2; Function<String, Integer> parseDouble = parse.andThen(doubl); // parse then double Function<String, Integer> doubleParse = doubl.compose(parse); // same result, different perspective parseDouble.apply("5"); // 10 // Supplier for lazy default Supplier<Connection> connFactory = () -> dataSource.getConnection(); Connection conn = cache.computeIfAbsent("main", k -> connFactory.get()); // Consumer chaining — log then save Consumer<Order> logOrder = o -> log.info("Processing: " + o.id()); Consumer<Order> saveOrder = o -> repo.save(o); Consumer<Order> pipeline = logOrder.andThen(saveOrder); orders.forEach(pipeline); // Custom functional interface @FunctionalInterface interface TriFunction<A,B,C,R> { R apply(A a, B b, C c); }
Common mistakes
!
Confusing Function.andThen vs Function.compose — f.andThen(g) = g(f(x)): f first, then g. f.compose(g) = f(g(x)): g first, then f. andThen reads left-to-right — much more intuitive. Prefer andThen for pipelines.
!
Not using primitive functional interfaces — IntPredicate, IntFunction<R>, ToIntFunction<T> etc. avoid boxing overhead. If you're working with int in a hot loop, Predicate<Integer> boxes each int. Use IntPredicate instead.
Interview questions
Distinguish
What is the difference between Supplier and Function with no arguments?
Conceptually both produce a value, but they serve different roles. Supplier<T> = () -> T: no input, produces output. It represents a deferred factory or lazy initialiser — the caller doesn't provide any input, the Supplier knows how to produce the value internally. Function<Void, T> is technically possible but awkward — you'd have to call f.apply(null). Supplier is the idiomatic choice for "produce a T without any input". In practice: orElseGet takes Supplier (you're providing a fallback producer), computeIfAbsent takes Function<K,V> (the missing key is passed as input).
Composition
How do Predicate's and/or/negate work internally?
These are default methods on the Predicate interface that return new Predicate instances wrapping the original. Predicate.and(other) returns a new Predicate whose test() calls this.test(t) && other.test(t). It uses short-circuit evaluation — if this.test() returns false, other.test() is never called. or() uses ||, negate() returns !this.test(t). This is the classic decorator pattern applied to functional interfaces — each composition wraps the previous in a new layer.

Java records
Java 16+ Immutable data carriers — what the compiler generates
Definition

A record is a concise class declaration for immutable data carriers. One line replaces 20+ lines of boilerplate: the compiler generates the constructor, accessors, equals, hashCode, and toString automatically. Records are transparent — all their data is exposed via their component accessors.

Without records (pre-Java 16)
Private final fields, all-args constructor, getters, equals/hashCode overrides, toString — 25+ lines for 3-field DTO.
With records
record Point(int x, int y) {} — 1 line, generates everything. The component list is the canonical constructor's parameter list.
What the compiler generates from record Point(int x, int y)
1
Final fields — private final int x; private final int y; Auto-generated, cannot be changed. Records are implicitly immutable for their components.
2
Canonical constructor — public Point(int x, int y) { this.x = x; this.y = y; } You can override it (compact constructor syntax) to add validation without repeating the assignments.
3
Accessors (not getters) — public int x() { return x; } and public int y() { return y; }. Note: no getX() — records use the component name directly. This breaks JavaBean conventions — be aware if frameworks like Jackson need configuration.
4
equals/hashCode — structural equality based on all components. Two records with same component values are equal. This is the correct semantic for value objects.
5
toString — Point[x=1, y=2]. Readable, component-based output automatically.
!
Records extend java.lang.Record implicitly — like enums extend Enum. They cannot extend any other class (but can implement interfaces). They are implicitly final — cannot be subclassed.
Code — records in practice
// Basic record record Point(int x, int y) {} Point p = new Point(3, 4); p.x(); // accessor — NOT getX() p.y(); p.toString(); // Point[x=3, y=4] // Compact canonical constructor — validation without repeating assignments record Money(long amount, String currency) { Money { // compact constructor — params implicitly assigned at the end if (amount < 0) throw new IllegalArgumentException("Negative amount"); currency = currency.toUpperCase(); // normalize before assignment } } // Custom methods on records record Range(int start, int end) { public boolean contains(int value) { return value >= start && value <= end; } public int size() { return end - start; } } // Records as DTOs with Jackson (needs @JsonProperty or jackson-module-parameter-names) record UserDto(String id, String name, String email) {} // Pattern matching with records (Java 21+) if (shape instanceof Circle(var radius)) { System.out.println("Area: " + Math.PI * radius * radius); }

Interview questions
Classic
What does the Java compiler generate for a record?
For record Point(int x, int y): (1) private final fields x and y, (2) canonical constructor assigning both, (3) accessors x() and y() (not JavaBean getX/getY), (4) equals() using all components for structural equality, (5) hashCode() consistent with equals, (6) toString() as "Point[x=1, y=2]". The class implicitly extends java.lang.Record, is implicitly final, and cannot extend other classes. You can add custom constructors, methods, and static members — you just can't make it mutable or subclassable.
Spring
Can you use a record as a Spring Bean or JPA Entity?
As a @Bean: yes — records can implement interfaces and be registered as Spring beans, especially for configuration properties (@ConfigurationProperties with Java 16+). As a JPA @Entity: no — JPA spec requires a no-arg constructor and mutable state. Use records for DTOs and value objects, not for JPA entities. Workaround: use a record as a JPA @Embeddable for embedded value types with some configuration, or use interface-based projections from Spring Data which can project onto records.

Sealed classes
Java 17+ Controlled inheritance · exhaustive pattern matching · ADTs in Java
Definition

A sealed class restricts which classes can extend or implement it, using a permits clause. The permitted subclasses must be in the same compilation unit (or explicitly listed). This closes the class hierarchy — the compiler knows every possible subtype, enabling exhaustive pattern matching in switch expressions without a default case.

Before sealed
An interface or abstract class could be extended by anyone. Pattern matching switch needed a default branch — the compiler couldn't know all cases.
With sealed
Compiler knows every permitted subtype. Switch on sealed type can omit default — the compiler verifies exhaustiveness at compile time. Missing a case = compile error.
Internals — permitted subclasses and their constraints
1
permits clause — sealed interface Shape permits Circle, Rectangle, Triangle. Only these three can implement Shape. Any other class attempting to extend Shape gets a compile error. Permitted subclasses must be in the same package or module.
2
Subclass must be final, sealed, or non-sealed — every permitted subclass must declare one of: (a) final — no further extension, (b) sealed — extends the hierarchy, (c) non-sealed — opens the hierarchy back up (escape hatch). You can't leave a permitted subclass without one of these — compile error.
3
Algebraic data types (ADT) — sealed + records = Haskell/Scala-style ADTs in Java. A sealed interface is a "sum type" (OR — can be any of these variants). A record is a "product type" (AND — has all of these fields). Together they express domain types like Result = Success(value) | Failure(error) with compiler-enforced exhaustiveness.
4
Compiler-enforced exhaustiveness — when you switch on a sealed type and cover all permitted subtypes, the compiler verifies no case is missing. Add a new permitted subtype → every switch that was exhaustive now fails to compile until you add the new case. No more forgotten branches.
Code — sealed interface + pattern switch + ADT result type
// Sealed interface with permitted subtypes sealed interface Shape permits Circle, Rectangle, Triangle {} record Circle(double radius) implements Shape {} record Rectangle(double w, double h) implements Shape {} final class Triangle implements Shape { ... } // Exhaustive switch — no default needed (compiler verifies) double area = switch (shape) { case Circle(var r) -> Math.PI * r * r; case Rectangle(var w, var h) -> w * h; case Triangle t -> t.area(); // No default — compiler knows all 3 cases are covered }; // ADT Result type — Railway-oriented error handling sealed interface Result<T> permits Result.Success, Result.Failure { record Success<T>(T value) implements Result<T> {} record Failure<T>(String err) implements Result<T> {} } Result<User> result = userService.getUser(id); String response = switch (result) { case Result.Success<User>(var user) -> "Hello, " + user.name(); case Result.Failure<?>(var err) -> "Error: " + err; };
Pattern matching features (Java 21+)
G
Guarded patterns — case Circle c when c.radius() > 10 -> "large". Add conditions to case branches using when keyword. Enables fine-grained discrimination within a type.
D
Deconstruction patterns — case Circle(var r). Extract record components directly in the case label. Removes manual accessor calls. Works with nested records too.
N
null handling — switch on sealed types can now have an explicit case null — finally making null-handling explicit in switch rather than a surprise NullPointerException before the switch.
Interview questions
Why
What problem do sealed classes solve?
They restore the ability to do exhaustive analysis on a type hierarchy. Before sealed classes: any code anywhere could add a new subclass of an interface, so the compiler could never know if a switch or if-else chain covered all cases. With sealed: the permitted subclasses are part of the type's definition. The compiler uses this to verify switch exhaustiveness — adding a new permitted subtype breaks all existing switches that were previously exhaustive, forcing you to update them. This eliminates the "forgotten branch" bug pattern and enables the ADT (algebraic data type) pattern from functional languages in Java.
Design
How would you model a payment status with sealed classes?
sealed interface PaymentStatus permits PaymentStatus.Pending, PaymentStatus.Completed, PaymentStatus.Failed, PaymentStatus.Refunded. Each is a record with relevant data: record Completed(String transactionId, Instant at) implements PaymentStatus {}. record Failed(String reason, int retryCount) implements PaymentStatus {}. Then anywhere you process a PaymentStatus, the switch is exhaustive — add Disputed later and every unhandled switch becomes a compile error. This is the correct Java 17+ alternative to string-based status enums for complex domain states.

var keyword
Java 10+ Local type inference — what the compiler does, where var is banned
Definition

var is a reserved type name (not a keyword) that asks the compiler to infer the type of a local variable from its initialiser. The inferred type is fixed at compile time — var is not dynamic typing (no JavaScript-style type changes). At runtime, there is no var — the class file contains the concrete type.

What var IS
Compile-time type inference for local variables. The type is determined once at declaration and is as strong and static as an explicit type declaration.
What var is NOT
Dynamic typing. A keyword (it's a reserved type name — you can still name a variable "var"). Available for fields, method params, or return types.
Where var is allowed vs forbidden
OK
Local variables with initialiser — var list = new ArrayList<String>(); Compiler infers ArrayList<String>. The type is fixed from that point.
OK
For-each loop variable — for (var user : users). Type inferred from the collection's element type.
OK
Try-with-resources variable — try (var conn = dataSource.getConnection()). Inferred from the expression type.
NO
Method parameters — void process(var x) is illegal. The compiler needs explicit types for method signatures (used by callers and overriding). Exception: lambda parameters in Java 11+ can use var for annotation support: (@NonNull var s) -> s.toUpperCase().
NO
Fields — instance and static fields cannot use var. Fields are part of the class API and need explicit types for reflection, serialisation, and documentation.
NO
Return types — method return types must be explicit. Callers need to know the return type without seeing the implementation.
NO
Without initialiser — var x; is illegal. The compiler has nothing to infer from. Also: var x = null; is illegal — null has no type.
!
Inferred type can be surprising — var list = new ArrayList<>(); infers ArrayList<Object> (not ArrayList<String>) because the diamond operator has nothing to infer from. Always provide a type argument with diamond when using var: var list = new ArrayList<String>();
Code — good uses and what to avoid

Anti-patterns
!
Using var when the type isn't obvious — var x = compute();. The reader has to look up what compute() returns. Use var only when the type is clear from the right-hand side or completely irrelevant.
!
var with raw generics — var list = new ArrayList<>(); infers ArrayList<Object>. You lose all generic type safety. Always specify the type parameter when using var with generics: var list = new ArrayList<String>();
!
var instead of interface type — var list = new ArrayList<String>(); infers ArrayList, not List. You're now coupled to the concrete implementation. For local-only variables this is minor, but be aware you're losing the interface contract.
Interview questions
Concept
Is var dynamic typing? What does the compiler do with it?
No — var is static typing with local type inference. The compiler infers the concrete type from the initialiser at compile time. The inferred type is permanent — you can't assign a value of a different type later (same rules as explicit type). The generated class file contains the fully explicit type — var doesn't exist at runtime. It's syntactic sugar for reducing verbosity, not a dynamic type system like JavaScript's var or Python's duck typing.
Limitations
Where can you NOT use var?
Cannot use var for: (1) method parameters — callers and overriders need explicit types, (2) method return types — callers need to know without reading the body, (3) fields — part of class API, needed for reflection/serialisation, (4) local variables without an initialiser — nothing to infer from, (5) null literal — null has no type. Can use for: local variables with initialisers, for-each loop variables, try-with-resources variables, and (Java 11+) lambda parameters when annotations are needed.
