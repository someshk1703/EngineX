Bubble · selection · insertion · merge · quick · heap — complexity + when to use each

Each algorithm — what makes it tick
B
Bubble sort — repeatedly compare adjacent elements, swap if out of order. The largest element "bubbles" to the end each pass. Optimised: early exit if no swap happened (O(n) best case for sorted). Never use. Exists to teach the concept of passes and comparisons.
S
Selection sort — find the minimum in the unsorted portion, swap it to the correct position. Always exactly n-1 swaps — useful when write operations are expensive (e.g. flash storage). Never stable by default.
I
Insertion sort — build the sorted array one element at a time by inserting each element into its correct position. Excellent for nearly-sorted data and small arrays. TimSort (Java's Arrays.sort for objects) uses insertion sort for runs under 32 elements because cache behaviour and small-n performance beat merge sort's overhead.
M
Merge sort — divide in half recursively until base case (size 1), then merge pairs in sorted order. Guaranteed O(n log n). Stable. Requires O(n) auxiliary space. Ideal for linked lists (no random access needed for merging). Used by Java's Arrays.sort for object arrays (TimSort).
Q
Quick sort — choose a pivot, partition into "less" and "greater", recurse. Average O(n log n) with great cache locality (in-place). Worst case O(n²) with bad pivot (sorted array + always pick first element). Fix: random pivot or median-of-three. Java's Arrays.sort for primitives uses dual-pivot Quicksort (Yaroslavskiy, 2009).
H
Heap sort — build a max-heap (O(n)), then repeatedly extract the maximum to build the sorted array in-place. Guaranteed O(n log n) worst case with O(1) space. Not cache-friendly (heap accesses jump around memory). Not stable. Use when you need guaranteed performance and O(1) space.
Code — merge sort + quick sort (interview-ready)
// MERGE SORT — stable, O(n log n) guaranteed void mergeSort(int[] arr, int lo, int hi) { if (lo >= hi) return; // base case: 0 or 1 element int mid = lo + (hi - lo) / 2; // avoid integer overflow mergeSort(arr, lo, mid); mergeSort(arr, mid + 1, hi); merge(arr, lo, mid, hi); } void merge(int[] arr, int lo, int mid, int hi) { int[] tmp = Arrays.copyOfRange(arr, lo, hi + 1); int i = 0, j = mid - lo + 1, k = lo; while (i <= mid - lo && j <= hi - lo) arr[k++] = (tmp[i] <= tmp[j]) ? tmp[i++] : tmp[j++]; // <= for stability while (i <= mid - lo) arr[k++] = tmp[i++]; while (j <= hi - lo) arr[k++] = tmp[j++]; } // QUICK SORT — average O(n log n), in-place void quickSort(int[] arr, int lo, int hi) { if (lo >= hi) return; int p = partition(arr, lo, hi); quickSort(arr, lo, p - 1); quickSort(arr, p + 1, hi); } int partition(int[] arr, int lo, int hi) { // Random pivot — prevents O(n²) on sorted input int pivotIdx = lo + (int)(Math.random() * (hi - lo + 1)); swap(arr, pivotIdx, hi); // move pivot to end int pivot = arr[hi], i = lo; for (int j = lo; j < hi; j++) if (arr[j] <= pivot) swap(arr, i++, j); swap(arr, i, hi); // place pivot in final position return i; } // Java's built-in sorts (know these for interviews) Arrays.sort(intArr); // dual-pivot Quicksort — O(n log n) avg Arrays.sort(objArr); // TimSort — stable O(n log n) Arrays.sort(arr, Comparator.reverseOrder()); // with comparator List.sort(Comparator.comparing(Person::getName));
Java's sort internals — what actually runs when you call Arrays.sort
P
Primitives — Dual-Pivot Quicksort: uses two pivots (p1 < p2) to partition into three regions: less than p1, between p1 and p2, greater than p2. Empirically faster than single-pivot on real data. Falls back to insertion sort for tiny sub-arrays (<47 elements). NOT stable — safe because primitive equality is value equality (no observable ordering).
O
Objects — TimSort: hybrid of merge sort + insertion sort. Finds "natural runs" (already sorted sequences) in the data. Merges pairs of runs. Best case O(n) for already-sorted data. Stable — required because object equality can be partial (two different objects with the same key must maintain their input order). Minimum run size is 32 elements.
!
Why different? Stability matters for objects (sort by name, then by age — the name sort must preserve age ordering among same-name entries). For primitives there's no observable distinction between "equal" values — stability is irrelevant, so the faster Quicksort wins.
Interview questions
Classic
What is the worst case of Quick Sort and how do you avoid it?
Worst case O(n²) occurs when the pivot is always the minimum or maximum element — every partition produces one sub-array of size n-1 and one empty. Classic trigger: sorted or reverse-sorted input with always-first-element pivot. Prevention: (1) random pivot — shuffle the input or pick a random element, reduces O(n²) probability to negligibly small. (2) Median-of-three — pick the median of first, middle, last elements as pivot. (3) 3-way partition (Dutch National Flag) — handles duplicate elements efficiently by creating three regions: less, equal, greater. Java's dual-pivot Quicksort uses two pivots and is empirically very fast in practice.
Choose
When do you use merge sort over quick sort?
Merge sort when: (1) stability required — equal elements must maintain relative order (sorting objects by one key when already sorted by another). (2) Linked lists — merging linked lists is O(1) extra space, no random access needed. (3) External sort — data too large for RAM; merge sort naturally divides work into chunks that fit in memory. Quick sort when: (1) in-place sorting needed (O(log n) stack vs O(n) for merge). (2) Cache performance matters — quick sort accesses memory sequentially in partitioning. (3) Average case matters more than worst case. Java uses merge sort (TimSort) for objects, dual-pivot quicksort for primitives — for exactly these reasons.
Depth
What is TimSort and why does Java use it for Arrays.sort on objects?
TimSort is a hybrid sort invented by Tim Peters for Python (2002), adopted by Java 7+ for object arrays. It exploits the fact that real-world data contains natural sorted runs. Algorithm: (1) scan left-to-right detecting natural ascending or descending runs. (2) If a run is too short (<32 elements), extend it using insertion sort. (3) Push runs onto a stack and merge adjacent runs when size ratios trigger a merge (maintains invariants to ensure O(n log n)). Result: O(n) on sorted/reverse-sorted data, O(n log n) in general. Stable. Java requires stability for objects because the contract of Comparable and Comparator says equal elements must maintain relative order.

Searching algorithms
Linear · binary · interpolation — and every binary search variant
Three search algorithms
L
Linear search — scan each element one by one. O(n) time, O(1) space. Use when: unsorted data, very small n, or searching a linked list (no random access). No prerequisites. Always correct.
B
Binary search — requires sorted array. Eliminate half the search space each step by comparing mid element to target. O(log n) time, O(1) space (iterative). The most important search algorithm. Masters: finding a value, finding first/last occurrence, finding insertion point, searching on answer space ("find minimum k such that condition(k) is true").
I
Interpolation search — like binary search but uses a formula to estimate the probe position based on value distribution: pos = lo + ((target - arr[lo]) * (hi - lo)) / (arr[hi] - arr[lo]). O(log log n) average for uniformly distributed data, O(n) worst case (exponential skew). Use for: large sorted arrays with uniform distribution (e.g. phone numbers, IDs). Rarely seen in interviews outside Google.
Binary search — the right template (no bugs)
!
The one bug that kills candidates: mid = (lo + hi) / 2 overflows when lo + hi exceeds Integer.MAX_VALUE. Always use: mid = lo + (hi - lo) / 2. This is a real production bug (found in Java's original Arrays.binarySearch in 2006 by Joshua Bloch).
1
Invariant-based thinking — the key to bug-free binary search is maintaining a clear loop invariant. Decide what [lo, hi] means: does it include both endpoints? Is hi exclusive? Then every update must maintain that invariant. The most robust template: lo = 0, hi = n-1, both inclusive, loop while lo <= hi.
2
When mid lands on answer — if you want the exact value: return mid. If you want first occurrence: save mid as candidate, search left (hi = mid - 1). If you want last occurrence: save mid, search right (lo = mid + 1). If searching on answer space: when condition(mid) is satisfied, save and narrow the range.
3
Binary search on answer space — the pattern: "find the minimum/maximum value that satisfies a condition." Instead of searching for a value in an array, search for the answer itself. Example: "minimum capacity to ship packages in D days" — binary search on capacity from max(weight) to sum(weights). For each candidate capacity, check if packages can be shipped in ≤ D days (O(n)). Total: O(n log(range)).
Code — the bulletproof binary search template + variants
// STANDARD — find exact value, return -1 if not found int binarySearch(int[] arr, int target) { int lo = 0, hi = arr.length - 1; while (lo <= hi) { int mid = lo + (hi - lo) / 2; // no overflow if (arr[mid] == target) return mid; else if (arr[mid] < target) lo = mid + 1; else hi = mid - 1; } return -1; } // FIRST OCCURRENCE — leftmost index where arr[i] == target int firstOccurrence(int[] arr, int target) { int lo = 0, hi = arr.length - 1, result = -1; while (lo <= hi) { int mid = lo + (hi - lo) / 2; if (arr[mid] == target) { result = mid; hi = mid - 1; } // save, go left else if (arr[mid] < target) lo = mid + 1; else hi = mid - 1; } return result; } // LOWER BOUND — first index where arr[i] >= target (insertion point) int lowerBound(int[] arr, int target) { int lo = 0, hi = arr.length; // hi = length (exclusive right) while (lo < hi) { int mid = lo + (hi - lo) / 2; if (arr[mid] < target) lo = mid + 1; else hi = mid; // include mid as candidate } return lo; // lo == hi == insertion point } // BINARY SEARCH ON ANSWER — min capacity to ship in D days int shipInDays(int[] weights, int D) { int lo = Arrays.stream(weights).max().getAsInt(); // must carry heaviest int hi = Arrays.stream(weights).sum(); // ship all in 1 day while (lo < hi) { int mid = lo + (hi - lo) / 2; if (canShip(weights, D, mid)) hi = mid; // feasible — try smaller else lo = mid + 1; } return lo; } boolean canShip(int[] w, int D, int cap) { int days = 1, cur = 0; for (int wt : w) { if (cur + wt > cap) { days++; cur = 0; } cur += wt; } return days <= D; }
Binary search variants — which template for which problem
1
Find exact value — standard template. Return mid when found, -1 if not. lo <= hi loop.
2
First/last occurrence of target — save result when found, then continue narrowing. hi = mid - 1 for first; lo = mid + 1 for last.
3
Lower/upper bound (insertion point) — lo = 0, hi = n (exclusive). Loop while lo < hi. Lower bound: arr[mid] < target → lo = mid + 1 else hi = mid. Returns first index >= target.
4
Answer space binary search — search on the range of possible answers, not on the array indices. Template: lo = min_possible, hi = max_possible. condition(mid) = is this answer feasible? If yes: hi = mid (try smaller). If no: lo = mid + 1. Return lo when loop ends.
5
Peak finding / rotated array — binary search by comparing mid with neighbours to decide which half to search. No explicit target — the property being searched is structural.
Interview questions
Bug
What is the integer overflow bug in binary search and how did it affect Java's standard library?
The bug: mid = (lo + hi) / 2. When lo and hi are both large (near Integer.MAX_VALUE = 2^31 - 1), their sum overflows to a negative number. mid becomes negative, causing ArrayIndexOutOfBoundsException. The fix: mid = lo + (hi - lo) / 2. Joshua Bloch documented this in 2006 — the bug existed in Java's Arrays.binarySearch for nearly a decade since JDK 1.0. It was considered a theoretical problem until binary search on arrays larger than 2^30 elements became realistic with modern hardware. This is why interviewers ask about it — it shows you read Bloch's Effective Java and think about edge cases.
Pattern
Explain binary search on the answer space with an example.
Instead of searching for a value in an array, you search for the answer itself. The pattern: identify what you're minimising or maximising, determine the feasible range [lo, hi], define a monotonic condition (if capacity=k is feasible, then capacity=k+1 is also feasible), binary search on [lo, hi]. Example: "minimum days to finish books reading at most pages/day" — binary search on pages_per_day from max(pages[i]) to sum(all pages). For each candidate, greedily simulate whether you can finish in D days. This converts an O(n * range) brute force into O(n log range). Common problems: capacity to ship packages, minimum days to make m bouquets, find kth smallest in matrix.

Recursion vs iteration
Call stack mechanics · tail recursion · when to convert · memoisation
Definition

Recursion: a function calls itself to solve a smaller sub-problem until reaching a base case. Iteration: a loop repeats a block of code. Both can solve the same problems but with different memory behaviour, readability trade-offs, and stack risk.

Recursion — use when
Problem naturally decomposes into identical sub-problems (trees, graphs, D&C). Code is cleaner and matches the mental model. Stack depth is bounded (not exceeding ~10k calls in Java by default).
Iteration — use when
Stack overflow risk (deep recursion). Performance-critical (function call overhead matters). Tail-recursive problems where the compiler won't optimise (Java doesn't do TCO). Simulating recursion with an explicit stack is the workaround.
Call stack internals — what happens when you recurse
1
Each call pushes a frame — every recursive call pushes a new stack frame containing: parameters, local variables, return address. For fibonacci(50), that's 50 frames deep. Java's default stack size is ~512KB–1MB per thread. Each frame costs ~50–500 bytes depending on local variables.
2
StackOverflowError — when the call stack exceeds the thread's stack size limit. Fibonacci(10000) or any deeply recursive algorithm on large input will blow the stack. Detection: deep recursion without a proper base case, or n > ~5000–10000 for most recursive functions. Fix: convert to iterative with an explicit stack, or use memoisation to limit recursion depth.
3
The two parts of a recursive call — (1) the call: push frame, jump to function start. (2) the return: pop frame, resume at return address with the return value. Understanding this makes it clear why recursion uses O(depth) space on the stack, and why iterating from bottom-up often uses O(1) space.
4
Recursion tree analysis — to find the time complexity of recursion, draw the tree. Each node is one function call. Count total nodes (time) and maximum depth (space). Fibonacci recursion tree has 2^n nodes = O(2^n) without memoisation, O(n) with it. Merge sort's tree has n log n nodes total = O(n log n).
Code — convert recursion to iteration (explicit stack)
// RECURSIVE fibonacci — clean but O(2^n) without memo int fib(int n) { if (n <= 1) return n; return fib(n - 1) + fib(n - 2); // two recursive branches } // MEMOISED recursion — O(n) time, O(n) space int[] memo = new int[n + 1]; Arrays.fill(memo, -1); int fibMemo(int n) { if (n <= 1) return n; if (memo[n] != -1) return memo[n]; return memo[n] = fibMemo(n-1) + fibMemo(n-2); } // ITERATIVE (bottom-up DP) — O(n) time, O(1) space, no stack risk int fibIter(int n) { if (n <= 1) return n; int prev2 = 0, prev1 = 1; for (int i = 2; i <= n; i++) { int cur = prev1 + prev2; prev2 = prev1; prev1 = cur; } return prev1; } // Converting tree traversal: recursive → iterative with explicit stack // RECURSIVE DFS void dfs(TreeNode node) { if (node == null) return; process(node.val); dfs(node.left); dfs(node.right); } // ITERATIVE DFS — explicit stack replaces call stack void dfsIterative(TreeNode root) { Deque<TreeNode> stack = new ArrayDeque<>(); if (root != null) stack.push(root); while (!stack.isEmpty()) { TreeNode node = stack.pop(); process(node.val); if (node.right != null) stack.push(node.right); // right first (LIFO) if (node.left != null) stack.push(node.left); } }
Tail recursion — and why Java doesn't optimise it
T
Tail recursion — a recursive call is "in tail position" if it's the very last operation before the function returns, with no computation after it. return f(n-1, acc + n) is tail-recursive. return n + f(n-1) is NOT — there's addition after the recursive call.
TCO
Tail call optimisation (TCO) — compilers that support TCO reuse the current stack frame for tail calls instead of pushing a new one, converting recursion to an O(1) space loop. Scala (on JVM), Kotlin (via trampolining), Haskell — all support TCO.
!
Java does NOT do TCO — by design. Each recursive call in Java always pushes a new stack frame. The JVM specification doesn't guarantee stack frame reuse. This is why tail-recursive Java code can still throw StackOverflowError. For tail-recursive Java code: convert it to an iterative loop manually. The compiler won't do it for you.
Interview questions
Classic
When would you convert a recursive solution to iterative?
Convert to iterative when: (1) stack overflow risk — recursion depth exceeds ~5000-10000 calls. (2) Performance — function call overhead matters in tight loops. (3) The recursive solution uses O(n) stack space unnecessarily — e.g. a simple linear scan written recursively. Conversion strategy: replace the call stack with an explicit ArrayDeque (stack). Push the "next state" instead of making a recursive call. Pop and process in the loop. This is exactly how DFS tree traversal converts to iterative. Keep recursion when it's cleaner and stack depth is bounded (tree depth = O(log n) for balanced trees).
Java
Why doesn't Java support tail call optimisation?
Primarily a design and JVM spec decision. (1) JVM security model depends on stack traces being complete — TCO would remove frames from stack traces, breaking stack-based access control checks (SecurityManager). (2) Reflection and stack inspection APIs (Thread.getStackTrace()) would behave differently. (3) JVM spec doesn't require TCO, so JVM implementations don't implement it for portability. (4) The JVM team has discussed it but it remains unimplemented. Workaround: trampolining (return a thunk instead of calling recursively, driver loop invokes thunks until done). In practice: just convert to iteration in Java.

Divide & conquer
Merge sort · quick sort · binary search · Master theorem
Definition

Divide and conquer breaks a problem into two or more sub-problems of the same type, solves them recursively, then combines the results. Three phases: divide (split the problem), conquer (recursively solve), combine (merge results). The combine step is what distinguishes D&C from simple recursion.

D&C pattern recognition
Problem decomposes into identical smaller versions. Sub-problems are independent (no overlap — overlapping means DP instead). A combine step produces the final answer from sub-answers.
Classic D&C problems
Merge sort (divide array, sort halves, merge). Quick sort (partition, sort sub-arrays). Binary search (discard half). Closest pair of points. Strassen matrix multiplication. Karatsuba multiplication.
Master theorem — solving T(n) = aT(n/b) + f(n)
!
The recurrence T(n) = aT(n/b) + f(n) means: problem of size n splits into a sub-problems of size n/b, and the combine step costs f(n). The critical exponent is c_crit = log_b(a).
1
Case 1: f(n) = O(n^c) where c < c_crit — sub-problems dominate. T(n) = Θ(n^c_crit). Example: Strassen matrix mult T(n) = 7T(n/2) + O(n²), c_crit = log₂7 ≈ 2.81 > 2, so T(n) = O(n^2.81).
2
Case 2: f(n) = Θ(n^c_crit) — work at each level is equal. T(n) = Θ(n^c_crit log n). Example: Merge sort T(n) = 2T(n/2) + O(n), c_crit = log₂2 = 1 = exponent of f(n). T(n) = O(n log n).
3
Case 3: f(n) = Ω(n^c) where c > c_crit — combine step dominates. T(n) = Θ(f(n)). Example: Binary search T(n) = 1·T(n/2) + O(1), c_crit = log₂1 = 0 < exponent 0 of f(n)... actually case 2: T(n) = O(log n).
Code — count inversions (classic D&C interview problem)
// Count inversions — pairs (i,j) where i < j but arr[i] > arr[j] // D&C approach: inversions = left_inversions + right_inversions + split_inversions // Split inversions are counted DURING the merge step of merge sort long countInversions(int[] arr, int lo, int hi) { if (lo >= hi) return 0; // base case int mid = lo + (hi - lo) / 2; long left = countInversions(arr, lo, mid); // DIVIDE & CONQUER left long right = countInversions(arr, mid+1, hi); // DIVIDE & CONQUER right return left + right + mergeCount(arr, lo, mid, hi); // COMBINE } long mergeCount(int[] arr, int lo, int mid, int hi) { int[] L = Arrays.copyOfRange(arr, lo, mid+1); int[] R = Arrays.copyOfRange(arr, mid+1, hi+1); int i = 0, j = 0, k = lo; long inv = 0; while (i < L.length && j < R.length) { if (L[i] <= R[j]) arr[k++] = L[i++]; else { arr[k++] = R[j++]; inv += L.length - i; // all remaining L elements form inversions with R[j] } } while (i < L.length) arr[k++] = L[i++]; while (j < R.length) arr[k++] = R[j++]; return inv; } // Time: O(n log n) — master theorem case 2: T(n)=2T(n/2)+O(n)
Interview questions
Classic
What distinguishes Divide & Conquer from Dynamic Programming?
Both break problems into sub-problems. Key distinction: in D&C, sub-problems are INDEPENDENT — solving the left half of a merge sort doesn't depend on the right half. In DP, sub-problems OVERLAP — computing fibonacci(5) requires fibonacci(4) which requires fibonacci(3)... and fibonacci(3) is computed multiple times without memoisation. D&C just recurse. DP memoises or tabulates to avoid recomputing. Signal: draw the recursion tree. If the same sub-problem appears multiple times = DP. If each sub-problem is unique = D&C. Merge sort recursion tree: each node is a unique sub-array. Fibonacci recursion tree: fib(3) appears in multiple branches = DP.
Apply
How would you find the closest pair of points using D&C?
Brute force: O(n²) — check all pairs. D&C: (1) Sort points by x-coordinate. (2) Divide: split into left and right halves. (3) Conquer: find closest pair in each half recursively — gives d = min(d_left, d_right). (4) Combine: check the strip of width 2d around the dividing line. Sort strip points by y and check each against at most 7 subsequent points (geometric argument — at most 8 points fit in a 2d×d rectangle). Time: T(n) = 2T(n/2) + O(n log n) for strip sort = O(n log² n). With a pre-sorted y-list: O(n log n). This shows the combine step is the tricky part in D&C and often determines the overall complexity.

Greedy algorithms
Activity selection · interval scheduling · coin change — when greedy works vs fails
Definition

A greedy algorithm makes the locally optimal choice at each step, hoping to arrive at the globally optimal solution. No backtracking. No consideration of future consequences. Fast (usually O(n log n) due to sorting), but only correct when the problem has the "greedy choice property."

Greedy works when
The problem has the greedy choice property: a locally optimal choice is always part of a globally optimal solution. Proof: exchange argument — show any optimal solution can be transformed to include the greedy choice without getting worse.
Greedy fails when
Local optimum doesn't lead to global optimum. Counter-example kills it. Use DP instead. Classic failure: coin change with non-standard denominations (coins = [1,3,4], target = 6: greedy picks 4+1+1 = 3 coins, DP finds 3+3 = 2 coins).
The two conditions for greedy correctness
1
Greedy choice property — there exists an optimal solution that includes the greedy choice. Proof technique: exchange argument. Assume there's an optimal solution that doesn't include our greedy pick. Show you can swap in the greedy choice without making things worse. Therefore, greedy choice is safe.
2
Optimal substructure — once you make the greedy choice, the remaining sub-problem has the same structure and can be solved optimally independently. This is the same condition DP requires — greedy just exploits it more aggressively by never revisiting choices.
!
The interview test: when asked if greedy works, always try to find a counter-example first. If you can't find one after trying a few cases, attempt an exchange argument proof. If you can find a counter-example: that's your signal to use DP.
Code — activity selection + interval scheduling + coin change comparison
// ACTIVITY SELECTION — max non-overlapping activities // Greedy: always pick the activity that ends earliest (leaves most room) int maxActivities(int[][] activities) { // activities[i] = {start, end} Arrays.sort(activities, (a, b) -> a[1] - b[1]); // sort by END time int count = 1, lastEnd = activities[0][1]; for (int i = 1; i < activities.length; i++) { if (activities[i][0] >= lastEnd) { // starts after last ends count++; lastEnd = activities[i][1]; } } return count; // O(n log n) sort + O(n) scan } // INTERVAL SCHEDULING — min meeting rooms (min platforms) // Greedy: track ongoing meetings with a min-heap of end times int minRooms(int[][] intervals) { Arrays.sort(intervals, (a, b) -> a[0] - b[0]); PriorityQueue<Integer> heap = new PriorityQueue<>(); // min-heap of end times for (int[] inv : intervals) { if (!heap.isEmpty() && heap.peek() <= inv[0]) heap.poll(); // room freed — reuse it heap.offer(inv[1]); } return heap.size(); } // COIN CHANGE — greedy FAILS for non-canonical denominations // Greedy works for [1,5,10,25] (US coins) but NOT for [1,3,4] int coinChangeGreedy(int[] coins, int amount) { // WRONG in general Arrays.sort(coins); int count = 0; for (int i = coins.length - 1; i >= 0; i--) { count += amount / coins[i]; amount %= coins[i]; } return (amount == 0) ? count : -1; } // Use DP instead for general coin change: // dp[i] = min coins to make i; dp[0]=0, dp[i]=min(dp[i-c]+1) for each coin c
When greedy fails — the counter-examples
1
Coin change (non-canonical denominations) — Coins [1, 3, 4], target = 6. Greedy picks 4 + 1 + 1 = 3 coins. Optimal: 3 + 3 = 2 coins. Greedy is wrong. Rule: greedy works for US coins because each coin "dominates" smaller combinations. For arbitrary denominations, use DP.
2
Fractional knapsack vs 0/1 knapsack — fractional: can take fraction of item → greedy by value/weight ratio works perfectly. 0/1 knapsack: must take whole item → greedy fails. Example: capacity 10, items {6:6, 5:5, 5:5}. Greedy picks {6:6} + part... but 0/1 optimal is {5:5} + {5:5} = 10 value. Use DP for 0/1.
3
Shortest path (negative edges) — Dijkstra is greedy and correct for non-negative edges. With negative edges, the greedy choice (smallest current distance) can be wrong. Use Bellman-Ford instead.
Interview questions
Classic
Why does the activity selection greedy algorithm sort by end time rather than start time or duration?
The goal is to maximise the number of non-overlapping activities. Sorting by end time and always picking the earliest-finishing activity maximises the remaining time window for future activities. Proof by exchange argument: suppose the optimal solution picks activity A_opt as its first activity instead of A_greedy (the earliest-finishing). Since A_greedy ends no later than A_opt, we can replace A_opt with A_greedy and the solution remains valid — activities after A_opt still don't overlap since A_greedy ends earlier. So any optimal solution can be transformed to include the greedy choice. Sorting by start time fails because a long early-starting activity blocks many shorter ones.
Vs DP
How do you decide between greedy and dynamic programming?
Greedy when: the problem has the greedy choice property (a local optimum is always part of a global optimum) provable by an exchange argument. Usually recognisable by: maximise/minimise non-overlapping intervals, fractional resource allocation, Huffman coding. DP when: greedy fails (coin change with arbitrary coins), or the sub-problems overlap (fibonacci, LCS, LIS). Practical approach: (1) Try greedy, find a counter-example with 3-4 items. (2) If you find one: DP. (3) If you can't: try exchange argument. (4) If the exchange argument holds: greedy is safe. The greedy-or-DP decision is one of the most important pattern recognitions in DSA.

Backtracking
N-Queens · sudoku · permutations · the prune-early imperative
Definition

Backtracking is a systematic method for generating all configurations of a search space by incrementally building candidates and abandoning ("pruning") a candidate as soon as it's determined it cannot lead to a valid solution. It's depth-first search with constraint checking. The key insight: fail fast.

Decision tree — what backtracking does
try choice A → try A1 → try A1a → INVALID → backtrack
                     → try A1b → VALID → add to result
           → try A2 → PRUNED (constraint violated) → backtrack
try choice B → ...
The universal backtracking template
// TEMPLATE — memorise this structure void backtrack(State current, List<Result> results) { // BASE CASE — found a valid complete solution if (isSolution(current)) { results.add(current.copy()); // COPY the state — don't store the reference return; } // TRY each possible next choice for (Choice c : getPossibleChoices(current)) { if (!isValid(current, c)) continue; // PRUNE — fail fast makeChoice(current, c); // modify state backtrack(current, results); // recurse undoChoice(current, c); // UNDO (backtrack) — restore state } } // The make/undo pair is the heart of backtracking. // State must be fully restored so the next branch starts clean.
Code — N-Queens + permutations
// N-QUEENS — place n queens so no two attack each other int n; List<List<String>> solveNQueens(int n) { this.n = n; List<List<String>> res = new ArrayList<>(); int[] queens = new int[n]; // queens[row] = col of queen in that row Arrays.fill(queens, -1); placeQueen(0, queens, res); return res; } void placeQueen(int row, int[] queens, List<List<String>> res) { if (row == n) { res.add(buildBoard(queens)); return; } // base case for (int col = 0; col < n; col++) { if (isValid(queens, row, col)) { // PRUNE early queens[row] = col; // make choice placeQueen(row + 1, queens, res); // recurse queens[row] = -1; // undo choice } } } boolean isValid(int[] queens, int row, int col) { for (int r = 0; r < row; r++) { int c = queens[r]; if (c == col) return false; // same column if (Math.abs(r - row) == Math.abs(c - col)) return false; // same diagonal } return true; } // PERMUTATIONS — all orderings of an array void permute(int[] nums, int start, List<int[]> res) { if (start == nums.length) { res.add(nums.clone()); return; } for (int i = start; i < nums.length; i++) { swap(nums, start, i); // make choice: put nums[i] at position start permute(nums, start + 1, res); // recurse on remaining swap(nums, start, i); // undo — restore array } } // n! permutations × n to build each = O(n × n!) time
Pruning strategies — how to make backtracking fast
Constraint check before recursing — don't recurse then validate. Validate first. If the current partial state already violates a constraint, prune immediately without going deeper.
Forward-looking pruning — check not just whether the current state is valid, but whether it CAN possibly lead to a solution given the remaining choices. Sudoku: if a cell has no valid digits, prune the entire branch immediately.
Ordering choices by most-constrained first — try the choice most likely to fail early. In N-Queens, try columns that cause fewer conflicts. In Sudoku, fill the cell with fewest valid digits first. Reduces the branching factor at the top of the tree.
Bit masking for O(1) constraint checking — in N-Queens, maintain three bitsets: columns used, left-diagonals used, right-diagonals used. Checking all three is O(1) with bitwise AND instead of O(n) linear scan. Reduces constant factor dramatically.
What pruning is NOT — returning early in the base case is not pruning. Pruning happens before a recursive call — you prevent an entire subtree from being explored, not just skip one leaf.
Interview questions
Classic
Explain the N-Queens backtracking solution and its time complexity.
Place queens one row at a time. For each row, try each column. Before placing, check: (1) no queen in the same column in any previous row, (2) no queen on the same diagonal. If valid: place the queen (queens[row] = col) and recurse to the next row. If we complete all n rows: add the board to results. Undo after each recursive call (queens[row] = -1). Time complexity: O(n!) in the worst case — the first queen has n choices, second has at most n-2 choices (columns on either side of first), etc. With pruning, empirically much better. For N=8: 8! / ~reduction ≈ 92 solutions found efficiently. Space: O(n) for the queens array + O(n) recursive stack depth.
Optimise
How would you use bit manipulation to speed up N-Queens?
Maintain three integers as bitmasks: cols (columns occupied), diag1 (left diagonals, row-col constant), diag2 (right diagonals, row+col constant). When placing queen at (row, col): cols |= (1 << col); diag1 |= (1 << (row-col+n)); diag2 |= (1 << (row+col)). Check validity: (cols & (1<<col)) == 0 && (diag1 & ...) == 0 && (diag2 & ...) == 0. All checks are O(1) bit operations vs O(n) linear scan. Available columns for the next row = ~(cols|diag1|diag2) & ((1<<n)-1). This classic optimisation appears in interviews at Google/Meta levels.

Bit manipulation
AND · OR · XOR · shifts · 15 essential tricks for interviews
Bitwise operators — what each does
a & b — AND: 1 only where both are 1. Used to mask / check bits.
a | b — OR: 1 where either is 1. Used to set bits.
a ^ b — XOR: 1 where exactly one is 1. The magic operator — cancels same values.
~a — NOT: flips all bits. ~0 = -1 (all 1s in two's complement).
a << k — left shift: multiply by 2^k. Fills with 0s from right.
a >> k — arithmetic right shift: divide by 2^k. Preserves sign bit.
a >>> k — logical right shift: fills with 0s (Java-specific). Use for unsigned right shift.

// COUNT SET BITS — Brian Kernighan's algorithm int countBits(int n) { int count = 0; while (n != 0) { n &= (n - 1); // removes lowest set bit each iteration count++; } return count; // O(number of set bits) — faster than O(32) } // Java shortcut: Integer.bitCount(n) // SINGLE NUMBER — find the one non-duplicate (XOR cancels pairs) int singleNumber(int[] nums) { int xor = 0; for (int n : nums) xor ^= n; // a ^ a = 0, a ^ 0 = a return xor; // only the unique element survives } // GENERATE ALL SUBSETS using bit mask List<List<Integer>> subsets(int[] nums) { List<List<Integer>> res = new ArrayList<>(); int n = nums.length; for (int mask = 0; mask < (1 << n); mask++) { // 2^n subsets List<Integer> subset = new ArrayList<>(); for (int i = 0; i < n; i++) if ((mask >> i & 1) == 1) subset.add(nums[i]); // bit i set = include res.add(subset); } return res; // O(n × 2^n) } // REVERSE BITS (32-bit integer) int reverseBits(int n) { int result = 0; for (int i = 0; i < 32; i++) { result = (result << 1) | (n & 1); // take LSB, shift result left n >>>= 1; // unsigned right shift } return result; }
XOR — the magic operator and why it works
P
XOR properties to memorise: a ^ 0 = a (identity), a ^ a = 0 (self-inverse), a ^ b = b ^ a (commutative), (a ^ b) ^ c = a ^ (b ^ c) (associative). The self-inverse property is the key: XOR-ing a value twice cancels it out.
1
Single number (XOR all) — if every number appears twice except one, XOR all elements. Duplicate pairs cancel (a^a=0). The unique number remains. This is O(n) time, O(1) space — the classic bit trick that blows away O(n) HashMap solutions.
2
Two numbers appearing once — XOR all: result = a^b (two different numbers). Find any set bit in result (they differ in that bit). Use that bit to partition all numbers into two groups and XOR each group separately — each group cancels to one number.
3
Find missing number — in array [0..n] with one missing: XOR all indices 0..n with all array values. Duplicates cancel, leaving the missing number. int miss = n; for(int i=0;i<n;i++) miss ^= i ^ nums[i];
Interview questions
Classic
How do you check if a number is a power of 2 using bits?
n > 0 && (n & (n-1)) == 0. Powers of 2 in binary are exactly one 1-bit: 1=0001, 2=0010, 4=0100, 8=1000. n-1 flips all bits from the rightmost 1 bit: 4=0100, 3=0011. AND gives 0000 — zero. For non-powers: n=6=0110, n-1=5=0101, n&(n-1)=0100 ≠ 0. The n > 0 check handles n=0 (which is not a power of 2 but 0 & -1 = 0 would pass without it). Variant: use this in a loop to count set bits — each iteration of n &= (n-1) removes exactly one 1-bit.
XOR
Find the single non-duplicate number in an array where every other number appears twice.
XOR all elements. XOR is commutative and associative, a^a=0, a^0=a. So XOR of all elements: pairs cancel to 0, the unique element XOR-s with 0 and survives. Code: int xor = 0; for(int n : nums) xor ^= n; return xor. O(n) time, O(1) space. Why better than HashMap: no memory allocation, single pass, works even with overflow concerns since XOR doesn't overflow. Follow-up: if two elements appear once and all others twice — XOR all to get a^b, find any set bit (where a and b differ), partition array by that bit and XOR each partition separately to get a and b individually.
Trick
How do you use bit manipulation to generate all subsets of a set?
For a set of n elements, there are 2^n subsets (each element is either included or not — binary choice). Iterate mask from 0 to 2^n - 1. For each mask, bit i set means include element i. Check: (mask >> i) & 1 == 1. This is the direct encoding of the subset selection as a binary number. Example: n=3, elements=[a,b,c]. mask=5=101 means include a and c (skip b). O(n × 2^n) total. More elegant than recursive backtracking for small n (<=20), and gives indices into the original array for free.
