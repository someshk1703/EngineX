JVM Architecture
ClassLoader subsystem · Runtime data areas · Execution Engine · JIT compiler
Definition

The JVM is a runtime engine that executes Java bytecode on any OS. It abstracts away the hardware. Its three major subsystems are: the ClassLoader (loads .class files), the Runtime Data Areas (memory regions), and the Execution Engine (interprets/JIT-compiles bytecode to native machine code).

ClassLoader
Finds, loads, links, and initialises .class files into the runtime. Hierarchical — Bootstrap → Extension → Application.
Runtime Data Areas
Memory regions: Method Area, Heap, JVM Stacks, PC Registers, Native Method Stacks.
Execution Engine
Interpreter for cold code + JIT compiler for hot code (HotSpot). Also drives the GC.
Internals — step by step
1
ClassLoader delegation model — Bootstrap ClassLoader (JVM native, loads rt.jar/java.base). Extension ClassLoader (loads ext/*.jar). Application ClassLoader (loads your classpath). Each delegates to parent first — ensures core JDK classes are never overridden by user code. You can write custom ClassLoaders (OSGi, app servers, hot-reload).
2
Linking phases — Verification: bytecode safety check (no stack overflows, valid types). Preparation: static fields get default zero values. Resolution: symbolic references (class names) resolved to direct memory references.
3
Method Area / Metaspace — stores class metadata (field/method descriptors, bytecode, constant pool, static variables). In Java 7 it lived on the heap (PermGen — fixed size, notorious for OutOfMemoryError: PermGen). Java 8 moved it to native memory as Metaspace — grows dynamically, capped with -XX:MaxMetaspaceSize.
4
PC Register — one per thread, holds the address of the currently executing bytecode instruction. For native methods the PC is undefined.
5
Native Method Stack — supports native (C/C++) method calls via JNI. Separate from JVM stacks.

JIT compiler — how HotSpot makes Java fast
1
Profiling phase — JVM starts in interpreted mode, counting method invocations and loop back-edges. A method is "hot" after ~10,000 invocations (tunable via -XX:CompileThreshold).
2
C1 (client compiler) — fast, simple compilation with basic optimisations (inlining, dead code elimination). Used for methods that become warm quickly. Produces fast-but-not-optimal native code.
3
C2 (server compiler) — deep profiling-guided optimisations. Loop unrolling, scalar replacement, escape analysis (objects that don't escape a method may be stack-allocated, skipping GC). This is why Java can match or beat C++ in long-running server workloads.
4
Inline caches — JIT remembers which concrete type is passed to a polymorphic call site. If always the same type (monomorphic), the virtual dispatch is replaced with a direct call. If 2–3 types (bimorphic/polymorphic), it checks an inline cache. 4+ types = megamorphic = back to vtable = slower.
5
Deoptimisation — if an optimistic assumption breaks (a new subclass is loaded), JVM deoptimises the compiled code back to the interpreter. This is why loading many subclasses at runtime can cause performance hiccups.
Common mistakes
!
Assuming JVM startup performance = steady-state performance — JVM starts slow (cold), speeds up as JIT kicks in. Benchmarks that measure only first-run performance are meaningless for production throughput. Use JMH (Java Microbenchmark Harness).
!
Thinking PermGen = Metaspace — they're not. PermGen was fixed-size on heap (Java ≤7). Metaspace is unbounded native memory (Java 8+). OutOfMemoryError: PermGen space = old JVM. OutOfMemoryError: Metaspace = modern JVM with MaxMetaspaceSize set or class loader leak.
!
Writing micro-benchmarks without JMH — JIT will dead-code-eliminate your benchmark loop if the result is unused. JMH prevents this with Blackhole consumers. Ad-hoc System.currentTimeMillis() benchmarks are always unreliable.
Interview questions
Architecture
Explain the JVM ClassLoader delegation model. Why does it exist?
Parent-delegation: when a class is requested, the ClassLoader asks its parent first. Bootstrap → Platform (Extension) → App. If parent finds it, done. If not, the current CL loads it. Purpose: (1) security — user code can't shadow java.lang.String with a malicious version because Bootstrap always wins, (2) consistency — same class won't be loaded twice by different loaders. Custom CL violates delegation deliberately for isolation (OSGi, hot-deploy, Java EE context isolation).
JIT
How does the JIT compiler make Java competitive with C in throughput?
Runtime profiling gives JIT information a static compiler (gcc/clang) can't have: which branch is taken 99% of the time, which type is passed to a virtual call. JIT can inline virtual methods (impossible statically due to polymorphism), unroll loops, eliminate dead branches, do escape analysis to stack-allocate short-lived objects. The trade-off is warmup time — C starts fast and stays flat; Java starts slow then surpasses it. This is why Java dominates long-running server workloads.
Metaspace
What is Metaspace and what causes OutOfMemoryError: Metaspace?
Metaspace stores class metadata (bytecode, field descriptors, constant pool, static field references) in native memory, outside the heap. Common causes of OOM: (1) class loader leak — frameworks that create new ClassLoaders without unloading old ones (hot redeploy in Tomcat, dynamic proxies, CGLib), (2) too many generated classes (Groovy closures, Jackson serializers per class, heavy reflection). Diagnose with heap dump + class histogram. Fix: -XX:MaxMetaspaceSize, find the leaking ClassLoader chain.

Heap vs Stack memory
What lives where · stack frames · object lifecycle · escape analysis
Definition

The stack is a per-thread, fixed-size LIFO structure that stores method call frames (local variables, operand stack, frame metadata). The heap is a single shared region where all objects live and where the GC operates. Primitives live on the stack when declared as local variables; objects always live on the heap (unless JIT escape-analysis stack-allocates them).

Stack — per thread
Method frames (locals + return address). Fixed size. Grows down. Freed when method returns. Very fast (just move a pointer). Stack overflow = too deep recursion. Default ~512KB–1MB per thread.
Heap — shared
All objects. Managed by GC. Young gen (Eden + Survivors) → Old gen. Slow relative to stack. Shared = thread-safety required. OOM = heap exhausted.
What lives where — complete rules
S
On stack — primitive local variables (int x = 5). Object references (the pointer itself, not the object). Method parameters. Return address. Operand stack for bytecode evaluation.
H
On heap — every object created with new. Arrays (even int[] — the array object is on heap, the ints are inline within it). Instance fields of any type. Static fields (their references; from Java 8 static field values are in heap-adjacent Metaspace). Interned strings (String pool).
!
Stack frame anatomy — each method call pushes a frame: (1) local variable array — indexed slots for parameters + locals, (2) operand stack — where bytecode pushes/pops values, (3) frame data — reference to runtime constant pool, return address. Frame is popped on method return or exception propagation.
E
Escape analysis — JIT analyses whether an object reference "escapes" the method (returned, stored in field, passed to another thread). If it doesn't escape, the JVM may allocate it on the stack — zero GC pressure. This is a hidden performance boost; you don't control it directly but write code that helps: short-lived objects, local scope, avoid unnecessary field storage.

Common mistakes
!
Thinking "primitives always live on the stack" — only when they're local variables. A primitive field inside an object lives on the heap, inside the object's memory layout. An int[] array body lives on the heap.
!
Confusing StackOverflowError with OutOfMemoryError — StackOverflow is infinite/deep recursion filling the per-thread call stack. OOM is heap (or Metaspace) exhaustion. They have completely different causes and fixes.
!
Creating too many threads thinking they're free — each thread has its own stack (default 512KB–1MB). 1000 threads = ~500MB–1GB of stack memory alone, before any heap objects. Use thread pools (ExecutorService) — always.
Interview questions
Classic
Where does a local variable int x = 5 live?
On the thread's JVM stack, in the local variable array of the current method's frame. When the method returns, the frame is popped and x ceases to exist — no GC needed. If x were an instance field inside an object, it would live on the heap inline within the object's memory block.
Deep
Can objects ever be allocated on the stack?
Yes, via JIT escape analysis. If the JIT can prove an object doesn't escape the current method (not returned, not stored to a field, not passed to another thread), it may allocate it on the stack. This is called stack allocation or scalar replacement (the object's fields become independent stack variables). Zero GC pressure. You can't force it, but you help it by keeping objects in local scope and short-lived.
Scenario
You see StackOverflowError in production. What caused it and how do you fix it?
Infinite or extremely deep recursion. Each recursive call adds a stack frame; the stack has a fixed size (typically 512KB–1MB). Causes: (1) recursive method without base case, (2) circular object graphs in toString/hashCode/equals, (3) mutually recursive calls. Fix: (1) convert recursion to iteration with explicit stack (ArrayDeque), (2) increase stack size -Xss (last resort, not scalable), (3) fix the logic. In production systems I've seen it from circular Jackson serialization — two objects referencing each other.

Garbage Collection
GC roots · reachability · mark-and-sweep · generational hypothesis · heap regions
Definition

GC automatically reclaims memory occupied by objects that are no longer reachable from any live thread. Java uses reachability (not reference counting) — an object is alive if and only if there exists a chain of references from a GC root to it. The GC traces this graph (mark phase) and reclaims unmarked objects (sweep/compact phase).

Generational hypothesis — why the heap is split

Empirically, most objects die young — 90%+ of objects become unreachable within milliseconds of creation (think: intermediate String in a web request, DTO parsed from JSON, temp loop variable). This means it's very efficient to collect the young generation frequently (minor GC) and only occasionally collect the long-lived old generation (major/full GC).

Mark-and-sweep-compact — full algorithm
1
Mark phase — start from GC roots (thread stacks, static fields, JNI refs). BFS/DFS traverse the object graph. Every reachable object gets a "live" mark bit set. Cost proportional to live set size.
2
Sweep phase — scan the heap linearly. Any object without the mark bit = garbage = reclaimed. Memory is returned to a free list.
3
Compact phase — slide all live objects to the start of the heap, eliminating fragmentation. Updates all references. Cost proportional to live set. Not all GCs compact (CMS didn't, which caused fragmentation OOM).
Y
Young gen (minor GC) — most frequent, stop-the-world but very short. Eden fills up → trigger minor GC. Live objects promoted to Survivor S0, garbage cleared. After N GCs in survivors, objects promoted to Old Gen. Eden is cleared entirely — cheap because most of it is dead.
O
Old gen (major/full GC) — less frequent, much longer pause. Triggered when old gen fills. Full GC = collect everything including young gen and Metaspace. The dreaded "stop the world" pause that causes request timeouts in production.
M
Metaspace GC — triggered when Metaspace fills or -XX:MaxMetaspaceSize is hit. Cleans dead ClassLoaders and their associated metadata. If classes are constantly loaded (dynamic proxies, code generation) without being unloaded, Metaspace grows without bound.

GC roots — the starting points of reachability

An object is reachable if and only if there is a path from at least one GC root to it. GC roots are the objects the JVM guarantees are live — everything they transitively reference is also live.

1
Thread stacks — every local variable and parameter in every active stack frame on every thread. The most common GC roots in a running application.
2
Static fields — class-level static references. As long as the class is loaded, its static fields are roots. Static fields holding large collections are a common source of memory leaks.
3
JNI references — objects referenced by native code via JNI. JVM can't collect these — if native code leaks a reference, the Java object is never collected.
4
Class objects of loaded classes — the Class object itself is a root while its ClassLoader is alive. When a ClassLoader is collected (all its loaded classes unreachable), their Class objects can be collected from Metaspace.
5
Synchronisation monitors — objects currently held under a synchronized lock are pinned as roots for the duration of the lock.
Common mistakes
!
Assuming null = immediately collected — setting a reference to null makes the object eligible for collection but the GC decides when. Calling System.gc() is a hint the JVM is free to ignore.
!
Not understanding that full GC = stop-the-world — in a full GC, all application threads are paused. For a 4GB heap with lots of live objects, this can be seconds. In production APIs this appears as request timeouts. Solution: tune GC, reduce allocation rate, or use low-pause collectors (G1, ZGC).
!
Large objects bypassing young gen — objects larger than -XX:PretenureSizeThreshold go directly to old gen, bypassing the fast Eden path. Frequent large object allocation (large byte arrays, strings) causes premature old gen filling and frequent full GCs.
Interview questions
Classic
How does Java GC know which objects to collect?
Via reachability analysis from GC roots (thread stacks, static fields, JNI references). The GC marks every object reachable from any root as live. Anything not marked is garbage. Java does NOT use reference counting (which suffers from circular reference leaks — Python does, Java doesn't). So circular references are NOT a problem in Java — if two objects reference each other but neither is reachable from any GC root, both are collected.
Generational
Why is Eden cleared entirely in a minor GC? Doesn't that waste work?
It's actually the opposite — it's why minor GC is so fast. Eden is cleared by copying the ~10% live objects to a Survivor space, then logically zeroing Eden in bulk. No marking-and-sweeping individual dead objects — the entire region is assumed dead except what was copied out. This is the "copying collector" strategy for young gen. It's only efficient because of the generational hypothesis: ~90%+ of Eden is dead by the time minor GC runs.
Production
You see frequent full GCs in production causing latency spikes. What's your approach?
(1) Capture GC logs: -Xlog:gc* in Java 9+. (2) Analyse with GCViewer or GCEasy — identify allocation rate, promotion rate, GC frequency, pause duration. (3) Common culprits: high allocation rate filling old gen (reduce object creation, pool objects), premature promotion (resize survivors), memory leak (heap dump + MAT). (4) If the algorithm is fundamentally stop-the-world, migrate to G1 or ZGC for sub-millisecond pauses without changing code.

GC Algorithms
Serial · Parallel · CMS · G1 · ZGC · Shenandoah — trade-offs for FAANG interviews
Four major GC algorithms
S
Serial GC (-XX:+UseSerialGC) — single-threaded, stop-the-world for all phases. Tiny footprint. Use only for small heap (<100MB), CLI tools, embedded systems. Never in production servers.
P
Parallel GC (-XX:+UseParallelGC, Java 8 default) — multi-threaded GC threads, still stop-the-world. Maximises throughput (total work/time) at the cost of longer pauses. Good for batch jobs where latency doesn't matter but total CPU efficiency does.
G1
G1 GC (-XX:+UseG1GC, Java 9+ default) — divides heap into equal-size regions (~1–32MB). Collects regions with most garbage first ("Garbage First"). Most phases concurrent (running alongside application). Targets a user-defined pause-time goal (-XX:MaxGCPauseMillis=200 default). Best balance of throughput and latency for heaps 4GB–32GB.
Z
ZGC (-XX:+UseZGC, Java 15+ production-ready) — almost entirely concurrent. Pause times under 1ms regardless of heap size (tested to 16TB). Uses load barriers and coloured pointers. Lower throughput than G1/Parallel (~5–10% overhead) but sub-millisecond latency. Use for latency-critical APIs, real-time systems.
G1 GC — how it actually works
1
Region-based heap — heap divided into 2048 equal-size regions. Each region is dynamically assigned a role: Eden, Survivor, Old, or Humongous (for objects > 50% of region size). No fixed young/old boundary — flexible allocation.
2
Remembered sets (RSets) — each region tracks which other regions have references into it. This allows G1 to collect a subset of regions without scanning the whole heap — key to meeting pause targets.
3
Mixed GC cycle — (a) Initial Mark (STW, piggybacks on minor GC) → (b) Concurrent Mark (runs with app, marks live objects) → (c) Remark (STW, finalises marking) → (d) Cleanup (STW, reclaims empty regions, selects candidates) → (e) Mixed GC (collects young + selected old regions with most garbage).
4
Humongous objects — objects larger than half a region go to Humongous regions directly (old gen). They're collected only during full GC or concurrent mark. Lots of large objects = poor G1 performance. Use -XX:G1HeapRegionSize to tune region size so large objects fit in normal regions.
ZGC — the sub-millisecond GC
1
Coloured pointers — ZGC stores metadata (GC phase bits, remapped/marked flags) in the upper unused bits of 64-bit pointers (Java objects are 64-bit). This allows GC to track object state without extra data structures and enables concurrent relocation.
2
Load barriers — code inserted at every object reference read. When you load a reference, a small barrier checks the colour bits. If the object was relocated, the barrier remaps the reference to the new location transparently. This is why ZGC can relocate objects concurrently while the app runs.
3
Why pauses are <1ms — the only stop-the-world phases are initial mark and final remark (each a few milliseconds for tiny root scanning). All heavy work (marking, relocation, remapping) runs concurrently. Heap size doesn't affect pause time — only the number of GC roots does (which is small and bounded).
4
When NOT to use ZGC — lower overall throughput than G1 (load barriers add ~5–10% overhead on every reference read). If your workload is batch/throughput-focused (not latency-sensitive), Parallel or G1 will process more total work per second.

Interview questions
Choose
Which GC would you use for a high-traffic Spring Boot REST API serving 10k req/s with p99 latency requirement < 50ms?
G1 with -XX:MaxGCPauseMillis=50 as the starting point. If that's not meeting the target after tuning (increase heap, adjust region size, monitor promotion rate), move to ZGC which guarantees sub-millisecond pauses regardless of heap size. I'd instrument with Micrometer/Prometheus and alert on GC pause duration before deciding. Parallel GC is out — its pauses will blow the latency budget.
Depth
How does ZGC achieve concurrent relocation? Why can't older GCs do this?
ZGC uses load barriers — every object reference read goes through a thin code stub that checks coloured pointer bits. If the object was relocated, the barrier transparently returns the new address and heals the reference. Older GCs (CMS, Parallel) can't do this because they don't instrument every reference read — adding load barriers would require significant JIT cooperation (which ZGC and Shenandoah have). The cost is ~5-10% barrier overhead on every read, which is the throughput trade-off.

Memory leaks in Java
How they happen · 5 common patterns · detection tools · heap dump analysis
Definition

A memory leak in Java is not a dangling pointer (Java has none) — it's an unintentional reachability chain: objects you no longer logically need are still reachable from a GC root, so the GC can't collect them. The heap grows over time, eventually causing OutOfMemoryError. Java's GC eliminates C-style leaks but introduces a new class: logical leaks through unintended references.

5 most common leak patterns in production Java
1
Static field accumulation — a static List, Map, or cache that grows without bound. Since static fields are GC roots, everything in them lives forever. Very common in singletons that accumulate state: event listeners, caches, session stores.
2
Listener / callback not deregistered — register an event listener on a publisher (Guava EventBus, Spring ApplicationEventPublisher, Swing component). Forget to deregister. Publisher holds a reference → listener lives → listener holds its closure/fields live. Classic in Android (Activity registered on Application-scoped object).
3
ThreadLocal leak — ThreadLocal values stored in thread-local maps keyed by Thread. In a thread pool (fixed threads that never die), ThreadLocal values accumulate per-task and are never cleared unless you call remove(). In app servers, this is catastrophic — every HTTP request leaks a value into the reused thread.
4
ClassLoader leak — in hot-deploy scenarios (Tomcat WAR redeploy, OSGI), the old ClassLoader should be garbage collected when the app is stopped. If anything in the new or global scope holds a reference to a class from the old loader (static field, ThreadLocal, cached Class object), the entire old ClassLoader and all its classes are kept alive. Metaspace grows with each redeploy.
5
HashMap with mutable keys or equals/hashCode broken — entries can't be found (wrong bucket) but still consume memory. The map grows without bound as "new" entries keep being added because lookups fail to find the "existing" one.
Code — leaks and fixes
// LEAK 1: Static cache grows without bound static Map<String, Report> cache = new HashMap<>(); // never evicts! // FIX: use bounded cache with eviction static Map<String, Report> cache = Collections.synchronizedMap( new LinkedHashMap<>(100, 0.75f, true) { // LRU protected boolean removeEldestEntry(Map.Entry e) { return size() > 100; } }); // LEAK 2: ThreadLocal — classic app server leak private static ThreadLocal<UserContext> ctx = new ThreadLocal<>(); void handleRequest(Request req) { ctx.set(new UserContext(req)); try { process(); } finally { ctx.remove(); // CRITICAL — always remove in finally block } } // LEAK 3: Inner class holds outer class reference class Service { List<byte[]> bigData = new ArrayList<>(); class Task implements Runnable { // non-static inner class! public void run() { /* Task holds implicit ref to Service.this */ } } // FIX: make Task a static class or a lambda that doesn't capture 'this' }
Detection workflow
1
Observe symptoms — heap usage grows over time (never flattens), full GC frequency increases, eventually OOM. Monitor with JVM metrics in Micrometer/Prometheus: jvm.memory.used, jvm.gc.pause.
2
Enable heap dump on OOM — -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/dumps/. Production-safe — dump only happens on crash, zero overhead otherwise. Do this for every production service.
3
Analyse with Eclipse MAT — open the .hprof dump. Run "Leak Suspects" report. Look at "Dominator Tree" — the objects retaining the most memory. Find the reference chain (shortest path to GC root) to understand WHY the object is alive.
4
VisualVM / JProfiler / async-profiler — live attach to a running JVM. Monitor heap in real-time. Take two heap snapshots minutes apart and compare object counts — the class that keeps growing is the leak.
5
JVM flight recorder (JFR) — -XX:+FlightRecorder, zero overhead production profiler. Records allocation sites, GC events, thread activity. Use with JDK Mission Control to find the stack trace that allocated the leaked objects.
Interview questions
Conceptual
Can you have memory leaks in Java? Explain how.
Yes — Java prevents C-style dangling pointer leaks (no manual free) but not logical leaks. A memory leak in Java is when you hold a reference to an object longer than needed, preventing GC from collecting it. Common patterns: unbounded static caches, underegistered listeners, ThreadLocal without remove(), ClassLoader leaks in hot-deploy. The GC collects everything unreachable — your job is to make sure things you're done with become unreachable.
Production
Walk me through how you'd diagnose and fix a memory leak in a Spring Boot service that OOMs after 48 hours.
(1) Add -XX:+HeapDumpOnOutOfMemoryError before next OOM. (2) Reproduce (or wait). (3) Open heap dump in Eclipse MAT — Leak Suspects report. (4) Dominator Tree shows largest retained objects. (5) Follow reference chain to find who's holding them. Common Spring findings: static ApplicationContext reference, @Bean singleton holding growing collections, @EventListener not cleaned up, Hibernate first-level cache in long-running thread. Fix based on findings. Add metric to alert before next OOM.

String interning and the String pool
Pool location · intern() · compile-time vs runtime strings · when to use
Definition

The String pool (interning pool) is a cache of String literals maintained by the JVM. When you write "hello" in source code, the JVM places it in the pool and reuses the same object for every subsequent "hello" literal. String.intern() lets you manually add runtime strings to the pool and get a canonical reference back.

Internals — pool mechanics
1
Pool location — Java 6 and earlier: PermGen (fixed-size, could cause OOM: PermGen with millions of interned strings). Java 7+: moved to main heap. Java 8+: heap, managed by GC like regular objects. This matters: in Java 7+ pooled strings can be collected if they become unreachable (no live literal reference and the classloader is gone).
2
Compile-time interning — string literals and constant expressions ("hello" + "world" where both are compile-time constants) are automatically interned by the compiler. The class file's constant pool entry is resolved to a shared String instance at class load time.
3
Runtime interning via intern() — "hello".intern() checks if the pool contains a string equal to "hello". If yes, returns the pooled reference. If no, adds this string to the pool and returns it. Comparing interned strings with == is safe (and faster than equals) — but fragile and rarely justified in application code.
!
new String("hello") — explicitly bypasses the pool. Creates a new heap object even if "hello" is already pooled. The literal "hello" arg is still pooled; the resulting new String() is not. new String("x").intern() gets back the pooled reference.
Code — the canonical == vs equals trap
String s1 = "hello"; // pooled literal String s2 = "hello"; // same pool ref String s3 = new String("hello"); // new heap object — NOT pool String s4 = "hel" + "lo"; // compile-time constant → pooled String part = "lo"; String s5 = "hel" + part; // runtime concat → NOT pooled s1 == s2; // true — same pool ref s1 == s3; // false — different object s1 == s4; // true — compile-time constant folded to pool s1 == s5; // false — runtime concat, new object s1.equals(s3); // true — value equal ALWAYS use equals! s3.intern() == s1; // true — intern() returns pool ref // When intern() is justified: massive datasets with repeated strings // e.g. parsing CSV with 10M rows all having same country codes String country = row.getCountry().intern(); // deduplicate in RAM
Interview questions
Classic
What does String.intern() do and when would you use it?
intern() returns the canonical pooled version of a string — if the pool already has an equal string, return that reference; otherwise add this string to the pool and return it. Use when: parsing large datasets with many repeated string values (country codes, enum-like strings) — interning converts O(n) string objects to O(unique values) strings, significant memory saving. Avoid for general code — pool lookup has synchronisation overhead in Java 6, and relying on == for string comparison is dangerous and confusing.
Trap
What is the output of: new String("hello") == "hello"?
false. new String("hello") bypasses the pool and creates a new heap object. The literal "hello" on the right is the pooled reference. They are different objects even though their values are equal. Always use .equals() for String comparison in production code. The only safe use of == on Strings is after explicit intern().
History
Why did String pool move from PermGen to heap in Java 7?
PermGen had a fixed size (-XX:MaxPermSize, default 64–256MB). Applications that intern many strings (large codebases, XML parsers, frameworks generating class names) would hit OutOfMemoryError: PermGen space. Moving to heap means the pool benefits from heap resizing (-Xmx) and GC can collect unused interned strings when their class is unloaded. It eliminated a whole category of PermGen OOM errors.

Java References
Strong · Soft · Weak · Phantom — reachability hierarchy and real-world use cases
Definition

Java provides four reference strengths that let you influence (not control) when the GC collects objects. The normal Java reference is "strong" — prevents collection entirely. Weaker references allow the GC to collect the referent when memory pressure demands it, enabling caches and listener registries that automatically clean up without manual intervention.

Four reference types — reachability hierarchy
St
Strong reference — the default. Object obj = new Object(). Object is never collected while any strong reference to it exists. Must be explicitly nulled or go out of scope. All your normal variables are strong references.
So
SoftReference<T> — GC collects the referent only when it needs memory (i.e., before throwing OOM). Perfect for memory-sensitive caches: the cache auto-shrinks under memory pressure instead of causing OOM. SoftReference<Image> imgRef = new SoftReference<>(img). Always check get() != null before use.
W
WeakReference<T> — GC collects the referent at the next GC cycle once no strong references remain, regardless of memory state. Use for canonicalising mappings (WeakHashMap) and listener registries where the listener's lifetime should be tied to the listener object, not the registry.
Ph
PhantomReference<T> — the weakest. get() always returns null — you can never access the referent through it. Use solely for post-mortem cleanup notification. When the referent is about to be collected, its PhantomReference is enqueued in a ReferenceQueue. You process the queue to free native resources. Replaced finalize() as the safe cleanup mechanism. Java 9 Cleaner wraps this pattern.
Reachability order: Strongly reachable → Softly reachable (no strong ref, but soft ref exists) → Weakly reachable → Phantom reachable → Unreachable
Code — all reference types
import java.lang.ref.*; // SoftReference — memory-sensitive cache SoftReference<byte[]> cache = new SoftReference<>(new byte[1024 * 1024]); byte[] data = cache.get(); // null if GC collected it under memory pressure if (data == null) data = loadFromDisk(); // reload if evicted // WeakReference — listener registry that auto-cleans WeakReference<UserSession> weakSession = new WeakReference<>(session); UserSession s = weakSession.get(); // null if session was GC'd if (s != null) s.notifyEvent(event); // WeakHashMap — cache that auto-evicts when key has no other refs Map<Widget, WidgetState> stateMap = new WeakHashMap<>(); // When Widget is no longer used elsewhere, entry auto-removed // PhantomReference + ReferenceQueue — native cleanup ReferenceQueue<Resource> queue = new ReferenceQueue<>(); PhantomReference<Resource> phantom = new PhantomReference<>(res, queue); // In cleanup thread: Reference<?> ref = queue.poll(); // non-null when res is phantom-reachable if (ref != null) closeNativeHandle(); // Java 9+ Cleaner — wraps PhantomReference nicely Cleaner cleaner = Cleaner.create(); cleaner.register(resource, () -> closeNativeHandle()); // runs when resource GC'd
ReferenceQueue — how cleanup notification works
1
Registration — create a Soft/Weak/PhantomReference with a ReferenceQueue as the second argument. When GC determines the referent's reachability crosses the threshold, the Reference object is enqueued into the queue.
2
Processing — a background thread polls/blocks on the queue. When it dequeues a Reference, it knows the referent was collected (or is about to be for Phantom) and performs cleanup: close file handles, release off-heap memory, remove from cache.
3
Why not finalizers? — finalize() runs on the finalizer thread (single thread — bottleneck), can delay GC (object needs two GC cycles to collect), can resurrect the object (dangerous), and cannot guarantee timely execution. PhantomReference + Cleaner are deterministic, thread-safe, and can't resurrect. Java 18 deprecated finalize().
Interview questions
Compare
When would you use SoftReference vs WeakReference?
SoftReference: memory-sensitive caches where you want to keep data in memory as long as possible but yield under pressure (image caches, computed result caches). GC only clears them before OOM. WeakReference: canonicalising maps and listener registries where the lifetime should track the natural reachability of the key — you don't WANT the reference to keep the object alive. WeakHashMap is the canonical use case. If no one else holds the key, the entry is automatically removed.
Depth
Why was finalize() deprecated and what replaced it?
Problems with finalize: (1) runs on a single finalizer thread — all objects with finalizers queue up, causing GC lag, (2) delays collection by one extra GC cycle (needs two cycles to collect a finalizable object), (3) can accidentally resurrect the object by storing 'this' somewhere in finalize(), (4) no execution guarantee — JVM may exit before finalizers run. Replacement: java.lang.ref.Cleaner (Java 9) uses PhantomReference + daemon thread internally. try-with-resources / AutoCloseable is the right answer for deterministic cleanup. Cleaner handles non-Java resources where you can't guarantee close() will be called.
Design
How would you implement a cache that automatically evicts entries when memory is low?
Use SoftReference values in a HashMap. When you get() a SoftReference and it returns null, the GC evicted that entry under memory pressure — reload from source and re-cache. For proper key cleanup use ReferenceQueue: when the SoftReference is enqueued, remove its key from the map. Or use Guava CacheBuilder with softValues() which does all this for you. This pattern gives you cache eviction without an explicit LRU mechanism — GC does the eviction based on actual memory conditions.
