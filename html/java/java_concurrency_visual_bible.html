Thread lifecycle
NEW → RUNNABLE → BLOCKED / WAITING / TIMED_WAITING → TERMINATED
Definition

A Java thread progresses through six states defined in Thread.State. The JVM transitions threads between states based on scheduling, locking, and waiting calls. Understanding these states is mandatory for debugging deadlocks, thread dumps, and concurrency bugs in production.

NEW → RUNNABLE ⇄ BLOCKED / WAITING / TIMED_WAITING → TERMINATED
Each state — what it means and what causes the transition
N
NEW — new Thread(r) called but start() not yet called. Thread object exists on heap. No OS thread exists yet. The thread has never run. Transition out: call start().
R
RUNNABLE — covers both "ready" (waiting for CPU slice) and "running" (currently on CPU). Java doesn't distinguish between them — from the JVM's perspective the thread is eligible to run. Also covers threads blocked on I/O at the OS level — the JVM sees them as RUNNABLE because the blocking happened inside native code. Transition out: try to acquire a lock, call wait/sleep/join.
B
BLOCKED — waiting to acquire an intrinsic monitor lock (synchronized). Another thread holds the lock. This is the state you see in deadlocks. Transition out: the lock becomes available and the thread wins the contest.
W
WAITING — indefinitely waiting for another thread to signal it. Caused by: Object.wait(), Thread.join() with no timeout, LockSupport.park(). Transition out: another thread calls notify()/notifyAll(), or the joined thread terminates.
T
TIMED_WAITING — waiting with a timeout. Caused by: Thread.sleep(ms), Object.wait(timeout), Thread.join(timeout), LockSupport.parkNanos(). Transition out: timeout expires, or the condition is met early.
X
TERMINATED — run() method returned normally or threw an uncaught exception. Thread is dead. Cannot be restarted (IllegalThreadStateException if you try). JVM removes the OS thread. The Thread object still exists on heap until GC'd.

Code — state transitions in practice
Thread t = new Thread(() -> { try { Thread.sleep(1000); // → TIMED_WAITING for 1s synchronized(lock) { // → BLOCKED if lock held; RUNNABLE if free lock.wait(); // → WAITING, releases lock } } catch (InterruptedException e) { Thread.currentThread().interrupt(); // restore interrupt flag } }); // t.getState() == NEW here t.start(); // t.getState() == RUNNABLE (immediately) // Read state at runtime (e.g. in monitoring / deadlock detector) Thread.State s = t.getState(); if (s == Thread.State.BLOCKED) { System.out.println("Potential deadlock: " + t.getName()); } // ThreadMXBean — get all thread states programmatically ThreadMXBean mx = ManagementFactory.getThreadMXBean(); ThreadInfo[] infos = mx.dumpAllThreads(true, true); for (ThreadInfo ti : infos) { System.out.println(ti.getThreadName() + ": " + ti.getThreadState()); }
Common mistakes
!
Confusing BLOCKED with WAITING — BLOCKED = waiting for a synchronized lock. WAITING = voluntarily waiting via wait()/join(). Huge difference in root cause: BLOCKED = lock contention (someone else holds it), WAITING = coordination protocol (waiting to be notified).
!
Thinking sleep() holds the lock — Thread.sleep() does NOT release any lock the thread holds. The thread stays TIMED_WAITING and keeps the lock. Only Object.wait() releases the lock. Sleeping while holding a lock is a common source of performance bugs.
!
Restarting a TERMINATED thread — calling start() on a terminated thread throws IllegalThreadStateException. Create a new thread or use a thread pool where the pool manages thread lifecycle.
Interview questions
Classic
What is the difference between BLOCKED and WAITING thread states?
BLOCKED: thread tried to enter a synchronized block/method but another thread holds the intrinsic monitor lock. Thread is not consuming CPU. Will become RUNNABLE when the lock is released and it wins. WAITING: thread explicitly gave up CPU by calling wait(), join(), or park(). It will only resume when another thread explicitly notifies it. BLOCKED is involuntary (couldn't get the lock). WAITING is voluntary (chose to wait for a condition). This distinction matters in thread dumps — a thread dump full of BLOCKED threads = lock contention; full of WAITING threads = likely waiting for work (normal in thread pools).
Production
How do you detect a deadlock from a thread dump?
Look for threads in BLOCKED state. Each BLOCKED thread shows "waiting to lock object X". If thread A waits for lock held by thread B, and thread B waits for lock held by thread A — that's a deadlock cycle. jstack or VisualVM will say "Found one Java-level deadlock:" and list the cycle explicitly. ThreadMXBean.findDeadlockedThreads() finds it programmatically. In Spring Boot, actuator /threaddump endpoint exposes this at runtime.

Creating threads
Thread · Runnable · Callable · Future — four approaches, one right choice
Four ways — hierarchy and when to use each
!
Extend Thread — override run(). Inflexible: your class can't extend anything else. Mixes task logic with threading mechanism. Almost never the right choice in modern Java.
R
Implement Runnable — pass to new Thread(r).start() or an ExecutorService. No return value, no checked exception propagation. Functional interface → usable as lambda. Use for fire-and-forget tasks.
C
Implement Callable<V> — like Runnable but returns a value and can throw checked exceptions. Submit to ExecutorService.submit() → returns a Future<V>. Use when you need a result from the thread.
F
Future<V> — handle to the result of an async computation. get() blocks until done. isDone() polls. cancel() interrupts. Limitation: can't chain, no non-blocking completion callback → CompletableFuture solves this.
Decision rule

In production code: never create raw threads. Always use ExecutorService. Choose Runnable for side-effect tasks and Callable for tasks that return values. Use CompletableFuture when you need async pipelines with chaining.

Internals
1
Thread.start() vs Thread.run() — run() just calls the method on the current thread — no new OS thread created. start() calls the JVM native method start0() which asks the OS to create a new thread and schedule it. The new OS thread will call your run(). Calling run() directly is the most common beginner mistake.
2
Callable<V> — functional interface: V call() throws Exception. ExecutorService.submit(callable) wraps it in a FutureTask<V> which implements both Runnable and Future<V>. The pool thread runs FutureTask.run() which calls callable.call() and stores the result or exception.
3
FutureTask internals — uses a single volatile int state field with CAS transitions (NEW→COMPLETING→NORMAL/EXCEPTIONAL/CANCELLED). Waiting threads queue on an internal linked list and are LockSupport.unpark()-ed when the state transitions to done.
Code — all four patterns
// 1. Extend Thread — almost never do this class MyThread extends Thread { public void run() { System.out.println("running"); } } new MyThread().start(); // 2. Runnable — preferred for no-return tasks Runnable r = () -> System.out.println("task"); new Thread(r).start(); // still raw thread — use executor instead // 3. Callable + Future — tasks that return values ExecutorService pool = Executors.newFixedThreadPool(4); Future<Integer> f = pool.submit(() -> heavyCompute()); // ... do other work ... int result = f.get(); // blocks until done; throws ExecutionException // 4. CompletableFuture — async pipeline (preferred modern approach) CompletableFuture.supplyAsync(() -> fetchUser(id)) .thenApply(user -> enrichWithProfile(user)) .thenAccept(enriched -> cache.put(enriched.id(), enriched)); // WRONG — calling run() directly — no new thread! new Thread(r).run(); // executes on CURRENT thread, defeats the purpose
Common mistakes
!
Calling run() instead of start() — executes the task synchronously on the calling thread. No new OS thread is created. Code appears to "work" (task executes) but never runs concurrently. Compile-time catch: none. Runtime symptom: sequential execution despite threading code.
!
Ignoring Future exceptions — if the Callable throws, the exception is wrapped in ExecutionException and stored. future.get() rethrows it. If you never call get(), the exception is silently lost. Always call get() or attach a CompletableFuture.exceptionally() handler.
!
Blocking on future.get() in the main thread — defeats the purpose of async. If you submit 10 tasks then call get() on each in order, you're back to sequential. Use CompletableFuture.allOf() or ExecutorCompletionService to process in completion order.
Interview questions
Classic
What is the difference between Runnable and Callable?
Runnable.run() returns void and cannot throw checked exceptions. Callable.call() returns a typed value V and declares throws Exception. Both are functional interfaces so both work as lambdas. Use Callable when you need (1) a return value from the background task, or (2) to propagate checked exceptions to the caller via Future.get()'s ExecutionException. Executors accept both via submit().
Gotcha
Why does calling thread.run() instead of thread.start() cause bugs that are hard to detect?
run() is just a normal method call — it executes synchronously on the calling thread, no OS thread is created. The code produces correct output (task runs) but runs sequentially, so the concurrency bug only manifests under load (expected parallelism doesn't happen). Static analysis won't catch it. The symptom is "my multi-threaded code is slow" — and the fix is replacing .run() with .start(). Senior answer: also mention you can catch this with Thread.currentThread().getName() inside run() — if it matches the main thread, you called run() not start().

synchronized keyword
Intrinsic locks · method vs block · object vs class lock · monitor semantics
Definition

synchronized provides mutual exclusion via an intrinsic monitor lock (every Java object has one). At most one thread can hold an object's monitor at a time. It provides both atomicity (no interleaving of the critical section) and visibility (changes made inside a synchronized block are visible to the next thread that acquires the same lock).

Synchronized method
Locks this for instance methods. Locks the Class object for static methods. Entire method body is the critical section.
Synchronized block
Locks an explicit object. Preferred — minimises the critical section, reducing contention. Allows different locks for different concerns in the same class.
How the JVM implements intrinsic locks
1
Object header — every Java object has a 2-word header. One word is the mark word which stores the object's identity hashCode, GC age, and lock state. The lock state transitions through: unlocked → biased (single thread, zero-cost locking) → thin lock (CAS-based, no OS involvement) → inflated (fat lock, OS mutex, used when there's contention).
2
Biased locking — (deprecated Java 15+) JVM assumes single-thread access. First lock: stamp the thread ID in the mark word — zero OS cost. Subsequent locks by same thread: one check, no CAS. Another thread contends: revoke bias (expensive — stop-the-world for that thread) → upgrade to thin lock.
3
Thin lock (CAS) — for lightly contended locks. Lock is encoded in the mark word. Acquire = CAS of mark word. If CAS fails, spin briefly, then inflate. No OS syscall — fast.
4
Fat lock (inflated) — on contention: inflate to an OS mutex. Waiting threads park via LockSupport.park() (or OS futex). Expensive OS context switch on each acquire/release. This is why high contention kills throughput.
5
Memory semantics — acquiring a lock = memory read barrier (flushes CPU caches, sees latest writes). Releasing a lock = memory write barrier (flushes pending writes to main memory). This is why synchronized gives visibility — not just mutual exclusion.
Code — method vs block, object vs class lock
class BankAccount { private double balance; private final Object lock = new Object(); // dedicated lock object // Synchronized method — locks 'this' for entire method public synchronized void deposit(double amount) { balance += amount; } // Synchronized block — preferred: minimises critical section public void transfer(BankAccount target, double amount) { validate(amount); // NOT locked — read-only check synchronized(lock) { // only the mutation is locked balance -= amount; } synchronized(target.lock) { target.balance += amount; } // WARNING: two locks = deadlock risk! } // Static synchronized — locks the Class object, not 'this' public static synchronized void audit() { // synchronized on BankAccount.class // ALL instances share this lock — major bottleneck } } // CLASSIC BUG: synchronizing on wrong object private Integer count = 0; synchronized(count) { count++; } // WRONG: count is reassigned after ++ // Integer auto-boxes to a new object → lock changes every iteration
Common mistakes
!
Synchronizing on a mutable field — synchronized(this.lock) where lock is reassigned elsewhere. The reference changes, threads lock different objects, no mutual exclusion. Always lock on final objects.
!
Synchronizing on String literals / Integer values — string literals and small Integers are pooled. synchronized("lock") uses the same lock as every other class using the same literal. Leads to unexpected contention between totally unrelated code paths.
!
Thinking synchronized = atomic for compound operations — synchronized makes the block atomic, not individual statements. If you need check-then-act to be atomic, both the check AND the act must be inside the same synchronized block.
Interview questions
Internals
What is an intrinsic lock and how does the JVM implement it?
Every Java object has a built-in lock encoded in the object header's mark word. The JVM uses a three-phase locking protocol: (1) biased locking — single-thread case, stamp the thread ID, zero-cost re-entry. (2) thin lock — CAS on the mark word, no OS involvement, fast. (3) fat lock — OS mutex via futex/pthread_mutex, used under contention. The JIT compiles monitorenter/monitorexit bytecodes into these lock operations. Memory-wise: monitorenter = acquire fence, monitorexit = release fence — guaranteeing visibility.
Design
When would you choose synchronized over ReentrantLock?
synchronized when: the critical section is simple and short, you don't need tryLock/timed-lock/interruptible lock, you want automatic lock release on exception (no finally needed). ReentrantLock when: you need fairness guarantees, tryLock to avoid blocking, lockInterruptibly for cancellable waits, multiple Condition objects (producer-consumer with two conditions). In practice: synchronized for simple mutual exclusion, ReentrantLock for advanced coordination patterns.

volatile keyword
Visibility guarantee · happens-before · when volatile is not enough
Definition

volatile guarantees two things: (1) visibility — writes to a volatile field are immediately flushed to main memory; reads always come from main memory, never from CPU caches. (2) ordering — volatile prevents reordering of instructions around the volatile access. It does not provide atomicity for compound operations.

volatile is enough when
One writer thread, multiple reader threads. Single-variable state flags. Double-checked locking pattern (with volatile). Long/double fields shared across threads.
volatile is NOT enough when
Read-modify-write: count++ is three steps (read, increment, write). Check-then-act: if(!done) done=true. You need synchronized or AtomicXxx for these.
How the CPU cache problem works — and how volatile fixes it
!
CPU cache problem — modern CPUs have L1/L2/L3 caches per core. Thread A on core 1 writes to variable running = false. That write goes to core 1's cache first. Thread B on core 2 may still read stale running = true from its own cache. Without a memory barrier, this can persist indefinitely — thread B runs forever even though thread A said stop.
V
volatile write — the JVM emits a store fence after every volatile write. This flushes the core's store buffer to main memory. Every other core's cache line for that address is invalidated. Cost: ~10-50ns depending on cache coherence protocol (x86 uses MESI).
V
volatile read — the JVM emits a load fence before every volatile read. This forces the CPU to fetch from main memory (or another core's cache), bypassing the local stale cache entry.
H
Happens-before edge — per the JMM: a write to a volatile field happens-before every subsequent read of that same field. This means ALL writes by thread A before the volatile write are visible to thread B after it reads the volatile field — even non-volatile writes. This is the key to the double-checked locking pattern.
Code — stop flag, DCL singleton, what volatile cannot do
// CORRECT use — stop flag with one writer, many readers class Worker implements Runnable { private volatile boolean running = true; // volatile: required! public void stop() { running = false; } public void run() { while (running) { process(); } // sees update immediately } } // Double-checked locking (DCL) — volatile is REQUIRED on instance class Singleton { private static volatile Singleton instance; // volatile prevents partial init public static Singleton get() { if (instance == null) { // first check (no lock) synchronized(Singleton.class) { if (instance == null) // second check (locked) instance = new Singleton(); } } return instance; } } // WRONG — volatile does NOT make compound ops atomic private volatile int count = 0; count++; // read → increment → write: another thread can interleave! // Fix: use AtomicInteger.incrementAndGet()
Common mistakes
!
Using volatile for count++ — the increment is not atomic: it's read + add + write. Another thread can interleave between the read and the write. Use AtomicInteger.
!
Omitting volatile in DCL singleton — without volatile on instance, the JIT can reorder object construction. Thread B may see a non-null but partially-constructed object. The volatile write barrier prevents this reordering.
!
Thinking volatile prevents all race conditions — it only guarantees visibility and ordering around that specific field. Race conditions on multi-step operations require synchronization or atomics.
Interview questions
Classic
What does volatile guarantee and what doesn't it guarantee?
Guarantees: (1) Visibility — writes immediately flush to main memory; reads always from main memory. (2) Ordering — prevents reordering of instructions around the volatile access (store/load fences). (3) JMM happens-before — a volatile write happens-before all subsequent reads of that field. Does NOT guarantee: atomicity. count++ on a volatile int is still three non-atomic operations. For atomicity: use synchronized or AtomicInteger.
DCL
Why does the double-checked locking singleton require volatile?
new Singleton() compiles to three steps: (1) allocate memory, (2) invoke constructor, (3) assign reference to field. The JIT can reorder steps 2 and 3 — assign the reference before the constructor runs. Thread B reads a non-null instance but sees a partially constructed object. volatile on the instance field inserts a store fence that prevents (3) from happening before (2). This is a subtle but real bug that caused actual production failures before volatile's JMM semantics were clarified in Java 5.

ReentrantLock vs synchronized
Fairness · tryLock · lockInterruptibly · Condition objects
Definition

ReentrantLock is an explicit lock (from java.util.concurrent.locks) that provides all the guarantees of synchronized plus advanced features: timed lock attempts, interruptible locking, fairness policy, and multiple Condition objects for fine-grained thread coordination. "Reentrant" means the same thread can acquire it multiple times without deadlocking.

Internals — built on AbstractQueuedSynchronizer
1
AbstractQueuedSynchronizer (AQS) — the backbone of most java.util.concurrent. Uses a single volatile int state field and a CLH queue (Craig–Landin–Hagersten queue — a variant of linked list) of waiting threads. Lock acquire = CAS on state. If CAS fails, enqueue the thread and LockSupport.park() it.
2
Reentrancy — state tracks the hold count. Each lock() increments state, each unlock() decrements. Thread ID stored to allow same-thread re-entry. Lock is released when state reaches 0.
3
Fairness — unfair (default): new thread can barge in and acquire the lock even if other threads are queued (allows JIT to do lock elision, better throughput). Fair: strict FIFO queue order. Fair is slower but prevents starvation. Default is unfair — choose explicitly.
4
Condition objects — lock.newCondition() creates a Condition backed by a separate wait queue within the same lock. condition.await() releases the lock and parks the thread. condition.signal() moves one thread from the condition queue back to the lock queue. Allows multiple producer/consumer conditions per lock — impossible with a single synchronized monitor.
Code — tryLock, fair lock, Condition producer-consumer
class BoundedBuffer<T> { private final Queue<T> queue = new ArrayDeque<>(); private final int maxSize; private final ReentrantLock lock = new ReentrantLock(); // unfair private final Condition notFull = lock.newCondition(); // producer waits here private final Condition notEmpty = lock.newCondition(); // consumer waits here public void put(T item) throws InterruptedException { lock.lock(); try { while (queue.size() == maxSize) notFull.await(); // releases lock queue.add(item); notEmpty.signal(); // wake ONE waiting consumer } finally { lock.unlock(); } // ALWAYS in finally } public T take() throws InterruptedException { lock.lock(); try { while (queue.isEmpty()) notEmpty.await(); T item = queue.poll(); notFull.signal(); return item; } finally { lock.unlock(); } } } // tryLock — non-blocking lock attempt with timeout if (lock.tryLock(100, TimeUnit.MILLISECONDS)) { try { doWork(); } finally { lock.unlock(); } } else { fallback(); } // didn't get lock in time

Interview questions
Compare
When do you use ReentrantLock over synchronized?
ReentrantLock when: (1) tryLock needed — can't afford to block indefinitely (circuit breaker pattern), (2) lockInterruptibly — want thread to respond to interruption while waiting for lock, (3) fairness required — prevent starvation (e.g., fair scheduling for shared resources), (4) multiple Condition objects — producer-consumer with two separate queues is cleaner with two Conditions than with synchronized+notifyAll (which wakes ALL waiters). synchronized when: simple mutual exclusion, shorter code, JIT can elide the lock entirely when single-threaded (not possible with ReentrantLock).
Depth
Explain AbstractQueuedSynchronizer — how does it underpin ReentrantLock, CountDownLatch, Semaphore?
AQS provides a framework for blocking locks and synchronisers. Core: volatile int state + CLH queue. Subclasses override tryAcquire/tryRelease to define what "acquire" means. ReentrantLock: state = hold count, tryAcquire = CAS if state==0 or same thread. Semaphore: state = permits, tryAcquire = decrement if >0. CountDownLatch: state = count, await blocks until state==0, countDown decrements. All the blocking/queuing/waking logic lives in AQS — subclasses only define the semantic of "can I proceed?". This is a textbook example of the Template Method pattern at the JVM level.

Atomic variables & CAS
AtomicInteger · AtomicReference · compare-and-swap · ABA problem
Definition

Atomic classes (AtomicInteger, AtomicLong, AtomicReference, etc.) provide thread-safe operations on single variables without locks. They use CPU-level Compare-And-Swap (CAS) — a single, indivisible hardware instruction that atomically checks a value and updates it only if it matches an expected value.

AtomicInteger
Thread-safe int ops. incrementAndGet, compareAndSet, addAndGet. Lock-free counters, sequence generators.
AtomicReference<V>
Thread-safe object reference. CAS on pointer. Used for lock-free data structures and non-blocking algorithms.
LongAdder
Higher throughput than AtomicLong under contention — stripes the counter across CPU slots. Use for high-frequency counters (request rate, byte counts).
How CAS works — the hardware instruction
1
CAS instruction — on x86: CMPXCHG. On ARM: LDREX/STREX. Semantics: "if memory[addr] == expected, atomically write newValue to memory[addr] and return true; otherwise return false". This is ONE hardware instruction — indivisible, even on multicore systems.
2
Spin loop (optimistic concurrency) — atomic ops use a retry loop: read current value → compute new value → CAS(current, new) → if CAS fails (another thread changed it), read again and retry. Under low contention this almost always succeeds on first try. Under high contention, retries burn CPU — this is the trade-off vs locks.
!
ABA problem — thread T1 reads value A. Thread T2 changes A → B → A. T1's CAS succeeds (sees A = expected) but the value went through B — intermediate state changes are invisible. Fix: AtomicStampedReference or AtomicMarkableReference — include a version stamp with each update so A→B→A becomes A@1 → B@2 → A@3, and CAS on (A,1) fails because the stamp is now 3.
V
VarHandle (Java 9+) — low-level access to fields and array elements with memory ordering semantics. Replaced sun.misc.Unsafe for most uses. Used internally by AtomicXxx classes. Allows CAS on any field type without boxing overhead.
Code — atomic counter, CAS loop, ABA solution
// AtomicInteger — lock-free counter AtomicInteger count = new AtomicInteger(0); count.incrementAndGet(); // atomic ++ count.addAndGet(5); // atomic += 5 count.compareAndSet(5, 10); // CAS: if current==5 set to 10 // Manual CAS spin loop — e.g. custom accumulator AtomicInteger ai = new AtomicInteger(); int old, updated; do { old = ai.get(); updated = compute(old); } while (!ai.compareAndSet(old, updated)); // retry if lost race // AtomicReference — non-blocking stack push AtomicReference<Node> top = new AtomicReference<>(); void push(Node n) { do { n.next = top.get(); } while (!top.compareAndSet(n.next, n)); // retry if top changed } // ABA fix — AtomicStampedReference AtomicStampedReference<String> asr = new AtomicStampedReference<>("A", 0); int[] stamp = new int[1]; String val = asr.get(stamp); // get value AND stamp atomically asr.compareAndSet(val, "B", stamp[0], stamp[0]+1); // update both // LongAdder — better throughput than AtomicLong under high contention LongAdder hits = new LongAdder(); hits.increment(); // add to a CPU-local cell hits.sum(); // sum all cells — slightly stale but fast
Common mistakes
!
Thinking Atomic = always faster than synchronized — under very high contention, CAS spin loops burn CPU with retries. Synchronized threads park and don't waste CPU. For high-contention scenarios, consider LongAdder (stripes counters) or synchronized.
!
Compound operations on separate AtomicXxx are not atomic together — two separate AtomicInteger updates are individually atomic but not jointly atomic. If you need to update two things as one transaction, you need synchronization.
Interview questions
Classic
How does compare-and-swap work and why is it called lock-free?
CAS is a single CPU instruction (CMPXCHG on x86) that atomically reads a memory location, compares it to an expected value, and writes a new value only if they match. It's "lock-free" because no thread ever blocks another — if a thread loses the CAS race, it retries rather than waiting. This means: (1) no deadlocks possible, (2) no priority inversion, (3) a delayed thread doesn't block others. Trade-off: under very high contention, retry loops burn CPU. Lock-free ≠ always faster.
ABA
What is the ABA problem and how do you fix it in Java?
ABA: thread reads value A. Another thread changes A→B→A. First thread's CAS succeeds even though the logical state changed. Classic example: lock-free stack — node with value A is removed (A→null) then a different node with same memory address and value A is pushed back. The CAS on the top pointer succeeds but now points to a different node than expected. Fix: AtomicStampedReference — include an integer stamp (version number) with every value. CAS must match both value AND stamp. A(stamp=1) → B(stamp=2) → A(stamp=3): CAS expecting A with stamp=1 now correctly fails.

Deadlock · Livelock · Starvation
Definitions · Coffman conditions · prevention · detection
Three distinct problems
D
Deadlock — two or more threads each hold a lock and each is waiting for a lock held by another. Nobody moves. Threads are in BLOCKED state forever. CPU usage drops to 0 for those threads. Classic: thread A locks X then tries to lock Y; thread B locks Y then tries to lock X.
L
Livelock — threads are not blocked but continuously change state in response to each other without making progress. CPU usage is HIGH (threads are running). Classic: two threads each detect a conflict and both back off, then both retry at the same time, detect conflict again, back off again — forever. Like two people in a hallway both stepping the same direction to let the other pass.
S
Starvation — a thread is perpetually denied access to a resource because other threads always get priority. The thread is alive and runnable but never gets CPU time or a lock. Classic: unfair lock where high-priority threads always win; a low-priority thread waits indefinitely.
Coffman conditions — all 4 must hold for deadlock
1
Mutual exclusion — at least one resource is held exclusively (can't be shared). For locks: always true by definition.
2
Hold and wait — a thread holds at least one resource while waiting to acquire more. If you can eliminate this (acquire all locks at once), deadlock is impossible.
3
No preemption — a lock cannot be forcibly taken from a thread. Threads must release voluntarily. tryLock(timeout) adds a form of preemption — give up if you can't get it.
4
Circular wait — a cycle exists in the resource-allocation graph. Thread A waits for B's lock, thread B waits for A's lock. Break this by imposing a global lock ordering.
Prevention strategy: eliminate at least one condition. The cheapest: eliminate circular wait by always acquiring locks in a consistent global order.
Code — deadlock, detection, prevention
// Classic deadlock — lock ordering inconsistency synchronized(lockA) { // Thread 1: A first synchronized(lockB) { ... } } synchronized(lockB) { // Thread 2: B first — DEADLOCK! synchronized(lockA) { ... } } // Prevention: consistent lock ordering using System.identityHashCode void transfer(Account from, Account to, double amt) { Object first = (from.id < to.id) ? from : to; // always lower ID first Object second = (from.id < to.id) ? to : from; synchronized(first) { synchronized(second) { doTransfer(from, to, amt); } } } // Prevention: tryLock with timeout — no hold-and-wait if (lockA.tryLock(100, TimeUnit.MILLISECONDS)) { try { if (lockB.tryLock(100, TimeUnit.MILLISECONDS)) { try { doWork(); } finally { lockB.unlock(); } } } finally { lockA.unlock(); } } // Detection at runtime: ThreadMXBean long[] deadlocked = ManagementFactory.getThreadMXBean() .findDeadlockedThreads(); // null if none
Prevention strategies — ranked by practicality
1
Lock ordering — always acquire multiple locks in the same global order (e.g. by object ID). Eliminates circular wait. Cheapest to implement, most effective. Java doesn't enforce this — it's your discipline.
2
tryLock with timeout — ReentrantLock.tryLock(timeout). If you can't get the lock in time, release what you hold and retry. Eliminates "no preemption" and "hold and wait". Introduces retry complexity and potential livelock if backoffs aren't randomised.
3
Single-lock or lock-free design — restructure to never need multiple locks simultaneously. Message passing (actor model, queue-based), or use a single orchestrator thread for writes. Eliminates the problem entirely.
4
Use higher-level concurrency abstractions — ConcurrentHashMap, BlockingQueue, database transactions. They manage locking internally with tested, correct ordering. Your service code never touches raw locks.
Interview questions
Classic
What are the four Coffman conditions for deadlock?
(1) Mutual exclusion — resource held exclusively. (2) Hold and wait — thread holds one resource while waiting for another. (3) No preemption — resources can't be forcibly taken. (4) Circular wait — cycle in the wait graph. ALL four must hold simultaneously. Prevention: eliminate any one of them. Most practical: break circular wait by enforcing lock ordering. In Java: use object ID / hashCode to determine lock acquisition order — always lock lower ID first.
Distinguish
How is livelock different from deadlock? How do you fix livelock?
Deadlock: threads are BLOCKED, no CPU usage, stuck forever waiting. Livelock: threads are RUNNABLE, consuming CPU, but making no net progress — each responds to the other's action and backs off in the same pattern. Fix for livelock: (1) randomise retry intervals (exponential backoff with jitter — the same fix as for network retry storms), (2) impose priority or a tie-breaker so one thread always wins the conflict, (3) timeout and fail-fast after N retries rather than retrying indefinitely.

ExecutorService & thread pools
FixedThreadPool · CachedThreadPool · ScheduledThreadPool · ForkJoinPool
Definition

ExecutorService decouples task submission from thread management. You submit Runnable/Callable tasks; the pool manages thread creation, reuse, and disposal. This avoids the cost of creating a new OS thread per task (expensive: ~1ms + 512KB stack) and allows controlled concurrency.

ThreadPoolExecutor internals
1
Core parameters — corePoolSize: minimum live threads even when idle. maximumPoolSize: max threads ever created. keepAliveTime: how long excess threads (above core) live while idle. workQueue: where tasks wait when all core threads are busy.
2
Task submission flow — (1) If active threads < corePoolSize: create a new thread. (2) Else if queue not full: enqueue the task. (3) Else if active threads < maxPoolSize: create new thread. (4) Else: apply rejection policy. This is why a FixedThreadPool with an unbounded queue can OOM — tasks pile up without bound.
3
Rejection policies — AbortPolicy (default, throws RejectedExecutionException), CallerRunsPolicy (caller thread runs the task — natural backpressure), DiscardPolicy (silently drops), DiscardOldestPolicy (drops oldest queued task).
4
shutdown() vs shutdownNow() — shutdown(): stop accepting new tasks, drain the queue, let running tasks complete. shutdownNow(): stop accepting, attempt to cancel queued tasks, interrupt running threads. Always shut down pools — leaked pools keep JVM alive and waste memory.
Four pool types — when to use each
F
FixedThreadPool(n) — corePoolSize = maxPoolSize = n, unbounded LinkedBlockingQueue. Predictable resource usage. Use for: CPU-bound tasks (n = Runtime.getRuntime().availableProcessors()). Danger: if tasks block on I/O, threads are held idle while queue grows unbounded → OOM.
C
CachedThreadPool — corePoolSize=0, maxPoolSize=Integer.MAX_VALUE, SynchronousQueue (no buffering — task must be taken immediately or new thread created). Threads cached 60s. Use for: many short-lived async tasks. Danger: task spike → unbounded thread creation → OOM. Never use for long-blocking I/O.
S
ScheduledThreadPool(n) — runs tasks with delay or periodically. Use for: polling jobs, health checks, cache refresh, retry schedulers. scheduleAtFixedRate vs scheduleWithFixedDelay — former fires every N ms from start; latter fires N ms after previous completes.
W
ForkJoinPool — work-stealing pool designed for divide-and-conquer. Each thread has a deque; idle threads steal from the back of busy threads' queues. Used internally by parallel streams and CompletableFuture.supplyAsync(). Use for: recursive decomposition tasks (parallel sort, merge). commonPool() = JVM-wide shared instance.
Code — custom pool, lifecycle, sizing
// Custom ThreadPoolExecutor — production pattern ExecutorService pool = new ThreadPoolExecutor( 4, // corePoolSize 8, // maxPoolSize 60, TimeUnit.SECONDS, // keepAlive for extra threads new ArrayBlockingQueue<>(100), // bounded queue! new ThreadFactory() { /* named threads */ }, new ThreadPoolExecutor.CallerRunsPolicy()); // backpressure // Correct shutdown pattern pool.shutdown(); try { if (!pool.awaitTermination(30, TimeUnit.SECONDS)) { pool.shutdownNow(); pool.awaitTermination(5, TimeUnit.SECONDS); } } catch (InterruptedException e) { pool.shutdownNow(); Thread.currentThread().interrupt(); } // Sizing guideline (Little's Law): // CPU-bound: threads = CPU cores // I/O-bound: threads = cores × (1 + wait_time/compute_time) // e.g. 4 cores, 80% wait time → 4 × (1 + 4) = 20 threads
Common mistakes
!
Using Executors.newFixedThreadPool with default unbounded queue — if producers outpace consumers, the LinkedBlockingQueue grows without bound → OOM. Always use a custom ThreadPoolExecutor with a bounded ArrayBlockingQueue and a rejection policy in production.
!
Not shutting down pools — ExecutorService pools hold non-daemon threads. The JVM won't exit until all non-daemon threads terminate. A "hello world" app that creates a pool and doesn't shut it down hangs forever.
!
Submitting blocking tasks to ForkJoinPool common pool — CompletableFuture.supplyAsync() uses the common ForkJoinPool. Blocking I/O inside it starves other tasks sharing the pool. Use a dedicated I/O executor or Virtual Threads (Java 21) for I/O-bound work.
Interview questions
Design
How do you size a thread pool for an I/O-bound web service?
Use Little's Law: pool_size = CPU_cores × (1 + wait_time/compute_time). For a typical REST API where 80% of time is I/O (DB, HTTP calls): 4-core machine → 4 × (1 + 4) = 20 threads. But this is a starting point. In practice: profile under load, monitor thread pool queue depth (should be near zero), watch for pool exhaustion causing request queuing. Also consider: Java 21 Virtual Threads for I/O-heavy workloads eliminate the sizing problem entirely — one virtual thread per request is now cheap.
Gotcha
What happens when you submit tasks faster than they can be processed in a FixedThreadPool?
Tasks queue up in the LinkedBlockingQueue (unbounded by default). Memory grows without bound. Eventually OutOfMemoryError. The JVM won't reject tasks — it silently keeps queuing. Solution: use a bounded queue (ArrayBlockingQueue) with a rejection policy. CallerRunsPolicy is often the best choice — it makes the calling thread run the task, which naturally slows down the producer without dropping work. This is backpressure.

CompletableFuture
thenApply · thenCompose · allOf · exceptionally · async pipelines
Definition

CompletableFuture<T> is a Promise/Future that you can explicitly complete and chain. Unlike raw Future, it supports non-blocking callbacks (thenApply, thenAccept), chaining dependent async stages (thenCompose), combining multiple futures (allOf, anyOf), and error handling (exceptionally, handle).

API map — the 4 most important operator pairs
1
thenApply vs thenCompose — thenApply(f) where f returns T (synchronous transform). thenCompose(f) where f returns CompletableFuture<T> (async flat-map — avoids CompletableFuture<CompletableFuture<T>>). Rule: use thenCompose when the next step is itself async.
2
thenAccept vs thenRun — thenAccept(consumer): receives the result, returns void. thenRun(runnable): doesn't receive the result, just runs an action after completion. Both return CompletableFuture<Void>.
3
allOf vs anyOf — allOf(cf1, cf2, cf3): completes when ALL complete. Returns CompletableFuture<Void> (no aggregated result — you fetch individually). anyOf(cf1,cf2,cf3): completes when ANY completes, result = first result.
4
exceptionally vs handle — exceptionally(e -> fallback): called only on exception, returns a recovery value. handle((result, ex) -> ...): called in both success and failure paths. whenComplete: side-effect only (logging), propagates the original result/exception unchanged.
Code — async pipeline, fan-out/fan-in, error handling
// 1. Basic async pipeline (runs on ForkJoinPool.commonPool by default) CompletableFuture.supplyAsync(() -> fetchUser(userId)) // async .thenApply(user -> enrichWithPermissions(user)) // sync transform .thenCompose(user -> fetchOrders(user.id())) // async flat-map .thenAccept(orders -> notifyFrontend(orders)) // terminal .exceptionally(ex -> { log.error(ex); return null; }); // 2. Fan-out then fan-in (call 3 services in parallel, join results) CompletableFuture<User> cfUser = supplyAsync(() -> getUser(id), pool); CompletableFuture<Orders> cfOrders = supplyAsync(() -> getOrders(id), pool); CompletableFuture<Profile> cfProfile = supplyAsync(() -> getProfile(id), pool); CompletableFuture.allOf(cfUser, cfOrders, cfProfile) .thenApply(v -> new Dashboard( cfUser.join(), // join() = get() but unchecked exception cfOrders.join(), cfProfile.join() )) .exceptionally(ex -> Dashboard.empty()); // 3. Timeout (Java 9+) CompletableFuture.supplyAsync(() -> slowService()) .orTimeout(500, TimeUnit.MILLISECONDS) // throws on timeout .completeOnTimeout(defaultVal, 500, TimeUnit.MILLISECONDS); // fallback // 4. Run on custom executor (not common pool) CompletableFuture.supplyAsync(() -> dbQuery(), ioExecutor) .thenApplyAsync(result -> transform(result), cpuExecutor);
Common mistakes
!
Blocking inside thenApply on the common pool — thenApply runs on the ForkJoinPool common pool by default. A blocking call inside it starves the pool. Use thenApplyAsync(fn, ioExecutor) to hand off to a dedicated I/O pool.
!
Using thenApply when thenCompose is needed — if your function returns a CompletableFuture, thenApply gives you CompletableFuture<CompletableFuture<T>>. You need thenCompose (the async equivalent of flatMap) to unwrap the nested future.
!
Not handling exceptions on every branch — in a fan-out pattern, if cfOrders fails and you have no exceptionally, the exception propagates to allOf's future silently. Each individual CF should handle exceptions, or the aggregated allOf should.
Interview questions
Compare
What is the difference between thenApply and thenCompose?
thenApply is like Stream.map — takes the result, applies a synchronous function, wraps the return value in a new CompletableFuture. thenCompose is like Stream.flatMap — the function returns a CompletableFuture itself; thenCompose unwraps it. If you use thenApply with an async function, you get CompletableFuture<CompletableFuture<T>> — nested and unusable without extra get(). Rule: if the next step returns a plain value → thenApply. If the next step returns a CompletableFuture → thenCompose.
Design
How would you call three microservices in parallel and aggregate the results?
Submit all three as CompletableFutures with supplyAsync on a shared executor. Use CompletableFuture.allOf(cf1, cf2, cf3) to wait for all. In the thenApply after allOf, call .join() on each individual future to get results (join() inside allOf's callback is safe — they're already done). Add exceptionally on each individual CF for per-service fallbacks, and a global exceptionally on the allOf for total failure. Also add .orTimeout() for latency SLO enforcement.

Java Memory Model (JMM)
Visibility · ordering · happens-before relationship · practical rules
Definition

The JMM defines when writes by one thread are guaranteed to be visible to reads by another. Without explicit synchronisation, the JVM and CPU are free to reorder instructions and cache values — two threads can see different values for the same variable. The JMM defines happens-before edges: if action A happens-before B, A's effects are guaranteed visible to B.

Visibility problem
Thread A writes x=5. Thread B reads x. Without happens-before, B may read 0 (stale cache). This is NOT a bug in Thread B — it's permitted by the JMM.
Ordering problem
JVM and CPU reorder instructions for performance. Thread A: x=1; flag=true. Another thread may see flag=true but x=0 if the write to x is reordered after flag. Happens-before prevents this.
The 7 happens-before rules you must know
1
Program order — within a single thread, each action happens-before every action that follows it in program order. (But the JVM can reorder between threads.)
2
Monitor lock — an unlock of a monitor happens-before every subsequent lock of that same monitor. This is why synchronized gives visibility: all writes before unlock are visible after the next lock.
3
Volatile write → read — a write to a volatile field happens-before every subsequent read of that field. Also: all writes before the volatile write are visible after the volatile read (transitive).
4
Thread start — Thread.start() happens-before any action in the started thread. The starting thread's state is fully visible to the new thread when it begins.
5
Thread join — all actions in a thread happen-before Thread.join() returns in the joining thread. Everything the child thread did is visible after join() returns.
6
Default values — writes of default values (0, null, false) to fields happen-before the first action in any thread. Objects are zeroed before any constructor runs.
7
Transitivity — if A hb B and B hb C, then A hb C. This allows happens-before chains to span multiple synchronisation actions.
Code — visibility bugs and fixes
// BUG: no happens-before between writer and reader threads boolean ready = false; int answer = 0; // Thread A: answer = 42; ready = true; // JVM may reorder: ready=true before answer=42 // Thread B: while (!ready) {} // may loop forever (stale cache) print(answer); // may print 0 even if !ready loop exited // FIX 1: volatile (establishes happens-before) volatile boolean ready = false; // FIX 2: synchronized (monitor lock rule) synchronized(lock) { ready = true; answer = 42; } synchronized(lock) { if(ready) print(answer); } // sees 42 // Safe publication — all these create happens-before: // • static initializer (class loading guarantees visibility) // • final field initialized in constructor (JMM freeze action) // • volatile field write before sharing // • synchronized block before sharing
Interview questions
Concept
What is the Java Memory Model and why does it exist?
The JMM is the specification that defines how threads interact through memory. It exists because modern CPUs have per-core caches and reorder instructions for performance — without a memory model, multi-threaded programs would behave non-deterministically across hardware. The JMM defines happens-before: a partial order on actions that guarantees visibility and ordering when the programmer establishes the relationship via synchronised, volatile, thread start/join, etc. Without happens-before, the JVM is free to show different values to different threads for the same variable.
Practical
Two threads share a variable. Thread A writes it, Thread B reads it. What ensures Thread B sees Thread A's write?
A happens-before relationship between the write and the read. Options: (1) synchronized — A writes inside synchronized(lock), B reads inside synchronized(lock). Monitor unlock (A) hb monitor lock (B). (2) volatile — A writes volatile field; B reads same field. Volatile write hb volatile read. (3) Thread.start() — if A sets the variable before starting thread B, start() creates happens-before. (4) Thread.join() — if A completes and main thread joins A, then reads: join() creates happens-before. Without one of these, Thread B may see a stale value — this is not a bug, it's permitted by the spec.

ConcurrentHashMap vs synchronizedMap
Segment locking · CAS on buckets · full map lock · when to use each
Definition

ConcurrentHashMap allows concurrent reads and fine-grained concurrent writes without a global lock. Collections.synchronizedMap wraps any map with a single synchronized mutex — every read and write acquires the same lock, making all operations sequential. At any meaningful concurrency level, ConcurrentHashMap wins on throughput by a large margin.

ConcurrentHashMap Java 8+ internals
1
Structure — same as HashMap: array of buckets, each bucket is a linked list (or Red-Black tree when chain ≥ 8). But locking is per-bucket.
2
Reads — entirely lock-free. Values and next-pointers are volatile. A read traverses the bucket list without acquiring any lock. Reads always see the latest committed write (volatile guarantees).
3
Writes — if the bucket is empty: CAS to set the first node (no lock). If the bucket has nodes: synchronized on the bucket head node (not the whole map). Two writes to different buckets execute truly concurrently.
4
Size — maintained via CounterCell array (like LongAdder) to avoid contention on a single counter. size() sums all cells — slightly stale but O(1) amortised.
!
Null keys/values forbidden — in ConcurrentHashMap you can't distinguish "key not present" from "key maps to null" in a concurrent context (check-then-act is not atomic). HashMap allows nulls; CHM doesn't. This is intentional.
5
Atomic compound operations — putIfAbsent(k, v), computeIfAbsent(k, fn), merge(k, v, fn), compute(k, fn) are all atomically executed per-bucket. Use these instead of get-then-put patterns which have race conditions even with a CHM.
Code — safe patterns and anti-patterns
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>(); // WRONG — not atomic even with ConcurrentHashMap if (!map.containsKey("key")) { map.put("key", 1); // race: another thread could put between check and put } // RIGHT — atomic putIfAbsent map.putIfAbsent("key", 1); // Word frequency counter — classic pattern map.merge(word, 1, Integer::sum); // atomic: if absent put 1; else fn(old, 1) // Compute pattern — build value on first access map.computeIfAbsent("group", k -> new CopyOnWriteArrayList<>()) .add("member"); // WRONG — check-then-act still has race Integer val = map.get("counter"); if (val != null) map.put("counter", val + 1); // race between get and put! // RIGHT — use compute map.compute("counter", (k, v) -> (v == null) ? 1 : v + 1); // synchronizedMap — needs external lock for iteration! Map<String,String> sm = Collections.synchronizedMap(new HashMap<>()); synchronized(sm) { // must hold lock for iteration for (String k : sm.keySet()) { ... } }

Interview questions
Classic
How is ConcurrentHashMap different from Hashtable and synchronizedMap?
Hashtable and synchronizedMap: single global lock on every operation — reads and writes all serialised. No concurrency at all. ConcurrentHashMap Java 8+: reads are fully lock-free (volatile fields). Writes lock only the affected bucket (one of 128+ buckets typically). Two concurrent writes to different keys almost always proceed without contention. Result: ConcurrentHashMap has near-linear scalability with reader threads and near-linear scaling for writer threads up to the number of buckets. Hashtable/synchronizedMap are effectively single-threaded despite being "thread-safe".
Gotcha
Why doesn't ConcurrentHashMap allow null keys or values?
Because get(key) returning null is ambiguous in a concurrent context: did the key not exist, or does it map to null? In a sequential map you can call containsKey() then get() safely. In a concurrent context, the key could be removed between those two calls. So ConcurrentHashMap prohibits null to keep the API unambiguous — if get() returns null, the key was definitely not present at the moment of the call. HashMap allows null because it's not designed for concurrent use, and the check-then-act race doesn't matter if you're the only thread.
