Single Responsibility Principle
S in SOLID
A class should have one, and only one, reason to change
Definition

A class should have one, and only one, reason to change. Robert C. Martin defines "reason to change" as a stakeholder or actor whose requirements could force a modification. If two different actors can demand a change to the same class — that class violates SRP.

One responsibility
The class does one thing well. Changes to that one concern only require touching this class. Easy to name, test, and reason about.
Multiple responsibilities
Changes to any one concern require touching the class. Changes intended for one actor can break behaviour for another. The class is hard to name without "and".
The real meaning — "reason to change" is about actors, not functions
1
SRP is NOT "a class should only do one thing" in the trivial sense. A UserService can have 10 methods — that's fine if they all serve the same business actor (user management). The question is: whose requirements can force this class to change?
2
Classic violation: a Report class with generateReport() AND formatForEmail() AND saveToDatabase(). Three actors: the analytics team, the marketing team, and the persistence team. Any one of them changing requirements forces you into this class, risking breaking the others.
3
The tell-tale sign: if you struggle to name the class without using "and" — UserParserAndValidator, OrderServiceAndEmailer — SRP is violated. A class named with "and" is confessing to two responsibilities.
4
In Spring Boot: the three-layer architecture (Controller → Service → Repository) enforces SRP at the architectural level. Controllers handle HTTP concerns, services handle business logic, repositories handle persistence. Each layer has one actor: the web team, the business team, the DBA team.
Violation vs fix — the OrderProcessor antipattern
Violation — one class, three actors
class OrderProcessor {
  void processOrder(Order o) { validate(); charge(); saveToDb(); sendEmail(); }
  void validate() { /* business rules team */ }
  void charge() { /* payments team */ }
  void saveToDb() { /* DBA team */ }
  void sendEmail() { /* marketing team */ }
}
↓ split by actor
Fixed — one class per actor
class OrderValidator { void validate(Order o) { ... } }
class PaymentService { void charge(Order o) { ... } }
class OrderRepository { void save(Order o) { ... } }
class OrderNotifier { void sendConfirmation(Order o) { ... } }
class OrderOrchestrator {
  // orchestrates — depends on the four above
}
Code — Spring Boot SRP in practice
// VIOLATION — UserService does everything @Service public class UserService { void register(UserDto dto) { validate(dto); // validation logic String hash = hash(dto.password()); // security logic User u = repo.save(toEntity(dto, hash)); // persistence emailService.sendWelcome(u.email()); // notification analytics.track("USER_REGISTERED", u.id()); // analytics } // changes for: validation rules, hashing algo, DB schema, email template, analytics schema // = 5 reasons to change in one class } // SRP-COMPLIANT — each class has one reason to change @Component public class UserRegistrationValidator { // actor: product team public void validate(UserDto dto) { /* rules */ } } @Component public class PasswordHasher { // actor: security team public String hash(String raw) { /* bcrypt */ return null; } } @Service public class UserRegistrationService { // actor: business team, orchestrates UserRegistrationValidator validator; PasswordHasher hasher; UserRepository repo; WelcomeEmailSender emailer; public User register(UserDto dto) { validator.validate(dto); String hash = hasher.hash(dto.password()); return repo.save(toEntity(dto, hash)); // emailer called async via @EventListener on UserRegisteredEvent } }
Spring Boot enforces SRP architecturally
@
@RestController — handles HTTP. Parses request, validates input format, delegates to service, maps response. Zero business logic. Reason to change: API contract.
@
@Service — business logic only. No HTTP, no SQL, no email formatting. Reason to change: business rules.
@
@Repository — persistence only. SQL, JPA, query logic. No business rules, no HTTP. Reason to change: schema or query optimisation.
!
The most common SRP violation in Spring Boot: business logic creeping into controllers (if (dto.amount() > 1000) applyTax()) or SQL being written in services. Keep the layers clean.
Interview questions
Classic
What does "one reason to change" mean? How do you identify SRP violations?
A "reason to change" = a stakeholder whose requirements could force a modification. SRP is violated when multiple different stakeholders can demand changes to the same class. Identify violations by: (1) the "and" test — if the class name needs "and", it has multiple responsibilities. (2) the actor test — list every person or team that might ask you to change this class. If more than one actor appears, split. (3) Change frequency — if different methods change for unrelated reasons, it's a sign of multiple responsibilities. Production signal: a class that's always in git diff no matter which feature is being built.
Spring
How does the three-layer Spring Boot architecture implement SRP?
The Controller layer handles HTTP concerns — its single reason to change is the API contract. The Service layer handles business logic — its reason to change is business rules. The Repository layer handles persistence — its reason to change is the data schema or query optimisation. Each layer depends only on the layer below, never on the layer above. This creates clean separation of actors: the frontend team drives controllers, the product team drives services, the DBA team drives repositories. No single class has to change for all three actors simultaneously.

Open/Closed Principle
O in SOLID
Software entities should be open for extension, closed for modification
Definition

Software entities (classes, modules, functions) should be open for extension — new behaviour can be added — but closed for modification — adding new behaviour should not require changing existing, tested code. New requirements should be met by writing new code, not editing old code.

Violation signal
Every time you add a new feature, you open an existing class and add another if/else or switch case. The class grows with every new requirement.
Correct approach
Define an abstraction (interface). Add new behaviour by creating a new implementation class. Existing code never touched — existing tests pass unchanged.
How to achieve OCP — the two mechanisms
1
Abstraction + polymorphism (runtime) — define an interface. Client code depends on the interface. New behaviours = new implementations. This is the Strategy, Factory, and Observer pattern in action. The client is "closed" because it never changes. The system is "open" because you add new implementations freely.
2
Template Method (compile-time) — base class defines the skeleton (closed). Subclasses override specific steps (open for extension). Same idea via inheritance. Java's HttpServlet is the template method; your doGet() override is the extension point.
!
OCP doesn't mean "never modify code". It means: for the anticipated axes of variation, design so those variations don't require modifying existing logic. You can't predict all change, but you can identify the most likely axes (e.g. "we'll add more payment providers") and protect against those.
The payment provider problem — violation vs fix
Violation — adding Razorpay means modifying PaymentService
class PaymentService {
  void process(Order o, String provider) {
    if ("STRIPE".equals(provider)) { stripeCall(); }
    else if ("PAYPAL".equals(provider)) { paypalCall(); }
    // Adding Razorpay = open this class = risk regression
  }
}
↓ extract abstraction
Fixed — adding Razorpay = new class, zero modification
interface PaymentGateway { Receipt charge(Order o); }
class StripeGateway implements PaymentGateway { ... }
class PaypalGateway implements PaymentGateway { ... }
class RazorpayGateway implements PaymentGateway { ... } // NEW — no edits

class PaymentService {
  private final PaymentGateway gateway; // injected — never changes
  Receipt process(Order o) { return gateway.charge(o); }
}
Code — OCP with Strategy + Spring DI
// Interface = extension point (the "open" seam) public interface DiscountPolicy { double apply(double basePrice, Order order); } // Existing implementations — NEVER touched when adding new policies @Component("noDiscount") class NoDiscountPolicy implements DiscountPolicy { public double apply(double p, Order o) { return p; } } @Component("seasonalDiscount") class SeasonalDiscountPolicy implements DiscountPolicy { public double apply(double p, Order o) { return p * 0.85; } } // New requirement: add flash sale — ZERO modification to existing code @Component("flashSale") class FlashSalePolicy implements DiscountPolicy { public double apply(double p, Order o) { return p * 0.6; } } // Consumer — closed for modification @Service class PricingService { private final Map<String, DiscountPolicy> policies; // Spring injects all double price(Order o, String policyName) { return policies.getOrDefault(policyName, policies.get("noDiscount")) .apply(o.basePrice(), o); // Adding a new policy never touches this method } }
Interview questions
Classic
What does "closed for modification" actually mean in practice?
It means existing, tested code should not need to be edited to support new requirements along anticipated axes of variation. "Closed" refers to the class's interface and existing behaviour — callers don't need to change. "Open" refers to the implementation's internal extension points — new behaviour is added by writing new code (new implementations, new subclasses) not by editing existing logic. The if-else chain that grows with every feature is the canonical violation — each addition modifies existing logic, which risks breaking existing tests and behaviour.
Design
How do you apply OCP without over-engineering?
Don't apply OCP speculatively to everything — that leads to interface explosion. Apply it reactively or when you can clearly identify the axis of variation. The rule of three: the first time you implement something, just do it. The second time you add a similar case, note the pattern. The third time, refactor to an abstraction. In Spring: every time you find yourself adding an if-else to a service based on a string/enum type parameter, that's the signal to extract a strategy interface and inject implementations by name or via a factory.

Liskov Substitution Principle
L in SOLID
Subtypes must be substitutable for their base types without breaking correctness
Definition

If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering the correctness of the program. A subclass must honour the contract of its parent. You must be able to use a subclass anywhere a parent is expected — without the caller needing to know it's a subclass.

The contract a subtype must honour
Pre
Preconditions cannot be strengthened — subclass can't require MORE than the parent required
Post
Postconditions cannot be weakened — subclass must deliver at LEAST what the parent promised
Inv
Invariants must be preserved — subclass cannot break the parent's guarantees about its state
The real meaning — behavioural substitutability, not just type hierarchy
1
LSP is about behaviour, not just type signatures. A Square IS-A Rectangle in mathematics. But in code, if Rectangle.setWidth(w) is expected to keep height unchanged, then Square.setWidth(w) (which also changes height to maintain equal sides) breaks LSP — even though the type hierarchy is technically valid.
2
LSP violation detection: if you find instanceof checks or if (shape instanceof Square) in code that calls a method on a Rectangle — that's the caller defending against broken substitutability. The client code should never need to know the concrete type.
3
The practical rule: if overriding a method requires adding a condition the parent doesn't have, or throwing an exception the parent doesn't throw, or returning values outside the parent's promised range — it's an LSP violation. The fix is usually to not use inheritance and instead use composition, or to redesign the abstraction.
Classic violations — square/rectangle + unsupported operations
Violation 1 — Square extends Rectangle
class Rectangle { setWidth(w) { this.w=w; } setHeight(h) { this.h=h; } }
class Square extends Rectangle {
  // LSP broken: setting width also sets height
  setWidth(w) { this.w=w; this.h=w; } // postcondition: h stays same — VIOLATED
}
// Client: rect.setWidth(5); rect.setHeight(3); assert area()==15; // FAILS for Square
Violation 2 — throwing UnsupportedOperationException
class ImmutableList extends ArrayList {
  void add(E e) { throw new UnsupportedOperationException(); }
// Violates: callers of List.add() expect it to work
// Collections.unmodifiableList() has the same issue — use with care
}
Fix — separate the abstraction
interface Shape { double area(); } // no setWidth/setHeight
class Rectangle implements Shape { ... } // own setters
class Square implements Shape { ... } // own setters
// Neither is a subtype of the other — no LSP issue
Code — LSP violation + correct design
// VIOLATION — PremiumBird violates Bird's fly() contract class Bird { void fly() { System.out.println("flying"); } } class Penguin extends Bird { void fly() { throw new UnsupportedOperationException("Penguins can't fly"); } // Caller: bird.fly() — works for Sparrow, throws for Penguin. LSP violated. } // FIX — redesign the hierarchy by capability interface Bird { void eat(); } interface Flyable { void fly(); } interface Swimmable { void swim(); } class Sparrow implements Bird, Flyable { ... } // can fly class Penguin implements Bird, Swimmable { ... } // can't fly, swims // Callers of Flyable only get birds that can fly — no surprise exception // Production LSP check — does your Spring @Repository subtype behave consistently? // If CachedUserRepository extends UserRepository but returns stale data // without callers knowing — that's a behavioural LSP violation // Fix: CachedUserRepository implements UserRepository (not extends) // and clearly documents its cache semantics in the interface contract
Interview questions
Classic
Explain the Square-Rectangle LSP violation.
Rectangle has setWidth(w) that keeps height unchanged, and setHeight(h) that keeps width unchanged. Square inherits from Rectangle but overrides setWidth to also set height (squares have equal sides). A caller that creates a Rectangle, sets width=5, height=3, and asserts area==15 will fail when given a Square — setting width to 5 also sets height to 5. The postcondition of Rectangle.setWidth (height unchanged) is violated by Square. The fix: don't model this with inheritance. Both Rectangle and Square should implement a Shape interface with area() but neither should be a subtype of the other — they have fundamentally different setter semantics.
Detection
How do you detect LSP violations in code reviews?
Key signals: (1) instanceof checks — if client code does if (animal instanceof Penguin) to avoid calling fly(), the abstraction is broken. (2) Overriding methods that throw UnsupportedOperationException — the base type promised this would work. (3) Overriding with stronger preconditions — "parent accepts any non-null string; subclass requires it to be numeric." (4) Methods that return null or empty when the parent's contract says it always returns a value. (5) Subclass methods that are no-ops for capabilities the parent advertised. Any of these means clients can't safely substitute the subtype — the abstraction is leaking.

Interface Segregation Principle
I in SOLID
Clients should not be forced to depend on methods they do not use
Definition

No client should be forced to depend on interfaces it does not use. Prefer many small, specific interfaces over one large general-purpose interface. When a class is forced to implement methods it doesn't need, changes to those unused methods force recompilation and retesting of the class — unnecessary coupling.

Fat interface
One interface with 10 methods. Implementors must provide all 10 even if they only need 3. Unused methods get empty stubs or UnsupportedOperationException — which also violates LSP.
Segregated interfaces
Multiple focused interfaces, each representing a specific capability. Classes implement only the interfaces for capabilities they actually have. Callers depend only on the methods they actually call.
The fat interface problem — Worker + Robot
Violation — Robot forced to implement eat()
interface Worker { void work(); void eat(); void sleep(); }
class Human implements Worker { ... } // all methods make sense
class Robot implements Worker {
  void work() { ... } // fine
  void eat() { throw new UnsupportedOperationException(); } // nonsense
  void sleep(){ throw new UnsupportedOperationException(); } // nonsense
// Robot is forced to depend on methods it will never use
}
↓ split by capability
Fixed — each interface = one capability
interface Workable { void work(); }
interface Feedable { void eat(); }
interface Restable { void sleep(); }

class Human implements Workable, Feedable, Restable { ... }
class Robot implements Workable { ... } // only what it can do
// WorkScheduler depends only on Workable — works with both
Code — ISP in Spring Boot repository pattern
// FAT INTERFACE — read-only services forced to compile against write methods interface UserRepository { User findById(String id); List<User> findAll(); void save(User u); void delete(String id); void bulkImport(List<User> users); void reindex(); // admin-only, forces all implementors to know this } // ISP-COMPLIANT — segregated by actor/capability interface UserReadRepository { User findById(String id); List<User> findAll(); } interface UserWriteRepository { void save(User u); void delete(String id); } interface UserAdminRepository { void bulkImport(List<User> users); void reindex(); } // Service dependencies are precise — no accidental write in read service @Service class UserSearchService { private final UserReadRepository repo; // can ONLY read } @Service class UserManagementService { private final UserReadRepository reader; private final UserWriteRepository writer; // explicit write access } // JpaUserRepository implements all three — one concrete class, multiple interfaces @Repository class JpaUserRepository implements UserReadRepository, UserWriteRepository, UserAdminRepository { ... }
Interview questions
Classic
Why is a "fat interface" a problem even if the client only calls some methods?
Two problems: (1) Forced coupling — if the fat interface changes a method the client doesn't use, the client still recompiles. In large systems, this ripples through unnecessary rebuild chains. (2) Implementation burden — classes implementing the fat interface must provide bodies for all methods, even irrelevant ones, often via empty stubs or UnsupportedOperationException (which also violates LSP). Segregating interfaces means: each client depends only on what it uses, each implementor only provides what it supports, and changes to one capability don't force recompilation of unrelated clients.
CQRS
How does ISP relate to the CQRS pattern?
CQRS (Command Query Responsibility Segregation) is ISP applied at the architectural level. The "Query" side of the system depends on read interfaces; the "Command" side depends on write interfaces. They're segregated so that the read model (potentially a materialized view or read replica) can change independently of the write model. A fat UserRepository that mixes reads and writes is the exact ISP violation CQRS solves at scale. In code: a UserReadRepository and UserWriteRepository are the ISP-compliant building blocks that CQRS formalises.

Dependency Inversion Principle
D in SOLID
High-level modules must not depend on low-level modules — both should depend on abstractions
Definition

High-level modules (business logic) should not depend on low-level modules (infrastructure, persistence, messaging). Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions. This is the principle that makes Spring's dependency injection architecturally sound.

The dependency direction before and after inversion

Without DIP — business depends on infrastructure

OrderService
  → MySqlOrderRepo (concrete!)
  → SendGridMailer (concrete!)
  → StripeGateway (concrete!)

Changing DB requires editing OrderService

With DIP — both depend on abstractions

OrderService
  → OrderRepository (interface)
  → EmailSender (interface)
  → PaymentGateway (interface)
↑ MySqlOrderRepo implements OrderRepository

Swap DB by swapping the implementation

What "inversion" means — the direction of dependency flips
1
In traditional layered code: high-level (business) creates instances of low-level (persistence). OrderService does new MySqlOrderRepository(). The dependency arrow points from business to infrastructure. Business knows the concrete type.
2
After inversion: OrderService depends on OrderRepository (interface). MySqlOrderRepository implements OrderRepository — the low-level module now depends on the abstraction, not the other way around. The dependency arrow from infrastructure to the interface is the "inversion".
3
Who resolves the abstraction to a concrete type? An external injector — Spring's IoC container, a factory, or the main method. This is the boundary between application code and configuration. Business logic is clean of all "which implementation" decisions.
!
DIP ≠ Dependency Injection. DIP is the principle. Dependency Injection is a technique to implement it. You can have DI without DIP (injecting concrete types). And you can satisfy DIP without a DI framework (factory pattern achieves the same dependency direction).
Code — without DIP vs with DIP + Spring
// WITHOUT DIP — high-level depends on low-level concrete class OrderService { private final MySqlOrderRepository repo; // concrete class! private final SendGridEmailer mail; // concrete class! OrderService() { repo = new MySqlOrderRepository(); // creates its own — untestable mail = new SendGridEmailer(); } // To test, you MUST have a MySQL DB and SendGrid credentials } // WITH DIP — depends on abstractions, injected from outside interface OrderRepository { Order save(Order o); Optional<Order> findById(String id); } interface EmailSender { void sendConfirmation(Order o); } @Service class OrderService { private final OrderRepository repo; // abstractions only private final EmailSender mail; // Spring injects the concrete implementations OrderService(OrderRepository repo, EmailSender mail) { this.repo = repo; this.mail = mail; } public Order place(Order o) { Order saved = repo.save(o); mail.sendConfirmation(saved); return saved; } } // In tests: inject mock implementations — zero DB, zero email class OrderServiceTest { OrderRepository mockRepo = Mockito.mock(OrderRepository.class); EmailSender mockMail = Mockito.mock(EmailSender.class); OrderService svc = new OrderService(mockRepo, mockMail); // Full business logic tests, no infrastructure needed }
How Spring DI implements DIP
1
@Autowired / constructor injection — Spring sees that OrderService needs an OrderRepository. It scans for a bean that implements that interface, creates it, and injects it. The service never calls new. The IoC container is the "external injector" that resolves abstractions to concretions.
2
Constructor injection (preferred) — Spring Boot 2.1+ does constructor injection without @Autowired annotation if there's one constructor. This makes dependencies explicit, enables immutable fields, and makes unit testing trivial (just pass mocks to the constructor).
3
Profiles for swapping implementations — @Profile("test") InMemoryOrderRepository vs @Profile("production") JpaOrderRepository. Same interface, different impl. The service code is identical in both profiles — DIP makes this zero-effort.
Interview questions
Classic
What is the difference between Dependency Inversion and Dependency Injection?
Dependency Inversion is the design principle: high-level modules should not depend on low-level modules; both should depend on abstractions. It's about the direction of dependency arrows in your design. Dependency Injection is a technique (one of several) to implement DIP: instead of a class creating its dependencies, they're provided from outside (via constructor, setter, or field). Spring's IoC container is a DI framework. You can satisfy DIP without Spring (using factories or manual construction). You can also use DI badly — injecting concrete classes rather than interfaces — which implements the technique but violates the principle.
Testing
How does DIP make unit testing possible?
When a class creates its own dependencies (new MySqlRepo()), you can't test it without those dependencies running — you need a real database. When a class receives its dependencies via interfaces (DIP + DI), you can inject test doubles (mocks, stubs) that don't need infrastructure. Mockito.mock(OrderRepository.class) creates an in-memory mock that satisfies the interface. The business logic is tested in complete isolation — fast, deterministic, no I/O. DIP is the principle that makes mocking meaningful: because the dependency is an abstraction, any conforming implementation (including a mock) can be substituted.

DRY · KISS · YAGNI
Three principles every senior engineer internalises — misapplication is as dangerous as violation
DRY — Don't Repeat Yourself

Every piece of knowledge must have a single, unambiguous, authoritative representation in a system. When you change a rule, you change it in one place. DRY is about knowledge duplication, not just code duplication.

!
DRY is about knowledge, not just literals. Two methods that look different but encode the same business rule — that's a DRY violation even if the code doesn't look like copy-paste. Two methods that look similar but represent genuinely different concepts are NOT a DRY violation.
E
Example: MAX_RETRY_COUNT = 3 defined in three different config files. When the team decides to change it to 5, they must find and update all three. That's the DRY violation — the "knowledge" (max retries) has three representations. Fix: extract to a single constant or configuration property.
!
Misapplication: creating a shared utility function for two methods that happen to look the same but represent different concepts. When one needs to change, you're forced to break the other or add a flag parameter. This is "Wrong Abstraction" — bad DRY. The rule: if removing the duplication requires adding parameters to distinguish cases, the methods aren't actually the same knowledge.
KISS — Keep It Simple, Stupid

Systems work best when they're kept simple. Avoid unnecessary complexity. The simplest solution that correctly solves the problem is the right solution. Complexity is the enemy of reliability and maintainability.

!
Simple ≠ easy. Simple means: one concept per unit, minimal state, minimal dependencies, obvious behaviour. A 5-line method can be complex (doing too much). A 30-line method can be simple (one clear thing, well-named).
1
KISS violations in Java: unnecessary inheritance chains, abstract classes with one subclass, factory patterns for a single type, 6-layer deep object hierarchies for a simple value holder. When a junior dev asks "why is this so complicated?" — KISS was violated.
2
The simplicity test: can a developer who didn't write this code understand what it does in under 2 minutes? If not, simplify. Good naming + simple structure > clever abstraction.
YAGNI — You Aren't Gonna Need It

Don't add functionality until it's needed. Implement only what you need now, not what you think you might need in the future. Speculative features add complexity without adding value — they're maintenance burden without ROI.

!
YAGNI violations: building a plugin system when you have one plugin, designing a multi-tenant schema when you have one tenant, adding a "mode" parameter to a method that will always be called with one value, abstracting a DB that will never change.
!
YAGNI isn't "never design for change." It means: don't pay the complexity cost today for a benefit that may never arrive. Do design obvious seams (like depending on interfaces per DIP) — that's cheap. Don't build elaborate plugin frameworks speculatively.
R
The economic argument: code you write today costs testing, documentation, maintenance, and understanding. Future requirements almost always differ from what you predicted. Writing YAGNI code often means throwing it away and rewriting when the real requirement arrives.
Misapplication — the traps senior engineers avoid
DRY misapplication — wrong abstraction
// calculateShippingCost and calculateTaxAmount look similar — DRY them?
// NO — they're different knowledge that happen to share a formula shape.
// When shipping rules change, you must not change tax calculation.
// "Duplication is far cheaper than the wrong abstraction" — Sandi Metz
KISS misapplication — "simple" at the wrong level
// Putting everything in one class is "simple" at the file level
// but complex at the understanding level.
// KISS operates on cognitive load, not line count.
YAGNI vs good engineering tension
// "We'll only ever have one payment provider" — YAGNI says: don't abstract.
// But DIP says: depend on an interface because swapping is cheap to set up.
// Resolution: create the interface (cheap seam) but don't build a plugin system.
// YAGNI prevents WORK, not STRUCTURE.
Code — DRY done right, DRY done wrong
// DRY VIOLATION — business rule duplicated in 3 places class OrderController { if(amount > 10000) applyGST(); } class InvoiceService { if(amount > 10000) addTaxLine(); } class ReportGenerator { if(amount > 10000) flagForAudit(); } // threshold 10000 is knowledge — lives in 3 places // DRY FIX — single source of truth class TaxPolicy { private static final long GST_THRESHOLD = 10_000L; public static boolean requiresGST(long amount) { return amount > GST_THRESHOLD; } } // Change: edit TaxPolicy.GST_THRESHOLD in one place // WRONG DRY — two different concepts forced into one method double calculate(boolean isShipping, double value) { return isShipping ? value * SHIPPING_RATE : value * TAX_RATE; } // "Duplication is far cheaper than the wrong abstraction" // CORRECT — separate methods for genuinely different knowledge double calculateShipping(double value) { return value * SHIPPING_RATE; } double calculateTax(double value) { return value * TAX_RATE; } // YAGNI violation — building for a "future" we're not sure about interface NotificationPlugin { void send(Event e); } class NotificationPluginRegistry { /* entire plugin system for 1 notification type */ } // YAGNI fix: just send the email. Refactor to a plugin when you have N>1 plugins.
Interview questions
DRY
Is it always wrong to have duplicate code? When is duplication acceptable?
Duplication of code that represents the same knowledge is wrong. Duplication of code that looks similar but represents different knowledge is fine and often preferable. Sandi Metz: "duplication is far cheaper than the wrong abstraction." The signal: if you need to add an if-statement or a boolean parameter to a shared function to distinguish between two use cases, the two use cases are not actually the same knowledge — they just look similar today. Premature deduplication creates accidental coupling. Also: in tests, some repetition makes tests clearer and more isolated — don't blindly DRY test setup code.
Tension
How do you balance YAGNI with good design principles like DIP?
YAGNI prevents unnecessary WORK, not necessary STRUCTURE. Depending on an interface (DIP) is cheap structural work that pays dividends immediately — it enables testing, enables clean layering. Building an entire plugin framework speculatively is YAGNI work that costs now for uncertain future benefit. The guideline: do the cheap, low-cost structural decisions (interfaces, clear naming, small methods) always. Defer expensive functionality (plugin systems, multi-tenant architecture, elaborate caching layers) until you have a concrete requirement for it.
Code Smells
Long method · God class · Primitive obsession · Shotgun surgery · Feature envy · Data clumps
What are code smells?

A code smell is a surface symptom that indicates a deeper problem. Smells don't break code — they indicate design problems that will cause bugs, make changes expensive, and accumulate technical debt. They're called "smells" because they alert you something is wrong before you know exactly what.

Smells are signals, not absolute rules
A long method isn't always wrong. A God class might be acceptable in certain contexts. The smell tells you: investigate here. The diagnosis tells you: here's why it's a problem.
Smells and SOLID
Most code smells are symptoms of SOLID violations: God class = SRP violation. Divergent change = SRP + OCP. Refused bequest = LSP. Feature envy = wrong responsibility distribution.
6 critical smells — what they are, what they signal, how to fix
1
Long method — a method that's hard to name in one phrase, that does multiple things, that requires comments to explain different sections. Signal: SRP violation inside a method. Fix: extract method. Name each extracted method after its intention, not its implementation. Rule of thumb: if you need a comment before a code block, that block is a method waiting to be extracted.
2
God class — one class that knows everything and does everything. Thousands of lines, dozens of unrelated methods. Signal: SRP violated at class scale. Fix: find the different "actors" whose requirements drive changes to this class. Extract a class per actor. Common in legacy services that accreted behaviour over years: UserService that now handles registration, authentication, billing, preferences, analytics, and audit logging.
3
Primitive obsession — using primitives (String, int, long) to represent domain concepts that deserve their own type. String email everywhere instead of Email email. long amountInPaisa instead of Money amount. Signal: domain logic that belongs in a class leaks into every caller. Fix: introduce value objects — small, immutable classes with validation in the constructor and domain methods on them.
4
Shotgun surgery — making one logical change requires editing many unrelated classes. Adding a new notification channel means touching 6 different files. Signal: OCP violation — the axis of variation isn't protected by an abstraction. Fix: consolidate the scattered logic behind an interface. One class, one responsibility. Adding the notification channel = adding one class.
5
Feature envy — a method in class A spends most of its time accessing data and methods from class B. The method wants to be in class B. Signal: wrong responsibility distribution. Fix: move the method to the class whose data it uses most. Ask: "whose data does this method need most?" — that's where it belongs.
6
Data clumps — three or more data items that always appear together: (String street, String city, String pincode) passing around as separate params everywhere. Signal: these belong in a class. Fix: introduce a class. Address address — now you have a cohesive object with validation and formatting methods.

Interview questions
Classic
What is primitive obsession and how do you fix it?
Primitive obsession is using primitives (String, int, long) to represent domain concepts that have their own rules and behaviours. A String email has no inherent validation or formatting — validation logic gets duplicated everywhere it's used. Fix: introduce value objects. Create an Email class (or record in Java 16+) with validation in the constructor. Now email is always valid by construction, normalisation happens once, and you can add domain methods like domain(), isDisposable() on the object. The benefit: the type system enforces correctness — a method signature taking Email instead of String is self-documenting and compiler-enforced.
Detection
How do you identify a God class in a Spring Boot codebase and what's your remediation approach?
Signals: a @Service class over ~400 lines, a class with more than 5-7 @Autowired dependencies, a class whose methods span unrelated concerns (authentication + order processing + reporting), a class that's always in the git diff regardless of which feature is being worked on. Remediation: (1) list every method in the class. (2) Group methods by actor — who would request a change to this method? (3) Extract a new service per actor group. (4) Use events (@ApplicationEvent) to decouple the extracted services instead of direct calls. (5) Extract value objects for primitive obsessions found during the process. This typically takes 2-3 iterations to stabilise.

SOLID — the full map
How the five principles connect, reinforce, and interact
How the principles reinforce each other
S+D
SRP → DIP: When you split a God class by responsibility (SRP), you naturally create seams. Those seams are the abstraction boundaries DIP says to depend on. SRP creates the motivation; DIP determines the dependency direction.
O+D
OCP + DIP: OCP says "depend on abstractions so you can extend without modifying." DIP says "high-level modules depend on abstractions." Both demand the same thing: an interface sits between the caller and the concrete type. They're two perspectives on the same design choice.
L+I
LSP + ISP: A fat interface (ISP violation) forces implementors to implement methods they don't need — often via UnsupportedOperationException — which violates LSP (callers can't substitute the subtype). ISP fixes the interface; LSP is the symptom that tells you ISP was violated.
All
All five together: SRP defines clean units. OCP protects them from change-driven modification. LSP ensures subtypes are trustworthy. ISP keeps dependencies precise. DIP keeps business logic clean of infrastructure. A codebase that honours all five is testable, extensible, and maintainable — the three outcomes SOLID aims for.
Quick reference card
S — SRPOne reason to change. If you need "and" to name it — split it.
O — OCPNew behaviour = new class. No editing existing tested code.
L — LSPSubtypes must honour the parent's contract. No surprise exceptions.
I — ISPClients depend only on what they use. No fat interfaces.
D — DIPBusiness depends on interfaces. Infrastructure implements them.
Spring Boot — where each principle shows up
S
Controller/Service/Repository layer split. Each @Component class serves one concern. EventListener for cross-cutting concerns instead of direct calls.
O
@Bean with interface type. Spring injects all implementations of an interface into a Map. Add a new implementation = register a @Component. Zero modification to the consumer.
L
Any class you mock in a @MockBean must honour its interface contract. If your mock works but the real impl throws — the real impl violates LSP.
I
Spring Data: CrudRepository vs JpaRepository. Your service injects only CrudRepository if it only needs save/findById. Doesn't need to know about flush(), merge(), etc.
D
Constructor injection of interfaces. @Profile("test") InMemoryImpl vs @Profile("prod") JpaImpl. The service never knows which it gets — that's DIP working.
Master interview questions
Classic
Walk me through all 5 SOLID principles with one cohesive example.
Using a payment processing service: SRP — PaymentService only orchestrates; PaymentValidator, PaymentNotifier, PaymentRepository are separate classes, each with one reason to change. OCP — PaymentGateway is an interface; adding Razorpay = new RazorpayGateway class, zero modification to PaymentService. LSP — RazorpayGateway honours the same contract as StripeGateway; calling charge() always returns a Receipt or throws PaymentException, never UnsupportedOperationException. ISP — PaymentService depends on PaymentGateway (charge + refund); ReportService depends on PaymentReadRepository (queries only) — not the full repository with write methods. DIP — PaymentService depends on PaymentGateway interface; the Spring container injects StripeGateway or RazorpayGateway based on config. Unit tests inject a mock gateway — zero network calls.
Trade-offs
Can following SOLID principles make code worse? When do you stop?
Yes. Over-application creates: interface explosion (an interface for everything, even things that will never have two implementations), premature abstraction (extracting classes for variations that never materialise), indirection fatigue (10 files to trace one request). SOLID is a set of principles, not rules. Apply them when: (1) you see concrete pain — tests are hard to write, changes ripple unexpectedly, classes are hard to understand. (2) You can identify a real axis of variation. The cost of not applying SOLID compounds over time. The cost of over-applying it is immediate complexity. The senior engineer judgment is knowing which cost is higher in the given context — and YAGNI is your guide for when NOT to apply them.
