The 4 OOP Pillars
Encapsulation · Inheritance · Polymorphism · Abstraction
Definition

OOP organises code around objects — bundles of state (fields) and behaviour (methods) — rather than procedures. Four principles govern how objects are designed, related, and replaced.

Encapsulation
Hide internal state. Expose only what callers need via a controlled public API. The object owns its own invariants — callers can't corrupt them.
Inheritance
A subclass acquires fields and methods from a superclass. Enables code reuse and "is-a" relationships. In Java, single-class inheritance only.
Polymorphism
One interface, many implementations. A reference of type Shape can hold a Circle or Rectangle — the correct method fires at runtime (dynamic dispatch).
Abstraction
Expose what an object does, hide how. Achieved via abstract classes and interfaces. Callers program to a contract, not a concrete type.
How it works internally
1
Encapsulation — the compiler enforces access modifiers (private, protected, public). The JVM itself doesn't care — it's a compile-time contract. Reflection can bypass it at runtime (deliberate escape hatch).
2
Inheritance — the JVM stores a reference to the parent class in the class metadata. Field lookup walks up the hierarchy. Method dispatch uses the virtual method table (vtable) — a per-class array of method pointers built at class-load time.
3
Polymorphism / dynamic dispatch — when you call shape.draw(), the JVM looks up the vtable of the actual object's class at runtime (not the reference type). This is the invokevirtual bytecode instruction. Costs one extra pointer dereference — negligible unless in a tight loop with many types (JIT's megamorphic call site problem).
4
Abstraction — abstract methods have no bytecode body; the JVM will throw AbstractMethodError if an unimplemented abstract class is somehow instantiated. Interfaces add an interface table (itable) lookup because a class can implement many interfaces.
Code example — all 4 pillars in one slice
// ABSTRACTION — contract only, no impl abstract class Shape { private String color; // ENCAPSULATION — private field public Shape(String color) { this.color = color; } public String getColor() { return color; } // controlled getter public abstract double area(); // subclass MUST provide impl public void describe() { // calls overridden area() at runtime — POLYMORPHISM System.out.println(color + " shape, area=" + area()); } } class Circle extends Shape { // INHERITANCE private double radius; public Circle(String c, double r) { super(c); this.radius = r; } @Override public double area() { return Math.PI * radius * radius; } } class Rect extends Shape { private double w, h; public Rect(String c, double w, double h) { super(c); this.w=w; this.h=h; } @Override public double area() { return w*h; } } // Main — polymorphic loop (vtable dispatch fires here) Shape[] shapes = { new Circle("Red", 5), new Rect("Blue", 4, 6) }; for (Shape s : shapes) s.describe(); // Red shape, area=78.53... / Blue shape, area=24.0
Real-world use cases
🏦
Encapsulation — Spring's @Service beans hide their repository dependencies. Callers invoke userService.createUser() with no knowledge of how DB writes happen.
🔗
Inheritance — Spring Data JPA: JpaRepository → PagingAndSortingRepository → CrudRepository. You extend JpaRepository and inherit 20+ methods for free.
⚙️
Polymorphism — Strategy pattern in payment systems: PaymentProcessor process(PaymentRequest) implemented by StripeProcessor, RazorpayProcessor, UPIProcessor. Spring injects the right bean by name.
📐
Abstraction — JDBC's Connection/Statement interfaces. Your code calls conn.prepareStatement(sql) — same code works with MySQL, Postgres, H2 without changing a line.
Common mistakes
⚠️
Making fields public "for convenience" — destroys encapsulation. Any class can corrupt state. Always start private, expose via methods.
⚠️
Overusing inheritance for code reuse — Inheritance models IS-A. If you're inheriting just to reuse a method, use composition. Violation leads to fragile base class problems.
⚠️
Forgetting @Override — without the annotation, a typo in the method signature creates a new method silently. The compiler won't catch it. Dynamic dispatch then calls the parent version — debugging nightmare.
⚠️
Calling overridable methods from constructors — the subclass hasn't initialized yet. The overridden method runs with uninitialized subclass fields → NullPointerException or wrong values.
Interview questions — click to reveal answers
Concept
What is the difference between encapsulation and abstraction?
Encapsulation is about hiding data — bundling state and protecting it from outside access. Abstraction is about hiding complexity — showing only the interface, not the implementation. You can have encapsulation without abstraction (a class with private fields and concrete methods), but abstraction implies encapsulation.
Deep dive
How does Java implement polymorphism at the bytecode level?
Via the invokevirtual bytecode instruction. Each class has a vtable — an array of method pointers. At runtime the JVM reads the actual object's class from the object header, indexes into its vtable, and calls the correct implementation. invokeinterface does similar but must handle multiple interface implementations (itable lookup).
Follow-up
Why can't you instantiate an abstract class?
Abstract methods have no bytecode body. If the JVM allowed instantiation, calling an abstract method would crash with no handler to jump to. The compiler enforces this. Note: you CAN create anonymous subclasses that implement all abstract methods — the JVM creates a concrete class behind the scenes.
Design
When would you choose composition over inheritance?
When the relationship is HAS-A, not IS-A. Also: (1) if you need to change behaviour at runtime (strategy), (2) if the parent class may change (fragile base class), (3) if you're tempted to inherit just for code reuse. Rule: "Favour composition over inheritance" — GoF, effective Java Item 18.

Interface vs Abstract Class
Contracts, default methods, functional interfaces
Definition

An interface is a pure contract — a list of method signatures that implementing classes must fulfil. An abstract class is a partial implementation — it can mix abstract methods with concrete ones and hold state. Java 8+ blurred the line with default and static methods on interfaces.

Interface
Defines WHAT. No state (except static final constants). Multiple implementation allowed. Use when unrelated classes share a capability (Comparable, Serializable).
Abstract Class
Defines WHAT + partially HOW. Can hold state. Single inheritance. Use when classes are related and share common base state/behaviour (HttpServlet, AbstractList).
How it works internally
1
Interfaces — compiled to a class file with ACC_INTERFACE flag. Method calls use invokeinterface bytecode (slower than invokevirtual — must scan itable). Since Java 8, default methods are real bytecode in the interface class file; implementing classes inherit them unless they override.
2
Abstract classes — normal class files with ACC_ABSTRACT flag + abstract methods with no Code attribute. JVM blocks new on them at runtime. Dispatch is invokevirtual — faster than interface dispatch.
3
Functional interfaces — any interface with exactly 1 abstract method. The @FunctionalInterface annotation is a compile-time guard (optional but recommended). Lambdas are syntactic sugar: the compiler generates an invokedynamic instruction that creates the implementing class lazily at first call via LambdaMetafactory.
4
Default method diamond conflict — if two interfaces provide default methods with the same signature, the implementing class must override to resolve ambiguity — compile error otherwise.

Code — default methods + functional interface
// Functional interface — exactly 1 abstract method @FunctionalInterface interface Validator<T> { boolean validate(T value); // default method — chaining validators (added Java 8) default Validator<T> and(Validator<T> other) { return v -> this.validate(v) && other.validate(v); } } // Abstract class — shared state + template abstract class BaseService { protected final String serviceName; // state — can't put in interface public BaseService(String name) { this.serviceName = name; } public abstract void process(); // subclass fills in public void log() { // shared concrete method System.out.println(serviceName + " processed"); } } // Usage — lambda satisfies functional interface Validator<String> notBlank = s -> !s.isBlank(); Validator<String> maxLen = s -> s.length() <= 50; Validator<String> combined = notBlank.and(maxLen); combined.validate("hello"); // true
Common mistakes
⚠️
Using abstract class when you mean interface — if you have no state and no shared behaviour, an abstract class just burns your single inheritance slot for no reason.
⚠️
Putting too much in default methods — default methods can't access instance state. Complex logic in defaults creates confusion about where behaviour lives. Keep defaults thin (e.g., chaining helpers).
⚠️
Forgetting @FunctionalInterface — without it, adding a second abstract method later silently breaks all lambda usages — no compile error at the interface definition site.
⚠️
Marker interfaces vs annotations — empty interfaces like Serializable were the old way to tag classes. For new code, use annotations (@Transactional, custom meta-annotations). Marker interfaces still exist in the JDK for historical reasons.
Interview questions
Classic
Can an interface extend another interface? Can an abstract class implement an interface?
Yes to both. An interface can extend multiple interfaces (interface multiple inheritance is fine). An abstract class can implement an interface without providing method bodies — the first concrete subclass must implement them.
Java 8+
What problem did default methods solve and what new problem did they create?
Solved: retroactively adding methods to interfaces (e.g. List.forEach) without breaking all existing implementations. Created: diamond problem — two interfaces with same default method signature forces the implementor to resolve the conflict explicitly via override.
Depth
Why is invokeinterface slower than invokevirtual?
vtable index is fixed per class — the JVM can jump straight to the right slot. itable is variable per class (a class may implement many interfaces in any order) — JVM must scan to find the right interface entry. JIT can optimise both away with inlining when the call site is monomorphic.

Generics
Generic methods · bounded wildcards · type erasure
Definition

Generics let you write a class or method once and parameterise it with a type — so List<String> and List<Integer> share one implementation but give you type safety at compile time. The type parameter is erased at runtime — the JVM only ever sees raw types.

Type erasure — what the JVM actually sees
1
At compile time, the compiler checks all generic types for correctness and inserts casts automatically where needed.
2
The compiler then erases all type parameters — List<String> becomes List. Bounded types (T extends Comparable) become their bound (Comparable). Unbounded become Object.
3
At runtime: List<String> and List<Integer> are the same class. new T() is impossible. instanceof List<String> is impossible. This is why you can't create generic arrays.
4
Reflection can reveal generic type info via ParameterizedType — because the compiler embeds it in the class file's Signature attribute. Jackson uses this to deserialize List<User>.
Wildcards — PECS rule
PECS: Producer Extends, Consumer Super
<? extends T> — upper bound
You can READ T from this collection. You cannot add to it (compiler doesn't know exact subtype). Use when the collection produces T values for you. Example: sum(List<? extends Number> list)
<? super T> — lower bound
You can WRITE T into this collection. You can only read Object out. Use when the collection consumes T values from you. Example: addNumbers(List<? super Integer> list)
// PECS in practice — Collections.copy signature static <T> void copy(List<? super T> dest, // consumer — super List<? extends T> src) { // producer — extends for (T t : src) dest.add(t); }
Code — generic class + method + type erasure demo
// Generic class class Pair<A, B> { private final A first; private final B second; public Pair(A a, B b) { first=a; second=b; } public A first() { return first; } public B second() { return second; } } // Generic method — T inferred from args public static <T extends Comparable<T>> T max(T a, T b) { return a.compareTo(b) >= 0 ? a : b; } // Type erasure — at runtime both are just List List<String> ls = new ArrayList<>(); List<Integer> li = new ArrayList<>(); System.out.println(ls.getClass() == li.getClass()); // true! // TypeToken pattern (Jackson / Gson) to survive erasure ObjectMapper mapper = new ObjectMapper(); List<User> users = mapper.readValue(json, new TypeReference<List<User>>(){}); // TypeReference creates anonymous subclass; compiler embeds // generic type in Signature attribute — readable at runtime
Common mistakes
⚠️
Using raw types — List list = new ArrayList() bypasses compile-time checks. A bad cast can hide for months until a ClassCastException at runtime.
⚠️
Creating generic arrays — new T[] is illegal because erasure makes it unsafe. Use List<T> or cast from Object[] with an unchecked warning you consciously suppress.
⚠️
Confusing <?> with <Object> — List<?> is a list of some unknown type (you can only read Object out, can't add). List<Object> accepts any object for writes but List<String> is NOT a subtype of it.
⚠️
Forgetting PECS direction — passing a List<Number> where List<? extends Number> is expected works. Reversing the wildcard doesn't — compiler error. PECS is the mental model to get this right every time.
Interview questions
Classic
What is type erasure and what are its practical consequences?
At compile time generics are checked and then removed — the JVM sees only raw types. Consequences: (1) can't use instanceof with generic type, (2) can't create generic arrays, (3) can't instantiate T with new, (4) two List<?> are the same class at runtime. Workaround: TypeToken / TypeReference pattern for reflection.
PECS
Why is List<String> not a subtype of List<Object>?
Because generics are invariant. If it were a subtype, you could pass a List<String> where List<Object> is expected and add an Integer into it — type safety breaks. Arrays ARE covariant (String[] IS-A Object[]) and that's exactly why arrays can throw ArrayStoreException. Generics chose invariance to catch this at compile time.
Depth
How does Jackson deserialize List<User> when type erasure hides the generic?
TypeReference creates an anonymous subclass. The compiler embeds the full generic signature in the class file's Signature attribute (erasure only removes it from bytecode method bodies, not from class-level declarations). Jackson reflects on this to recover List<User> and instantiate User for each element.

Immutability
String pool · StringBuilder vs StringBuffer · custom immutable classes
Definition

An immutable object's state cannot change after construction. Java's String is immutable — every "modification" creates a new object. Immutability is a cornerstone of thread safety: if an object can't change, no synchronisation is needed.

String (immutable)
Safe to share across threads. Can be interned (pooled). Used as HashMap key safely. Cost: new object on every change.
StringBuilder (mutable)
Not thread-safe. Best for single-thread string construction. Backed by a resizable char array.
StringBuffer (mutable)
Thread-safe via synchronized methods. Slower than StringBuilder. Legacy — rarely needed now; prefer explicit synchronisation.
Why String is immutable — 5 design reasons
1
String pool / caching — if strings could change, sharing them in a pool would corrupt all references. Immutability makes pooling safe.
2
HashMap keys — hashCode is computed once and cached in the String object. If the value changed, the hash would be wrong and the key would be unfindable in a HashMap.
3
Thread safety for free — immutable objects need no locks. Strings passed between threads are safe without synchronisation.
4
Security — class loading, file paths, network connections all use String. If mutable, an attacker could change a path after a security check but before use (TOCTOU attack).
5
Implementation — String is final class + private final char[] value. No setters. No subclassing allowed. Since Java 9, uses byte[] with a coder flag for compact strings (LATIN1 = 1 byte/char).

Code — custom immutable class + StringBuilder perf
// Custom immutable class — recipe public final class Money { // final — no subclassing private final long amount; // final fields private final String currency; public Money(long amount, String currency) { this.amount = amount; this.currency = currency; // String already immutable } public long getAmount() { return amount; } public String getCurrency(){ return currency; } public Money add(Money other) { // return new object, don't mutate if (!currency.equals(other.currency)) throw new IllegalArgumentException(); return new Money(amount + other.amount, currency); } } // StringBuilder vs String concatenation // BAD — O(n²) — each += creates a new String object String result = ""; for (int i = 0; i < 1000; i++) result += i; // GOOD — O(n) — single resizable buffer StringBuilder sb = new StringBuilder(); for (int i = 0; i < 1000; i++) sb.append(i); String r = sb.toString();
Common mistakes
⚠️
Mutable fields in "immutable" class — if your immutable class holds a List, callers can mutate the list via the getter. Fix: return Collections.unmodifiableList() or copy on return.
⚠️
String concatenation in loops — s += x in a loop is O(n²) because each concat allocates a new String. Use StringBuilder. Note: Java compiler optimises single-expression concat like "a" + b + "c" to StringBuilder automatically — only loops are a problem.
⚠️
Using == for String comparison — compares references, not values. Pool interning makes it work sometimes, which creates subtle bugs that only appear in production (where strings come from DB, not literals). Always use .equals().
Interview questions
Classic
Why is String immutable in Java?
5 reasons: (1) String pool safety — shared references need stable values, (2) HashMap key integrity — cached hashCode must stay valid, (3) thread safety for free, (4) security — class loading and file paths use String; mutability = TOCTOU attacks, (5) design philosophy — strings are value-like, not object-like.
Differentiate
StringBuilder vs StringBuffer — when do you actually use StringBuffer?
Almost never in new code. StringBuffer's synchronised methods are coarse-grained and slow. If you need thread-safe string building, you usually want to confine StringBuilder to a single thread anyway, or use a higher-level abstraction. StringBuffer is a Java 1.0 relic — StringBuilder (Java 5) replaced it in single-threaded contexts.
Design
How would you make a class with a List field truly immutable?
Copy the list in the constructor (defensive copy), store as a private final field, and return Collections.unmodifiableList() or List.copyOf() from the getter. Java 16+ records do this cleanly. Also: mark the class final to prevent mutable subclasses.

Pass by Value vs Reference
Java is always pass-by-value — but what that means for objects is subtle
Definition

Java is always pass-by-value. For primitives, the value itself is copied. For objects, the reference (memory address) is copied — not the object. So the callee gets its own copy of the pointer, not a pointer-to-pointer. You can mutate the object the reference points to, but you cannot make the caller's variable point to a different object.

Memory model — what actually happens on the stack
1
Primitive (int x = 5) — value 5 lives on the caller's stack frame. Passing to a method copies 5 to the callee's stack frame. Changing it in the callee has zero effect on the caller.
2
Object (Dog d = new Dog("Rex")) — d on the stack holds a reference (think: memory address, e.g. 0x7f4a). Passing d copies the address 0x7f4a to the callee. Both caller and callee now hold 0x7f4a — same dog on the heap. Mutating dog.name = "Max" via the callee's copy is visible to the caller. But doing dog = new Dog("other") in the callee only updates the callee's local copy of the address — caller's d still points to 0x7f4a.
Code — the classic demo
class Dog { String name; Dog(String n){ name=n; } } void tryReassign(Dog d) { d.name = "Max"; // mutates the heap object — VISIBLE to caller d = new Dog("Other"); // reassigns LOCAL copy of reference — NOT visible } Dog rex = new Dog("Rex"); tryReassign(rex); System.out.println(rex.name); // "Max" — mutation visible // NOT "Other" — reassign not visible // Primitive void tryChange(int x) { x = 99; } int v = 5; tryChange(v); System.out.println(v); // still 5
Common mistakes
⚠️
Thinking Java objects are pass-by-reference — they're not. You pass a copy of the reference. The distinction matters when you want to swap two objects or make a method return a new object to the caller — you can't do it via parameter alone.
⚠️
Mutating parameters unintentionally — passing a List into a helper, modifying it, and not realising the caller's List is now changed. Use defensive copies when you don't intend mutation.
Interview questions
Classic
Is Java pass-by-value or pass-by-reference?
Always pass-by-value. For primitives that's the primitive value. For objects that's the value of the reference (the address). The confusion arises because you can mutate the object via the copied reference — but you can't make the caller's variable point elsewhere. True pass-by-reference (like C++ &) would allow that.
Follow-up
How would you write a swap(a, b) method in Java that actually works?
You can't swap primitive local variables via a method in Java. For objects you can wrap them in an array or a holder object: pass int[] arr = {a, b} and swap arr[0]/arr[1]. Or return a Pair. This is one of the most common Java gotchas — the language simply doesn't support true pass-by-reference.

Type Casting
Widening · narrowing · ClassCastException · instanceof pattern matching
Definition

Type casting converts a value from one type to another. Widening (implicit) goes up the type hierarchy — safe, no data loss, done automatically. Narrowing (explicit) goes down — potentially lossy, requires a cast operator, can throw ClassCastException at runtime if the object isn't actually of the target type.

Primitive widening chain (automatic)
byte→ short→ int→ long→ float→ double (narrowing goes ← — explicit cast required)
How it works internally
1
Primitive widening — compiler emits conversion bytecodes (i2l, l2f, etc.). No runtime cost. int → float can lose precision for very large ints (float has 23-bit mantissa).
2
Primitive narrowing — explicit cast like (int) 3.9 emits d2i bytecode. Truncates, doesn't round. (byte) 300 wraps: 300 & 0xFF = 44.
3
Reference widening — upcasting (Animal a = new Dog()). Always safe, done implicitly. No bytecode instruction — just changes the compile-time type of the reference.
4
Reference narrowing — downcasting ((Dog) animal). JVM emits checkcast bytecode: reads the actual class from the object header and compares to the target type. If mismatch → ClassCastException. Always guard with instanceof or use pattern matching (Java 16+).
Code — widening, narrowing, checkcast, pattern matching
// Primitive widening — implicit int i = 42; long l = i; // int → long, safe double d = i; // int → double, safe // Primitive narrowing — explicit, lossy double pi = 3.14159; int truncated = (int) pi; // 3 — truncated, not rounded byte b = (byte) 300; // 44 — wrap-around (300 & 0xFF) // Reference widening — upcast, implicit Animal a = new Dog("Rex"); // Dog IS-A Animal // Reference narrowing — downcast, DANGEROUS without check if (a instanceof Dog) { Dog dog = (Dog) a; // safe — checkcast passes dog.fetch(); } // Java 16+ pattern matching — cleaner, no double-cast if (a instanceof Dog dog) { dog.fetch(); // dog is already Dog type here } // Java 21 switch pattern matching String result = switch (a) { case Dog dog -> "Dog: " + dog.getName(); case Cat cat -> "Cat: " + cat.purr(); default -> "Unknown animal"; };
Common mistakes
⚠️
Casting without instanceof check — leads to ClassCastException at runtime. The compiler can't always detect this (it only knows the declared type, not the actual runtime type).
⚠️
Assuming int → double is always safe — it is, but int → float can lose precision for large ints. (float) 123456789 gives 1.23456792E8 — slightly different due to float's 23-bit mantissa.
⚠️
Integer autoboxing type confusion — Integer x = 127; Integer y = 127; x == y is true (cached). Integer x = 128; Integer y = 128; x == y is false (different objects). Always use .equals() for object comparison.
Interview questions
Classic
What causes ClassCastException and how do you prevent it?
ClassCastException is thrown by the checkcast JVM instruction when the actual runtime type of an object is not compatible with the cast target. Prevent with: (1) instanceof check before casting, (2) Java 16+ pattern matching (combines check + cast), (3) generics to catch invalid types at compile time. Never suppress ClassCastException — it means your type model is wrong.
Subtle
What's the output of (int)(char)(byte) -1 and why?
-1 as int → byte: -1 (fits, all 8 bits = 0xFF = 11111111). byte -1 → char: char is unsigned 16-bit — byte -1 (0xFF) sign-extends to int -1 (0xFFFFFFFF) then masked to 16 bits = 0xFFFF = 65535. char 65535 → int: 65535. Output: 65535. Tests whether you understand sign extension and unsigned char.
Modern Java
How does Java 16+ pattern matching instanceof improve on the old approach?
Old: you check with instanceof, then cast again — two operations that must be kept in sync. If you accidentally cast to a different type than checked, ClassCastException at runtime. New: if (obj instanceof Dog dog) — single operation that checks AND binds the variable. The compiler guarantees the binding is type-safe. Cleaner, safer, and enables sealed class exhaustiveness checking.
